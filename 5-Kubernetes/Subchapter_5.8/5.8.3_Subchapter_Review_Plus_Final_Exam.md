## Cheatsheet: Troubleshooting and Monitoring

### Pod Status Quick Reference

| Status                | Meaning                    | First Command                |
| --------------------- | -------------------------- | ---------------------------- |
| **Pending**           | Scheduling issue           | `kubectl describe pod`       |
| **ContainerCreating** | Image pull or volume issue | `kubectl describe pod`       |
| **CrashLoopBackOff**  | Container crashing         | `kubectl logs --previous`    |
| **ImagePullBackOff**  | Cannot pull image          | `kubectl describe pod`       |
| **OOMKilled**         | Out of memory              | `kubectl describe pod`       |
| **Evicted**           | Node pressure              | `kubectl describe pod`       |
| **Terminating**       | Deletion stuck             | `kubectl delete pod --force` |

### Troubleshooting Commands

| Command                                         | Purpose                 |
| ----------------------------------------------- | ----------------------- |
| `kubectl get pods -w`                           | Watch pod status        |
| `kubectl describe pod NAME`                     | Events + details        |
| `kubectl logs NAME`                             | Current logs            |
| `kubectl logs NAME --previous`                  | Previous container logs |
| `kubectl exec -it NAME -- /bin/sh`              | Interactive shell       |
| `kubectl top pod NAME`                          | Resource usage          |
| `kubectl get events --sort-by='.lastTimestamp'` | Cluster events          |

### Node Troubleshooting

```bash
# Check node status
kubectl get nodes
kubectl describe node NAME

# Check node conditions
kubectl get node NAME -o jsonpath='{.status.conditions}'

# Drain node for maintenance
kubectl cordon NODE
kubectl drain NODE --ignore-daemonsets --delete-emptydir-data
kubectl uncordon NODE
```

### Question 1: Cluster Failure (15 minutes)

**Scenario:** A production cluster has 3 master nodes (stacked etcd) and 5 worker nodes. One master node fails completely (hardware failure, cannot recover). `kubectl` commands still work.

**Questions:**

1. Is the cluster still operational? Why?
2. What is the immediate impact on cluster operations?
3. How would you restore the failed master node?
4. What is the risk if another master node fails before recovery?

**Answer:**

1. **Yes, the cluster is still operational.** With 3 master nodes, quorum is 2. One failure leaves 2 healthy masters, maintaining quorum. The API server load balancer routes traffic to the remaining masters.

2. **Immediate impact:**

   * Reduced fault tolerance (only 2 masters left; another failure would lose quorum)

   * `kubectl` commands still work (via remaining masters)

   * New pods can still be scheduled

   * Existing workloads continue running

3. **Restore failed master:**

```bash
# On a new machine (same IP or new IP)
# 1. Install kubeadm, kubelet, kubectl
# 2. Join as control plane node
kubeadm join lb.example.com:6443 --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash> \
  --control-plane --certificate-key <key>

# 3. Verify cluster
kubectl get nodes
# All 3 masters should be Ready
```

1. **Risk of second master failure:** If another master fails before recovery, quorum would be lost (1 out of 3 remaining). The cluster would become unavailable for any write operations (deployments, scaling, config changes). Existing pods would continue running, but no new changes would be possible.

### Question 2: Application Deployment Failure (15 minutes)

**Scenario:** A developer deploys a new version of their application using `kubectl apply -f deployment.yaml`. The deployment shows:

```bash
kubectl get pods
# NAME                        READY   STATUS             RESTARTS   AGE
# myapp-7d8f9b6c4d-abc12      0/1     ImagePullBackOff   0          30s
```

**Questions:**

1. What does ImagePullBackOff indicate?
2. How would you diagnose the root cause?
3. List 4 possible causes and their fixes.
4. How would you verify the fix?

**Answer:**

1. **ImagePullBackOff indicates:** Kubernetes cannot pull the container image from the registry. The pod is in a backoff state, retrying with increasing delays.

2. **Diagnose:**

```bash
# Describe pod for detailed error
kubectl describe pod myapp-7d8f9b6c4d-abc12 | grep -A 10 Events

# Check image name
kubectl get pod myapp-7d8f9b6c4d-abc12 -o jsonpath='{.spec.containers[0].image}'
```

1. **Four possible causes and fixes:**

| Cause                         | Fix                                                                                            |
| ----------------------------- | ---------------------------------------------------------------------------------------------- |
| Wrong image name or tag       | `kubectl set image deployment/myapp myapp=correct-image:tag`                                   |
| Private registry without auth | `kubectl create secret docker-registry regcred --docker-server=...` and add `imagePullSecrets` |
| Registry unreachable          | Check network connectivity, CoreDNS, firewall                                                  |
| Image doesn't exist           | Build and push image to registry                                                               |

1. **Verify fix:**

```bash
# Watch pods
kubectl get pods -w

# Check rollout status
kubectl rollout status deployment/myapp

# Verify pod is Running
kubectl get pods | grep myapp
```

### Question 3: Pod CrashLoopBackOff (15 minutes)

**Scenario:** A pod is stuck in CrashLoopBackOff. You run `kubectl logs mypod` and see:

```
Error: cannot connect to database at postgres.default.svc.cluster.local:5432
```

**Questions:**

1. What is the most likely cause?
2. How would you verify the database service is running?
3. How would you test connectivity from the crashing pod?
4. What are two ways to fix this issue?

**Answer:**

1. **Most likely cause:** The database service is not running or not reachable. The application fails on startup because it cannot connect to its required dependency.

2. **Verify database service:**

```bash
# Check if PostgreSQL pods are running
kubectl get pods -l app=postgres

# Check service endpoints
kubectl get svc postgres
kubectl get endpoints postgres

# Check service DNS resolution
kubectl run test --rm -it --image=busybox -- nslookup postgres.default.svc.cluster.local
```

1. **Test connectivity from crashing pod:**

```bash
# Option 1: Create debug pod in same namespace
kubectl run debug --rm -it --image=nicolaka/netshoot -- /bin/bash
nslookup postgres.default.svc.cluster.local
nc -zv postgres.default.svc.cluster.local 5432

# Option 2: Use ephemeral container (if supported)
kubectl debug mypod -it --image=nicolaka/netshoot -- /bin/bash
```

1. **Two ways to fix:**

**Fix A: Ensure database starts first (initContainer)**

```yaml
spec:
  initContainers:
  - name: wait-for-db
    image: busybox
    command: ['sh', '-c', 'until nslookup postgres; do sleep 2; done;']
  containers:
  - name: app
    image: myapp
```

**Fix B: Add retry logic in application**

```python
# Python example
import time
import psycopg2

def connect_with_retry(max_retries=10, delay=5):
    for i in range(max_retries):
        try:
            conn = psycopg2.connect(host="postgres", port=5432)
            return conn
        except Exception as e:
            print(f"Attempt {i+1} failed: {e}")
            time.sleep(delay)
    raise Exception("Could not connect to database")
```

### Question 4: Node NotReady (10 minutes)

**Scenario:** A worker node shows `NotReady` status. You SSH into the node and find the disk is 95% full.

**Questions:**

1. What command would you run to check disk usage on the node?
2. What are the main culprits for disk space consumption on a Kubernetes node?
3. How would you free up space without rebooting?
4. After fixing, how would you verify the node becomes Ready?

**Answer:**

1. **Check disk usage:**

```bash
df -h
# Check /var (where Docker/containerd stores data)
```

1. **Main culprits for disk space:**

   * Docker/containerd images (`/var/lib/docker` or `/var/lib/containerd`)

   * Container logs (JSON logs)

   * Orphaned volumes

   * Old kernel images (`/boot`)

   * Application logs on host

2. **Free up space without rebooting:**

```bash
# Clean Docker/containerd
docker system prune -a -f
# or
crictl rmi --prune

# Clean journal logs
journalctl --vacuum-size=500M

# Clean old container logs
truncate -s 0 /var/log/containers/*.log

# Find and remove large files
find /var -type f -size +100M -exec ls -lh {} \;
```

1. **Verify node becomes Ready:**

```bash
# From control plane
kubectl get nodes -w
# Wait for node status to change from NotReady to Ready

# Check node conditions
kubectl describe node worker-1 | grep -A 5 Conditions
```

### Question 5: HPA Not Scaling (10 minutes)

**Scenario:** An HPA is configured for a deployment, but it's not scaling when CPU usage increases. `kubectl get hpa` shows:

```
NAME        REFERENCE              TARGETS   MINPODS   MAXPODS   REPLICAS   AGE
myapp-hpa   Deployment/myapp       0%/50%    2         10        2          5m
```

**Questions:**

1. What prerequisite is required for HPA to work with CPU metrics?
2. How would you check if metrics are available?
3. What could cause TARGETS to show 0% even under load?
4. How would you fix it?

**Answer:**

1. **Prerequisite:** metrics-server must be installed to provide CPU/memory metrics.

2. **Check metrics availability:**

```bash
# Check if metrics-server is running
kubectl get pods -n kube-system | grep metrics-server

# Test metrics API
kubectl top nodes
kubectl top pods

# Check API availability
kubectl get --raw /apis/metrics.k8s.io/v1beta1/nodes | jq .
```

1. **Causes for 0% even under load:**

   * **No CPU requests defined** in deployment (HPA requires requests)

   * metrics-server not installed or not working

   * Pods not actually using CPU (check with `kubectl top pods`)

   * Metric collection delay (can take 1-2 minutes)

2. **Fix:**

```bash
# Add CPU requests to deployment
kubectl patch deployment myapp -p '{"spec":{"template":{"spec":{"containers":[{"name":"myapp","resources":{"requests":{"cpu":"100m"}}}]}}}}'

# Install metrics-server if missing
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml

# Generate load to test
kubectl run load-generator --image=busybox -- /bin/sh -c "while true; do wget -q -O- http://myapp; done"
```

### Question 6: StatefulSet Recovery (10 minutes)

**Scenario:** A StatefulSet for a database has 3 replicas (db-0, db-1, db-2). Pod db-1 is deleted accidentally.

**Questions:**

1. Will the StatefulSet automatically recreate db-1? What will happen to its data?
2. How would you manually restore db-1 if automatic recovery fails?
3. What is the correct order of operations if you need to delete the entire StatefulSet but preserve data?

**Answer:**

1. **Automatic recovery:** Yes, the StatefulSet controller will recreate db-1 automatically. The PVC `data-db-1` still exists, so the new pod will bind to the same PVC and data will be preserved.

2. **Manual restore if automatic recovery fails:**

```bash
# Check if PVC still exists
kubectl get pvc data-db-1

# Delete the stuck pod (will be recreated)
kubectl delete pod db-1

# If pod is stuck in Terminating
kubectl delete pod db-1 --force --grace-period=0

# If PVC is missing, restore from backup
kubectl apply -f pvc-backup.yaml
```

1. **Delete StatefulSet but preserve data:**

```bash
# Delete StatefulSet without deleting PVCs
kubectl delete statefulset db --cascade=orphan

# PVCs remain with data
kubectl get pvc

# To delete PVCs (data loss!)
kubectl delete pvc data-db-0 data-db-1 data-db-2
```

### Question 7: Network Policy Isolation (10 minutes)

**Scenario:** A security team requires that only the frontend pods can access the backend API. The backend runs on port 8080.

**Questions:**

1. Write a NetworkPolicy that allows only frontend pods to access backend.
2. Write a default deny policy for the namespace.
3. How would you verify the policy is working?

**Answer:**

1. **Allow frontend to backend:**

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
```

1. **Default deny all ingress:**

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
spec:
  podSelector: {}
  policyTypes:
  - Ingress
```

1. **Verify policy:**

```bash
# Get pod IPs
kubectl get pods -o wide

# Test from frontend (should succeed)
kubectl exec frontend-pod -- curl http://<backend-ip>:8080

# Test from other pod (should fail)
kubectl run test --rm -it --image=busybox -- wget -O- http://<backend-ip>:8080
# Should timeout or be rejected
```

### Question 8: ConfigMap Update (10 minutes)

**Scenario:** A ConfigMap is updated with new values. Pods using the ConfigMap via environment variables do not see the new values.

**Questions:**

1. Why aren't the new values reflected?
2. How can you force pods to pick up the new configuration without downtime?
3. What alternative method would allow automatic updates?

**Answer:**

1. **Why not reflected:** ConfigMaps mounted as environment variables are injected when the pod starts and never change. Kubernetes does not update environment variables dynamically.

2. **Force pods to pick up new config without downtime (rolling restart):**

```bash
# For deployment
kubectl rollout restart deployment myapp

# For StatefulSet
kubectl rollout restart statefulset myapp

# For manual pod deletion (if not managed by controller)
kubectl delete pod mypod
```

1. **Alternative for automatic updates:** Mount ConfigMap as a volume instead of environment variables.

```yaml
spec:
  containers:
  - name: app
    volumeMounts:
    - name: config
      mountPath: /etc/config
      readOnly: true
  volumes:
  - name: config
    configMap:
      name: app-config
```

With volume mounts, files update automatically (kubelet sync \~1 minute). Applications need to watch for file changes or reload periodically.
