## Cheatsheet: Configuration, Secrets, and Autoscaling

### ConfigMap Creation

| Method    | Command                                                  |
| --------- | -------------------------------------------------------- |
| Literal   | `kubectl create configmap NAME --from-literal=KEY=value` |
| File      | `kubectl create configmap NAME --from-file=path/to/file` |
| Directory | `kubectl create configmap NAME --from-file=./configs/`   |
| Env file  | `kubectl create configmap NAME --from-env-file=app.env`  |

### ConfigMap YAML

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  DB_HOST: postgres.default.svc.cluster.local
  DB_PORT: "5432"
  app.properties: |
    cache.ttl=3600
    max.connections=100
```

### Secret Creation

| Method   | Command                                                          |
| -------- | ---------------------------------------------------------------- |
| Literal  | `kubectl create secret generic NAME --from-literal=KEY=value`    |
| File     | `kubectl create secret generic NAME --from-file=path/to/file`    |
| Env file | `kubectl create secret generic NAME --from-env-file=secrets.env` |
| TLS      | `kubectl create secret tls NAME --cert=cert.pem --key=key.pem`   |
| Docker   | `kubectl create secret docker-registry NAME --docker-server=...` |

### Secret YAML

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
data:
  username: YWRtaW4=      # base64 encoded
  password: UzNjcjN0UGFzcw==
---
apiVersion: v1
kind: Secret
metadata:
  name: tls-secret
type: kubernetes.io/tls
data:
  tls.crt: LS0tLS1CRU...
  tls.key: LS0tLS1CRU...
```

### Using ConfigMaps/Secrets in Pods

| Method                    | YAML Snippet                                               |
| ------------------------- | ---------------------------------------------------------- |
| Environment variable      | `valueFrom: { configMapKeyRef: { name: NAME, key: KEY } }` |
| All environment variables | `envFrom: [{ configMapRef: { name: NAME } }]`              |
| Volume mount              | `volumes: [{ name: vol, configMap: { name: NAME } }]`      |

### Secret Types

| Type                             | Purpose                     |
| -------------------------------- | --------------------------- |
| `Opaque`                         | Generic secret (default)    |
| `kubernetes.io/tls`              | TLS certificate + key       |
| `kubernetes.io/dockerconfigjson` | Docker registry credentials |
| `kubernetes.io/basic-auth`       | Basic authentication        |
| `kubernetes.io/ssh-auth`         | SSH authentication          |

### Horizontal Pod Autoscaler (HPA)

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 50
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300
    scaleUp:
      stabilizationWindowSeconds: 0
```

### HPA Commands

| Command                                                               | Purpose      |
| --------------------------------------------------------------------- | ------------ |
| `kubectl autoscale deployment NAME --cpu-percent=50 --min=1 --max=10` | Create HPA   |
| `kubectl get hpa`                                                     | List HPAs    |
| `kubectl describe hpa NAME`                                           | Show details |
| `kubectl delete hpa NAME`                                             | Delete HPA   |

### Vertical Pod Autoscaler (VPA)

```yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: myapp-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  updatePolicy:
    updateMode: "Auto"  # Off, Initial, Recreate, Auto
  resourcePolicy:
    containerPolicies:
    - containerName: "*"
      minAllowed:
        cpu: "100m"
        memory: "256Mi"
      maxAllowed:
        cpu: "2"
        memory: "4Gi"
```

### VPA Update Modes

| Mode       | Behavior             |
| ---------- | -------------------- |
| `Off`      | Recommendations only |
| `Initial`  | Set at pod creation  |
| `Recreate` | Evict pod on change  |
| `Auto`     | Recreate when needed |

### Cluster Autoscaler (CA)

| Parameter                            | Default | Purpose                             |
| ------------------------------------ | ------- | ----------------------------------- |
| `--scale-down-delay-after-add`       | 10m     | Wait after scale up                 |
| `--scale-down-unneeded-time`         | 10m     | How long node must be underutilized |
| `--scale-down-utilization-threshold` | 0.5     | Utilization threshold               |
| `--max-node-provision-time`          | 15m     | Max time to provision node          |
| `--balance-similar-node-groups`      | false   | Balance across node groups          |

### Cluster Autoscaler Expanders

| Strategy      | Behavior                 |
| ------------- | ------------------------ |
| `random`      | Random node group        |
| `most-pods`   | Maximize pods scheduled  |
| `least-waste` | Minimize resource waste  |
| `priority`    | Priority-based selection |
| `price`       | Lowest price (cloud)     |

## Comparison Tables

### ConfigMap vs Secret

| Feature             | ConfigMap            | Secret                   |
| ------------------- | -------------------- | ------------------------ |
| **Encoding**        | Plain text           | Base64                   |
| **etcd encryption** | No (by default)      | Yes (can be enabled)     |
| **Size limit**      | 1MiB                 | 1MiB                     |
| **Immutable**       | Yes                  | Yes                      |
| **Use case**        | Non-sensitive config | Passwords, tokens, certs |

### HPA vs VPA vs Cluster Autoscaler

| Feature            | HPA               | VPA                 | Cluster Autoscaler        |
| ------------------ | ----------------- | ------------------- | ------------------------- |
| **Scales**         | Pod count         | Pod resources       | Node count                |
| **Pod restart**    | No                | Yes (Auto/Recreate) | N/A                       |
| **Metrics source** | metrics-server    | metrics-server      | Pending pods              |
| **Best for**       | Stateless apps    | Stateful apps       | All workloads             |
| **Requires**       | Resource requests | Resource requests   | Cloud provider/autoscaler |

### HPA Metric Types

| Type       | Source             | Example                  |
| ---------- | ------------------ | ------------------------ |
| `Resource` | CPU/Memory         | `averageUtilization: 50` |
| `Pods`     | Custom pod metrics | `averageValue: 100`      |
| `Object`   | Other K8s objects  | `value: "10k"`           |
| `External` | External metrics   | `averageValue: 5`        |

## Interview Questions (Scenario-Based)

These questions assume only knowledge from Subchapter 5.6. Answers reference only concepts from 5.6.1 and 5.6.2.

### Question 1

**Scenario:** A developer creates a Secret using `kubectl create secret generic my-secret --from-literal=password=secret`. They then view the secret with `kubectl get secret my-secret -o yaml` and see the password is base64 encoded.

**Question:** Is the password secure? What additional security measures should be implemented for production secrets?

**Answer:**

**Is the password secure?** No, base64 is encoding, not encryption. Anyone with access to etcd or the Kubernetes API can decode it.

**Why base64 is not secure:**

* Anyone with `get secret` permission can decode: `echo "c2VjcmV0" | base64 -d`

* etcd stores secrets in plaintext (unless encryption is configured)

* Secrets are not encrypted at rest by default

**Production security measures:**

**1. Enable etcd encryption at rest:**

```yaml
# encryption-config.yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
- resources:
  - secrets
  providers:
  - aescbc:
      keys:
      - name: key1
        secret: <base64-32-byte-key>
```

**2. Use external secrets management:**

```yaml
# ExternalSecrets Operator
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: aws-secret
spec:
  secretStoreRef:
    name: aws-secretsmanager
    kind: SecretStore
  target:
    name: my-k8s-secret
  data:
  - secretKey: password
    remoteRef:
      key: production/db/password
```

**3. Use SealedSecrets for GitOps:**

```bash
# Encrypt secret for Git
kubeseal --format yaml < secret.yaml > sealed-secret.yaml
# Sealed secret can be committed to Git
```

**4. RBAC restrictions:**

```yaml
# Only specific service accounts can read secrets
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: secret-reader
rules:
- apiGroups: [""]
  resources: ["secrets"]
  verbs: ["get", "list"]
  resourceNames: ["my-secret"]  # Limit to specific secret
```

**5. Use immutable secrets for static values:**

```yaml
metadata:
  name: static-secret
immutable: true
```

**6. Audit logging:**

```bash
# Enable audit logs to track secret access
# --audit-policy-file=/etc/kubernetes/audit-policy.yaml
```

### Question 2

**Scenario:** An application running in Kubernetes experiences CPU spikes during peak traffic. The ops team manually scales the deployment from 3 to 10 replicas. This happens daily.

**Question:** How would you automate this using HPA? What metrics would you use? What are the prerequisites?

**Answer:**

**Solution:** Implement HPA based on CPU utilization.

**Prerequisites:**

1. **metrics-server installed:**

```bash
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
kubectl top nodes  # Verify
```

1. **Resource requests defined in deployment:**

```yaml
spec:
  containers:
  - name: app
    resources:
      requests:
        cpu: 200m   # Required for HPA
        memory: 256Mi
```

**HPA Configuration:**

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 3
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 0      # Scale up immediately
      policies:
      - type: Percent
        value: 100
        periodSeconds: 15
    scaleDown:
      stabilizationWindowSeconds: 300    # Wait 5 min before scaling down
      policies:
      - type: Percent
        value: 10
        periodSeconds: 60
```

**Create HPA:**

```bash
# Via YAML
kubectl apply -f hpa.yaml

# Or via command line
kubectl autoscale deployment myapp --cpu-percent=70 --min=3 --max=20
```

**Additional metrics for better scaling:**

```yaml
metrics:
# CPU-based scaling
- type: Resource
  resource:
    name: cpu
    target:
      type: Utilization
      averageUtilization: 70
# Memory-based scaling
- type: Resource
  resource:
    name: memory
    target:
      type: Utilization
      averageUtilization: 80
# Custom metrics (requires Prometheus)
- type: Pods
  pods:
    metric:
      name: requests_per_second
    target:
      type: AverageValue
      averageValue: 100
```

**Monitoring HPA:**

```bash
# Watch HPA status
kubectl get hpa -w

# View detailed events
kubectl describe hpa app-hpa

# Check current metrics
kubectl top pods
```

### Question 3

**Scenario:** A ConfigMap is updated with new configuration values. The pod using the ConfigMap via environment variables does not see the new values, even after waiting 10 minutes.

**Question:** Why aren't the new values reflected? How can you force the pod to pick up the new configuration?

**Answer:**

**Why values aren't reflected:** ConfigMaps mounted as environment variables are **not** updated automatically. Environment variables are injected when the pod starts and never change afterward.

**ConfigMap update behavior:**

| Consumption Method         | Auto-Update | Delay        |
| -------------------------- | ----------- | ------------ |
| **Environment variables**  | No          | N/A          |
| **Volume mount (file)**    | Yes         | \~60 seconds |
| **Volume mount (subPath)** | No          | N/A          |

**Solutions:**

**Solution 1: Restart the pod (simplest)**

```bash
# For deployment
kubectl rollout restart deployment myapp

# For pod (deletes and recreates)
kubectl delete pod mypod
```

**Solution 2: Use volume mount instead of environment variables**

```yaml
# Update pod spec to mount ConfigMap as volume
spec:
  containers:
  - name: app
    volumeMounts:
    - name: config
      mountPath: /etc/config
      readOnly: true
  volumes:
  - name: config
    configMap:
      name: app-config
```

**Solution 3: Use reloader tool for automatic restart**

```bash
# Install Reloader
kubectl apply -f https://raw.githubusercontent.com/stakater/Reloader/master/deployments/kubernetes/reloader.yaml

# Add annotation to deployment
metadata:
  annotations:
    reloader.stakater.com/match: "true"
# Reloader will restart pod when ConfigMap changes
```

**Solution 4: Add checksum annotation (manual)**

```bash
# Update deployment with ConfigMap hash
kubectl patch deployment myapp -p "{\"spec\":{\"template\":{\"metadata\":{\"annotations\":{\"configmap-hash\":\"$(kubectl get configmap app-config -o jsonpath='{.data}' | sha256sum | cut -d ' ' -f1)\"}}}}}"
```

**Solution 5: Read config from file and watch for changes (application-level)**

```python
# Python example – watch for file changes
import time
from watchdog.observers import Observer
from watchdog.events import FileSystemEventHandler

class ConfigHandler(FileSystemEventHandler):
    def on_modified(self, event):
        if event.src_path == '/etc/config/app.properties':
            reload_config()

observer = Observer()
observer.schedule(ConfigHandler(), path='/etc/config')
observer.start()
```

**Best practice recommendation:**

* Use volume mounts for configuration that changes frequently

* Use environment variables for static configuration

* Use Reloader for automatic pod restarts when ConfigMaps change

### Question 4

**Scenario:** A cluster runs a web application with variable traffic. During peak hours, pods are scaled up by HPA, but nodes become saturated and new pods remain pending due to insufficient resources.

**Question:** What is missing from the autoscaling setup? How would you fix it so nodes also scale automatically?

**Answer:**

**Missing component:** Cluster Autoscaler (CA) – it adds nodes when pods are pending due to resource constraints.

**Problem:**

* HPA scales pods based on CPU/metrics

* When pods scale up, existing nodes may not have capacity

* Pods enter `Pending` state (insufficient CPU/memory)

* Without Cluster Autoscaler, no new nodes are added

**Solution: Install and configure Cluster Autoscaler**

**AWS EKS example:**

```bash
# Install Cluster Autoscaler
kubectl apply -f https://raw.githubusercontent.com/kubernetes/autoscaler/master/cluster-autoscaler/cloudprovider/aws/examples/cluster-autoscaler-autodiscover.yaml

# Edit deployment with cluster name
kubectl -n kube-system edit deployment cluster-autoscaler
# Add: --node-group-auto-discovery=asg:tag=k8s.io/cluster-autoscaler/enabled,k8s.io/cluster-autoscaler/<cluster-name>
```

**GKE example:**

```bash
# Enable Cluster Autoscaler on node pool
gcloud container clusters update my-cluster \
  --enable-autoscaling \
  --min-nodes=1 \
  --max-nodes=10 \
  --node-pool=default-pool
```

**Azure AKS example:**

```bash
# Enable Cluster Autoscaler
az aks update \
  --resource-group my-rg \
  --name my-cluster \
  --enable-cluster-autoscaler \
  --min-count 1 \
  --max-count 10
```

**Complete autoscaling stack:**

```yaml
# 1. HPA for pod scaling
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: webapp-hpa
spec:
  minReplicas: 3
  maxReplicas: 50
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70

# 2. VPA for resource recommendations (Off mode)
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: webapp-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: webapp
  updatePolicy:
    updateMode: "Off"

# 3. Cluster Autoscaler configured on node pools
# (Infrastructure as Code – Terraform/CloudFormation)
```

**Verify Cluster Autoscaler is working:**

```bash
# Check CA logs
kubectl logs -n kube-system deployment/cluster-autoscaler

# Check for pending pods
kubectl get pods --field-selector=status.phase=Pending

# Check CA status
kubectl get cm -n kube-system cluster-autoscaler-status -o yaml
```

**Expected behavior:**

1. Traffic increases → HPA scales pods to 20 replicas
2. Nodes run out of resources → pods pending
3. Cluster Autoscaler detects pending pods → adds new node
4. Pending pods schedule → application scales

### Question 5

**Scenario:** A deployment has VPA with `updateMode: Auto`. HPA is also configured for the same deployment. The team notices that pods are frequently restarting and replica counts are fluctuating unexpectedly.

**Question:** What is the likely conflict? How would you resolve it?

**Answer:**

**Likely conflict:** VPA and HPA conflict when both are active on the same deployment.

**Why they conflict:**

* **HPA** scales number of replicas based on metrics

* **VPA** (Auto mode) restarts pods to adjust CPU/memory requests

* When VPA restarts a pod, HPA may see a temporary metric drop and scale down

* HPA may also scale up based on metrics before VPA finishes adjusting

**Symptoms:**

* Frequent pod restarts

* Unstable replica count

* Unexpected scaling events

* Resource thrashing

**Resolution options:**

**Option 1: Set VPA to** **`Off`** **(recommended)**

```yaml
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: myapp-vpa
spec:
  updatePolicy:
    updateMode: "Off"  # Only recommendations, no changes
```

Use VPA only for recommendations, HPA for actual scaling.

**Option 2: Use VPA for initial sizing only**

```yaml
updatePolicy:
  updateMode: "Initial"  # Only set at pod creation
```

**Option 3: Use HPA with custom metrics that VPA doesn't affect**

**Option 4: For stateful workloads, use VPA without HPA**

```yaml
# Use VPA Auto, no HPA
updatePolicy:
  updateMode: "Auto"
# Remove HPA entirely
```

**Best practice combination:**

```yaml
# VPA in Off mode (recommendations only)
apiVersion: autoscaling.k8s.io/v1
kind: VerticalPodAutoscaler
metadata:
  name: myapp-vpa
spec:
  targetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  updatePolicy:
    updateMode: "Off"

# HPA for actual scaling
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  minReplicas: 2
  maxReplicas: 20
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
```

**View VPA recommendations:**

```bash
kubectl get vpa myapp-vpa -o yaml | grep -A 15 recommendation
# Output shows recommended CPU/memory values
# Manually update deployment with recommended values
```
