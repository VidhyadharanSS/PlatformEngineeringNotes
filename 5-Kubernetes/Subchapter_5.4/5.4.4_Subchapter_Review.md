## Cheatsheet: Kubernetes Networking

### Service Types

| Type             | Access                          | Use Case                         | DNS           |
| ---------------- | ------------------------------- | -------------------------------- | ------------- |
| **ClusterIP**    | Internal only                   | Service-to-service communication | Yes           |
| **NodePort**     | Node IP:high port (30000-32767) | External access (dev/test)       | Yes           |
| **LoadBalancer** | Cloud LB                        | Production external access       | Yes           |
| **ExternalName** | External DNS CNAME              | Access external services         | Yes (CNAME)   |
| **Headless**     | `clusterIP: None`               | StatefulSet pod discovery        | Yes (pod IPs) |

### Service YAML Template

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  type: ClusterIP  # or NodePort, LoadBalancer
  selector:
    app: myapp
  ports:
  - port: 80
    targetPort: 8080
    protocol: TCP
    name: http
  - port: 9090
    targetPort: 9090
    name: metrics
```

### Ingress Controller Installation

```bash
# Nginx Ingress (bare metal)
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.9.0/deploy/static/provider/baremetal/deploy.yaml

# Nginx Ingress (cloud)
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/controller-v1.9.0/deploy/static/provider/cloud/deploy.yaml
```

### Ingress YAML Templates

**Basic:**

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
spec:
  rules:
  - host: example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: web-service
            port:
              number: 80
```

**With TLS:**

```yaml
spec:
  tls:
  - hosts:
    - secure.example.com
    secretName: tls-secret
  rules:
  - host: secure.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: secure-service
            port:
              number: 80
```

**Path-based routing:**

```yaml
spec:
  rules:
  - host: example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: api-service
            port:
              number: 8080
      - path: /
        pathType: Prefix
        backend:
          service:
            name: web-service
            port:
              number: 80
```

| Annotation                                       | Purpose           |
| ------------------------------------------------ | ----------------- |
| `nginx.ingress.kubernetes.io/rewrite-target`     | URL rewriting     |
| `nginx.ingress.kubernetes.io/ssl-redirect`       | Force HTTPS       |
| `nginx.ingress.kubernetes.io/limit-rps`          | Rate limiting     |
| `nginx.ingress.kubernetes.io/auth-type`          | Basic auth        |
| `nginx.ingress.kubernetes.io/proxy-body-size`    | Max upload size   |
| `nginx.ingress.kubernetes.io/websocket-services` | WebSocket support |

### Gateway API Resources

| Resource         | Scope     | Purpose                     |
| ---------------- | --------- | --------------------------- |
| **GatewayClass** | Cluster   | Controller definition       |
| **Gateway**      | Namespace | Load balancer configuration |
| **HTTPRoute**    | Namespace | HTTP routing rules          |

### Network Policy Types

| Type      | Controls         |
| --------- | ---------------- |
| `Ingress` | Incoming traffic |
| `Egress`  | Outgoing traffic |
| Both      | Both directions  |

### Network Policy YAML Templates

**Deny all ingress:**

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all-ingress
spec:
  podSelector: {}
  policyTypes:
  - Ingress
```

**Allow from specific pod:**

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-from-frontend
spec:
  podSelector:
    matchLabels:
      app: backend
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 80
```

**Allow from namespace:**

```yaml
ingress:
- from:
  - namespaceSelector:
      matchLabels:
        kubernetes.io/metadata.name: monitoring
```

**Allow egress to external:**

```yaml
egress:
- to:
  - ipBlock:
      cidr: 0.0.0.0/0
  ports:
  - protocol: TCP
    port: 443
```

## Comparison Tables

### Service Types Comparison

| Feature         | ClusterIP | NodePort      | LoadBalancer | Headless    |
| --------------- | --------- | ------------- | ------------ | ----------- |
| External access | No        | Yes (node IP) | Yes (LB DNS) | No          |
| Port range      | Any       | 30000-32767   | Any          | Any         |
| Cost            | Free      | Free          | Cloud cost   | Free        |
| Use case        | Internal  | Dev/test      | Production   | StatefulSet |

### Ingress vs Gateway API

| Feature               | Ingress     | Gateway API                      |
| --------------------- | ----------- | -------------------------------- |
| **Resources**         | Single      | GatewayClass, Gateway, HTTPRoute |
| **Role separation**   | Limited     | Full                             |
| **Traffic splitting** | Annotations | Native weights                   |
| **Cross-namespace**   | Limited     | Yes                              |
| **Protocols**         | HTTP/HTTPS  | HTTP, HTTPS, TCP, UDP, TLS       |

### Network Policy Selectors

| Selector Type       | Scope                 | Example                                          |
| ------------------- | --------------------- | ------------------------------------------------ |
| `podSelector`       | Same namespace        | `podSelector: {matchLabels: {app: web}}`         |
| `namespaceSelector` | All pods in namespace | `namespaceSelector: {matchLabels: {name: prod}}` |
| `ipBlock`           | External IPs          | `ipBlock: {cidr: 10.0.0.0/8}`                    |
| Combined            | Cross-namespace pods  | `namespaceSelector: {...}, podSelector: {...}`   |

### CNI Network Policy Support

| CNI     | Network Policy | L7 Policies      |
| ------- | -------------- | ---------------- |
| Calico  | Full           | No               |
| Cilium  | Full           | Yes (HTTP, gRPC) |
| Weave   | Full           | No               |
| Flannel | No             | No               |
| Antrea  | Full           | Yes              |

## Interview Questions (Scenario-Based)

These questions assume only knowledge from Subchapter 5.4. Answers reference only concepts from 5.4.1, 5.4.2, and 5.4.3.

### Question 1

**Scenario:** A developer creates a ClusterIP service for their application. They can access it from within the cluster using `curl http://my-service:8080`, but they cannot access it from their laptop.

**Question:** Why can't they access the service from outside? What changes would you recommend to expose it externally? What are the trade-offs of each option?

**Answer:**

**Why it can't be accessed:** ClusterIP services are only accessible within the cluster. They have no external IP or port mapping.

**Options to expose externally:**

| Option           | Command                                                     | Pros                                        | Cons                                     |
| ---------------- | ----------------------------------------------------------- | ------------------------------------------- | ---------------------------------------- |
| **NodePort**     | `kubectl edit svc my-service` (change type to NodePort)     | Simple, works on any cluster                | Node IP required, port range 30000-32767 |
| **LoadBalancer** | `kubectl edit svc my-service` (change type to LoadBalancer) | External IP, standard port                  | Cloud cost, cloud provider required      |
| **Ingress**      | Create Ingress resource                                     | Single IP for multiple services, L7 routing | Requires Ingress controller              |
| **Port-forward** | `kubectl port-forward svc/my-service 8080:8080`             | Quick testing                               | Not for production, one-time             |

**Recommended solution for production (cloud):**

```bash
# Change to LoadBalancer
kubectl patch svc my-service -p '{"spec":{"type":"LoadBalancer"}}'

# Get external IP
kubectl get svc my-service -w
```

**Recommended solution for on-premise:**

```yaml
# Use NodePort with Ingress
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  type: NodePort
  selector:
    app: myapp
  ports:
  - port: 8080
    nodePort: 30080

---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
spec:
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: my-service
            port:
              number: 8080
```

### Question 2

**Scenario:** A security team requires that only the frontend pods can access the backend database. All other pods should be blocked. The database runs on port 5432.

**Question:** How would you implement this using Kubernetes network policies? Provide the complete YAML.

**Answer:**

```yaml
# backend-network-policy.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: database-network-policy
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: database
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 5432
```

**Additional policy to deny all other ingress (optional but recommended):**

```yaml
# default-deny-all.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  # No ingress rules = deny all
```

**Verification steps:**

```bash
# Label frontend pods (if not already labeled)
kubectl label pod frontend-xxx app=frontend

# Test from frontend (should succeed)
kubectl exec frontend-xxx -- nc -zv database-service 5432

# Test from other pod (should fail)
kubectl run test --rm -it --image=busybox -- nc -zv database-service 5432
```

**Prerequisites:**

* CNI must support Network Policies (Calico, Cilium, Weave)

* Check with: `kubectl get pods -n kube-system | grep -E "calico|cilium|weave"`

### Question 3

**Scenario:** An Ingress resource is created with the following YAML, but requests to `http://api.example.com/v1/users` return 404 Not Found.

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress
spec:
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /v1
        pathType: Prefix
        backend:
          service:
            name: api-v1-service
            port:
              number: 80
      - path: /v2
        pathType: Prefix
        backend:
          service:
            name: api-v2-service
            port:
              number: 80
```

**Question:** What is the likely issue? How would you fix it? What annotation might be needed?

**Answer:**

**Likely issue:** The backend services expect the full path `/v1/users`, but the Ingress forwards the path as-is. The services may not have endpoints at `/v1/users`.

**Fix:** Use rewrite annotation to strip the path prefix.

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: api-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /$2
spec:
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /v1(/|$)(.*)
        pathType: Prefix
        backend:
          service:
            name: api-v1-service
            port:
              number: 80
      - path: /v2(/|$)(.*)
        pathType: Prefix
        backend:
          service:
            name: api-v2-service
            port:
              number: 80
```

**How it works:**

* Request: `/v1/users` → captures `$2` = `users`

* Rewritten to: `/users` before forwarding to service

**Alternative approach (without rewrite):**

* Deploy services that listen on the exact paths (e.g., `/v1/users`)

* Or configure the application to handle the prefix

**Debugging steps:**

```bash
# Check Ingress status
kubectl describe ingress api-ingress

# Check Ingress controller logs
kubectl logs -n ingress-nginx deployment/ingress-nginx-controller | grep "api.example.com"

# Test directly from Ingress controller
kubectl port-forward -n ingress-nginx service/ingress-nginx-controller 8080:80
curl -H "Host: api.example.com" http://localhost:8080/v1/users
```

### Question 4

**Scenario:** A company has three microservices: `web`, `api`, and `db`. They want:

* `web` can talk to `api`

* `api` can talk to `db`

* `web` cannot talk directly to `db`

* No other pods can talk to any of these services

**Question:** Write the complete NetworkPolicy YAMLs for this scenario.

**Answer:**

```yaml
# default-deny-all.yaml
# Deny all ingress to all pods in the namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
spec:
  podSelector: {}
  policyTypes:
  - Ingress

---
# allow-web-to-api.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-web-to-api
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: web
    ports:
    - protocol: TCP
      port: 8080

---
# allow-api-to-db.yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-api-to-db
spec:
  podSelector:
    matchLabels:
      app: db
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: api
    ports:
    - protocol: TCP
      port: 5432

---
# allow-dns-egress.yaml (optional, for DNS resolution)
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns-egress
spec:
  podSelector: {}
  policyTypes:
  - Egress
  egress:
  - to:
    - namespaceSelector: {}
      podSelector:
        matchLabels:
          k8s-app: kube-dns
    ports:
    - protocol: UDP
      port: 53
    - protocol: TCP
      port: 53
```

**Verification:**

```bash
# Label pods appropriately
kubectl label pod web-xxx app=web
kubectl label pod api-xxx app=api
kubectl label pod db-xxx app=db

# Get pod IPs
kubectl get pods -o wide

# Test web -> api (should succeed)
kubectl exec web-xxx -- curl http://<api-ip>:8080

# Test api -> db (should succeed)
kubectl exec api-xxx -- nc -zv <db-ip> 5432

# Test web -> db (should fail)
kubectl exec web-xxx -- nc -zv <db-ip> 5432
```

### Question 5

**Scenario:** A cluster uses Flannel as the CNI plugin. You create NetworkPolicy resources, but they don't seem to have any effect – all traffic still flows freely.

**Question:** Why aren't the NetworkPolicies working? How would you fix it without replacing the entire CNI?

**Answer:**

**Problem:** Flannel does not support NetworkPolicy natively. It only provides pod networking (IP assignment), not network security policies.

**Fix:** Install Calico in policy-only mode alongside Flannel (Canal).

```bash
# Install Calico for network policies only (Canal)
kubectl apply -f https://raw.githubusercontent.com/projectcalico/calico/v3.27/manifests/canal.yaml

# Canal uses Flannel for networking, Calico for policies
```

**Alternative solutions:**

| Solution                               | Complexity | Downtime | Notes                    |
| -------------------------------------- | ---------- | -------- | ------------------------ |
| **Canal**                              | Low        | None     | Flannel + Calico policy  |
| **Replace Flannel with Calico**        | High       | Some     | Complete replacement     |
| **Switch to Cilium**                   | High       | Some     | Feature-rich, eBPF-based |
| **Install Cilium in policy-only mode** | Medium     | None     | Cilium ClusterMesh       |

**Verify Calico is running:**

```bash
# Check Calico pods
kubectl get pods -n calico-system

# Check if policies are now enforced
kubectl get netpol
kubectl describe netpol my-policy
```

**Why Flannel lacks policy support:**

* Flannel only implements the CNI specification for networking

* NetworkPolicy requires additional control plane components (Felix in Calico, Cilium agent)

* Flannel's minimal design doesn't include policy enforcement

**Prevention:**

* When setting up a new cluster, choose a CNI with policy support (Calico, Cilium, Weave)

* For kubeadm: `--pod-network-cidr=10.244.0.0/16` with Calico

* For managed clusters: Most support NetworkPolicy (EKS with Calico, GKE Dataplane V2)
