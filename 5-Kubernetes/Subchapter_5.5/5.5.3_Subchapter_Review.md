## Cheatsheet: Kubernetes Storage

### Ephemeral Volume Types

| Volume Type           | Lifetime      | Use Case                          |
| --------------------- | ------------- | --------------------------------- |
| **emptyDir**          | Pod lifetime  | Temporary cache, shared tmp space |
| **emptyDir (Memory)** | Pod lifetime  | Fast, RAM-backed cache            |
| **hostPath**          | Node lifetime | Node debugging, daemon logs       |
| **configMap**         | Pod lifetime  | Configuration injection           |
| **secret**            | Pod lifetime  | Credentials, TLS certs            |
| **downwardAPI**       | Pod lifetime  | Pod metadata exposure             |

### emptyDir YAML

```yaml
volumes:
- name: cache
  emptyDir:
    medium: ""        # "" = disk, "Memory" = RAM
    sizeLimit: 1Gi    # Optional limit
```

### hostPath YAML

```yaml
volumes:
- name: host-logs
  hostPath:
    path: /var/log
    type: Directory    # Directory, File, Socket, DirectoryOrCreate
```

### hostPath Types

| Type                | Behavior          |
| ------------------- | ----------------- |
| `DirectoryOrCreate` | Create if missing |
| `Directory`         | Must exist        |
| `FileOrCreate`      | Create if missing |
| `File`              | Must exist        |
| `Socket`            | Unix socket       |
| `CharDevice`        | Character device  |
| `BlockDevice`       | Block device      |

### DownwardAPI YAML

```yaml
volumes:
- name: podinfo
  downwardAPI:
    items:
    - path: "pod-name"
      fieldRef:
        fieldPath: metadata.name
    - path: "labels"
      fieldRef:
        fieldPath: metadata.labels
```

### PersistentVolume YAML

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: manual
  hostPath:
    path: /mnt/data
```

### PersistentVolumeClaim YAML

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
  storageClassName: manual
```

### StorageClass YAML

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast
provisioner: kubernetes.io/aws-ebs
parameters:
  type: gp3
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true
```

### Access Modes

| Mode             | Abbr | Description            |
| ---------------- | ---- | ---------------------- |
| ReadWriteOnce    | RWO  | Single node read/write |
| ReadOnlyMany     | ROX  | Many nodes read-only   |
| ReadWriteMany    | RWX  | Many nodes read/write  |
| ReadWriteOncePod | RWOP | Single pod (alpha)     |

### PV Reclaim Policies

| Policy    | Behavior                   |
| --------- | -------------------------- |
| `Retain`  | PV kept after PVC deletion |
| `Delete`  | PV and storage deleted     |
| `Recycle` | Deprecated                 |

### StorageClass Volume Binding Modes

| Mode                   | Behavior                      |
| ---------------------- | ----------------------------- |
| `Immediate`            | PV created immediately        |
| `WaitForFirstConsumer` | PV created when pod scheduled |

### Cloud Storage Provisioners

| Cloud      | Provisioner                    | Example Parameters                |
| ---------- | ------------------------------ | --------------------------------- |
| AWS EBS    | `kubernetes.io/aws-ebs`        | `type: gp3`                       |
| GCE PD     | `kubernetes.io/gce-pd`         | `type: pd-ssd`                    |
| Azure Disk | `kubernetes.io/azure-disk`     | `storageaccounttype: Premium_LRS` |
| vSphere    | `kubernetes.io/vsphere-volume` | `diskformat: thin`                |

### CSI Drivers

| Vendor   | Driver                  | Features                   |
| -------- | ----------------------- | -------------------------- |
| AWS EBS  | `ebs.csi.aws.com`       | Snapshot, resize, topology |
| GCP PD   | `pd.csi.storage.gke.io` | Snapshot, resize           |
| NFS      | `nfs.csi.k8s.io`        | RWX volumes                |
| Ceph RBD | `rbd.csi.ceph.com`      | RWO, RWX                   |

## Comparison Tables

### Ephemeral vs Persistent Storage

| Feature                  | emptyDir   | hostPath            | PersistentVolume         |
| ------------------------ | ---------- | ------------------- | ------------------------ |
| **Lifetime**             | Pod        | Node                | Beyond pod/node          |
| **Data on pod delete**   | Lost       | Lost (on same node) | Preserved                |
| **Data on node failure** | Lost       | Lost                | Depends on backend       |
| **Shared across nodes**  | No         | No                  | Yes (with RWX)           |
| **Use case**             | Cache, tmp | Node debugging      | Databases, stateful apps |

### Static vs Dynamic Provisioning

| Feature          | Static Provisioning | Dynamic Provisioning |
| ---------------- | ------------------- | -------------------- |
| **PV creation**  | Admin manually      | StorageClass auto    |
| **Scalability**  | Limited             | Unlimited            |
| **StorageClass** | Not required        | Required             |
| **Use case**     | On-prem, NFS        | Cloud, automated     |

### Access Modes Comparison

| Mode                           | RWO | ROX | RWX | RWOP |
| ------------------------------ | --- | --- | --- | ---- |
| Same node, multiple pods       | No  | Yes | Yes | No   |
| Different nodes, multiple pods | No  | Yes | Yes | No   |
| Single pod                     | Yes | Yes | Yes | Yes  |
| Databases                      | ✓   | ✗   | ✗   | ✓    |
| Web servers (read-only)        | ✗   | ✓   | ✓   | ✗    |
| Shared storage (NFS)           | ✗   | ✓   | ✓   | ✗    |

## Interview Questions (Scenario-Based)

These questions assume only knowledge from Subchapter 5.5. Answers reference only concepts from 5.5.1 and 5.5.2.

### Question 1

**Scenario:** A developer creates a pod with an emptyDir volume to store temporary data. When the pod is restarted (not deleted), the data disappears.

**Question:** Is this expected behavior? Explain the lifecycle of emptyDir. How would you modify the setup to preserve data across pod restarts?

**Answer:**

**Expected behavior?** Yes, emptyDir volumes are tied to the pod's lifetime. When a pod restarts (due to crash or manual restart), a new pod instance is created, and the emptyDir is recreated empty.

**emptyDir lifecycle:**

* Created when pod is assigned to a node

* Exists as long as the pod is running (including across container restarts within the same pod)

* **Deleted** when the pod is removed from the node (not just restarted)

**What happens on pod restart:**

* `kubectl delete pod mypod` → Pod deleted, emptyDir deleted

* New pod created → New emptyDir (empty)

**To preserve data across pod restarts:** Use a PersistentVolumeClaim.

```yaml
# persistent-storage.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: data-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
---
apiVersion: v1
kind: Pod
metadata:
  name: mypod
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: data
      mountPath: /data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: data-pvc
```

**When to use emptyDir vs PVC:**

| Use Case             | Recommended |
| -------------------- | ----------- |
| Temporary cache      | emptyDir    |
| Build artifacts (CI) | emptyDir    |
| Database data        | PVC         |
| User uploads         | PVC         |
| Logs (if important)  | PVC         |

### Question 2

**Scenario:** A StatefulSet for PostgreSQL is deployed with 3 replicas. Each pod has its own PVC. One of the database nodes fails and is recreated.

**Question:** What happens to the PVC when a StatefulSet pod is deleted and recreated? Will the new pod get a new empty volume or reuse the existing data?

**Answer:**

**When a StatefulSet pod is deleted:**

* The pod is removed, but the PVC **remains** (by design)

* Data is preserved in the PVC

**When the pod is recreated (by StatefulSet controller):**

* The new pod with the same ordinal index (e.g., `postgres-1`) is created

* It binds to the **existing PVC** (same name: `data-postgres-1`)

* Data is preserved and reused

**Verification:**

```bash
# Check PVCs before deletion
kubectl get pvc
# NAME               STATUS   VOLUME
# data-postgres-0    Bound    pv-0
# data-postgres-1    Bound    pv-1
# data-postgres-2    Bound    pv-2

# Delete pod
kubectl delete pod postgres-1

# PVC still exists
kubectl get pvc
# data-postgres-1    Bound    pv-1   (still there!)

# StatefulSet recreates pod
kubectl get pods
# postgres-1    Running

# New pod uses same PVC
kubectl describe pod postgres-1 | grep -A 5 Volumes
```

**Important distinction:**

| Action                    | PVC Behavior                     |
| ------------------------- | -------------------------------- |
| Pod deletion (individual) | PVC preserved                    |
| StatefulSet scale down    | PVC preserved (data not deleted) |
| StatefulSet deletion      | PVC preserved (by default)       |
| PVC deletion              | Data lost                        |

**To delete PVCs when deleting StatefulSet:**

```bash
# Delete StatefulSet and its PVCs
kubectl delete statefulset postgres --cascade=orphan  # Leaves PVCs
kubectl delete pvc --all  # Delete PVCs separately

# Or use Helm (if deployed with Helm)
helm uninstall postgres --delete-pvcs
```

### Question 3

**Scenario:** A PVC is created with `storage: 10Gi`. The pod using it runs out of space. You want to increase the storage to 20Gi without downtime.

**Question:** What are the requirements for expanding PVCs? How would you perform the expansion? What happens to existing data?

**Answer:**

**Requirements for PVC expansion:**

1. **StorageClass must support expansion:**

```yaml
allowVolumeExpansion: true
```

1. **Volume mode must be Filesystem (not Block)**

2. **CSI driver or in-tree provisioner must support resize**

**Perform expansion (zero downtime):**

```bash
# Step 1: Edit PVC
kubectl edit pvc my-pvc
# Change storage: 10Gi to storage: 20Gi

# OR
kubectl patch pvc my-pvc -p '{"spec":{"resources":{"requests":{"storage":"20Gi"}}}}'

# Step 2: Check status
kubectl get pvc my-pvc
# NAME     STATUS   VOLUME   CAPACITY   ACCESS MODES   STORAGECLASS
# my-pvc   Bound    pv-1     10Gi       RWO            gp2
# (capacity still shows 10Gi initially)

# Step 3: Watch for expansion
kubectl get pvc my-pvc -w
# After some time, capacity updates to 20Gi
```

**What happens to existing data:**

* Data is **preserved** (filesystem is resized online)

* No downtime required (pod continues running)

**Verification after expansion:**

```bash
# Check PVC capacity
kubectl get pvc my-pvc
# CAPACITY: 20Gi

# Check filesystem inside pod
kubectl exec my-pod -- df -h /mount-path
# Should show ~20Gi

# If filesystem not resized automatically:
kubectl exec my-pod -- resize2fs /dev/xxx  # ext4
# or
kubectl exec my-pod -- xfs_growfs /mount-path  # XFS
```

**Limitations:**

| Limitation                   | Behavior                                      |
| ---------------------------- | --------------------------------------------- |
| Shrinking PVC                | Not supported (can only expand)               |
| Expand while pod not running | Works, but pod will see new size when started |
| Expand with ReadWriteOnce    | Works (single node)                           |
| Expand with ReadWriteMany    | Depends on storage backend                    |

### Question 4

**Scenario:** A cluster has multiple StorageClasses: `fast-ssd` (high performance, expensive) and `slow-hdd` (cheap, slow). Developers are creating PVCs without specifying `storageClassName`.

**Question:** Which StorageClass will be used? How can you control the default StorageClass? How can you ensure production workloads use the fast SSD?

**Answer:**

**Default StorageClass behavior:**

* If a PVC doesn't specify `storageClassName`, the **default StorageClass** is used

* Only one StorageClass can be default

**Check default StorageClass:**

```bash
kubectl get storageclass
# NAME        PROVISIONER             AGE   DEFAULT
# fast-ssd    kubernetes.io/aws-ebs   10d   true
# slow-hdd    kubernetes.io/aws-ebs   10d
```

**Set default StorageClass:**

```bash
# Remove existing default annotation
kubectl patch storageclass slow-hdd -p '{"metadata": {"annotations":{"storageclass.kubernetes.io/is-default-class":"false"}}}'

# Set new default
kubectl patch storageclass fast-ssd -p '{"metadata": {"annotations":{"storageclass.kubernetes.io/is-default-class":"true"}}}'
```

**Ensure production workloads use fast SSD:**

**Option 1: Explicit StorageClass in PVC**

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: production-pvc
spec:
  storageClassName: fast-ssd  # Explicit
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 100Gi
```

**Option 2: Admission Webhook (advanced)**

* Mutating webhook to inject `storageClassName` based on namespace labels

* Example: `production` namespace gets `fast-ssd`, `development` gets `slow-hdd`

**Option 3: Namespace default StorageClass override (using admission)**

```yaml
# Create a mutating admission webhook configuration
# Or use Open Policy Agent (OPA) Gatekeeper
```

**Option 4: RBAC to restrict StorageClass usage**

```yaml
# Limit which StorageClasses developers can use
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: production
  name: storage-class-restriction
rules:
- apiGroups: ["storage.k8s.io"]
  resources: ["storageclasses"]
  verbs: ["get", "list", "watch"]
  resourceNames: ["fast-ssd"]  # Only allow fast-ssd
```

### Question 5

**Scenario:** A pod is stuck in `ContainerCreating` with the event:

```
MountVolume.SetUp failed for volume "pv-abc123" : mount failed: exit status 32
```

**Question:** What does this error typically indicate? How would you diagnose and resolve it?

**Answer:**

**Error interpretation:**

* Exit status 32 typically indicates a mount error

* The underlying storage (NFS, EBS, etc.) is not accessible from the node

**Common causes and diagnostics:**

**Cause 1: NFS export not accessible**

```bash
# On the node where pod is scheduled
sudo mount -t nfs <nfs-server>:/export /mnt/test
# Check error message

# Check NFS server is reachable
ping <nfs-server>
showmount -e <nfs-server>
```

**Cause 2: EBS volume in different availability zone**

```bash
# Check PV zone affinity
kubectl describe pv pv-abc123 | grep "Node Affinity"

# Check pod's node zone
kubectl get node <node> -o yaml | grep failure-domain.beta.kubernetes.io/zone

# If zones don't match, volume won't mount
```

**Cause 3: Permission issues**

```bash
# For hostPath, check permissions
kubectl debug node/<node> -it --image=busybox -- ls -la /mnt/data

# For NFS, check export permissions
cat /etc/exports  # On NFS server
```

**Diagnosis workflow:**

```bash
# 1. Get pod details
kubectl describe pod my-pod

# 2. Get node where pod is scheduled
NODE=$(kubectl get pod my-pod -o jsonpath='{.spec.nodeName}')

# 3. Check kubelet logs on that node (requires SSH or node debugging)
kubectl debug node/${NODE} -it --image=busybox -- tail -100 /var/log/kubelet.log

# 4. Check PV/PVC status
kubectl get pv
kubectl get pvc

# 5. Check if volume is already mounted elsewhere
kubectl get pods --all-namespaces -o wide | grep <volume-name>
```

**Resolution steps:**

| Cause                  | Fix                                    |
| ---------------------- | -------------------------------------- |
| NFS unreachable        | Check NFS server, network, firewall    |
| EBS zone mismatch      | Delete PVC, recreate in correct zone   |
| Permission denied      | Fix permissions on hostPath/NFS export |
| Volume already mounted | Delete pod using volume elsewhere      |

**Quick recovery:**

```bash
# Delete the pod (may reschedule to a node that can mount the volume)
kubectl delete pod my-pod

# If pod keeps failing, delete PVC (warning: data loss!)
kubectl delete pvc my-pvc
kubectl delete pv my-pv
```
