# 5.5.1 Ephemeral Volumes: emptyDir, hostPath, and Ephemeral Storage

#### Why Ephemeral Storage Matters

Not all data needs to persist beyond a pod's lifetime. Ephemeral storage is useful for:

* **Shared temporary space** between containers in the same pod (e.g., sidecar logging)

* **Cache data** that can be regenerated

* **Node-level storage** for debugging or performance

* **Empty directories** that start empty each pod start

This note covers ephemeral volume types. Note 5.5.2 covers PersistentVolumes and StorageClasses; note 5.5.3 is the subchapter review.

**Backward references:** Pods from 5.3.1; container volumes from Module 4 (bind mounts, tmpfs); Linux filesystem from Module 1.

***

## Part 1: Volume Types Overview

| Volume Type          | Lifetime      | Persistence                          | Use Case                          |
| -------------------- | ------------- | ------------------------------------ | --------------------------------- |
| **emptyDir**         | Pod lifetime  | Lost when pod deleted                | Temporary cache, shared tmp space |
| **hostPath**         | Node lifetime | Survives pod deletion (on same node) | Node-level debugging, daemon logs |
| **configMap/secret** | Pod lifetime  | Read-only config                     | Configuration injection           |
| **downwardAPI**      | Pod lifetime  | Pod metadata                         | Expose pod info to containers     |
| **PersistentVolume** | Beyond pod    | Survives node failures               | Databases, stateful apps          |

***

## Part 2: emptyDir – Temporary Pod Storage

emptyDir volumes are created when a pod is assigned to a node and exist as long as the pod runs. They start empty and are deleted when the pod is removed.

```yaml
# pod-emptydir.yaml
apiVersion: v1
kind: Pod
metadata:
  name: emptydir-demo
spec:
  containers:
  - name: writer
    image: busybox
    command: ['sh', '-c', 'echo "Hello from writer" > /cache/data.txt && sleep 3600']
    volumeMounts:
    - name: shared-cache
      mountPath: /cache
  - name: reader
    image: busybox
    command: ['sh', '-c', 'sleep 3600']
    volumeMounts:
    - name: shared-cache
      mountPath: /data
      readOnly: true
  volumes:
  - name: shared-cache
    emptyDir: {}
```

### emptyDir with Medium (Memory Backing)

```yaml
# pod-emptydir-memory.yaml
volumes:
- name: tmpfs-storage
  emptyDir:
    medium: Memory   # Use tmpfs (RAM-backed)
    sizeLimit: 256Mi # Limit to 256MB
```

### Use Cases for emptyDir

| Use Case                  | Example                                  |
| ------------------------- | ---------------------------------------- |
| **Shared scratch space**  | Two containers sharing temporary files   |
| **Cache**                 | Downloaded assets, compiled binaries     |
| **Log aggregation**       | Sidecar reading logs from main container |
| **Sort/merge operations** | Large temporary datasets                 |

### emptyDir vs tmpfs (Docker)

| Feature            | emptyDir (disk)   | emptyDir (memory)     |
| ------------------ | ----------------- | --------------------- |
| **Backing**        | Node disk         | Node RAM              |
| **Speed**          | Disk speed        | Very fast             |
| **Persistence**    | Pod lifetime      | Pod lifetime          |
| **Capacity limit** | Yes (`sizeLimit`) | Yes (`sizeLimit`)     |
| **Use case**       | Large temp files  | Fast, ephemeral cache |

## Part 3: hostPath – Node Filesystem Access

hostPath mounts a file or directory from the host node's filesystem into the pod.

```yaml
# pod-hostpath.yaml
apiVersion: v1
kind: Pod
metadata:
  name: hostpath-demo
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: host-logs
      mountPath: /host/var/log
    - name: docker-socket
      mountPath: /var/run/docker.sock
  volumes:
  - name: host-logs
    hostPath:
      path: /var/log
      type: Directory
  - name: docker-socket
    hostPath:
      path: /var/run/docker.sock
      type: Socket
```

### hostPath Types

| Type                | Behavior                       |
| ------------------- | ------------------------------ |
| `Directory`         | Directory must exist           |
| `DirectoryOrCreate` | Create directory if not exists |
| `File`              | File must exist                |
| `FileOrCreate`      | Create file if not exists      |
| `Socket`            | Unix socket must exist         |
| `CharDevice`        | Character device must exist    |
| `BlockDevice`       | Block device must exist        |

### Use Cases for hostPath

| Use Case               | Example                     |
| ---------------------- | --------------------------- |
| **Node monitoring**    | Access `/proc`, `/sys`      |
| **Log collection**     | Read `/var/log`             |
| **Docker socket**      | Access Docker daemon (DinD) |
| **Device access**      | GPU, FPGA, USB devices      |
| **Node configuration** | Read `/etc/kubernetes/pki`  |

### Security Warning

```yaml
# DANGEROUS – Pod can read host's root filesystem
volumes:
- name: host-root
  hostPath:
    path: /
    type: Directory
```

**Risks:**

* Pod can read/write any host file

* Can escape container isolation

* Use only for trusted workloads

* Consider using `readOnly: true`

```yaml
# Safer – read-only hostPath
volumeMounts:
- name: host-logs
  mountPath: /host/var/log
  readOnly: true
```

## Part 4: ConfigMap and Secret Volumes

ConfigMaps and Secrets can be mounted as volumes, updating automatically when the ConfigMap/Secret changes.

```yaml
# pod-configmap-volume.yaml
apiVersion: v1
kind: Pod
metadata:
  name: configmap-demo
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: config
      mountPath: /etc/config
      readOnly: true
  volumes:
  - name: config
    configMap:
      name: app-config
      items:
      - key: nginx.conf
        path: nginx.conf
      - key: app.properties
        path: app.properties
```

```yaml
# pod-secret-volume.yaml
apiVersion: v1
kind: Pod
metadata:
  name: secret-demo
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: secrets
      mountPath: /etc/secrets
      readOnly: true
  volumes:
  - name: secrets
    secret:
      secretName: app-secrets
      defaultMode: 0400  # Read-only for non-root
```

### ConfigMap/Secret Volume Features

| Feature                | Description                                                              |
| ---------------------- | ------------------------------------------------------------------------ |
| **Auto-update**        | Files update when ConfigMap/Secret changes (kubelet sync period \~1 min) |
| **Items filter**       | Mount only specific keys                                                 |
| **Path customization** | Map keys to different filenames                                          |
| **Permission mode**    | `defaultMode` for all files                                              |
| **Optional**           | `optional: true` if ConfigMap may not exist                              |

## Part 5: DownwardAPI Volume

DownwardAPI exposes pod metadata (labels, annotations, pod IP, namespace) to containers via files.

```yaml
# pod-downwardapi.yaml
apiVersion: v1
kind: Pod
metadata:
  name: downwardapi-demo
  labels:
    app: myapp
    environment: production
  annotations:
    build-id: "12345"
spec:
  containers:
  - name: app
    image: busybox
    command: ['sh', '-c', 'cat /etc/podinfo/* && sleep 3600']
    volumeMounts:
    - name: podinfo
      mountPath: /etc/podinfo
      readOnly: true
  volumes:
  - name: podinfo
    downwardAPI:
      items:
      - path: "pod-name"
        fieldRef:
          fieldPath: metadata.name
      - path: "pod-namespace"
        fieldRef:
          fieldPath: metadata.namespace
      - path: "pod-ip"
        fieldRef:
          fieldPath: status.podIP
      - path: "labels"
        fieldRef:
          fieldPath: metadata.labels
      - path: "annotations"
        fieldRef:
          fieldPath: metadata.annotations
      - path: "cpu-limit"
        resourceFieldRef:
          containerName: app
          resource: limits.cpu
          divisor: 1m
      - path: "memory-limit"
        resourceFieldRef:
          containerName: app
          resource: limits.memory
          divisor: 1Mi
```

### DownwardAPI Available Fields

| Field                       | Description          |
| --------------------------- | -------------------- |
| `metadata.name`             | Pod name             |
| `metadata.namespace`        | Pod namespace        |
| `metadata.labels`           | All pod labels       |
| `metadata.annotations`      | All pod annotations  |
| `status.podIP`              | Pod IP address       |
| `status.hostIP`             | Node IP address      |
| `spec.nodeName`             | Node name            |
| `spec.serviceAccountName`   | Service account name |
| `resources.limits.cpu`      | CPU limit            |
| `resources.limits.memory`   | Memory limit         |
| `resources.requests.cpu`    | CPU request          |
| `resources.requests.memory` | Memory request       |

## Part 6: Ephemeral Volume for Logging Sidecar

```yaml
# pod-logging-sidecar.yaml
apiVersion: v1
kind: Pod
metadata:
  name: logging-sidecar-demo
spec:
  containers:
  - name: app
    image: busybox
    command: ['sh', '-c', 'while true; do echo "$(date): Application log" >> /var/log/app.log; sleep 2; done']
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log
  - name: log-collector
    image: busybox
    command: ['sh', '-c', 'tail -f /var/log/app.log']
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log
  volumes:
  - name: shared-logs
    emptyDir: {}
```

## Part 7: Ephemeral Storage Requests and Limits

Kubernetes 1.8+ supports ephemeral storage requests/limits for `emptyDir` volumes.

```yaml
# pod-ephemeral-storage.yaml
apiVersion: v1
kind: Pod
metadata:
  name: ephemeral-storage-demo
spec:
  containers:
  - name: app
    image: nginx
    resources:
      requests:
        ephemeral-storage: "1Gi"
      limits:
        ephemeral-storage: "2Gi"
    volumeMounts:
    - name: cache
      mountPath: /cache
  volumes:
  - name: cache
    emptyDir:
      sizeLimit: 1Gi
```

```bash
# Check ephemeral storage usage
kubectl exec ephemeral-storage-demo -- df -h /cache

# Check pod's ephemeral storage consumption
kubectl describe pod ephemeral-storage-demo | grep -A 5 "Ephemeral"
```

## Part 8: Troubleshooting Ephemeral Volumes

### Common Issues

**Issue 1: Pod fails to start due to hostPath missing**

```bash
# Check if path exists on node
kubectl debug node/<node-name> -it --image=busybox -- ls -la /path

# Use DirectoryOrCreate type
hostPath:
  path: /path
  type: DirectoryOrCreate
```

**Issue 2: emptyDir filling up node disk**

```bash
# Set sizeLimit
emptyDir:
  sizeLimit: 1Gi

# Check usage
kubectl exec pod -- df -h /mount-path
```

**Issue 3: ConfigMap not updating in volume**

```bash
# Check kubelet sync period (default 1 minute)
# Force restart pod to get latest ConfigMap
kubectl rollout restart deployment myapp
```

**Issue 4: Permission denied on hostPath**

```bash
# Check host directory permissions
ls -la /host/path

# Run container with privileged or specific user
securityContext:
  runAsUser: 0  # root
```

## Summary Table: Ephemeral Volume Types

| Volume Type     | Lifetime | Persistence              | Use Case                    |
| --------------- | -------- | ------------------------ | --------------------------- |
| **emptyDir**    | Pod      | Lost on pod delete       | Temporary cache, shared tmp |
| **hostPath**    | Node     | Survives pod (same node) | Node debugging, daemon logs |
| **configMap**   | Pod      | Read-only config         | Configuration injection     |
| **secret**      | Pod      | Read-only secrets        | Credentials, TLS certs      |
| **downwardAPI** | Pod      | Pod metadata             | Expose pod info             |

### Volume Syntax Comparison

| Volume Type       | Volume Definition                               | Mount Example             |
| ----------------- | ----------------------------------------------- | ------------------------- |
| emptyDir          | `emptyDir: {}`                                  | `mountPath: /cache`       |
| emptyDir (memory) | `emptyDir: {medium: Memory, sizeLimit: 256Mi}`  | `mountPath: /tmpfs`       |
| hostPath          | `hostPath: {path: /host/path, type: Directory}` | `mountPath: /host`        |
| configMap         | `configMap: {name: my-config}`                  | `mountPath: /etc/config`  |
| secret            | `secret: {secretName: my-secret}`               | `mountPath: /etc/secrets` |

### hostPath Types

| Type                | Behavior          |
| ------------------- | ----------------- |
| `DirectoryOrCreate` | Create if missing |
| `Directory`         | Must exist        |
| `FileOrCreate`      | Create if missing |
| `File`              | Must exist        |
| `Socket`            | Unix socket       |
| `CharDevice`        | Character device  |
| `BlockDevice`       | Block device      |
