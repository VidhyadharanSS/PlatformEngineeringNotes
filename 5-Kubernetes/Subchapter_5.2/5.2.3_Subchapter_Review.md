## Cheatsheet: High Availability and etcd Management

### HA Architecture Options

| Architecture      | etcd Location            | Pros                                  | Cons             |
| ----------------- | ------------------------ | ------------------------------------- | ---------------- |
| **Stacked etcd**  | Same nodes as API server | Simpler setup, fewer nodes            | Shared resources |
| **External etcd** | Dedicated nodes          | Better isolation, independent scaling | More complex     |

### Control Plane HA Requirements

| Component              | HA Strategy                       | Failure Tolerance    |
| ---------------------- | --------------------------------- | -------------------- |
| **API Server**         | Multiple replicas + load balancer | All but one can fail |
| **etcd**               | Raft consensus (odd number)       | (N-1)/2 failures     |
| **Scheduler**          | Leader election                   | Active-standby       |
| **Controller Manager** | Leader election                   | Active-standby       |

### etcd Quorum

| Cluster Size | Quorum | Failures Tolerated |
| ------------ | ------ | ------------------ |
| 1            | 1      | 0                  |
| 3            | 2      | 1                  |
| 5            | 3      | 2                  |
| 7            | 4      | 3                  |

### kubeadm HA Commands

| Command                                                        | Purpose                    |
| -------------------------------------------------------------- | -------------------------- |
| `kubeadm init --control-plane-endpoint LB:6443 --upload-certs` | HA init with load balancer |
| `kubeadm join --control-plane --certificate-key KEY`           | Join additional master     |
| `kubeadm token create --print-join-command`                    | Generate join token        |

### etcdctl Commands

| Command                                                             | Purpose              |
| ------------------------------------------------------------------- | -------------------- |
| `etcdctl snapshot save /path/backup.db`                             | Create snapshot      |
| `etcdctl snapshot restore /path/backup.db --data-dir=/var/lib/etcd` | Restore snapshot     |
| `etcdctl snapshot status /path/backup.db`                           | Verify snapshot      |
| `etcdctl endpoint health`                                           | Check member health  |
| `etcdctl endpoint status`                                           | Show Raft info, size |
| `etcdctl member list`                                               | List cluster members |
| `etcdctl alarm list`                                                | List alarms          |
| `etcdctl alarm disarm`                                              | Clear alarms         |
| `etcdctl compact N`                                                 | Compact history      |
| `etcdctl defrag`                                                    | Defragment database  |

### etcd Environment Variables

```bash
export ETCDCTL_API=3
export ETCDCTL_CACERT=/etc/kubernetes/pki/etcd/ca.crt
export ETCDCTL_CERT=/etc/kubernetes/pki/etcd/server.crt
export ETCDCTL_KEY=/etc/kubernetes/pki/etcd/server.key
```

### etcd Data Directory Structure

```
/var/lib/etcd/member/
├── snap/
│   └── db          # Snapshot database file
└── wal/
    └── *.wal       # Write-ahead logs
```

### Pod Disruption Budget (PDB)

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: app-pdb
spec:
  minAvailable: 2   # or maxUnavailable: 1
  selector:
    matchLabels:
      app: myapp
```

### Control Plane Taints

| Taint                                              | Effect            | Purpose               |
| -------------------------------------------------- | ----------------- | --------------------- |
| `node-role.kubernetes.io/control-plane:NoSchedule` | No pods scheduled | Protect control plane |

### HAProxy Configuration (Layer 4)

```haproxy
frontend kubernetes-api
    bind *:6443
    default_backend kubernetes-masters

backend kubernetes-masters
    balance roundrobin
    server master-1 10.0.0.10:6443 check
    server master-2 10.0.0.11:6443 check
    server master-3 10.0.0.12:6443 check
```

## Comparison Tables

### Stacked etcd vs External etcd

| Aspect                 | Stacked etcd    | External etcd               |
| ---------------------- | --------------- | --------------------------- |
| **Nodes required**     | 3 or 5          | 5-8 (3 etcd + 2+ API)       |
| **Resource isolation** | Shared          | Separate                    |
| **Setup complexity**   | Low             | High                        |
| **Maintenance**        | Together        | Independent                 |
| **Use case**           | Most production | Large scale, separate teams |

### HA Setup Methods

| Method                    | Complexity | Production Ready | Cloud Compatible |
| ------------------------- | ---------- | ---------------- | ---------------- |
| **kubeadm stacked etcd**  | Medium     | Yes              | Yes              |
| **kubeadm external etcd** | High       | Yes              | Yes              |
| **Managed (EKS/GKE/AKS)** | Low        | Yes              | Yes (cloud only) |
| **K3s HA**                | Medium     | Yes              | Yes              |

### etcd Failure Scenarios

| Scenario                  | Impact                | Recovery Method                 |
| ------------------------- | --------------------- | ------------------------------- |
| Single member down        | Cluster still healthy | Restart member                  |
| Two members down (3-node) | Quorum lost           | Restore from snapshot           |
| Data corruption           | Cluster unstable      | Restore from snapshot           |
| Disk full                 | NOSPACE alarm         | Compact, defrag, increase quota |
| Network partition         | Split brain (rare)    | Raft heals after partition      |

## Interview Questions (Scenario-Based)

These questions assume only knowledge from Subchapter 5.2. Answers reference only concepts from 5.2.1 and 5.2.2.

### Question 1

**Scenario:** A production cluster has 3 control plane nodes. One master node fails completely (hardware failure, cannot recover). You have a backup of etcd from 6 hours ago.

**Question:** Is the cluster still operational? What is the immediate impact? How would you recover the failed master? What data loss might occur?

**Answer:**

**Is the cluster still operational?**

* **Yes** – With 3 masters, quorum is 2. One failure leaves 2 healthy masters, maintaining quorum.

* API server load balancer routes traffic to the two remaining masters.

* Existing workloads continue running normally.

**Immediate impact:**

* Reduced fault tolerance (only 2 masters left; another failure would lose quorum)

* `kubectl` commands still work (via remaining masters)

* New pods can still be scheduled

**Recovery steps for failed master:**

```bash
# On a new machine (same IP or new IP)

# 1. Install kubeadm, kubelet, kubectl
# 2. Join as control plane node
kubeadm join lb.example.com:6443 --token <token> \
  --discovery-token-ca-cert-hash sha256:<hash> \
  --control-plane --certificate-key <key>

# 3. Verify cluster
kubectl get nodes
# All 3 masters should be Ready
```

**If you need to restore from backup (corruption scenario):**

```bash
# On the failed master (or new node)
# 1. Stop kubelet
systemctl stop kubelet

# 2. Restore etcd snapshot
ETCDCTL_API=3 etcdctl snapshot restore /backup/etcd-6hours.db \
  --data-dir=/var/lib/etcd

# 3. Start kubelet
systemctl start kubelet

# 4. Verify
kubectl get nodes
```

**Data loss:**

* **6 hours of changes lost** (deployments, configmaps, secrets, pod state)

* Resources created in the last 6 hours will be missing

* Existing pods (running before backup) continue running but are not recorded in etcd

* Recommendation: Increase backup frequency (every hour) for production

### Question 2

**Scenario:** A `kubectl get pods` command hangs and eventually times out. You suspect etcd is unhealthy. The cluster has 3 masters with stacked etcd.

**Question:** What commands would you run to diagnose etcd health? How would you identify which master is the leader? What if the leader is the failed node?

**Answer:**

**Diagnosis commands:**

```bash
# On any master node

# 1. Check etcd member list
ETCDCTL_API=3 etcdctl \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  member list

# 2. Check endpoint health (shows each member)
ETCDCTL_API=3 etcdctl endpoint health --cluster
# https://10.0.0.10:2379 is healthy
# https://10.0.0.11:2379 is unhealthy (timeout)
# https://10.0.0.12:2379 is healthy

# 3. Check endpoint status (shows leader)
ETCDCTL_API=3 etcdctl endpoint status --cluster -w table
# +------------------+------------------+---------+---------+-----------+--------+------------+
# |     ENDPOINT     |        ID        | VERSION | DB SIZE | IS LEADER | RAFT INDEX | RAFT TERM |
# +------------------+------------------+---------+---------+-----------+--------+------------+
# | 10.0.0.10:2379   | 1a2b3c4d5e6f7g8h | 3.5.10  | 12 MB   | true      | 12345      | 3          |
# | 10.0.0.11:2379   | 2b3c4d5e6f7g8h9i | 3.5.10  | 12 MB   | false     | 12344      | 3          |
# | 10.0.0.12:2379   | 3c4d5e6f7g8h9i0j | 3.5.10  | 12 MB   | false     | 12344      | 3          |
```

**Identifying the leader:** Look for `IS LEADER = true` in the endpoint status output.

**If the leader is the failed node:**

```bash
# On the failed node (if accessible) or on remaining masters

# 1. Check if leader is reachable
ping 10.0.0.10

# 2. If leader is down, remaining nodes will elect a new leader automatically
# Wait up to election timeout (default 1000ms)

# 3. Verify new leader
ETCDCTL_API=3 etcdctl endpoint status --cluster | grep "true"

# 4. If no new leader elected (quorum lost), restore from backup
```

**Common etcd issues and fixes:**

| Issue               | Symptom             | Fix                        |
| ------------------- | ------------------- | -------------------------- |
| Disk latency high   | Slow reads/writes   | Move etcd to SSD           |
| Network partition   | Split brain         | Check network connectivity |
| Leader unreachable  | No new leader       | Check firewall (port 2380) |
| Database full       | NOSPACE alarm       | Compact and defrag         |
| Certificate expired | TLS handshake fails | Renew certificates         |

### Question 3

**Scenario:** You need to perform a scheduled maintenance on a master node (reboot for kernel update). The cluster has 3 masters with stacked etcd.

**Question:** How would you safely drain the master node? What commands would you use? What happens to etcd quorum during the maintenance? How do you rejoin the node after reboot?

**Answer:**

**Step 1: Check current cluster health**

```bash
# Verify all masters are healthy
kubectl get nodes
kubectl get pods -n kube-system -o wide | grep -E "etcd|kube-apiserver"

# Check etcd members
ETCDCTL_API=3 etcdctl member list
# Ensure 3 members, all healthy
```

**Step 2: Cordon the master node (prevent new pods)**

```bash
kubectl cordon master-1
# node/master-1 cordoned

# Verify scheduling disabled
kubectl describe node master-1 | grep Unschedulable
# Unschedulable: true
```

**Step 3: Drain the master node (evict pods)**

```bash
# Drain control plane node (ignore daemonsets, delete local data)
kubectl drain master-1 --ignore-daemonsets --delete-emptydir-data

# Output:
# node/master-1 already cordoned
# evicting pod kube-system/calico-node-xxx
# evicting pod kube-system/kube-proxy-xxx
# ... (control plane static pods are not evicted)
```

**What happens to etcd quorum:**

* During reboot, master-1's etcd member will be marked as unreachable

* Remaining 2 masters maintain quorum (2/3 = majority)

* Cluster remains operational

**Step 4: Reboot the node**

```bash
sudo reboot
```

**Step 5: After reboot, uncordon the node**

```bash
# Wait for node to become Ready
kubectl get nodes
# master-1   NotReady   ... (waiting for kubelet to start)

# Uncordon to allow scheduling
kubectl uncordon master-1
# node/master-1 uncordoned

# Verify etcd member rejoined
ETCDCTL_API=3 etcdctl member list
# All 3 members should be healthy
```

**Alternative: Using** **`kubeadm`** **to restart control plane**

```bash
# If control plane components don't restart automatically
sudo kubeadm init phase control-plane all --config kubeadm-config.yaml
```

**Post-maintenance verification:**

```bash
# Check all nodes Ready
kubectl get nodes

# Check all control plane pods running
kubectl get pods -n kube-system | grep -E "etcd|apiserver|scheduler|controller"

# Test etcd health
ETCDCTL_API=3 etcdctl endpoint health --cluster
```

### Question 4

**Scenario:** Your etcd database has grown to 3.5GB. The default quota is 2GB. You start seeing `etcdserver: mvcc: database space exceeded` errors. `kubectl` commands are failing.

**Question:** What is causing this? How would you resolve it without losing data? What preventive measures would you implement?

**Answer:**

**Cause:** etcd database has exceeded its quota (default 2GB for embedded etcd in kubeadm). This typically happens due to:

* Many revisions of objects (every update creates a new revision)

* Large objects (secrets, configmaps)

* High churn (frequent deployments, pod restarts)

**Immediate resolution (without data loss):**

```bash
# Step 1: Check current database size
ETCDCTL_API=3 etcdctl endpoint status
# DB SIZE: 3.5 GB

# Step 2: Check alarms
ETCDCTL_API=3 etcdctl alarm list
# memberID:123456789 NOSPACE

# Step 3: Get current revision
REV=$(ETCDCTL_API=3 etcdctl endpoint status --write-out="json" | jq '.[0].Status.header.revision')
echo $REV
# 1234567

# Step 4: Compact old revisions (keep recent history)
# Keep last 1000 revisions or compact to a specific revision
ETCDCTL_API=3 etcdctl compact $((REV - 1000))

# Step 5: Defragment the database (reclaims space)
ETCDCTL_API=3 etcdctl defrag

# Step 6: Disarm the alarm
ETCDCTL_API=3 etcdctl alarm disarm

# Step 7: Verify
ETCDCTL_API=3 etcdctl endpoint status
# DB SIZE: 1.8 GB (reduced)
```

**If compaction doesn't free enough space, increase quota:**

```bash
# Step 1: Edit etcd manifest on each master
sudo vi /etc/kubernetes/manifests/etcd.yaml

# Add or modify:
spec:
  containers:
  - command:
    - etcd
    - --quota-backend-bytes=8589934592   # 8GB

# Step 2: etcd pod will restart automatically
# Step 3: Verify new quota
ETCDCTL_API=3 etcdctl endpoint status
```

**Preventive measures:**

1. **Enable automatic compaction:**

```yaml
# /etc/kubernetes/manifests/etcd.yaml
spec:
  containers:
  - command:
    - etcd
    - --auto-compaction-retention=1000   # Keep 1000 revisions
    - --auto-compaction-mode=revision
    # OR time-based:
    - --auto-compaction-retention=24h    # Keep 24 hours
```

1. **Set up regular defragmentation cron job:**

```bash
#!/bin/bash
# /usr/local/bin/etcd-defrag.sh
ETCDCTL_API=3 etcdctl compact $(ETCDCTL_API=3 etcdctl endpoint status --write-out="json" | jq '.[0].Status.header.revision') --physical
ETCDCTL_API=3 etcdctl defrag
```

1. **Monitor etcd size with Prometheus:**

```yaml
# Prometheus alert
- alert: EtcdDatabaseSizeHigh
  expr: etcd_debugging_mvcc_db_total_size_in_bytes > 1.5e9
  for: 5m
  annotations:
    summary: "etcd database size is high"
```

1. **Regular cleanup of old resources:**

```bash
# Delete old completed jobs
kubectl delete jobs --field-selector status.successful=1 --all-namespaces

# Delete old pods in Completed state
kubectl delete pods --field-selector status.phase=Succeeded --all-namespaces
```

### Question 5

**Scenario:** You have a 5-node etcd cluster. Two nodes simultaneously fail due to a network partition. The remaining 3 nodes cannot communicate with each other (split into two partitions: 2 nodes and 1 node).

**Question:** Can the cluster still make progress? What is the impact? How would you recover?

**Answer:**

**Can the cluster still make progress?**

* **No** – With 5 nodes, quorum is 3. No partition has 3 nodes:

  * Partition A: 2 nodes (not enough for quorum)

  * Partition B: 1 node (not enough for quorum)

  * Partition C: 2 nodes? Wait, total is 5. If 2 failed, 3 remain. But the scenario says the 3 remaining cannot communicate with each other.

* Actually, if the 3 remaining are split (2+1), neither partition has quorum.

* **Result:** etcd cluster loses quorum, becomes unavailable for writes AND reads.

**Impact:**

* `kubectl` commands fail (cannot read or write to etcd)

* New pod scheduling stops

* Existing pods continue running (kubelet has local state)

* No new deployments, scaling, or configuration changes

**Recovery steps:**

```bash
# On the node that is isolated but has the most recent data
# (identify by checking Raft term and index)

# Step 1: Stop kubelet on that node
systemctl stop kubelet

# Step 2: Stop etcd
mv /etc/kubernetes/manifests/etcd.yaml /tmp/

# Step 3: Force etcd to start as single-node cluster
# Edit etcd.yaml to add --force-new-cluster flag
cp /tmp/etcd.yaml /etc/kubernetes/manifests/etcd.yaml
# Add: - --force-new-cluster

# Step 4: etcd will start with this node as sole member
# Wait for etcd to become healthy

# Step 5: Remove force-new-cluster flag
vi /etc/kubernetes/manifests/etcd.yaml
# Remove --force-new-cluster

# Step 6: Add back other nodes
ETCDCTL_API=3 etcdctl member add master-2 --peer-urls=https://10.0.0.11:2380
ETCDCTL_API=3 etcdctl member add master-3 --peer-urls=https://10.0.0.12:2380

# Step 7: On other nodes, restore from snapshot or rejoin
# On master-2:
systemctl stop kubelet
rm -rf /var/lib/etcd
ETCDCTL_API=3 etcdctl snapshot restore /backup/snapshot.db --data-dir=/var/lib/etcd \
  --initial-cluster=master-1=https://10.0.0.10:2380,master-2=https://10.0.0.11:2380,master-3=https://10.0.0.12:2380 \
  --initial-cluster-token=etcd-cluster \
  --initial-advertise-peer-urls=https://10.0.0.11:2380
systemctl start kubelet

# Repeat for master-3
```

**Prevention:**

* Run etcd on reliable network infrastructure

* Use 5 nodes for better fault tolerance (can tolerate 2 failures)

* Monitor network partitions with etcd metrics

* Configure etcd with proper heartbeat intervals for your network

## Bridge Concepts (Not in Notes but Added for Clarity)

| Concept             | Explanation                                                            |
| ------------------- | ---------------------------------------------------------------------- |
| **Raft**            | Consensus algorithm used by etcd. Ensures consistency across replicas. |
| **Quorum**          | Minimum number of nodes that must agree (N/2 + 1). 3 nodes → quorum 2. |
| **Term**            | Logical time in Raft. Increases during leader elections.               |
| **Raft Index**      | Number of log entries committed. Used to compare data freshness.       |
| **Revision**        | etcd's logical clock. Increments on every write.                       |
| **Compaction**      | Removes old revisions to free space.                                   |
| **Defragmentation** | Reclaims space after compaction. Requires etcd restart.                |
| **Leader election** | Process where followers elect a new leader when current leader fails.  |
