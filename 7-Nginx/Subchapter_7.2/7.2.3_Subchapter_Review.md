## Cheatsheet: Reverse Proxy, Load Balancing, and Security

### Reverse Proxy Directives

| Directive | Purpose | Example |
|-----------|---------|---------|
| `proxy_pass` | Forward request to backend | `proxy_pass http://backend:8080` |
| `proxy_set_header` | Set request headers | `proxy_set_header Host $host` |
| `proxy_buffering` | Enable/disable buffering | `proxy_buffering off` |
| `proxy_connect_timeout` | Connection timeout | `proxy_connect_timeout 5s` |
| `proxy_read_timeout` | Read timeout | `proxy_read_timeout 60s` |

### Load Balancing Algorithms

| Algorithm | Directive | Use Case |
|-----------|-----------|----------|
| Round Robin | (default) | Equal traffic distribution |
| Least Connections | `least_conn;` | Variable request duration |
| IP Hash | `ip_hash;` | Sticky sessions (no cookies) |
| Random | `random two least_conn;` | Large backend pools |

### Upstream Server Options

| Option | Default | Purpose |
|--------|---------|---------|
| `weight=1` | 1 | Traffic weight |
| `max_fails=1` | 1 | Failures to mark down |
| `fail_timeout=10s` | 10s | Down time |
| `backup` | - | Backup server |
| `down` | - | Mark as down |

### SSL Directives

| Directive | Purpose | Example |
|-----------|---------|---------|
| `ssl_certificate` | Certificate file | `/etc/nginx/ssl/cert.pem` |
| `ssl_certificate_key` | Private key | `/etc/nginx/ssl/key.pem` |
| `ssl_protocols` | TLS versions | `TLSv1.2 TLSv1.3` |
| `ssl_ciphers` | Cipher suites | `ECDHE-ECDSA-AES128-GCM-SHA256` |
| `ssl_session_cache` | Session cache | `shared:SSL:10m` |

### HTTP to HTTPS Redirect

```nginx
server {
    listen 80;
    server_name example.com;
    return 301 https://$server_name$request_uri;
}
```

### Cache Directives

| Directive | Purpose | Example |
|-----------|---------|---------|
| `proxy_cache_path` | Define cache | `keys_zone=mycache:10m` |
| `proxy_cache` | Enable cache | `proxy_cache mycache` |
| `proxy_cache_key` | Cache key | `$scheme$host$uri` |
| `proxy_cache_valid` | Validity per code | `200 302 60m` |
| `proxy_cache_bypass` | Bypass conditions | `$http_cache_control` |

### Cache Status Headers

| Status | Meaning |
|--------|---------|
| `HIT` | Served from cache |
| `MISS` | Not in cache, fetched from backend |
| `EXPIRED` | Expired, fetched from backend |
| `BYPASS` | Bypassed (explicitly) |

### Rate Limiting Directives

| Directive | Purpose | Example |
|-----------|---------|---------|
| `limit_req_zone` | Define rate limit zone | `zone=api:10m rate=10r/s` |
| `limit_req` | Apply rate limit | `limit_req zone=api burst=20` |
| `limit_conn_zone` | Define connection limit | `zone=conn:10m` |
| `limit_conn` | Apply connection limit | `limit_conn conn 10` |
| `limit_req_status` | Custom status code | `429` |

### Gzip Directives

| Directive | Default | Purpose |
|-----------|---------|---------|
| `gzip on` | off | Enable compression |
| `gzip_comp_level` | 6 | Compression level (1-9) |
| `gzip_min_length` | 20 | Minimum size to compress |
| `gzip_types` | text/html | MIME types to compress |

### Security Headers

| Header | Purpose | Value |
|--------|---------|-------|
| `Strict-Transport-Security` | Force HTTPS | `max-age=31536000` |
| `X-Frame-Options` | Prevent clickjacking | `SAMEORIGIN` |
| `X-Content-Type-Options` | Prevent MIME sniffing | `nosniff` |
| `X-XSS-Protection` | XSS protection | `1; mode=block` |

## Comparison Tables

### Forward Proxy vs Reverse Proxy

| Aspect | Forward Proxy | Reverse Proxy |
|--------|--------------|---------------|
| **Client knows** | Yes (configured) | No (transparent) |
| **Server sees** | Proxy IP | Proxy IP |
| **Use case** | Corporate internet, privacy | Load balancing, security |

### Load Balancing Algorithms

| Algorithm | Sticky | Overhead | Best For |
|-----------|--------|----------|----------|
| Round Robin | No | Low | Equal capacity |
| Least Connections | No | Medium | Variable request duration |
| IP Hash | Yes | Low | Simple stickiness |
| Random | No | Low | Large pools |

## Interview Questions (Scenario-Based)

These questions assume only knowledge from Subchapter 7.2. Answers reference only concepts from 7.2.1 and 7.2.2.

### Question 1

**Scenario:** A web application is slow. Users report that pages take 3-5 seconds to load. The backend application is fast (under 100ms). Static assets (CSS, JS, images) are served by the same backend.

**Question:** How would you use Nginx to improve performance? What configuration changes would you make?

**Answer:**

**Solutions:**

1. **Move static assets to Nginx (bypass backend):**
```nginx
server {
    root /var/www/example;
    
    # Serve static assets directly
    location ~* \.(css|js|jpg|jpeg|png|gif|ico|svg)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
        access_log off;
    }
    
    # Dynamic requests go to backend
    location / {
        proxy_pass http://backend;
    }
}
```

2. **Add caching for backend responses:**
```nginx
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=web_cache:100m max_size=1g;

server {
    location / {
        proxy_cache web_cache;
        proxy_cache_key "$scheme$request_method$host$request_uri";
        proxy_cache_valid 200 302 60m;
        add_header X-Cache-Status $upstream_cache_status;
        proxy_pass http://backend;
    }
}
```

3. **Enable Gzip compression:**
```nginx
http {
    gzip on;
    gzip_comp_level 6;
    gzip_types text/plain text/css text/xml text/javascript application/json application/javascript;
}
```

**Expected improvement:**
- Static assets: 100-1000x faster (no backend)
- Cached pages: 10-100x faster
- Compressed responses: 70-90% smaller

### Question 2

**Scenario:** A login API endpoint is being brute-forced. Attackers are sending thousands of login requests per second from many IP addresses.

**Question:** How would you configure Nginx to protect this endpoint? Write the configuration.

**Answer:**

**Solution: Rate limiting on the login endpoint**

```nginx
# Define rate limit zones (in http block)
# 5 requests per minute per IP (strict)
limit_req_zone $binary_remote_addr zone=login_limit:10m rate=5r/m;

# Also limit by username to prevent distributed attacks
limit_req_zone $arg_username zone=user_limit:10m rate=2r/m;

server {
    location /api/login {
        # Apply rate limits
        limit_req zone=login_limit burst=5 nodelay;
        limit_req zone=user_limit burst=2;
        limit_req_status 429;
        
        # Log rate-limited requests
        limit_req_log_level warn;
        
        # Return custom error page
        error_page 429 /429.html;
        
        proxy_pass http://auth_backend;
    }
    
    location = /429.html {
        internal;
        return 429 "Too Many Requests - Please try again later\n";
        add_header Content-Type text/plain;
    }
}
```

**Additional protections:**

```nginx
# Block known bad user agents
if ($http_user_agent ~* (bot|crawler|scraper|curl|wget)) {
    return 403;
}

# Allow only from specific countries (using geoip module)
geoip_country /etc/nginx/geoip/GeoIP.dat;
map $geoip_country_code $allowed_country {
    default 0;
    US 1;
    CA 1;
}

location /api/login {
    if ($allowed_country = 0) {
        return 403;
    }
    # ... rest of config
}
```

### Question 3

**Scenario:** A client reports that their API requests occasionally fail with "502 Bad Gateway" when the backend takes more than 60 seconds to respond.

**Question:** What is causing this? How would you fix it?

**Answer:**

**Cause:** Nginx's `proxy_read_timeout` defaults to 60 seconds. If the backend takes longer, Nginx closes the connection and returns 502.

**Fix: Increase timeout for slow endpoints**

```nginx
location /api/slow-endpoint {
    proxy_pass http://backend;
    proxy_read_timeout 300s;  # 5 minutes
    proxy_connect_timeout 10s;
    proxy_send_timeout 300s;
}

# Or globally for all endpoints
location /api/ {
    proxy_pass http://backend;
    proxy_read_timeout 120s;
    proxy_connect_timeout 10s;
}
```

**Alternative: Use proxy_buffering off for streaming**

```nginx
location /api/stream {
    proxy_pass http://backend;
    proxy_buffering off;  # Don't buffer, stream immediately
    proxy_read_timeout 300s;
}
```

**Best practice: Different timeouts for different endpoints**

```nginx
# Fast APIs
location /api/fast/ {
    proxy_read_timeout 10s;
    proxy_pass http://backend;
}

# Slow report generation
location /api/reports/ {
    proxy_read_timeout 300s;
    proxy_pass http://backend;
}

# WebSocket (long-lived)
location /ws/ {
    proxy_pass http://ws_backend;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_read_timeout 3600s;
}
```

### Question 4

**Scenario:** A website uses HTTP only. The team wants to enable HTTPS with a free certificate and redirect all HTTP traffic to HTTPS.

**Question:** How would you configure Nginx to achieve this? Include certificate generation.

**Answer:**

**Step 1: Install Certbot and obtain certificate**

```bash
# Install certbot
sudo apt update
sudo apt install certbot python3-certbot-nginx

# Obtain certificate (auto-configures Nginx)
sudo certbot --nginx -d example.com -d www.example.com
```

**Step 2: Certbot automatically creates configuration:**

```nginx
# /etc/nginx/sites-available/example.com

# HTTP redirect (created by certbot)
server {
    listen 80;
    server_name example.com www.example.com;
    return 301 https://$server_name$request_uri;
}

# HTTPS server (created by certbot)
server {
    listen 443 ssl;
    server_name example.com www.example.com;
    
    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;
    
    # Additional security (add manually)
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;
    
    # HSTS
    add_header Strict-Transport-Security "max-age=31536000" always;
    
    location / {
        proxy_pass http://backend;
    }
}
```

**Step 3: Set up auto-renewal**

```bash
# Test renewal
sudo certbot renew --dry-run

# Certbot adds cron job automatically
sudo systemctl list-timers | grep certbot
```

### Question 5

**Scenario:** A microservices architecture has 5 backend services. One service (payment) is failing health checks and being marked down by Nginx. Other services are healthy.

**Question:** How does Nginx detect backend failures? What configuration controls this behavior? How would you configure a backup server?

**Answer:**

**How Nginx detects failures:** Passive health checks using `max_fails` and `fail_timeout`.

```nginx
upstream backend {
    server payment1:8080 max_fails=3 fail_timeout=30s;
    server payment2:8080 max_fails=3 fail_timeout=30s;
    server api1:8080;
    server api2:8080;
}
```

**Parameters explained:**

| Parameter | Default | Meaning |
|-----------|---------|---------|
| `max_fails` | 1 | Number of failed attempts to mark server down |
| `fail_timeout` | 10s | Time server is considered down; also time window for counting failures |

**Behavior:**
- Nginx counts failures within `fail_timeout` window
- After `max_fails` failures, server is marked down for `fail_timeout`
- Failed requests are retried on other servers

**Backup server configuration:**

```nginx
upstream backend {
    # Primary servers
    server payment1:8080 max_fails=3 fail_timeout=30s;
    server payment2:8080 max_fails=3 fail_timeout=30s;
    
    # Backup (only used when all primaries fail)
    server payment-backup:8080 backup;
    
    # API servers (no special health check)
    server api1:8080;
    server api2:8080;
}
```

**Active health checks (Nginx Plus):**

```nginx
upstream backend {
    zone backend 64k;
    
    server payment1:8080;
    server payment2:8080;
    
    health_check interval=5s fails=3 passes=2 uri=/health;
}
```

**Monitoring down servers:**

```bash
# Check upstream status (Nginx Plus)
curl http://localhost/upstream_conf?upstream=backend

# View logs for failures
tail -f /var/log/nginx/error.log | grep "upstream"
```
