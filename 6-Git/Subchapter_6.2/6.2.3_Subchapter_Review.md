## Cheatsheet: Branching, Merging, and Remotes

### Branch Commands

| Command | Purpose |
|---------|---------|
| `git branch` | List branches |
| `git branch <name>` | Create branch |
| `git checkout -b <name>` | Create and switch |
| `git switch -c <name>` | Create and switch (modern) |
| `git checkout <name>` | Switch branch |
| `git switch <name>` | Switch branch (modern) |
| `git branch -d <name>` | Delete branch (safe) |
| `git branch -D <name>` | Delete branch (force) |
| `git branch -vv` | Show tracking info |

### Merge Types

| Type | Condition | Result |
|------|-----------|--------|
| **Fast-forward** | Current branch has all commits | Pointer moves forward |
| **3-way** | Branches diverged | Merge commit with 2 parents |
| **Squash** | `--squash` flag | Single commit with all changes |

### Merge Commands

| Command | Purpose |
|---------|---------|
| `git merge <branch>` | Merge branch into current |
| `git merge --no-ff <branch>` | Merge with commit (no fast-forward) |
| `git merge --squash <branch>` | Squash commits before merge |
| `git merge --abort` | Abort merge |
| `git mergetool` | Launch merge tool |

### Conflict Resolution

| Option | Effect |
|--------|--------|
| Manual edit | Edit file, remove markers |
| `git checkout --ours` | Keep current branch version |
| `git checkout --theirs` | Keep incoming branch version |
| `git mergetool` | Use external merge tool |

### Remote Commands

| Command | Purpose |
|---------|---------|
| `git remote -v` | List remotes |
| `git remote add <name> <url>` | Add remote |
| `git remote remove <name>` | Remove remote |
| `git fetch <remote>` | Download without merging |
| `git fetch --prune` | Fetch and clean deleted branches |
| `git pull` | Fetch + merge |
| `git pull --rebase` | Fetch + rebase |
| `git push` | Upload changes |
| `git push -u` | Push and set upstream |
| `git push --force-with-lease` | Safer force push |

### Branch Naming Conventions

| Type | Convention | Example |
|------|------------|---------|
| Feature | `feature/description` | `feature/user-auth` |
| Bugfix | `bugfix/description` | `bugfix/login-error` |
| Hotfix | `hotfix/description` | `hotfix/critical-patch` |
| Release | `release/version` | `release/v1.2.0` |

### Workflow Comparison

| Workflow | Complexity | Branches | Best For |
|----------|------------|----------|----------|
| **Git Flow** | High | main, develop, feature, release, hotfix | Scheduled releases |
| **GitHub Flow** | Low | main + feature | Continuous delivery |
| **Trunk-Based** | Very Low | main (+ short-lived) | High-velocity teams |

### Remote States

| State | Meaning | Command |
|-------|---------|---------|
| `ahead` | Local commits not in remote | `git push` |
| `behind` | Remote commits not local | `git pull` |
| `diverged` | Both have unique commits | `git pull --rebase` |

## Comparison Tables

### Merge Strategies

| Strategy | Command | When to Use |
|----------|---------|-------------|
| Fast-forward | `git merge` (default) | Linear history, no diverged branches |
| No-fast-forward | `git merge --no-ff` | Preserve branch history |
| Squash | `git merge --squash` | Clean history, one commit per feature |
| Rebase | `git rebase` (pre-merge) | Linear history without merge commits |

### Git Flow vs GitHub Flow

| Aspect | Git Flow | GitHub Flow |
|--------|----------|-------------|
| **Release branches** | Yes | No |
| **Hotfix branches** | Yes | No (feature from main) |
| **Develop branch** | Yes | No |
| **Complexity** | High | Low |
| **Release cadence** | Scheduled | Continuous |
| **Use case** | Enterprise, multiple versions | Web apps, SaaS |

### Remote vs Local Branches

| Branch Type | Location | Reference |
|-------------|----------|-----------|
| Local branch | `.git/refs/heads/` | `main` |
| Remote branch | `.git/refs/remotes/origin/` | `origin/main` |
| Tracking | Link between local and remote | `main -> origin/main` |

## Interview Questions (Scenario-Based)

These questions assume only knowledge from Subchapter 6.2. Answers reference only concepts from 6.2.1 and 6.2.2.

### Question 1

**Scenario:** A developer runs `git merge feature` on the `main` branch. Git creates a merge commit even though there were no new commits on `main` since the feature branch was created.

**Question:** Why did Git create a merge commit instead of doing a fast-forward? How can you avoid this?

**Answer:**

**Why a merge commit was created:** Git created a merge commit because fast-forward merging was disabled, either:
- `git merge --no-ff` was used explicitly
- The repository has `merge.ff` configuration set to `false`
- The feature branch was created with `--no-ff` behavior

**How to avoid (do fast-forward):**

```bash
# Method 1: Use --ff-only (fail if not possible)
git merge --ff-only feature

# Method 2: Use default merge (fast-forwards when possible)
git merge feature
# If fast-forward is possible, it will happen

# Method 3: Rebase first, then fast-forward
git rebase feature
git checkout main
git merge feature  # now fast-forward
```

**Check current merge settings:**
```bash
git config --global merge.ff
# false (if set to always create merge commits)
git config --global merge.ff true
```

**When you might want `--no-ff`:**
- Preserve feature branch history (see that a feature branch existed)
- Team convention to always have merge commits
- Integration with tools that expect merge commits

### Question 2

**Scenario:** A team uses Git Flow. A developer finishes a feature branch and merges it into `develop`. Later, they realize the feature has a bug and needs to be reverted.

**Question:** How can they revert the feature merge without affecting other work that happened after the merge? What are the implications?

**Answer:**

**Revert the merge commit:**

```bash
# Find the merge commit SHA
git log --oneline --merges develop
# a1b2c3d Merge feature/user-auth

# Revert the merge (requires -m flag)
git checkout develop
git revert -m 1 a1b2c3d
```

**The `-m` flag:** Specifies which parent to keep (1 = first parent, the branch you were on when merging).

**Implications:**
- The revert creates a new commit that undoes the merge
- The feature branch commits are still in history
- If you later want to re-merge the same feature (after fixing), you need to revert the revert first

**To re-merge after fixing:**
```bash
# First revert the revert
git revert <revert-commit-sha>

# Then merge the fixed feature
git merge feature/user-auth-fixed
```

**Alternative approach (if history is clean):**
```bash
# Reset to commit before merge (DANGEROUS if others have pulled)
git reset --hard <commit-before-merge>
git push --force-with-lease
```

**Best practice:** Use `git revert` for shared branches; use `git reset` only for local or feature branches.

### Question 3

**Scenario:** A developer pushes a commit to `main`, then realizes they accidentally committed a sensitive file (e.g., `.env` with passwords).

**Question:** How can they remove the sensitive file from the remote repository? What are the risks?

**Answer:**

**Immediate action (if no one has pulled):**

```bash
# Option 1: Remove file and amend commit
git rm --cached .env
git commit --amend --no-edit
git push --force-with-lease origin main
```

**Option 2: Remove from entire history (if file was in older commits):**

```bash
# Use git filter-repo (modern, recommended)
git filter-repo --path .env --invert-paths

# Force push
git push --force-with-lease origin main

# Clean local references
git reflog expire --expire=now --all
git gc --prune=now --aggressive
```

**Option 3: Use BFG Repo-Cleaner (alternative):**
```bash
java -jar bfg.jar --delete-files .env repo.git
git push --force
```

**Risks and mitigation:**
| Risk | Mitigation |
|------|------------|
| Others have pulled the commit | Notify team immediately; they must reset or re-clone |
| Broken history for collaborators | Use `--force-with-lease` instead of `--force` |
| CI/CD may have cached the file | Rotate secrets immediately |
| GitHub may still have the commit in PRs | Also clean PR branches |

**Immediately rotate the secret:**
- Change passwords/tokens in the `.env` file
- Revoke any exposed API keys
- Check if anyone could have accessed the commit

**Prevention:**
- Add `.env` to `.gitignore` before first commit
- Use `git-secrets` or pre-commit hooks to scan for secrets
- Use environment variables or secret managers, not committed files

### Question 4

**Scenario:** A developer is working on a feature branch. They need to get the latest changes from `main` but want to keep a linear history without merge commits.

**Question:** Should they use `git merge main` or `git rebase main`? What are the trade-offs?

**Answer:**

**Recommendation: Use `git rebase main` for linear history**

```bash
git checkout feature
git rebase main
```

**Trade-offs:**

| Aspect | `git merge main` | `git rebase main` |
|--------|------------------|-------------------|
| **History** | Non-linear (merge commits) | Linear (no merge commits) |
| **Commit SHAs** | Preserved | New SHAs |
| **Conflict resolution** | Once | Once per commit (can be many) |
| **Safety on shared branches** | Safe | Dangerous (rewrites history) |
| **Ease of understanding** | Shows when branch was updated | Hides when updates happened |

**When to use merge:**
- Feature branch is shared with others
- You want to preserve exact timeline
- Team prefers explicit merge commits

**When to use rebase:**
- Feature branch is local only
- You want clean, linear history
- Before opening a Pull Request

**Golden rule of rebasing:** Do not rebase branches that others have based work on.

**Best practice workflow:**
```bash
# Update local main
git checkout main
git pull origin main

# Rebase feature branch (local only)
git checkout feature
git rebase main

# Resolve conflicts if any
git add .
git rebase --continue

# Force push (if already pushed, with caution)
git push --force-with-lease origin feature
```

### Question 5

**Scenario:** A developer forks a repository on GitHub, clones it locally, and makes changes. They want to contribute back to the original repository.

**Question:** What remote configuration is needed? What commands would they use to keep their fork in sync with the original?

**Answer:**

**Remote configuration:**

```bash
# After cloning your fork
git clone https://github.com/yourname/repo.git
cd repo

# Add upstream remote (original repository)
git remote add upstream https://github.com/original/repo.git

# Verify remotes
git remote -v
# origin    https://github.com/yourname/repo.git (fetch)
# origin    https://github.com/yourname/repo.git (push)
# upstream  https://github.com/original/repo.git (fetch)
# upstream  https://github.com/original/repo.git (push)
```

**Sync fork with upstream:**

```bash
# Fetch upstream changes
git fetch upstream

# Update local main
git checkout main
git merge upstream/main

# Push to your fork (origin)
git push origin main

# Update feature branch with upstream
git checkout feature-branch
git rebase main
git push --force-with-lease origin feature-branch
```

**Complete contribution workflow:**

```bash
# 1. Sync main with upstream
git checkout main
git fetch upstream
git merge upstream/main
git push origin main

# 2. Create feature branch
git checkout -b feature/new-feature

# 3. Make changes, commit, push to fork
git add .
git commit -m "Add new feature"
git push -u origin feature/new-feature

# 4. Open Pull Request on GitHub (from your fork to upstream)

# 5. After PR is merged, sync again
git checkout main
git fetch upstream
git merge upstream/main
git push origin main
git branch -d feature/new-feature
```

**One-liner to sync fork (after adding upstream):**
```bash
git fetch upstream && git checkout main && git merge upstream/main && git push origin main
```
