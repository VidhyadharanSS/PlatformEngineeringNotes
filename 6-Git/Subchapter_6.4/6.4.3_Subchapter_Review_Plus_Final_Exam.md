## Cheatsheet: Undoing Mistakes, Git LFS, Submodules

### Undo Commands

| Operation | Command | Safety |
|-----------|---------|--------|
| Discard working changes | `git restore file` | Safe |
| Unstage file | `git restore --staged file` | Safe |
| Undo commit (keep staged) | `git reset --soft HEAD~1` | Safe (local) |
| Undo commit (unstage) | `git reset HEAD~1` | Safe (local) |
| Discard commits and changes | `git reset --hard HEAD~1` | Dangerous |
| Safe undo (shared branch) | `git revert HEAD` | Safe |
| Recover lost commits | `git reflog` | Safe |

### Reset Modes

| Mode | Working Dir | Staging | Command |
|------|-------------|---------|---------|
| `--soft` | Unchanged | Unchanged | `git reset --soft HEAD~1` |
| `--mixed` (default) | Unchanged | Reset | `git reset HEAD~1` |
| `--hard` | Reset | Reset | `git reset --hard HEAD~1` |

### Git LFS Commands

| Command | Purpose |
|---------|---------|
| `git lfs install` | Initialize LFS |
| `git lfs track "*.ext"` | Track file pattern |
| `git lfs ls-files` | List LFS files |
| `git lfs pull` | Download LFS files |
| `git lfs migrate import` | Convert to LFS |

### Submodule Commands

| Command | Purpose |
|---------|---------|
| `git submodule add <url> <path>` | Add submodule |
| `git submodule init` | Initialize |
| `git submodule update` | Update submodules |
| `git clone --recursive` | Clone with submodules |

### Subtree Commands

| Command | Purpose |
|---------|---------|
| `git subtree add --prefix=<path> <url> <branch>` | Add subtree |
| `git subtree pull --prefix=<path> <url> <branch>` | Update |
| `git subtree push --prefix=<path> <url> <branch>` | Push changes |

### Tool Selection Guide

| Requirement | Tool |
|-------------|------|
| Large binary files | Git LFS |
| Shared library, independent updates | Submodule |
| Simple dependency, occasional updates | Subtree |
| Vendor code you modify | Subtree or copy |

### Question 1: Rebase vs Merge (10 minutes)

**Scenario:** A team has two long-running branches: `main` and `develop`. A developer finishes a feature on `feature/login` and wants to integrate it into `develop`. The team values a clean, linear history.

**Questions:**
1. Should they use `git merge` or `git rebase`?
2. What command would they use?
3. After the feature is merged into `develop`, how should `develop` be integrated into `main` for release?

**Answer:**

1. **Use `git rebase`** – For a linear history on a feature branch before merging into `develop`, rebase creates a clean, linear history without merge commits.

2. **Commands:**
```bash
# Update feature branch with latest develop
git checkout feature/login
git rebase develop

# Resolve any conflicts
git add .
git rebase --continue

# Fast-forward merge (since rebase makes it linear)
git checkout develop
git merge feature/login --ff-only
```

3. **For release (develop → main):** Use `git merge --no-ff` to create a merge commit marking the release point.
```bash
git checkout main
git merge develop --no-ff -m "Release v1.0.0"
git tag v1.0.0
```

### Question 2: Recovering Lost Commits (10 minutes)

**Scenario:** A developer ran `git reset --hard HEAD~5` to discard some commits, but immediately realized they needed one of the discarded commits. They haven't pushed.

**Questions:**
1. How can they recover the lost commit?
2. What command would they use?
3. How long will this recovery method work?

**Answer:**

1. **Use `git reflog`** – The reflog records all movements of HEAD, including commits that are no longer reachable.

2. **Commands:**
```bash
# View reflog to find the lost commit
git reflog
# a1b2c3d HEAD@{0}: reset: moving to HEAD~5
# e4f5g6h HEAD@{1}: commit: Important feature
# i7j8k9l HEAD@{2}: commit: WIP

# Recover the lost commit (create branch or cherry-pick)
git checkout -b recovered e4f5g6h
# OR cherry-pick to current branch
git cherry-pick e4f5g6h
```

3. **Recovery window:** Reflog entries expire after **90 days** for reachable commits and **30 days** for unreachable commits. After that, `git gc` may remove them permanently.

### Question 3: Git LFS Implementation (10 minutes)

**Scenario:** A team is developing a game with large asset files (`.psd`, `.fbx`, `.wav`). Their Git repository has grown to 5GB and cloning takes 10 minutes.

**Questions:**
1. What tool should they use to reduce repository size?
2. How would they migrate existing large files to this tool?
3. What configuration is needed for team members?

**Answer:**

1. **Use Git LFS (Large File Storage)** – Stores large files as pointers in Git, actual files on LFS server.

2. **Migrate existing repository:**
```bash
# Install and initialize LFS
git lfs install

# Track file types
git lfs track "*.psd"
git lfs track "*.fbx"
git lfs track "*.wav"
git lfs track "*.mp3"

# Migrate existing files to LFS
git lfs migrate import --include="*.psd,*.fbx,*.wav,*.mp3" --everything

# Push migrated history
git push --force --all
git push --force --tags
```

3. **Team configuration:**
```bash
# Each team member needs to:
git lfs install

# Re-clone the repository (simplest)
git clone <repo-url>

# OR update existing clone
git lfs pull
```

### Question 4: Submodule Management (10 minutes)

**Scenario:** A project uses a shared library as a Git submodule. A developer updates the library to a newer version.

**Questions:**
1. What commands do they need to run to update the submodule?
2. What happens to the main repository after the update?
3. What must they push, and in what order?

**Answer:**

1. **Update submodule commands:**
```bash
# Go to submodule directory
cd libs/shared-library

# Fetch and checkout new version
git fetch origin
git checkout v2.0.0
# OR update to latest main
git pull origin main

# Go back to main repo
cd ../..

# Stage the submodule change
git add libs/shared-library
git commit -m "Update shared-library to v2.0.0"
```

2. **What happens to main repo:** The main repository records the **new commit SHA** of the submodule, not the file contents. This is stored as a special entry in the index.

3. **Push order:**
```bash
# First push submodule changes (if any local commits)
cd libs/shared-library
git push origin v2.0.0

# Then push main repository change
cd ../..
git push origin main
```

**Important:** Always push submodule changes before pushing the main repository, or others will get a broken reference.

### Question 5: Cherry-pick and Revert (10 minutes)

**Scenario:** A hotfix was applied to the `main` branch (commit `a1b2c3d`). The team needs this fix on the `release/v1.0` branch, but they don't want to merge the entire `main` branch.

**Questions:**
1. What command would apply only the hotfix to the release branch?
2. After applying, they discover the hotfix introduced a new bug. How do they undo it on the release branch without affecting the fix on main?
3. Later, they want to re-apply a corrected version of the hotfix. How?

**Answer:**

1. **Use `git cherry-pick`:**
```bash
git checkout release/v1.0
git cherry-pick a1b2c3d
```

2. **Undo the hotfix on release branch (safe way):**
```bash
# Use revert (creates new commit that undoes changes)
git revert HEAD
# Or revert the specific cherry-picked commit
git revert a1b2c3d
```

3. **Re-apply corrected version:**
```bash
# First revert the revert (if you used revert)
git revert <revert-commit-sha>

# OR cherry-pick the corrected commit from main
git cherry-pick <corrected-commit-sha>
```

**Why revert instead of reset:** Revert is safe for shared branches because it doesn't rewrite history.

### Question 6: Merge Conflict Resolution (10 minutes)

**Scenario:** Two developers modified the same lines in `config.yml` on different branches. When merging `feature` into `main`, Git reports a conflict.

**Questions:**
1. What does the conflict look like in the file?
2. What commands show which commits caused the conflict?
3. How do they resolve the conflict and complete the merge?

**Answer:**

1. **Conflict markers:**
```yaml
<<<<<<< HEAD
database: production
=======
database: staging
>>>>>>> feature
```

2. **Commands to see conflicting commits:**
```bash
# Show merge base (common ancestor)
git merge-base main feature

# Show commits on both branches
git log --oneline main...feature

# Show who changed the conflicting lines
git blame config.yml

# Show conflict details
git diff --merge
```

3. **Resolve and complete merge:**
```bash
# Edit config.yml to desired content (remove markers)
# For example, keep one version:
database: production

# Stage resolved file
git add config.yml

# Complete merge
git commit -m "Merge feature: resolve config conflict"
```

### Question 7: Interactive Rebase (10 minutes)

**Scenario:** A developer has 5 commits on their feature branch:
```
a1b2c3d Add login UI
e4f5g6h Fix typo in login
i7j8k9l Add login validation
m9n0o1p WIP - debugging
q1r2s3t Add login API call
```

They want to:
- Squash "Fix typo" into "Add login UI"
- Drop the "WIP" commit
- Reorder so API call comes before validation

**Questions:**
1. What command starts interactive rebase?
2. How should the rebase todo list be edited?
3. What happens to commit SHAs after rebase?

**Answer:**

1. **Start interactive rebase:**
```bash
git rebase -i HEAD~5
```

2. **Edit the todo list:**
```
pick a1b2c3d Add login UI
squash e4f5g6h Fix typo in login
pick q1r2s3t Add login API call      # moved up
pick i7j8k9l Add login validation
drop m9n0o1p WIP - debugging          # removed
```

3. **What happens to SHAs:** All commit SHAs change (except possibly the first if unchanged). Interactive rebase creates new commit objects with new SHAs because the parent relationships change.

### Question 8: Stash Workflow (10 minutes)

**Scenario:** A developer is working on a feature branch and has uncommitted changes. A critical bug needs immediate fixing on `main`.

**Questions:**
1. How do they save their work, fix the bug, and resume?
2. What if they need to keep their work for more than a day?
3. How do they recover a stash if they accidentally drop it?

**Answer:**

1. **Stash workflow:**
```bash
# Save current work
git stash push -m "WIP: feature in progress"

# Fix bug on main
git checkout main
git checkout -b hotfix/bug
# Fix bug
git add .
git commit -m "Fix critical bug"
git push origin hotfix/bug

# Return to feature
git checkout feature
git stash pop
```

2. **Keep stash for longer:** Stashes persist indefinitely until explicitly dropped. Name them clearly:
```bash
git stash push -m "WIP: feature - in progress - Jan 15"
git stash list
```

3. **Recover dropped stash:**
```bash
# Find lost stash commits
git fsck --unreachable | grep commit | cut -d' ' -f3 | xargs git show --oneline

# Create branch from recovered commit
git branch recovered-stash a1b2c3d
```
