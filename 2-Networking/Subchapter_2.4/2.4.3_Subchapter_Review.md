### Layer 4 vs Layer 7 Load Balancing

| Feature                 | Layer 4            | Layer 7                    |
| ----------------------- | ------------------ | -------------------------- |
| **Based on**            | IP, port           | HTTP headers, path, cookie |
| **Protocols**           | TCP, UDP           | HTTP, HTTPS, gRPC          |
| **SSL termination**     | Limited            | Full offload               |
| **Content routing**     | No                 | Yes (path, host)           |
| **Performance**         | Very fast          | Fast (with overhead)       |
| **Session persistence** | Source IP hash     | Cookie                     |
| **Use cases**           | Databases, any TCP | Web APIs, microservices    |

| Least Time        | Fastest response          | No     |
| IP Hash           | Hash of client IP         | Yes    |
| Weighted          | Based on server weights   | No     |

## Interview Questions (Scenario-Based)

These questions assume only knowledge from Subchapter 2.4. Answers reference only concepts from 2.4.1 and 2.4.2.

### Question 1

**Scenario:** You are debugging a microservice that calls `https://api.payment.example.com/v1/charge`. The call occasionally fails with `504 Gateway Timeout`. The payment service has three backends behind a load balancer.

**Question:** What does a 504 error indicate? Which component is likely the problem? How would you investigate using curl and other tools?

**Answer:**

**What 504 Gateway Timeout means:**

* The load balancer (gateway) sent a request to a backend

* The backend did not respond within the timeout period

* The load balancer returned 504 to the client

**Which component is likely the problem:**

* **Backend servers** (most likely) – one or more are slow or stuck

* **Network** – congestion between LB and backend

* **Load balancer timeout** – configured too low for normal backend processing

**Investigation with curl and tools:**

```bash
# 1. Test each backend directly (bypass load balancer)
curl -w "Total: %{time_total}s\n" -o /dev/null -s https://10.0.0.1:443/v1/charge -d '{"test":1}'
curl -w "Total: %{time_total}s\n" -o /dev/null -s https://10.0.0.2:443/v1/charge -d '{"test":1}'
curl -w "Total: %{time_total}s\n" -o /dev/null -s https://10.0.0.3:443/v1/charge -d '{"test":1}'

# 2. Check if specific backend is slow
# If one backend shows > 30s and others < 1s, that backend is the problem

# 3. Check load balancer timeout configuration (example for HAProxy)
grep "timeout" /etc/haproxy/haproxy.cfg
# timeout server 30s  <-- if this is too low, increase it

# 4. Check backend health from load balancer
# On the load balancer, check health check status
# HAProxy stats page or:
echo "show stat" | socat stdio /var/run/haproxy.sock | grep payment_backend

# 5. Capture traffic to see slow responses
sudo tcpdump -i eth0 host 10.0.0.2 and port 443 -w slow-backend.pcap
```

**Fixes:**

| Cause                      | Fix                                     |
| -------------------------- | --------------------------------------- |
| Slow backend (one server)  | Remove unhealthy backend from pool      |
| Slow backend (all servers) | Optimize application, add more backends |
| Timeout too low            | Increase `timeout server` in LB config  |
| Network congestion         | Check network, increase bandwidth       |

### Question 2

**Scenario:** You are deploying a new API service. The service must handle 10,000 requests per second. You have 10 backend servers. The API does not store session data (stateless). You need to choose a load balancing algorithm.

**Question:** Which algorithm would you choose and why? Would you use Layer 4 or Layer 7? Explain the trade-offs.

**Answer:**

**Algorithm choice: Round Robin or Least Connections**

| Algorithm         | Pros                 | Cons                      | Best for this scenario?         |
| ----------------- | -------------------- | ------------------------- | ------------------------------- |
| Round Robin       | Simple, predictable  | May overload slow servers | **Yes** (if servers equal)      |
| Least Connections | Balances load better | Slightly more overhead    | **Yes** (if request times vary) |
| Least Time        | Fastest response     | Complex                   | Overkill                        |
| IP Hash           | Sticky               | Not needed (stateless)    | No                              |

**Layer choice: Layer 4 (TCP) for this scenario**

**Why Layer 4 is better for high throughput:**

* Lower overhead (no HTTP parsing)

* Higher performance (10K RPS easily)

* No need for content-based routing

* Stateless API → no need for sticky sessions

**Layer 7 trade-offs:**

* Would allow future features (rate limiting by path, authentication)

* But adds CPU overhead (TLS termination, HTTP parsing)

* May become bottleneck at 10K RPS

**Hybrid approach for production:**

```
Internet → Layer 4 LB (NLB) → Layer 7 LBs (HAProxy) → Backends
```

* Layer 4 handles high throughput

* Layer 7 provides advanced features for a subset of traffic

**Configuration example (HAProxy Layer 4):**

```haproxy
frontend api_frontend
    bind *:443
    mode tcp
    default_backend api_backend

backend api_backend
    mode tcp
    balance roundrobin  # or leastconn
    server api1 10.0.0.1:443 check
    server api2 10.0.0.2:443 check
    server api3 10.0.0.3:443 check
    # ... up to 10 servers
```

### Question 3

**Scenario:** A user reports that after refreshing a web page, they are sometimes logged out and sometimes logged in. The web application stores session data in local memory on each backend server (not a shared session store). The load balancer is using round robin.

**Question:** What is the problem? How would you fix it without changing application code?

**Answer:**

**Problem:** Session data is stored locally on each backend server. Round robin sends different requests from the same user to different backends. When a request goes to a backend that doesn't have the user's session, the user appears logged out.

**Solution: Session persistence (sticky sessions)**

Without changing application code, configure the load balancer to send all requests from the same user to the same backend.

**Fix using HAProxy (cookie insertion):**

```haproxy
backend web_backend
    # Insert cookie to track backend
    cookie SERVERID insert indirect nocache
    
    # Each server gets a unique cookie value
    server web1 10.0.0.1:80 cookie web1 check
    server web2 10.0.0.2:80 cookie web2 check
    server web3 10.0.0.3:80 cookie web3 check
```

**How it works:**

1. First request: LB picks backend (e.g., web1) and inserts `Set-Cookie: SERVERID=web1`
2. Browser stores cookie
3. Subsequent requests: Browser sends `Cookie: SERVERID=web1`
4. LB reads cookie and sends request to web1

**Fix using Nginx (sticky cookie):**

```nginx
upstream web_backend {
    sticky cookie srv_id expires=1h domain=.example.com path=/;
    server 10.0.0.1:80;
    server 10.0.0.2:80;
    server 10.0.0.3:80;
}
```

**Alternative: Source IP hash (less reliable)**

```nginx
upstream web_backend {
    ip_hash;  # Hash of client IP
    server 10.0.0.1:80;
    server 10.0.0.2:80;
    server 10.0.0.3:80;
}
```

**Drawback of source IP hash:** Multiple users behind the same NAT gateway share the same IP, causing all to go to the same backend (uneven load).

**Long-term solution:** Move session storage to a shared store (Redis, Memcached, database) so any backend can handle any request.

### Question 4

**Scenario:** A curl command `curl https://api.example.com/data` returns:

```
curl: (60) SSL certificate problem: certificate has expired
```

**Question:** What is the problem? How would you verify the certificate expiry date? How would you fix it? When is it acceptable to use `-k`?

**Answer:**

**Problem:** The SSL/TLS certificate presented by `api.example.com` has passed its expiration date. Browsers and curl will reject it as insecure.

**Verify certificate expiry:**

```bash
# Method 1: Using openssl
openssl s_client -connect api.example.com:443 -servername api.example.com 2>/dev/null | \
    openssl x509 -noout -dates
# notBefore=Jan 15 00:00:00 2023 GMT
# notAfter=Jan 15 23:59:59 2024 GMT  <-- if this is in the past, expired

# Method 2: Using curl with verbose
curl -v https://api.example.com/data 2>&1 | grep -A 5 "Server certificate"
# * SSL certificate verify result: expired (10)

# Method 3: Using curl with --cacert and --cert-status
curl --cert-status https://api.example.com/data -v 2>&1 | grep -i expire
```

**Fix the issue:**

| Role                              | Action                                                              |
| --------------------------------- | ------------------------------------------------------------------- |
| **Certificate owner**             | Renew certificate from CA, install new cert on server               |
| **System administrator**          | Update certificate files on server, reload web server/load balancer |
| **Client (temporary workaround)** | Update local CA bundle, or use `-k` (not recommended)               |

**When is** **`-k`** **acceptable (skip verification)?**

| Scenario                                    | Acceptable?         | Why                     |
| ------------------------------------------- | ------------------- | ----------------------- |
| Production API with real users              | **No**              | Security risk           |
| Internal testing with self-signed cert      | **Yes**             | Controlled environment  |
| Debugging certificate issues                | **Yes (temporary)** | To isolate the problem  |
| Automated CI/CD talking to test environment | **Maybe**           | Use internal CA instead |

**Better than** **`-k`** **for internal testing:**

```bash
# Use custom CA certificate
curl --cacert internal-ca.pem https://internal.api.example.com

# Add certificate to system trust store
sudo cp internal-ca.pem /usr/local/share/ca-certificates/
sudo update-ca-certificates
```

### Question 5

**Scenario:** You have two backend servers: Server A can handle 1000 RPS, Server B can handle 500 RPS. You are using round robin load balancing.

**Question:** What problem will occur? How would you configure the load balancer to handle the capacity difference?

**Answer:**

**Problem with round robin:**

* Round robin sends equal traffic to both servers (50% each)

* Server A gets 750 RPS (75% of its capacity) – healthy

* Server B gets 750 RPS (150% of its capacity) – overloaded!

* Server B will become slow, time out, or crash

**Solution: Weighted load balancing**

Assign higher weight to the more powerful server.

**HAProxy configuration (weighted round robin):**

```haproxy
backend web_backend
    balance roundrobin
    # Weight: A gets 3x traffic of B
    server serverA 10.0.0.1:80 weight 3 check
    server serverB 10.0.0.2:80 weight 1 check
```

**Traffic distribution:**

* Total weight = 3 + 1 = 4

* Server A receives 3/4 = 75% of traffic (750 RPS at 1000 RPS total)

* Server B receives 1/4 = 25% of traffic (250 RPS at 500 RPS capacity)

* Both operate within capacity

**Nginx configuration (weighted round robin):**

```nginx
upstream web_backend {
    server 10.0.0.1:80 weight=3;
    server 10.0.0.2:80 weight=1;
}
```

**Alternative: Least Connections with weights**

```haproxy
backend web_backend
    balance leastconn
    server serverA 10.0.0.1:80 weight 3 check
    server serverB 10.0.0.2:80 weight 1 check
```

**Verification:**

```bash
# Check traffic distribution (from load balancer stats)
echo "show stat" | socat stdio /var/run/haproxy.sock | grep web_backend

# Monitor backend load
curl http://serverA/metrics
curl http://serverB/metrics
```

**Dynamic adjustment:** If capacity changes (e.g., Server B gets upgraded), update weights without downtime:

```bash
# HAProxy runtime weight change
echo "set weight web_backend/serverB 2" | socat stdio /var/run/haproxy.sock
```
