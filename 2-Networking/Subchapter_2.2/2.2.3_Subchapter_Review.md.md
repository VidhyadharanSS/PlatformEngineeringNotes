## Cheatsheet: Core Networking Tools and DNS

### Ping – Connectivity Testing

| Command             | Purpose                          |
| ------------------- | -------------------------------- |
| `ping host`         | Continuous ping (Ctrl+C to stop) |
| `ping -c 4 host`    | Send 4 packets                   |
| `ping -i 0.5 host`  | 0.5 second interval              |
| `ping -s 1400 host` | 1400 byte packet size            |
| `ping -I eth0 host` | Specify source interface         |

**Output fields:** `icmp_seq` (sequence), `ttl` (time to live), `time` (RTT in ms), packet loss %

### Traceroute / MTR – Path Discovery

| Command                    | Purpose                        |
| -------------------------- | ------------------------------ |
| `traceroute host`          | UDP traceroute (Linux default) |
| `traceroute -I host`       | ICMP traceroute                |
| `traceroute -T -p 80 host` | TCP to port 80                 |
| `mtr host`                 | Continuous interactive         |
| `mtr -r -c 10 host`        | Report mode (10 packets)       |

### SS – Socket Statistics (Modern)

| Command                   | Purpose                            |
| ------------------------- | ---------------------------------- |
| `ss -tln`                 | Listening TCP ports (numeric)      |
| `ss -tlnp`                | Listening TCP ports with processes |
| `ss -uln`                 | Listening UDP ports                |
| `ss -tn`                  | Established TCP connections        |
| `ss -s`                   | Socket statistics summary          |
| `ss -t state established` | Filter by state                    |

### IP Command – Interface and Routing

| Command                                | Purpose             |
| -------------------------------------- | ------------------- |
| `ip addr show`                         | Show IP addresses   |
| `ip route show`                        | Show routing table  |
| `ip neigh show`                        | Show ARP cache      |
| `ip link set eth0 up/down`             | Interface control   |
| `ip route add default via 192.168.1.1` | Add default gateway |

### ARP – Address Resolution

| Command              | Purpose            |
| -------------------- | ------------------ |
| `arp -a`             | Show ARP cache     |
| `arp -n`             | Numeric (no DNS)   |
| `arp -d 192.168.1.1` | Delete ARP entry   |
| `ip neigh show`      | Modern alternative |

### Dig – DNS Lookup (Gold Standard)

| Command               | Purpose                  |
| --------------------- | ------------------------ |
| `dig domain`          | Basic A record query     |
| `dig domain AAAA`     | IPv6 address             |
| `dig domain MX`       | Mail exchange records    |
| `dig domain NS`       | Name servers             |
| `dig domain TXT`      | Text records (SPF, DKIM) |
| `dig @8.8.8.8 domain` | Query specific server    |
| `dig domain +short`   | Terse output             |
| `dig domain +trace`   | Full resolution path     |
| `dig -x 8.8.8.8`      | Reverse DNS (PTR)        |

### Host – Simple DNS Lookup

| Command             | Purpose              |
| ------------------- | -------------------- |
| `host domain`       | Basic lookup         |
| `host -t MX domain` | Specific record type |
| `host 8.8.8.8`      | Reverse lookup       |

### Nslookup – Legacy DNS Tool

| Command                    | Purpose               |
| -------------------------- | --------------------- |
| `nslookup domain`          | Basic lookup          |
| `nslookup -type=MX domain` | Specific record type  |
| `nslookup domain 8.8.8.8`  | Query specific server |

### DNS Configuration Files

| File                 | Purpose                           |
| -------------------- | --------------------------------- |
| `/etc/resolv.conf`   | DNS resolver configuration        |
| `/etc/hosts`         | Local static resolution           |
| `/etc/nsswitch.conf` | Name service order (files vs dns) |

### DNS Record Types Summary

| Type  | Purpose          | Example Query          |
| ----- | ---------------- | ---------------------- |
| A     | IPv4 address     | `dig domain A`         |
| AAAA  | IPv6 address     | `dig domain AAAA`      |
| CNAME | Alias            | `dig www.domain CNAME` |
| MX    | Mail server      | `dig domain MX`        |
| NS    | Name server      | `dig domain NS`        |
| TXT   | Text (SPF, DKIM) | `dig domain TXT`       |
| PTR   | Reverse DNS      | `dig -x IP`            |
| SOA   | Zone metadata    | `dig domain SOA`       |

### DNS Resolution Flow

```
1. Check /etc/hosts (if files first in nsswitch.conf)
2. Check local DNS cache
3. Query nameserver 1 from /etc/resolv.conf
4. If timeout, query nameserver 2
5. Apply search domains (if short name)
6. Return result or NXDOMAIN
```

### Common DNS Error Messages

| Message               | Meaning                                    |
| --------------------- | ------------------------------------------ |
| `NXDOMAIN`            | Domain does not exist                      |
| `SERVFAIL`            | DNS server error                           |
| `REFUSED`             | Server refused to answer                   |
| `NOERROR + 0 answers` | Domain exists, no record of requested type |

## Comparison Tables

### Ping vs Traceroute vs MTR

| Tool         | Best For                    | Shows                | Continuous         |
| ------------ | --------------------------- | -------------------- | ------------------ |
| `ping`       | Basic reachability, latency | RTT, loss            | Yes (until Ctrl+C) |
| `traceroute` | Path discovery (once)       | Each hop, 3 probes   | No                 |
| `mtr`        | Path + latency monitoring   | Each hop, statistics | Yes                |

### SS vs Netstat

| Feature         | `ss`                   | `netstat`                   |
| --------------- | ---------------------- | --------------------------- |
| Speed           | Fast (reads /proc/net) | Slow (iterates through all) |
| Color output    | Yes                    | No                          |
| Modern features | Yes (timers, filters)  | Limited                     |
| Availability    | Modern Linux           | Legacy, still available     |
| Recommendation  | **Use** **`ss`**       | Use only on old systems     |

### Dig vs Host vs Nslookup

| Tool       | Verbosity | Best For            | Scriptable |
| ---------- | --------- | ------------------- | ---------- |
| `dig`      | High      | Debugging, learning | Yes        |
| `host`     | Low       | Quick lookups       | Yes        |
| `nslookup` | Medium    | Legacy systems      | Moderate   |

### DNS Record TTL Guidelines

| Record Type       | Typical TTL    | Use Case                         |
| ----------------- | -------------- | -------------------------------- |
| A/AAAA (critical) | 60-300 sec     | Failover, blue/green deployments |
| A/AAAA (normal)   | 300-3600 sec   | Production services              |
| CNAME             | 300-3600 sec   | Aliases                          |
| MX                | 3600-86400 sec | Email (changes rare)             |
| NS                | 86400+ sec     | Name servers (very stable)       |
| TXT (SPF)         | 300-3600 sec   | Email authentication             |

## Interview Questions (Scenario-Based)

These questions assume only knowledge from Subchapter 2.2. Answers reference only concepts from 2.2.1 and 2.2.2.

### Question 1

**Scenario:** A user reports that `curl https://api.internal.company.com` returns "Could not resolve host: api.internal.company.com". However, `ping 10.0.0.5` works, and you know that `api.internal.company.com` should resolve to `10.0.0.5`.

**Question:** Walk through your systematic diagnosis. What commands would you run, and what would each tell you?

**Answer:**

**Step-by-step diagnosis:**

```bash
# 1. Verify the DNS resolution failure
dig api.internal.company.com
# If NXDOMAIN or no answer, DNS resolution is failing

# 2. Check if this is a search domain issue
# The domain is internal.company.com – check if resolver is configured for this domain
cat /etc/resolv.conf
# Look for: search internal.company.com or company.com

# 3. Query the specific DNS server directly (bypass local config)
dig @10.0.0.1 api.internal.company.com
# Replace 10.0.0.1 with the internal DNS server IP

# 4. Check if the record exists at all (query authoritative NS)
dig api.internal.company.com +trace

# 5. Check if /etc/hosts has an entry (may override DNS)
cat /etc/hosts | grep api

# 6. Check nsswitch.conf to confirm resolution order
cat /etc/nsswitch.conf | grep hosts
# Should show: hosts: files dns (or similar)
```

**Common causes and fixes:**

| Finding                                     | Cause                                  | Fix                                          |
| ------------------------------------------- | -------------------------------------- | -------------------------------------------- |
| `dig` returns NXDOMAIN                      | Record doesn't exist in DNS            | Create A record for api.internal.company.com |
| `dig` times out                             | DNS server unreachable                 | Check network connectivity to DNS server     |
| `dig @10.0.0.1` works but plain `dig` fails | Wrong DNS server in /etc/resolv.conf   | Update resolv.conf with correct DNS server   |
| Search domain missing                       | Using short name `api` instead of FQDN | Add search domain or use FQDN                |
| `/etc/hosts` has wrong entry                | Local override                         | Correct or remove /etc/hosts entry           |

**Quick workaround (if DNS fix takes time):**

```bash
# Add to /etc/hosts temporarily
echo "10.0.0.5 api.internal.company.com" | sudo tee -a /etc/hosts
```

### Question 2

**Scenario:** You run `ss -tlnp` and see:

```
LISTEN 0 128 0.0.0.0:80 0.0.0.0:* users:(("nginx",pid=1234,fd=6))
LISTEN 0 128 127.0.0.1:3306 0.0.0.0:* users:(("mysqld",pid=5678,fd=20))
LISTEN 0 128 :::22 :::* users:(("sshd",pid=9012,fd=4))
```

**Question:** For each listening socket, answer: Is it accessible from other machines? Why or why not? What does `0.0.0.0` vs `127.0.0.1` vs `:::` mean?

**Answer:**

| Socket                   | Accessible from other machines? | Why?                                                                 |
| ------------------------ | ------------------------------- | -------------------------------------------------------------------- |
| `0.0.0.0:80` (nginx)     | **Yes**                         | Listens on all IPv4 interfaces (including external)                  |
| `127.0.0.1:3306` (mysql) | **No**                          | Listens only on loopback (localhost only)                            |
| `:::22` (sshd)           | **Yes**                         | `:::` means all IPv6 interfaces (includes IPv4 via mapped addresses) |

**Explanation of address meanings:**

| Address                             | Meaning                        | Accessibility                               |
| ----------------------------------- | ------------------------------ | ------------------------------------------- |
| `0.0.0.0`                           | All IPv4 interfaces (wildcard) | Accessible from any network interface       |
| `127.0.0.1`                         | Loopback only                  | Only localhost, not accessible from network |
| `::`                                | All IPv6 interfaces (wildcard) | Accessible from any IPv6 interface          |
| `::1`                               | IPv6 loopback                  | Only localhost                              |
| Specific IP (e.g., `192.168.1.100`) | Only that interface            | Only on that specific IP                    |

**Why MySQL is on 127.0.0.1:**

* Security best practice – database should not be exposed to network

* Applications must run on same server or use SSH tunnel

* To make accessible from other servers, change `bind-address` in MySQL config

**IPv6 note:** `:::22` with two colons means:

* First `::` = all IPv6 addresses (wildcard)

* `:22` = port 22

* This also accepts IPv4 connections if `net.ipv6.bindv6only` is 0 (default on most Linux)

**Fixing MySQL accessibility (if needed):**

```bash
# Edit /etc/mysql/mysql.conf.d/mysqld.cnf
# Change: bind-address = 0.0.0.0
sudo systemctl restart mysql
# Also update firewall to restrict access to trusted IPs
```

### Question 3

**Scenario:** You are debugging a slow web application. Users report that `curl https://api.example.com/health` takes 5 seconds sometimes, but other times it's fast. You suspect DNS is involved.

**Question:** How would you measure DNS resolution time independently? What commands would you use to isolate DNS as the culprit? How would you confirm?

**Answer:**

**Step 1: Measure DNS resolution time in isolation**

```bash
# Using dig (shows query time)
dig api.example.com
# Query time: 2345 msec  ← This is the DNS resolution time!

# Multiple queries to see variance
for i in {1..10}; do dig api.example.com | grep "Query time"; done

# Using drill (alternative)
drill api.example.com | grep "Query time"

# Using time command with host
time host api.example.com
```

**Step 2: Compare with curl's DNS time**

```bash
# curl with verbose shows each stage
curl -w "DNS lookup: %{time_namelookup}s\nTotal: %{time_total}s\n" \
     -o /dev/null -s https://api.example.com/health

# Output example:
# DNS lookup: 2.345s  ← If high, DNS is slow
# Total: 2.500s
```

**Step 3: Isolate the issue**

```bash
# Bypass DNS by using IP directly
IP=$(dig api.example.com +short)
curl -w "Total: %{time_total}s\n" -o /dev/null -s https://$IP/health

# If IP direct is fast (e.g., 0.1s) but hostname is slow (2.3s), DNS is the problem
# If both are slow, problem is network or application
```

**Step 4: Identify which DNS server is slow**

```bash
# Check current DNS servers
cat /etc/resolv.conf
# nameserver 10.0.0.1
# nameserver 8.8.8.8

# Test each nameserver individually
time dig @10.0.0.1 api.example.com
time dig @8.8.8.8 api.example.com

# Check if first nameserver is timing out
# Default timeout is 5 seconds per nameserver
```

**Step 5: Verify DNS over TCP vs UDP**

```bash
# DNS uses UDP by default (fast but unreliable for large responses)
# Force TCP (more reliable but slightly slower)
dig +tcp api.example.com

# If UDP is failing/dropping packets, DNS will fall back to TCP (slower)
```

**Common DNS slow causes and fixes:**

| Cause                               | Diagnostic                     | Fix                                    |
| ----------------------------------- | ------------------------------ | -------------------------------------- |
| First nameserver unreachable        | `time dig @10.0.0.1` times out | Reorder resolv.conf (working NS first) |
| High latency to DNS server          | `ping 10.0.0.1` shows high RTT | Use closer DNS server                  |
| Packet loss to DNS server           | `mtr 10.0.0.1` shows loss      | Investigate network                    |
| Large DNS response (over 512 bytes) | Use `dig +bufsize=512`         | Truncation forces TCP (slower)         |
| DNSSEC validation                   | `dig +dnssec` shows validation | Disable DNSSEC if not needed           |

**Temporary workaround:**

```bash
# Add static entry to /etc/hosts
echo "192.168.1.100 api.example.com" | sudo tee -a /etc/hosts
```

**Permanent fix (if using systemd-resolved):**

```bash
# Configure specific DNS server for a domain
sudo mkdir -p /etc/systemd/resolved.conf.d/
sudo tee /etc/systemd/resolved.conf.d/example.conf << EOF
[Resolve]
DNS=10.0.0.1
Domains=~example.com
EOF
sudo systemctl restart systemd-resolved
```

### Question 4

**Scenario:** You have just deployed a new website `newsite.example.com`. From inside your office network, `ping newsite.example.com` works and returns the correct IP. From your home internet (using Google DNS 8.8.8.8), `ping newsite.example.com` returns "unknown host".

**Question:** What is the most likely configuration issue? How would you verify using DNS tools? How would you fix it?

**Answer:**

**Most likely cause:** Split-horizon (split-view) DNS configuration. The DNS server inside the office network has the record, but public DNS servers (like 8.8.8.8) do not. The domain likely has no public A record.

**Verification using DNS tools:**

```bash
# From inside office network
dig newsite.example.com
# Should return IP address

# From outside (using public DNS)
dig @8.8.8.8 newsite.example.com
# Likely returns NXDOMAIN or NOERROR with 0 answers

# Check if the domain has ANY public records
dig @8.8.8.8 example.com NS
# Verify the authoritative nameservers for the domain

# Query the authoritative nameserver directly
dig @ns1.example.com newsite.example.com
# If this works but @8.8.8.8 doesn't, the public NS doesn't have the record

# Check the SOA record to see the zone
dig @8.8.8.8 example.com SOA
```

**Alternative cause (less likely):**

* DNS propagation delay (if record was just added, may take hours)

* TTL caching (but `+trace` would bypass cache)

**Fixing the issue:**

**Option 1: Add public A record (if site should be public)**

```bash
# Go to DNS hosting provider (e.g., Route53, Cloudflare, GoDaddy)
# Add A record:
# Name: newsite
# Type: A
# Value: (public IP address)
# TTL: 300 (5 minutes for testing, then increase)
```

**Option 2: If site should only be internal (no public access)**

* The current behavior is correct – external users cannot resolve

* No fix needed (design intent)

**Option 3: Use different subdomain for internal vs external**

```bash
# Internal: newsite.internal.example.com
# External: newsite.example.com (public)
# Or use separate zones
```

**Testing after fix:**

```bash
# Wait for TTL (or use +trace to bypass caches)
dig +trace newsite.example.com

# Verify from multiple public DNS servers
dig @8.8.8.8 newsite.example.com
dig @1.1.1.1 newsite.example.com
dig @9.9.9.9 newsite.example.com

# Check propagation from multiple locations (using online tools)
# dig web interface: https://digwebinterface.com
```

**Understanding TTL and propagation:**

* After adding/updating a record, old TTL values cause caching

* Default TTL often 3600 seconds (1 hour)

* For critical changes, lower TTL to 60 seconds beforehand, wait, make change, then increase TTL

### Question 5

**Scenario:** A junior engineer asks: "Why does `ping google.com` work, but `ping -c 1 google.com | grep "time="` sometimes shows `1000 ms` but other times shows `20 ms`? And why does `mtr google.com` show packet loss at hop 3, but no loss at the final destination?"

**Question:** Explain these phenomena. What do they indicate about the network?

**Answer:**

**Part 1: High latency variance (jitter)**

**Phenomenon:** RTT varies from 20ms to 1000ms.

**Explanation:** This is called **jitter** – variance in packet delay. Causes include:

* Network congestion (buffers filling up)

* Route changes (different paths taken)

* Wireless interference (if WiFi involved)

* CPU load on routers (processing delay)

**What it indicates:**

* Unstable network connection

* Potential bufferbloat (excessive buffering in routers)

* Congestion somewhere along the path

**Diagnosis:**

```bash
# Measure jitter over time
ping -c 100 google.com | grep "time=" | awk -F'time=' '{print $2}' | awk '{print $1}'

# Use mtr to see which hop introduces jitter
mtr google.com
# Look for high StDev (standard deviation) at a specific hop
```

**Part 2: Packet loss at intermediate hop but not at destination**

**Phenomenon:** Hop 3 shows 20% loss, but final destination shows 0% loss.

**Explanation:** This is **normal and common**. Causes:

* Intermediate routers prioritize forwarding over responding to ICMP (rate limiting)

* Router control plane (CPU) handles ICMP replies separately from data plane (fast path)

* The router may drop ICMP echo requests but still forward data packets

**What it indicates:**

* NOT a problem – it's a feature of router design

* The router is configured to rate-limit ICMP responses

* Data traffic is still flowing correctly

**How to confirm:**

```bash
# Use TCP traceroute (more realistic, not rate-limited as often)
traceroute -T -p 443 google.com

# Use MTR in TCP mode
mtr -T -P 443 google.com

# Compare ICMP vs TCP loss
# If ICMP shows loss but TCP doesn't, it's ICMP rate limiting
```

**When to worry about packet loss:**

| Loss Pattern                                      | Concern Level       | Action                     |
| ------------------------------------------------- | ------------------- | -------------------------- |
| Loss only at intermediate hops, 0% at destination | Low (normal)        | None needed                |
| Loss at intermediate AND destination              | High                | Investigate network        |
| Loss increases as hops increase                   | Medium              | Possible congestion        |
| Loss at specific hop only (but not later)         | Low (rate limiting) | Verify with TCP traceroute |

**Complete diagnosis script:**

```bash
#!/bin/bash
# network_quality.sh

TARGET="google.com"

echo "=== ICMP Ping Test ==="
ping -c 20 $TARGET | tail -1

echo -e "\n=== ICMP MTR (loss/latency per hop) ==="
mtr -r -c 20 $TARGET

echo -e "\n=== TCP MTR (bypasses ICMP rate limiting) ==="
mtr -T -r -c 20 $TARGET

echo -e "\n=== Jitter Analysis ==="
ping -c 50 $TARGET | grep "time=" | sed 's/.*time=\([0-9.]*\) ms.*/\1/' | awk '{sum+=$1; sumsq+=$1*$1; count++} END {mean=sum/count; jitter=sqrt(sumsq/count - mean*mean); print "Mean:", mean, "ms\nJitter (stddev):", jitter, "ms"}'
```
