## Cheatsheet: Packet Capture and Firewalling

### Tcpdump – Packet Capture

| Command                                | Purpose                   |
| -------------------------------------- | ------------------------- |
| `sudo tcpdump -i eth0`                 | Capture on interface eth0 |
| `sudo tcpdump -i eth0 -c 100`          | Capture 100 packets       |
| `sudo tcpdump -i eth0 -n`              | No DNS resolution         |
| `sudo tcpdump -i eth0 -A`              | Show packet data in ASCII |
| `sudo tcpdump -i eth0 -w capture.pcap` | Write to file             |
| `tcpdump -r capture.pcap`              | Read from file            |

### Tcpdump Filters (BPF)

| Filter             | Example                                          |
| ------------------ | ------------------------------------------------ |
| Host               | `host 192.168.1.1`                               |
| Source/Destination | `src host 192.168.1.1` or `dst host 192.168.1.1` |
| Network            | `net 192.168.1.0/24`                             |
| Port               | `port 80`                                        |
| Protocol           | `tcp`, `udp`, `icmp`, `arp`                      |
| Complex            | `host 10.0.0.1 and port 80`                      |
| TCP flags          | `'tcp[tcpflags] & tcp-syn != 0'`                 |

### TCP Flags

| Flag    | Symbol | Meaning          |
| ------- | ------ | ---------------- |
| SYN     | `[S]`  | Start connection |
| SYN-ACK | `[S.]` | Acknowledge sync |
| ACK     | `[.]`  | Acknowledge      |
| PSH     | `[P.]` | Push data        |
| FIN     | `[F]`  | Finish close     |
| RST     | `[R]`  | Reset connection |

### Iptables – Firewall (Legacy)

| Command                                                            | Purpose           |
| ------------------------------------------------------------------ | ----------------- |
| `iptables -L -v -n`                                                | List rules        |
| `iptables -F`                                                      | Flush all rules   |
| `iptables -P INPUT DROP`                                           | Default policy    |
| `iptables -A INPUT -p tcp --dport 22 -j ACCEPT`                    | Allow SSH         |
| `iptables -A INPUT -s 203.0.113.50 -j DROP`                        | Block IP          |
| `iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT` | Allow established |
| `iptables-save > /etc/iptables.rules`                              | Save rules        |
| `iptables-restore < /etc/iptables.rules`                           | Restore rules     |

### Iptables Chains (filter table)

| Chain   | Direction          | Typical Use                 |
| ------- | ------------------ | --------------------------- |
| INPUT   | To local process   | Allow/deny inbound services |
| FORWARD | Through server     | Routing between interfaces  |
| OUTPUT  | From local process | Allow/deny outbound         |

### Iptables NAT (nat table)

| Chain       | Direction      | Typical Use            |
| ----------- | -------------- | ---------------------- |
| PREROUTING  | Before routing | DNAT (port forwarding) |
| POSTROUTING | After routing  | SNAT (masquerade)      |

### Common Iptables Rule Options

| Option     | Meaning          | Example               |
| ---------- | ---------------- | --------------------- |
| `-p tcp`   | Protocol         | `-p udp`, `-p icmp`   |
| `--dport`  | Destination port | `--dport 80`          |
| `--sport`  | Source port      | `--sport 1024:65535`  |
| `-s`       | Source IP        | `-s 192.168.1.0/24`   |
| `-d`       | Destination IP   | `-d 10.0.0.1`         |
| `-i`       | Input interface  | `-i eth0`             |
| `-o`       | Output interface | `-o eth0`             |
| `-m state` | Connection state | `--state ESTABLISHED` |

### UFW – Simplified Firewall (Ubuntu/Debian)

| Command                             | Purpose          |
| ----------------------------------- | ---------------- |
| `sudo ufw enable`                   | Enable firewall  |
| `sudo ufw disable`                  | Disable firewall |
| `sudo ufw status verbose`           | Show status      |
| `sudo ufw allow 22/tcp`             | Allow SSH        |
| `sudo ufw allow from 192.168.1.100` | Allow from IP    |
| `sudo ufw deny from 203.0.113.50`   | Block IP         |
| `sudo ufw delete allow 80/tcp`      | Delete rule      |

### Nftables – Modern Firewall

| Command                                                                   | Purpose          |
| ------------------------------------------------------------------------- | ---------------- |
| `nft list ruleset`                                                        | List all rules   |
| `nft flush ruleset`                                                       | Flush all rules  |
| `nft add table inet filter`                                               | Create table     |
| `nft add chain inet filter input { type filter hook input priority 0\; }` | Create chain     |
| `nft add rule inet filter input tcp dport 22 accept`                      | Add allow rule   |
| `nft list ruleset -a`                                                     | Show handles     |
| `nft delete rule inet filter input handle 3`                              | Delete by handle |
| `nft list ruleset > /etc/nftables.conf`                                   | Save rules       |

### Netfilter Packet Flow

```
Incoming → PREROUTING → Routing → INPUT (to local) OR FORWARD (through)
Outgoing (local) → OUTPUT → POSTROUTING
Routing (forward) → PREROUTING → FORWARD → POSTROUTING
```

## Comparison Tables

### Tcpdump vs Tshark vs Wireshark

| Tool        | Interface | Best For                                  |
| ----------- | --------- | ----------------------------------------- |
| `tcpdump`   | CLI       | Quick captures, remote debugging, scripts |
| `tshark`    | CLI       | Analysis, stream reassembly, statistics   |
| `Wireshark` | GUI       | Deep inspection, filters, visual analysis |

### Iptables vs Nftables vs UFW

| Feature        | Iptables        | Nftables                           | UFW              |
| -------------- | --------------- | ---------------------------------- | ---------------- |
| Complexity     | High            | Medium                             | Low              |
| Performance    | Slower          | Faster                             | Same as iptables |
| IPv4/IPv6      | Separate tables | Unified (inet)                     | Unified          |
| Default on     | Legacy systems  | RHEL 8+, Debian 10+, Ubuntu 22.04+ | Ubuntu, Debian   |
| Learning curve | Steep           | Moderate                           | Shallow          |
| Use case       | Complex rules   | Modern deployments                 | Simple rules     |

### Netfilter Tables

| Table  | Purpose                     | Chains                                 |
| ------ | --------------------------- | -------------------------------------- |
| filter | Packet filtering            | INPUT, FORWARD, OUTPUT                 |
| nat    | Network address translation | PREROUTING, INPUT, OUTPUT, POSTROUTING |
| mangle | Packet modification         | All 5 chains                           |
| raw    | Connection tracking bypass  | PREROUTING, OUTPUT                     |

## Interview Questions (Scenario-Based)

These questions assume only knowledge from Subchapter 2.3. Answers reference only concepts from 2.3.1 and 2.3.2.

### Question 1

**Scenario:** A user reports that `curl http://api.example.com` hangs and eventually times out. You suspect a firewall is blocking the connection.

**Question:** How would you use `tcpdump` and `iptables` to diagnose and confirm? Provide the step-by-step process.

**Answer:**

**Step 1: Capture traffic to see what's happening**

```bash
# On the server (or client, depending on where you have access)
sudo tcpdump -i eth0 -n host api.example.com and port 80 -c 20
```

**What to look for:**

| Tcpdump Output                       | Meaning                                   |
| ------------------------------------ | ----------------------------------------- |
| `Flags [S]` sent, no `[S.]` response | Firewall dropping SYN (no response)       |
| `Flags [S]` sent, `[R]` received     | Firewall actively rejecting (RST)         |
| No packets at all                    | Routing issue or wrong interface          |
| Packets sent but no response         | Firewall dropping or server not listening |

**Step 2: Check server-side firewall**

```bash
# On the server (api.example.com)
# Check if port 80 is allowed in iptables
sudo iptables -L INPUT -v -n | grep :80

# Check if default policy is DROP
sudo iptables -L INPUT -n | grep "Chain INPUT"

# Check for any DROP rules
sudo iptables -L INPUT -v -n | grep DROP
```

**Step 3: Temporarily add logging to identify dropped packets**

```bash
# Add logging rule before DROP
sudo iptables -I INPUT 1 -j LOG --log-prefix "FIREWALL INPUT: " --log-level 4

# Check logs
sudo tail -f /var/log/kern.log | grep "FIREWALL INPUT"

# After test, remove logging rule
sudo iptables -D INPUT 1
```

**Step 4: Test by temporarily allowing the traffic**

```bash
# Add allow rule (temporarily)
sudo iptables -I INPUT 1 -p tcp --dport 80 -j ACCEPT

# Test curl again
curl http://api.example.com

# If works, firewall was the issue
# Remove test rule
sudo iptables -D INPUT 1
```

**Step 5: Fix permanently**

```bash
# Add permanent allow rule
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT

# Save rules
sudo iptables-save > /etc/iptables/rules.v4
```

### Question 2

**Scenario:** Your tcpdump output shows:

```
10:00:01.234567 IP 192.168.1.100.54321 > 93.184.216.34.80: Flags [S], seq 123456789, win 65535, length 0
10:00:02.345678 IP 192.168.1.100.54321 > 93.184.216.34.80: Flags [S], seq 123456789, win 65535, length 0
10:00:04.456789 IP 192.168.1.100.54321 > 93.184.216.34.80: Flags [S], seq 123456789, win 65535, length 0
10:00:08.567890 IP 192.168.1.100.54321 > 93.184.216.34.80: Flags [S], seq 123456789, win 65535, length 0
```

**Question:** What is happening? Why is there no response? What does the pattern of retransmissions tell you?

**Answer:**

**What is happening:**

* Client is sending TCP SYN packets (connection requests) to port 80

* No SYN-ACK response is received

* Client retransmits SYN at increasing intervals (1s, 2s, 4s, 8s) – this is TCP's exponential backoff

**Why no response (most likely):**

* **Firewall is dropping the packets** (no response at all)

* Alternative possibilities:

  * Server is down (but would usually send RST)

  * Routing issue (packets never reach server)

  * Server has no route back

**What the pattern tells you:**

| Observation                          | Meaning                                          |
| ------------------------------------ | ------------------------------------------------ |
| SYN sent, no response                | Packet likely dropped (not rejected)             |
| Exponential backoff (1s, 2s, 4s, 8s) | Normal TCP retransmission behavior               |
| Same source port (54321)             | Same connection attempt, not new connections     |
| No SYN-ACK or RST                    | Firewall is silently dropping (DROP, not REJECT) |

**Distinguishing DROP vs REJECT:**

| Firewall Action | Tcpdump Response                 |
| --------------- | -------------------------------- |
| `DROP`          | No response (timeout)            |
| `REJECT`        | ICMP port unreachable or TCP RST |

**Next diagnostic steps:**

```bash
# 1. On the server, verify if packets are arriving
sudo tcpdump -i eth0 -n host 192.168.1.100 and port 80

# 2. Check server firewall for DROP rules
sudo iptables -L INPUT -v -n | grep DROP

# 3. Check if server is listening on port 80
sudo ss -tlnp | grep :80

# 4. Temporarily disable firewall for testing
sudo iptables -P INPUT ACCEPT
# Then retest with tcpdump

# 5. Check network path
traceroute -n 93.184.216.34
```

### Question 3

**Scenario:** You need to set up a firewall on a production web server that:

* Allows SSH only from the management network `10.0.0.0/8`

* Allows HTTP/HTTPS from anywhere

* Allows ICMP (ping) from anywhere

* Drops everything else

* Logs dropped packets for auditing

**Question:** Write the complete iptables ruleset. Then write the equivalent nftables ruleset.

**Answer:**

**Iptables ruleset:**

```bash
#!/bin/bash
# production_web_firewall.sh

# Flush existing rules
iptables -F
iptables -t nat -F
iptables -t mangle -F

# Set default policies
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

# Allow loopback
iptables -A INPUT -i lo -j ACCEPT

# Allow established connections
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Allow SSH from management network only
iptables -A INPUT -p tcp --dport 22 -s 10.0.0.0/8 -j ACCEPT

# Allow HTTP and HTTPS from anywhere
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# Allow ICMP (ping) from anywhere
iptables -A INPUT -p icmp --icmp-type echo-request -j ACCEPT

# Log dropped packets (rate limited)
iptables -A INPUT -m limit --limit 5/min -j LOG --log-prefix "iptables DROP: " --log-level 4

# Save rules (Debian/Ubuntu)
iptables-save > /etc/iptables/rules.v4
```

**Nftables ruleset (`/etc/nftables.conf`):**

```bash
#!/usr/sbin/nft -f

flush ruleset

table inet filter {
    chain input {
        type filter hook input priority 0; policy drop;
        
        # Allow loopback
        iif lo accept
        
        # Allow established connections
        ct state established,related accept
        
        # Allow SSH from management network
        ip saddr 10.0.0.0/8 tcp dport 22 accept
        
        # Allow HTTP/HTTPS
        tcp dport {80, 443} accept
        
        # Allow ICMP (ping)
        ip protocol icmp icmp type echo-request accept
        
        # Log dropped packets (rate limited)
        log prefix "nftables DROP: " limit rate 5/minute counter drop
    }
    
    chain forward {
        type filter hook forward priority 0; policy drop;
    }
    
    chain output {
        type filter hook output priority 0; policy accept;
    }
}
```

**Enable and test:**

```bash
# For iptables
sudo chmod +x production_web_firewall.sh
sudo ./production_web_firewall.sh

# For nftables
sudo nft -f /etc/nftables.conf
sudo systemctl enable nftables
sudo systemctl start nftables

# Test
# Should succeed:
curl http://localhost
ping localhost
ssh -p 22 user@server  # from 10.0.0.0/8

# Should fail:
ssh -p 22 user@server  # from outside 10.0.0.0/8
telnet localhost 25
```

### Question 4

**Scenario:** You are troubleshooting a Kubernetes node. Some pods cannot communicate with each other across nodes. You capture packets with tcpdump on the source node and see:

```
10:00:01.234567 IP 10.244.1.5.54321 > 10.244.2.5.80: Flags [S], seq 123456789, win 65535, length 0
10:00:01.345678 IP 10.244.1.5.54321 > 10.244.2.5.80: Flags [S], seq 123456789, win 65535, length 0
```

But on the destination node, you see no packets at all.

**Question:** Where is the packet being dropped? What is the most likely cause in a Kubernetes environment? How would you verify?

**Answer:**

**Where the packet is being dropped:**

* Packets are leaving the source node (tcpdump shows them)

* Packets are not arriving at the destination node (no tcpdump output)

* Therefore, packet is being dropped **somewhere between the nodes**

**Most likely causes in Kubernetes:**

| Cause                                      | Likelihood | Verification                       |
| ------------------------------------------ | ---------- | ---------------------------------- |
| Network policy blocking cross-node traffic | High       | Check Calico/Cilium/Weave policies |
| CNI plugin misconfiguration                | Medium     | Check CNI logs                     |
| Overlay network (VXLAN, IPIP) not working  | Medium     | Check tunnel interfaces            |
| Firewall on host blocking overlay ports    | High       | Check iptables on both nodes       |
| Routing table missing route to pod CIDR    | Medium     | Check `ip route show`              |

**Verification steps:**

**Step 1: Check if nodes can ping each other**

```bash
# From source node to destination node IP (not pod IP)
ping <destination_node_ip>

# If fails, basic network issue
```

**Step 2: Check pod CIDR routing**

```bash
# On source node
ip route show | grep 10.244
# Should show route to destination pod CIDR via destination node IP

# Example:
# 10.244.2.0/24 via 192.168.1.20 dev eth0
```

**Step 3: Check overlay network**

```bash
# For VXLAN (common in Flannel, Calico)
ip -d link show flannel.1
ip neigh show | grep flannel

# For IPIP
ip -d link show tunl0
```

**Step 4: Check iptables on both nodes**

```bash
# Check if FORWARD chain allows pod-to-pod traffic
sudo iptables -L FORWARD -v -n

# Check if Kubernetes network policies are enforced
sudo iptables -L -v -n | grep -i "networkpolicy"
```

**Step 5: Capture on overlay interface**

```bash
# On source node, capture on overlay interface (e.g., flannel.1, vxlan.calico)
sudo tcpdump -i flannel.1 -n

# If packets seen on overlay but not on physical interface, overlay is working
```

**Step 6: Check CNI logs**

```bash
# Calico
kubectl logs -n kube-system calico-node-xxxxx

# Flannel
kubectl logs -n kube-system kube-flannel-xxxxx

# Cilium
kubectl logs -n kube-system cilium-xxxxx
```

**Most common fix:**

```bash
# Ensure overlay ports are open in firewall
# VXLAN: UDP 4789
# IPIP: IP protocol 4
# Geneve: UDP 6081

# On both nodes
sudo iptables -A INPUT -p udp --dport 4789 -j ACCEPT
```

### Question 5

**Scenario:** You need to set up port forwarding so that incoming traffic to `203.0.113.10:8080` is forwarded to an internal server `192.168.1.100:80`. The internal server is not directly accessible from the internet.

**Question:** Write the complete iptables ruleset for this NAT configuration. What additional kernel parameter is required?

**Answer:**

**Iptables NAT ruleset:**

```bash
#!/bin/bash
# port_forwarding_nat.sh

# Enable IP forwarding (required for NAT)
echo 1 > /proc/sys/net/ipv4/ip_forward
# Make permanent
echo "net.ipv4.ip_forward=1" >> /etc/sysctl.conf

# Flush existing rules
iptables -F
iptables -t nat -F
iptables -t mangle -F

# Set default policies for FORWARD (allow forwarding)
iptables -P FORWARD ACCEPT

# DNAT: Forward incoming port 8080 to internal server port 80
iptables -t nat -A PREROUTING -p tcp --dport 8080 -d 203.0.113.10 -j DNAT --to-destination 192.168.1.100:80

# Also handle traffic originating from the firewall itself (if needed)
iptables -t nat -A OUTPUT -p tcp --dport 8080 -d 203.0.113.10 -j DNAT --to-destination 192.168.1.100:80

# Allow forwarding of the traffic in filter table
iptables -A FORWARD -p tcp -d 192.168.1.100 --dport 80 -j ACCEPT
iptables -A FORWARD -p tcp -s 192.168.1.100 --sport 80 -j ACCEPT

# Optional: Source NAT (masquerade) so internal server sees traffic from firewall
# This is needed if internal server doesn't have route back to client
iptables -t nat -A POSTROUTING -d 192.168.1.100 -p tcp --dport 80 -j MASQUERADE

# Save rules
iptables-save > /etc/iptables/rules.v4
```

**Required kernel parameter:**

```bash
# IP forwarding must be enabled
net.ipv4.ip_forward = 1

# Verify
sysctl net.ipv4.ip_forward
# Should return: net.ipv4.ip_forward = 1
```

**Testing the configuration:**

```bash
# From external client
curl http://203.0.113.10:8080

# On the firewall, check NAT statistics
iptables -t nat -L -v -n | grep -A 5 PREROUTING

# On the internal server, verify traffic is coming from firewall IP
sudo tcpdump -i eth0 -n port 80
# Source IP should be firewall's internal IP (192.168.1.1 or similar)

# Check connection tracking
conntrack -L | grep 8080
```

**Alternative using nftables:**

```bash
#!/usr/sbin/nft -f

flush ruleset

table ip nat {
    chain prerouting {
        type nat hook prerouting priority -100; policy accept;
        tcp dport 8080 dnat to 192.168.1.100:80
    }
    
    chain postrouting {
        type nat hook postrouting priority 100; policy accept;
        ip daddr 192.168.1.100 tcp dport 80 masquerade
    }
}

table ip filter {
    chain forward {
        type filter hook forward priority 0; policy accept;
        ip daddr 192.168.1.100 tcp dport 80 accept
    }
}
```

**Troubleshooting NAT:**

```bash
# Check if packets are reaching DNAT rule
iptables -t nat -L PREROUTING -v -n

# Check if packets are being forwarded
iptables -L FORWARD -v -n

# Check NAT conntrack entries
conntrack -L | grep dnat

# Test with curl from firewall itself
curl http://127.0.0.1:8080
```

## Bridge Concepts (Not in Notes but Added for Clarity)

| Concept                           | Explanation                                                                                |
| --------------------------------- | ------------------------------------------------------------------------------------------ |
| BPF (Berkeley Packet Filter)      | The filtering language used by tcpdump. Allows powerful packet matching.                   |
| `conntrack`                       | Connection tracking system in Linux. Used by NAT and stateful firewalls.                   |
| Overlay network                   | Virtual network on top of physical network (VXLAN, IPIP, Geneve). Used by Kubernetes CNIs. |

| `MASQUERADE`                      | Special form of SNAT that uses the outgoing interface's IP. Good for dynamic IPs.          |
