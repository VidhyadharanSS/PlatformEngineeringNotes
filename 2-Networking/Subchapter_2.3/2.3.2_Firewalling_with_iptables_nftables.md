# 2.3.2 Firewalling with iptables/nftables: The Linux Packet Filter

#### Why Firewalling Matters

Every production server needs a firewall. Linux has built-in packet filtering capabilities that allow you to:

* Restrict access to services (e.g., only allow SSH from specific IPs)

* Block malicious traffic (DDoS, port scans)

* Perform NAT (Network Address Translation) for containers and VMs

* Log dropped packets for security auditing

This note covers `iptables` (legacy, still common) and `nftables` (modern replacement). Note 2.3.1 covered packet capture; note 2.3.3 is the subchapter review.

***

## Part 1: Netfilter Architecture – How Linux Filters Packets

Linux firewalling is built on the **Netfilter** framework, which hooks into the networking stack at key points.

```mermaid
graph LR
    A[Network Interface<br/>eth0] --> B[PREROUTING]
    B --> C[Routing Decision]
    C -->|Local process| D[INPUT]
    C -->|Forward| E[FORWARD]
    D --> F[Local process<br/>(nginx, sshd)]
    F --> G[OUTPUT]
    E --> H[POSTROUTING]
    G --> H
    H --> I[Network Interface]
    
    style B fill:#f99,stroke:#333
    style D fill:#9f9,stroke:#333
    style E fill:#9cf,stroke:#333
    style G fill:#ff9,stroke:#333
    style H fill:#f9f,stroke:#333
```

### Netfilter Hooks (Chains)

| Chain           | Point in Path      | Traffic Type               | Typical Use                 |
| --------------- | ------------------ | -------------------------- | --------------------------- |
| **PREROUTING**  | Before routing     | Incoming                   | DNAT (port forwarding)      |
| **INPUT**       | To local process   | Incoming to server         | Allow/deny inbound services |
| **FORWARD**     | Through server     | Routing between interfaces | Router/gateway rules        |
| **OUTPUT**      | From local process | Outgoing from server       | Allow/deny outbound         |
| **POSTROUTING** | After routing      | Outgoing                   | SNAT (masquerading)         |

### Packet Flow Summary

1. **Incoming packet** → PREROUTING → Routing Decision → INPUT (if for server) OR FORWARD (if routing)
2. **Outgoing packet** from local process → OUTPUT → POSTROUTING
3. **Routing packet** → PREROUTING → FORWARD → POSTROUTING

## Part 2: Iptables – The Legacy Firewall (Still Everywhere)

### Tables and Chains

Iptables organizes rules into **tables** (by purpose) and **chains** (by hook point).

| Table      | Purpose                        | Chains                                 | Default Policy |
| ---------- | ------------------------------ | -------------------------------------- | -------------- |
| **filter** | Packet filtering (allow/deny)  | INPUT, FORWARD, OUTPUT                 | ACCEPT         |
| **nat**    | Network address translation    | PREROUTING, INPUT, OUTPUT, POSTROUTING | ACCEPT         |
| **mangle** | Packet modification (TTL, TOS) | All 5 chains                           | ACCEPT         |
| **raw**    | Connection tracking exemptions | PREROUTING, OUTPUT                     | ACCEPT         |

**90% of your work will be in the** **`filter`** **table.**

### Basic Iptables Commands

```bash
# View current rules (filter table)
sudo iptables -L
sudo iptables -L -v -n   # verbose, numeric (no DNS)

# View with line numbers (for deletion)
sudo iptables -L --line-numbers

# View specific chain
sudo iptables -L INPUT -v -n

# View NAT table
sudo iptables -t nat -L -v -n

# Flush all rules (delete)
sudo iptables -F
sudo iptables -t nat -F

# Delete specific rule by line number
sudo iptables -D INPUT 3

# Set default policy
sudo iptables -P INPUT DROP
sudo iptables -P FORWARD DROP
sudo iptables -P OUTPUT ACCEPT
```

### Allow/Deny Rules

```bash
# Allow SSH from specific IP
sudo iptables -A INPUT -p tcp --dport 22 -s 192.168.1.100 -j ACCEPT

# Allow HTTP and HTTPS from anywhere
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# Allow ICMP (ping) from anywhere
sudo iptables -A INPUT -p icmp --icmp-type echo-request -j ACCEPT

# Allow established connections (important!)
sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Block an IP address
sudo iptables -A INPUT -s 10.0.0.100 -j DROP

# Block a specific port from a specific IP
sudo iptables -A INPUT -p tcp --dport 22 -s 203.0.113.50 -j DROP

# Log dropped packets (before DROP rule)
sudo iptables -A INPUT -j LOG --log-prefix "DROPPED: " --log-level 4
```

### Rule Matching Options

| Option         | Meaning          | Example                   |
| -------------- | ---------------- | ------------------------- |
| `-p tcp`       | Protocol         | `-p udp`, `-p icmp`       |
| `--dport`      | Destination port | `--dport 80`              |
| `--sport`      | Source port      | `--sport 1024:65535`      |
| `-s`           | Source IP        | `-s 192.168.1.0/24`       |
| `-d`           | Destination IP   | `-d 10.0.0.1`             |
| `-i`           | Input interface  | `-i eth0`                 |
| `-o`           | Output interface | `-o eth0`                 |
| `-m state`     | Connection state | `--state ESTABLISHED`     |
| `-m multiport` | Multiple ports   | `--dports 80,443`         |
| `-m comment`   | Add comment      | `--comment "web traffic"` |

### Practical Iptables Example: Web Server

```bash
#!/bin/bash
# web_server_firewall.sh

# Flush existing rules
iptables -F
iptables -t nat -F

# Default policies
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

# Allow loopback
iptables -A INPUT -i lo -j ACCEPT

# Allow established connections
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Allow SSH (rate limited to prevent brute force)
iptables -A INPUT -p tcp --dport 22 -m state --state NEW -m recent --set
iptables -A INPUT -p tcp --dport 22 -m state --state NEW -m recent --update --seconds 60 --hitcount 4 -j DROP
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# Allow web traffic
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# Allow ICMP (ping)
iptables -A INPUT -p icmp --icmp-type echo-request -j ACCEPT

# Log dropped packets (rate limited)
iptables -A INPUT -m limit --limit 5/min -j LOG --log-prefix "iptables DROP: " --log-level 4

# Save rules (Debian/Ubuntu)
iptables-save > /etc/iptables/rules.v4

# Save rules (RHEL/Rocky)
service iptables save
```

### NAT with Iptables

**Source NAT (Masquerade)** – Share a single public IP for outbound traffic:

```bash
# Masquerade all traffic from 192.168.1.0/24 going out eth0
sudo iptables -t nat -A POSTROUTING -s 192.168.1.0/24 -o eth0 -j MASQUERADE

# Enable IP forwarding (required for NAT)
sudo sysctl -w net.ipv4.ip_forward=1
echo "net.ipv4.ip_forward=1" | sudo tee -a /etc/sysctl.conf
```

**Destination NAT (Port Forwarding)** – Forward incoming port to internal server:

```bash
# Forward incoming port 8080 to internal server port 80
sudo iptables -t nat -A PREROUTING -p tcp --dport 8080 -j DNAT --to-destination 192.168.1.100:80

# Also allow forwarding (filter table)
sudo iptables -A FORWARD -p tcp -d 192.168.1.100 --dport 80 -j ACCEPT
```

### Saving and Restoring Iptables Rules

```bash
# Save current rules
sudo iptables-save > /etc/iptables.rules

# Restore rules
sudo iptables-restore < /etc/iptables.rules

# Debian/Ubuntu persistent
sudo apt install iptables-persistent
sudo netfilter-persistent save
sudo netfilter-persistent reload

# RHEL/Rocky persistent
sudo service iptables save
sudo systemctl enable iptables
```

## Part 3: UFW – Uncomplicated Firewall (Simplified Iptables)

UFW provides a user-friendly wrapper around iptables, common on Ubuntu/Debian.

```bash
# Enable UFW
sudo ufw enable

# Check status
sudo ufw status verbose

# Allow SSH
sudo ufw allow 22/tcp
sudo ufw allow ssh   # by service name (reads /etc/services)

# Allow HTTP/HTTPS
sudo ufw allow 80/tcp
sudo ufw allow 443/tcp

# Allow from specific IP
sudo ufw allow from 192.168.1.100 to any port 22

# Allow subnet
sudo ufw allow from 192.168.1.0/24

# Deny an IP
sudo ufw deny from 203.0.113.50

# Delete rule
sudo ufw delete allow 80/tcp

# Show numbered rules (for deletion)
sudo ufw status numbered
sudo ufw delete 3

# Reset (disable all rules)
sudo ufw reset

# Disable UFW
sudo ufw disable
```

**UFW configuration files:**

* `/etc/default/ufw` – Main configuration

* `/etc/ufw/before.rules` – Rules processed before defaults

* `/etc/ufw/after.rules` – Rules processed after defaults

## Part 4: Nftables – The Modern Replacement

Nftables is the successor to iptables, with simpler syntax and better performance. It's default on RHEL 8+, Debian 10+, Ubuntu 22.04+.

### Basic Nftables Commands

```bash
# List all rules
sudo nft list ruleset

# List specific table
sudo nft list table inet filter

# Flush all rules
sudo nft flush ruleset

# Create table
sudo nft add table inet filter

# Create chains
sudo nft add chain inet filter input { type filter hook input priority 0\; }
sudo nft add chain inet filter forward { type filter hook forward priority 0\; }
sudo nft add chain inet filter output { type filter hook output priority 0\; }

# Add rules
sudo nft add rule inet filter input tcp dport 22 accept
sudo nft add rule inet filter input tcp dport 80 accept
sudo nft add rule inet filter input tcp dport 443 accept
sudo nft add rule inet filter input ct state established,related accept
sudo nft add rule inet filter input drop

# Insert rule at position
sudo nft insert rule inet filter input position 0 tcp dport 22 accept

# Delete rule (by handle)
sudo nft list ruleset -a   # show handles
sudo nft delete rule inet filter input handle 3
```

### Nftables Configuration File

```bash
# /etc/nftables.conf
#!/usr/sbin/nft -f

flush ruleset

table inet filter {
    chain input {
        type filter hook input priority 0; policy drop;
        
        # Allow loopback
        iif lo accept
        
        # Allow established connections
        ct state established,related accept
        
        # Allow SSH
        tcp dport 22 accept
        
        # Allow HTTP/HTTPS
        tcp dport {80, 443} accept
        
        # Allow ICMP (ping)
        ip protocol icmp accept
        
        # Log dropped packets
        log prefix "nftables DROP: " counter drop
    }
    
    chain forward {
        type filter hook forward priority 0; policy drop;
    }
    
    chain output {
        type filter hook output priority 0; policy accept;
    }
}
```

### Enable Nftables

```bash
# Start and enable
sudo systemctl enable nftables
sudo systemctl start nftables

# Test configuration
sudo nft -f /etc/nftables.conf

# Save current rules
sudo nft list ruleset > /etc/nftables.conf
```

## Part 5: Iptables vs Nftables Comparison

| Feature        | Iptables                                                             | Nftables                           |
| -------------- | -------------------------------------------------------------------- | ---------------------------------- |
| Syntax         | Multiple commands (`iptables`, `ip6tables`, `arptables`, `ebtables`) | Single `nft` command               |
| Performance    | Slower (linear rule evaluation)                                      | Faster (concatenation, maps)       |
| Rule updates   | Replace entire ruleset                                               | Incremental updates                |
| IPv4/IPv6      | Separate tables                                                      | Unified (inet family)              |
| Default on     | Older systems                                                        | RHEL 8+, Debian 10+, Ubuntu 22.04+ |
| Learning curve | Steep                                                                | Moderate                           |

### When to Use Which

| Scenario                                | Recommended            |
| --------------------------------------- | ---------------------- |
| Maintaining legacy servers              | Iptables               |
| New deployments (RHEL 8+/Ubuntu 22.04+) | Nftables               |
| Simple rules (allow/deny ports)         | UFW (if Debian/Ubuntu) |
| Complex NAT/routing                     | Iptables or nftables   |
| Kubernetes network policies             | Iptables (kube-proxy)  |

## Part 6: Common Firewall Scenarios

### Scenario 1: Docker Host

Docker manipulates iptables directly. Don't flush all rules!

```bash
# Allow SSH from management network
sudo iptables -I INPUT 1 -p tcp --dport 22 -s 10.0.0.0/8 -j ACCEPT

# Docker will handle container traffic automatically
# Don't set default DROP on FORWARD chain
```

### Scenario 2: Kubernetes Node

Kubernetes kube-proxy manages iptables rules.

```bash
# Allow kubelet API
sudo iptables -A INPUT -p tcp --dport 10250 -j ACCEPT

# Allow NodePort services (30000-32767)
sudo iptables -A INPUT -p tcp --dport 30000:32767 -j ACCEPT

# Allow overlay network (e.g., Flannel)
sudo iptables -A INPUT -p udp --dport 8472 -j ACCEPT
```

### Scenario 3: Rate Limiting SSH to Prevent Brute Force

```bash
# Iptables
sudo iptables -A INPUT -p tcp --dport 22 -m state --state NEW -m recent --set
sudo iptables -A INPUT -p tcp --dport 22 -m state --state NEW -m recent --update --seconds 60 --hitcount 4 -j DROP
sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# Nftables
sudo nft add rule inet filter input tcp dport 22 ct state new limit rate 4/minute accept
```

### Scenario 4: GeoIP Blocking (Using ipsets)

```bash
# Install ipset
sudo apt install ipset   # Debian/Ubuntu
sudo dnf install ipset   # RHEL/Rocky

# Create ipset for blocked countries
sudo ipset create blocklist hash:net

# Add networks (e.g., China, Russia)
sudo ipset add blocklist 1.0.0.0/8
sudo ipset add blocklist 2.0.0.0/8

# Block in iptables
sudo iptables -A INPUT -m set --match-set blocklist src -j DROP
```

## Part 7: Connection Tracking (conntrack) Deep Dive

Stateful firewalls rely on **connection tracking** to allow return traffic. Understanding `conntrack` is essential for debugging production issues.

### What conntrack Does

The `conntrack` system tracks every connection passing through the firewall. When you write `--state ESTABLISHED,RELATED`, the kernel checks the conntrack table.

```bash
# Install conntrack tools
sudo apt install conntrack   # Debian/Ubuntu
sudo dnf install conntrack-tools  # RHEL/Rocky

# View active connections
sudo conntrack -L
# tcp 6 431999 ESTABLISHED src=10.0.0.5 dst=10.0.0.10 sport=54321 dport=80 ...

# Count total tracked connections
sudo conntrack -C
# 8432

# Check maximum conntrack table size
cat /proc/sys/net/netfilter/nf_conntrack_max
# 262144

# View current usage vs max
echo "$(sudo conntrack -C) / $(cat /proc/sys/net/netfilter/nf_conntrack_max)"
```

### Conntrack Table Overflow — A Silent Production Killer

When the conntrack table fills up, **new connections are silently dropped**. This causes random connection failures with no clear error.

```bash
# Check for dropped connections (critical monitoring!)
dmesg | grep "nf_conntrack: table full"
# nf_conntrack: table full, dropping packet

# Monitor in real time
sudo journalctl -k -f | grep conntrack
```

**Fixing conntrack overflow:**

```bash
# Increase table size (temporary)
sudo sysctl -w net.netfilter.nf_conntrack_max=524288

# Decrease timeout for finished connections (free entries faster)
sudo sysctl -w net.netfilter.nf_conntrack_tcp_timeout_time_wait=30
sudo sysctl -w net.netfilter.nf_conntrack_tcp_timeout_established=3600

# Make permanent in /etc/sysctl.conf
echo "net.netfilter.nf_conntrack_max=524288" | sudo tee -a /etc/sysctl.conf
echo "net.netfilter.nf_conntrack_tcp_timeout_time_wait=30" | sudo tee -a /etc/sysctl.conf
```

**Conntrack states and their timeouts:**

| State          | Default Timeout | Meaning                              |
| -------------- | --------------- | ------------------------------------ |
| ESTABLISHED    | 432000s (5 days)| Active connection                    |
| TIME_WAIT      | 120s            | Connection closed, awaiting cleanup  |
| SYN_SENT       | 120s            | Outgoing connection attempt          |
| NEW            | 30s             | First packet of new connection       |

> **Warning:** In Kubernetes, kube-proxy uses conntrack heavily. A node handling thousands of services can easily exhaust the default 262144 table. Always monitor `nf_conntrack_count` vs `nf_conntrack_max` in production.

## Part 8: Troubleshooting Firewalls

### Check if Firewall is Blocking

```bash
# Check current rules
sudo iptables -L -v -n
sudo nft list ruleset

# Check packet counters (look for dropped packets)
sudo iptables -L -v -n | grep -A 10 "Chain INPUT"

# Temporarily disable firewall (for testing)
# Iptables
sudo iptables -P INPUT ACCEPT
sudo iptables -P FORWARD ACCEPT
sudo iptables -F

# UFW
sudo ufw disable

# Nftables
sudo nft flush ruleset

# Re-enable after test
```

### Check if Port is Open (From Outside)

```bash
# From another machine
nc -zv target_ip 80
telnet target_ip 80

# Check if connection reaches server (on server)
sudo tcpdump -i eth0 port 80
```

### Logging Dropped Packets

```bash
# Iptables
sudo iptables -A INPUT -j LOG --log-prefix "FIREWALL DROP: " --log-level 4
sudo tail -f /var/log/kern.log | grep "FIREWALL DROP"

# Nftables
sudo nft add rule inet filter input log prefix "nftables DROP: " counter drop
sudo journalctl -f -k | grep "nftables DROP"
```

## Quick Task: Firewall Practice

*Practice configuring firewalls in a test environment (not production!).*

1. Using iptables, allow only SSH (port 22) from your current IP. Drop everything else.
2. Test by trying to ping google.com (should fail).
3. Add a rule to allow HTTP (port 80) to any destination.
4. Flush all rules and restore normal operation.
5. (Optional) Repeat using nftables or UFW.

> **Ready Solution:**
>
> ```bash
> # Task 1: Iptables restrictive
> sudo iptables -P INPUT DROP
> sudo iptables -P FORWARD DROP
> sudo iptables -P OUTPUT ACCEPT
> sudo iptables -A INPUT -p tcp --dport 22 -s $(curl -s ifconfig.me) -j ACCEPT
> sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
>
> # Task 2: Test ping (should fail)
> ping -c 2 google.com
>
> # Task 3: Allow HTTP outbound
> sudo iptables -I OUTPUT 1 -p tcp --dport 80 -j ACCEPT
>
> # Task 4: Flush rules
> sudo iptables -P INPUT ACCEPT
> sudo iptables -P FORWARD ACCEPT
> sudo iptables -F
> ```

## Summary Table: Iptables vs Nftables vs UFW

| Tool         | Complexity | Best For                       | Default Policy       |
| ------------ | ---------- | ------------------------------ | -------------------- |
| **Iptables** | High       | Legacy systems, complex rules  | Manual configuration |
| **Nftables** | Medium     | Modern systems, performance    | Manual configuration |
| **UFW**      | Low        | Ubuntu/Debian desktops/servers | `deny incoming`      |

| Operation     | Command                                         |
| ------------- | ----------------------------------------------- |
| List rules    | `iptables -L -v -n`                             |
| Allow SSH     | `iptables -A INPUT -p tcp --dport 22 -j ACCEPT` |
