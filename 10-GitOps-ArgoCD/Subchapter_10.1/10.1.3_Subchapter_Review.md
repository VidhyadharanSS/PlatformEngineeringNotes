## Cheatsheet: GitOps and ArgoCD

### GitOps Core Principles

| Principle | Description |
|-----------|-------------|
| **Declarative** | Entire system described in Git |
| **Versioned** | Git as single source of truth |
| **Pulled** | Operator pulls changes, not pushed |
| **Continuous reconciliation** | Constant drift correction |
| **Immutable** | Changes only through Git |

### Push vs Pull Comparison

| Aspect | Push CI/CD | GitOps (Pull) |
|--------|-----------|---------------|
| **Deployment method** | CI pushes to cluster | Operator pulls from Git |
| **Credentials location** | CI system | Cluster (operator) |
| **Drift handling** | None | Automatic correction |
| **Rollback** | Re-run pipeline | `git revert` |
| **Audit trail** | CI logs | Git history |
| **Multi-cluster** | Multiple pipelines | Single GitOps operator per cluster |

### ArgoCD Components

| Component | Purpose | Port |
|-----------|---------|------|
| **API Server** | gRPC/REST API, UI | 8080, 443 |
| **Repo Server** | Git clone, render manifests | 8081 |
| **Application Controller** | Reconcile loop | 8082 |
| **Redis** | Cache | 6379 |
| **Dex** | OIDC authentication | 5556, 5557 |

### ArgoCD Installation

```bash
# Create namespace
kubectl create namespace argocd

# Install
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

# Wait for pods
kubectl get pods -n argocd -w

# Get admin password
kubectl -n argocd get secret argocd-initial-admin-secret -o jsonpath="{.data.password}" | base64 -d
```

### ArgoCD CLI Commands

| Command | Purpose |
|---------|---------|
| `argocd login <server>` | Login to ArgoCD |
| `argocd cluster add <ctx>` | Add cluster |
| `argocd repo add <url>` | Add Git repo |
| `argocd app list` | List applications |
| `argocd app get <app>` | Show app details |
| `argocd app sync <app>` | Sync application |
| `argocd app rollback <app>` | Rollback to revision |

### Access Methods

| Method | Command | Use Case |
|--------|---------|----------|
| Port-forward | `kubectl port-forward svc/argocd-server -n argocd 8080:443` | Local testing |
| NodePort | `kubectl patch svc argocd-server -n argocd -p '{"spec": {"type": "NodePort"}}'` | Simple external access |
| Ingress | Ingress resource with TLS | Production |

## Comparison Tables

### Push vs Pull CI/CD

| Aspect | Push CI/CD | GitOps |
|--------|-----------|--------|
| **Security** | CI needs cluster credentials | Operator has limited RBAC |
| **Auditability** | CI logs (often rotated) | Git history (permanent) |
| **Drift** | Manual correction | Automatic self-healing |
| **Rollback speed** | Minutes (re-run pipeline) | Seconds (`git revert`) |
| **Multi-cluster** | Multiple pipelines | Single operator per cluster |

### ArgoCD vs Flux

| Feature | ArgoCD | Flux |
|---------|--------|------|
| **Web UI** | Yes | Limited |
| **Multi-cluster** | Yes (from one instance) | Yes (per cluster) |
| **SSO/RBAC** | Yes (Dex) | Limited |
| **Sync windows** | Yes | No |
| **Resource hooks** | Yes | No |
| **Learning curve** | Moderate | Low |

### GitOps Tools Comparison

| Tool | Best For | Key Features |
|------|----------|--------------|
| ArgoCD | Enterprise, complex environments | UI, multi-cluster, SSO, RBAC |
| Flux | Smaller teams, simplicity | Lightweight, integrated with Helm |
| Flagger | Progressive delivery | Canary deployments |
| Jenkins X | All-in-one CI/CD + GitOps | Full pipeline + GitOps |

## Interview Questions (Scenario-Based)

These questions assume only knowledge from Subchapter 10.1. Answers reference only concepts from 10.1.1 and 10.1.2.

### Question 1

**Scenario:** A team's CI/CD pipeline has cluster credentials stored in Jenkins. A security audit finds that Jenkins has admin access to the production cluster.

**Question:** How would GitOps improve this security posture? What changes would you recommend?

**Answer:**

**How GitOps improves security:**
- CI/CD pipeline no longer needs cluster credentials
- GitOps operator (ArgoCD) runs inside the cluster with limited RBAC
- Jenkins only needs Git push access
- No credentials stored in CI system

**Recommended changes:**

**Before (Push CI/CD):**
```yaml
# Jenkins had kubeconfig with admin access
kubectl apply -f deployment.yaml  # Admin credentials in Jenkins
```

**After (GitOps):**

1. **Remove cluster credentials from Jenkins**
2. **Install ArgoCD in cluster** (limited RBAC)
3. **Update CI pipeline** to only build and push images, then update Git:
```yaml
# Jenkins pipeline (GitHub Actions)
- name: Build and push image
  run: |
    docker build -t myapp:${{ github.sha }} .
    docker push myapp:${{ github.sha }}
    
- name: Update manifest
  run: |
    sed -i "s|myapp:.*|myapp:${{ github.sha }}|" k8s/deployment.yaml
    git commit -m "Update image to ${{ github.sha }}"
    git push
```

4. **ArgoCD** pulls the updated manifest and applies it

**Result:** CI/CD has no cluster access. Only ArgoCD (in-cluster) has RBAC permissions.

### Question 2

**Scenario:** A developer manually edits a deployment in the production cluster using `kubectl edit`. Later, ArgoCD reverts the change.

**Question:** Why did ArgoCD revert the change? How can developers make temporary changes for debugging?

**Answer:**

**Why ArgoCD reverted:** ArgoCD continuously reconciles the cluster state with Git. When a manual change is detected, ArgoCD sees drift and corrects it by reapplying the desired state from Git.

**This is a feature, not a bug** – it prevents configuration drift.

**Options for temporary debugging changes:**

**Option 1: Use `argocd app sync --dry-run` to preview changes**
```bash
argocd app get myapp
# Shows diff between Git and cluster
```

**Option 2: Override sync for specific resource (if absolutely necessary)**
```bash
# Pause auto-sync temporarily
argocd app set myapp --sync-policy none
# Make manual changes
kubectl edit deployment myapp
# Debug...
# Resume auto-sync
argocd app set myapp --sync-policy automated
argocd app sync myapp
```

**Option 3: Use ephemeral debug pod (better practice)**
```bash
# Instead of editing production deployment
kubectl run debug-pod --image=busybox -it --rm -- /bin/sh
# Debug from within the cluster
```

**Option 4: Create a separate debug deployment**
```yaml
# debug-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-debug
spec:
  replicas: 1
  template:
    spec:
      containers:
      - name: debug
        image: myapp:debug
```

### Question 3

**Scenario:** A team has 5 Kubernetes clusters (dev, staging, prod-us, prod-eu, prod-asia). They want to manage all from a single control plane.

**Question:** How would ArgoCD handle multi-cluster management? What needs to be configured?

**Answer:**

**ArgoCD multi-cluster architecture:**
- Single ArgoCD instance (control plane) runs in one cluster
- All 5 clusters are registered with ArgoCD
- ArgoCD can deploy to all clusters from one UI/CLI

**Configuration steps:**

**Step 1: Install ArgoCD in management cluster**
```bash
kubectl create namespace argocd
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml
```

**Step 2: Register each cluster**
```bash
# Add dev cluster
argocd cluster add dev-context --name dev

# Add staging cluster
argocd cluster add staging-context --name staging

# Add production clusters
argocd cluster add prod-us-context --name prod-us
argocd cluster add prod-eu-context --name prod-eu
argocd cluster add prod-asia-context --name prod-asia
```

**Step 3: Verify clusters**
```bash
argocd cluster list
# SERVER                          NAME        VERSION
# https://dev-k8s:6443            dev         v1.29
# https://staging-k8s:6443        staging     v1.29
# https://prod-us-k8s:6443        prod-us     v1.29
# https://prod-eu-k8s:6443        prod-eu     v1.29
# https://prod-asia-k8s:6443      prod-asia   v1.29
```

**Step 4: Deploy applications to specific clusters**
```yaml
# app-dev.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp-dev
spec:
  destination:
    server: https://dev-k8s:6443  # dev cluster
    namespace: myapp
  source:
    repoURL: https://github.com/myorg/config
    path: overlays/dev
```

### Question 4

**Scenario:** A security breach occurs. The team needs to know exactly who changed a deployment and when.

**Question:** How does GitOps provide better auditability than traditional CI/CD?

**Answer:**

**GitOps audit trail:**
- Every change is a Git commit with:
  - Author (who)
  - Timestamp (when)
  - Commit message (why)
  - Full diff (what changed)

**Traditional CI/CD audit trail problems:**
- CI logs may be rotated/deleted
- CI logs often don't capture who triggered the deployment
- Multiple people may use same CI service account
- No built-in diff between versions

**GitOps audit commands:**

```bash
# See who changed what
git log --oneline k8s/deployment.yaml
# a1b2c3d (alice) Update image to v2.1.0
# e4f5g6h (bob)   Increase replicas to 5
# i7j8k9l (alice) Add health check

# See full diff
git show a1b2c3d
# diff --git a/k8s/deployment.yaml b/k8s/deployment.yaml
# - image: myapp:v2.0.0
# + image: myapp:v2.1.0

# See ArgoCD sync history
argocd app history myapp
# ID  DATE          REVISION
# 3   2024-01-15    a1b2c3d (alice) Update image
# 2   2024-01-14    e4f5g6h (bob)   Scale replicas
# 1   2024-01-13    i7j8k9l (alice) Initial deploy
```

**Compliance benefits:**
- SOC2: Audit trail of all changes
- HIPAA: Know who accessed/modified infrastructure
- PCI: Track all deployment changes
- GDPR: Prove data handling changes

### Question 5

**Scenario:** A team wants to adopt GitOps but is concerned about the learning curve and operational overhead.

**Question:** What are the minimal requirements to start with GitOps? What's the migration path?

**Answer:**

**Minimal requirements:**

| Requirement | Description |
|-------------|-------------|
| **Git repository** | One repo for Kubernetes manifests |
| **Kubernetes cluster** | Target cluster (can start with dev) |
| **ArgoCD** | Single installation |
| **One application** | Start with non-critical app |

**Migration path (gradual adoption):**

**Phase 1: Shadow (no risk)**
```yaml
# Deploy alongside existing CI/CD, but don't sync
spec:
  syncPolicy:
    automated: null  # manual sync only
```
- ArgoCD installed but manual sync only
- Compare Git vs cluster state (no automatic changes)
- Team learns ArgoCD UI and CLI

**Phase 2: Single non-critical app**
```yaml
# Migrate one app to GitOps
spec:
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
```
- First app fully managed by ArgoCD
- CI pipeline stops deploying this app
- Monitor for issues

**Phase 3: Expand to more apps**
- Add more applications gradually
- Set up RBAC for different teams
- Configure sync windows for prod

**Phase 4: Full GitOps**
- All apps managed by ArgoCD
- CI only builds images and updates Git
- Direct kubectl access disabled for production

**Estimated timeline:** 2-4 weeks for full migration

**Operational overhead:**
- New component to monitor (ArgoCD itself)
- Need Git discipline (no direct kubectl)
- Learning curve for developers (2-3 days)

**ROI:**
- Faster rollbacks (seconds vs minutes)
- No more drift
- Better audit trail
- Reduced CI complexity

## Bridge Concepts (Not in Notes but Added for Clarity)

| Concept | Explanation |
|---------|-------------|
| **Dex** | OpenID Connect (OIDC) provider used by ArgoCD for authentication. Supports GitHub, Google, LDAP, etc. |
| **Application Controller** | Core ArgoCD component that runs the reconcile loop (checks Git vs cluster every 3 minutes by default). |
| **Repo Server** | Clones Git repositories and renders Helm/Kustomize manifests. Caches results for performance. |
| **Sync window** | Time-based restrictions for sync operations (e.g., "no deployments to prod between 6 PM and 6 AM"). |
| **Self-heal** | ArgoCD feature that automatically corrects drift when cluster state differs from Git. |
