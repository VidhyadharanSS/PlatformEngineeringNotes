## Cheatsheet: Container Internals and Docker Basics

### VMs vs Containers

| Aspect            | Virtual Machines    | Containers             |
| ----------------- | ------------------- | ---------------------- |
| Isolation         | Full OS (strong)    | Process-level (faster) |
| Boot time         | Minutes             | Milliseconds           |
| Size              | GBs                 | MBs                    |
| Kernel            | Each VM has its own | Shared with host       |
| Resource overhead | High                | Low                    |

### Linux Namespaces (What Containers See)

| Namespace | Isolates                    | Why It Matters                      |
| --------- | --------------------------- | ----------------------------------- |
| **PID**   | Process IDs                 | Container processes can't see host  |
| **NET**   | Network interfaces, ports   | Each container gets its own IP      |
| **UTS**   | Hostname                    | Container can have its own hostname |
| **MNT**   | Filesystem mounts           | Container has its own rootfs        |
| **IPC**   | Inter-process communication | Prevents container interference     |
| **USER**  | User/group IDs              | Map container root to non-root host |

### Linux Cgroups (What Containers Use)

| Resource     | Docker Flag     | Example                          |
| ------------ | --------------- | -------------------------------- |
| CPU cores    | `--cpus`        | `--cpus=1.5`                     |
| CPU shares   | `--cpu-shares`  | `--cpu-shares=512`               |
| Memory limit | `--memory`      | `--memory=512m`                  |
| Memory+swap  | `--memory-swap` | `--memory=512m --memory-swap=1g` |
| PIDs limit   | `--pids-limit`  | `--pids-limit=100`               |

### Docker Architecture Components

| Component     | Role                           |
| ------------- | ------------------------------ |
| Docker Client | CLI command (`docker`)         |
| Docker Daemon | Background service (`dockerd`) |
| containerd    | Container runtime              |
| runc          | Low-level container creator    |
| Registry      | Image storage (Docker Hub)     |

## Interview Questions (Scenario-Based)

These questions assume only knowledge from Subchapter 4.1. Answers reference only concepts from 4.1.1 and 4.1.2.

### Question 1

**Scenario:** A developer asks: "Why can't I see my host's running processes when I run `ps aux` inside a container?"

**Question:** Explain which Linux kernel feature provides this isolation and how it works. What would happen if this isolation didn't exist?

**Answer:**

**The feature:** **PID namespace** – one of the Linux namespaces that Docker uses.

**How it works:**

* Each container gets its own PID namespace

* Inside the namespace, processes are assigned PIDs starting from 1

* The host's processes are in a different PID namespace (the root namespace)

* The container's PID 1 maps to a high-numbered PID on the host (e.g., 12345), but the container can't see that mapping

**Without PID namespace:**

* Processes inside the container could see all host processes

* Container could accidentally (or maliciously) kill host processes (e.g., `kill -9 1` would kill systemd)

* Process IDs would conflict (two processes claiming PID 1)

* No isolation between containers

**Demonstration:**

```bash
# On host, see many processes
ps aux | wc -l
# 150+

# Inside container, see only container processes
docker run alpine ps aux
# PID 1: ps command itself
# No host processes visible

# Show namespace differences
docker run --pid=host alpine ps aux
# Now sees host processes (sharing host PID namespace)
```

**Real-world impact:**

* Security: Prevents container escape via process manipulation

* Stability: Container can't crash host by killing critical processes

* Isolation: Each container's "PID 1" is its own init process

### Question 2

**Scenario:** A containerized Java application consumes all available CPU on the host, affecting other containers. The host has 8 CPU cores.

**Question:** How would you limit this container to use at most 2 CPU cores? What cgroup feature would you use? Show the Docker command.

**Answer:**

**Solution:** Use the `--cpus` flag to limit CPU cores.

```bash
# Limit to 2 CPU cores
docker run --cpus=2.0 java-app

# Or limit to 1.5 cores
docker run --cpus=1.5 java-app
```

**Cgroup feature:** `cpu` cgroup controller, specifically the CFS (Completely Fair Scheduler) quota and period.

**How it works under the hood:**

* `--cpus=2.0` sets `cpu.cfs_quota_us = 200000` and `cpu.cfs_period_us = 100000`

* Container gets 200,000 microseconds of CPU time per 100,000 microsecond period (2 cores worth)

**Alternative approaches:**

```bash
# Using CPU shares (relative weight)
docker run --cpu-shares=512 java-app
# Default is 1024, so this gets half the CPU of default containers

# Using CPU quota directly (more precise)
docker run --cpu-quota=50000 --cpu-period=100000 java-app
# 50% of one core (0.5 CPU)
```

**Verification:**

```bash
# Check container's cgroup settings on host
docker inspect java-app | grep -i pid
# Get container's PID, then:
cat /proc/<PID>/cgroup

# Or use docker stats
docker stats java-app
```

**Best practices for production:**

* Always set CPU and memory limits for production containers

* Use `--cpus` for simple core limits

* Use `--cpu-shares` for relative priority between containers

* Monitor with `docker stats` or Prometheus

### Question 3

**Scenario:** A junior engineer runs `docker run ubuntu` and sees nothing happen. They expected an interactive shell.

**Question:** Why didn't they get a shell? What's the difference between `docker run ubuntu`, `docker run -it ubuntu`, and `docker run -it ubuntu bash`?

**Answer:**

**Why** **`docker run ubuntu`** **shows nothing:**

* The default command for the Ubuntu image is `bash` (or `/bin/bash`)

* But without `-it` (interactive + TTY), bash has no input or output

* The container starts, bash runs with no terminal, then exits immediately

**Differences explained:**

| Command                      | Interactive | TTY | Command           | Result                      |
| ---------------------------- | ----------- | --- | ----------------- | --------------------------- |
| `docker run ubuntu`          | No          | No  | `bash` (default)  | Starts, exits immediately   |
| `docker run -it ubuntu`      | Yes         | Yes | `bash` (default)  | Interactive bash shell      |
| `docker run -it ubuntu bash` | Yes         | Yes | `bash` (explicit) | Same as above               |
| `docker run ubuntu bash`     | No          | No  | `bash`            | Starts, exits (no terminal) |

**Why** **`-it`** **is needed:**

* `-i` (interactive): Keep STDIN open (allows you to type)

* `-t` (TTY): Allocate a pseudo-terminal (provides shell features like prompt, line editing)

**Demonstration:**

```bash
# No output, container exits immediately
docker run ubuntu
docker ps -a | grep ubuntu
# STATUS: Exited (0)

# Interactive shell – works
docker run -it ubuntu
# root@abc123:/# _ (prompt appears)

# Run a command and exit
docker run ubuntu echo "hello"
# hello (container exits after command)

# Run without TTY (no prompt, but command works)
docker run -i ubuntu bash
# (no prompt, but commands still work)
ls
exit
```

**When to use each:**

* `docker run -it IMAGE` – Interactive debugging, shell access

* `docker run IMAGE COMMAND` – Run a single command and exit

* `docker run -d IMAGE` – Run in background as a service

### Question 4

**Scenario:** Two containers both need to listen on port 80. On a host without Docker, this would fail with "Address already in use".

**Question:** How can two containers both use port 80 simultaneously? Explain which Linux kernel feature enables this.

**Answer:**

**The feature:** **NET namespace** – each container gets its own network namespace with its own port space.

**How it works:**

* Each container has its own isolated network stack

* Port 80 inside Container A is completely separate from port 80 inside Container B

* The host maps each container's port 80 to different host ports (or uses different IPs)

**Implementation:**

```bash
# Map Container A's port 80 to host port 8080
docker run -d --name web1 -p 8080:80 nginx

# Map Container B's port 80 to host port 8081
docker run -d --name web2 -p 8081:80 nginx

# Both containers listen on port 80 inside their own namespaces
```

**Alternative: Use different host IPs**

```bash
# Bind to different host IP addresses
docker run -d --name web1 -p 192.168.1.10:80:80 nginx
docker run -d --name web2 -p 192.168.1.11:80:80 nginx
```

**How Docker implements this:**

1. Creates veth pair (virtual Ethernet) for each container
2. One end in container's net namespace (appears as `eth0`)
3. Other end bridged to host network (appears as `veth...`)
4. Port forwarding uses `iptables` NAT rules (from Module 2)

**Verification:**

```bash
# See network namespaces on host
ls -la /var/run/docker/netns/

# See iptables NAT rules
sudo iptables -t nat -L -n | grep -A 5 "DOCKER"

# Inspect container's network
docker inspect web1 | grep -A 10 "NetworkSettings"
```

**Why this is powerful:**

* No port conflicts between containers

* Same application can run multiple instances on same host

* Foundation for microservices architecture

### Question 5

**Scenario:** A production server runs out of disk space. Investigation shows `/var/lib/docker` is 80GB, with many stopped containers and unused images.

**Question:** How would you safely clean up Docker disk usage? What commands would you run? What should you check before cleaning?

**Answer:**

**Step 1: Check disk usage**

```bash
# See Docker disk usage breakdown
docker system df
# TYPE            TOTAL     ACTIVE    SIZE      RECLAIMABLE
# Images          15        5         30GB      20GB (66%)
# Containers      20        3         10GB      8GB (80%)
# Volumes         5         2         40GB      30GB (75%)

# Detailed view
docker system df -v
```

**Step 2: Identify what's safe to remove**

```bash
# List stopped containers
docker ps -a --filter "status=exited"

# List unused images
docker images --filter "dangling=true"

# List unused volumes
docker volume ls --filter "dangling=true"
```

**Step 3: Clean up safely**

```bash
# Remove all stopped containers
docker container prune

# Remove unused images (dangling)
docker image prune

# Remove unused images including those without containers
docker image prune -a

# Remove unused volumes (DANGEROUS – may delete data!)
docker volume prune

# Remove unused networks
docker network prune

# Remove everything unused (one command)
docker system prune -a --volumes
# WARNING: This removes all stopped containers, unused images, unused volumes
```

**Step 4: More targeted cleanup**

```bash
# Remove containers older than 7 days
docker container prune --filter "until=168h"

# Remove images without tags
docker image prune --filter "dangling=true"

# Remove specific old image
docker rmi ubuntu:18.04

# Remove all stopped containers and their volumes
docker rm $(docker ps -a -q -f status=exited) -v
```

**What to check before cleaning:**

| Check              | Command                                                 | Why                                                 |
| ------------------ | ------------------------------------------------------- | --------------------------------------------------- |
| Running containers | `docker ps`                                             | Don't stop production containers                    |
| Active volumes     | `docker volume ls`                                      | Don't delete volumes with live data                 |
| Container age      | `docker ps -a --format "table {{.Names}}\t{{.Status}}"` | Some stopped containers may be needed for debugging |
| Image usage        | `docker image ls`                                       | Some images are base layers for others              |

**Preventive measures:**

```bash
# Add to daemon.json for automatic log rotation
{
  "log-opts": {
    "max-size": "10m",
    "max-file": "3"
  }
}

# Set up cron job for periodic cleanup
0 2 * * 0 docker system prune -f --filter "until=168h"
```
