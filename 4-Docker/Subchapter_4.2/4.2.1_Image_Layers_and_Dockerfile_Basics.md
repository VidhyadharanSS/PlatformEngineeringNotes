# 4.2.1 Image Layers and Dockerfile Basics: Building Custom Containers

#### Why Dockerfiles Matter

Images are immutable templates for containers. While you can `docker commit` a running container into an image, the professional approach is a **Dockerfile** – a declarative recipe for building images. Dockerfiles are:

* **Versionable** – Store in Git alongside your code

* **Reproducible** – Same Dockerfile builds same image

* **Automated** – CI/CD pipelines can build and push images

This note covers image layering and Dockerfile instructions. Note 4.2.2 covers image optimization and multi-stage builds; note 4.2.3 is the subchapter review.

***

## Part 1: Union Filesystems and Image Layers

### How Docker Images Are Stored

Docker images are not single files – they are **layers** stacked on top of each other using a union filesystem (overlay2 on modern Linux).

```mermaid
graph TD
    A[Container Layer<br/>Read-Write] --> B[Image Layer 3<br/>Add app code]
    B --> C[Image Layer 2<br/>Install dependencies]
    C --> D[Image Layer 1<br/>Base OS (Ubuntu)]
    
    style A fill:#f99,stroke:#333,stroke-dasharray: 5 5
    style B fill:#9f9,stroke:#333
    style C fill:#9cf,stroke:#333
    style D fill:#ff9,stroke:#333
```

### Layer Characteristics

| Layer Type              | Read/Write | Persistence                         | Created By                  |
| ----------------------- | ---------- | ----------------------------------- | --------------------------- |
| **Base image layers**   | Read-only  | Permanent                           | Dockerfile `FROM`           |
| **Intermediate layers** | Read-only  | Permanent                           | Each Dockerfile instruction |
| **Container layer**     | Read-write | Temporary (until container removed) | `docker run`                |

### Why Layers Matter for Performance

```bash
# Build an image – layers are cached
docker build -t myapp .

# Rebuild after changing only the last line – cached layers reused!
docker build -t myapp .
# Using cache for: FROM ubuntu
# Using cache for: RUN apt update
# Using cache for: RUN apt install -y python3
# Running: COPY app.py /app/  (not cached – rebuilt)
```

### Viewing Image Layers

```bash
# See image layers
docker history nginx:latest
# IMAGE          CREATED       CREATED BY                                      SIZE
# abc123...      2 weeks ago   /bin/sh -c #(nop)  CMD ["nginx" "-g" "daemon…   0B
# def456...      2 weeks ago   /bin/sh -c #(nop)  STOPSIGNAL SIGQUIT           0B
# 789abc...      2 weeks ago   /bin/sh -c #(nop)  EXPOSE 80                    0B

# More detailed (full command)
docker history --no-trunc nginx:latest

# Show layer sizes (find largest layers)
docker history --human nginx:latest
```

### Union Filesystem (overlay2)

```bash
# Check Docker's storage driver
docker info | grep "Storage Driver"
# Storage Driver: overlay2

# Layer locations (on host)
ls /var/lib/docker/overlay2/
# Each subdirectory is a layer
```

## Part 2: Dockerfile Instructions – The Building Blocks

### Complete Dockerfile Example

```dockerfile
# Stage 1: Build stage (multi-stage – covered in 4.2.2)
FROM golang:1.21-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN go build -o myapp .

# Stage 2: Runtime stage
FROM alpine:3.18
# Metadata
LABEL maintainer="platform@example.com"
LABEL version="1.0"
LABEL description="My Go application"

# Set environment variables
ENV APP_PORT=8080
ENV APP_ENV=production

# Set working directory
WORKDIR /app

# Copy from builder stage
COPY --from=builder /app/myapp .

# Copy configuration files
COPY --chown=nobody:nogroup config.yaml ./config.yaml

# Expose port (documentation only)
EXPOSE 8080

# Run as non-root user (security)
USER nobody

# Health check
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:8080/health || exit 1

# Default command
CMD ["./myapp"]
```

### Dockerfile Instruction Reference

| Instruction   | Purpose                           | Example                                     |
| ------------- | --------------------------------- | ------------------------------------------- |
| `FROM`        | Base image                        | `FROM ubuntu:22.04`                         |
| `RUN`         | Execute command (build time)      | `RUN apt update && apt install -y curl`     |
| `COPY`        | Copy files from host              | `COPY app.py /app/`                         |
| `ADD`         | Copy + auto-extract archives      | `ADD app.tar.gz /app/`                      |
| `WORKDIR`     | Set working directory             | `WORKDIR /app`                              |
| `ENV`         | Set environment variable          | `ENV NODE_ENV=production`                   |
| `ARG`         | Build-time variable               | `ARG VERSION=latest`                        |
| `EXPOSE`      | Document port                     | `EXPOSE 8080`                               |
| `CMD`         | Default command (runtime)         | `CMD ["node", "server.js"]`                 |
| `ENTRYPOINT`  | Main command (harder to override) | `ENTRYPOINT ["/app/entrypoint.sh"]`         |
| `USER`        | Run as non-root                   | `USER appuser`                              |
| `LABEL`       | Metadata                          | `LABEL version="1.0"`                       |
| `VOLUME`      | Create mount point                | `VOLUME /data`                              |
| `HEALTHCHECK` | Check container health            | `HEALTHCHECK CMD curl -f http://localhost/` |
| `SHELL`       | Change default shell              | `SHELL ["/bin/bash", "-c"]`                 |

## Part 3: Shell Form vs Exec Form — Signal Handling (Critical)

Every `RUN`, `CMD`, and `ENTRYPOINT` instruction can use either **shell form** or **exec form**. The choice directly impacts whether your container can shut down gracefully.

| Form | Syntax | PID 1 Process | Receives SIGTERM? |
|------|--------|---------------|-------------------|
| **Exec form** | `CMD ["node", "server.js"]` | `node` is PID 1 | ✅ Yes |
| **Shell form** | `CMD node server.js` | `/bin/sh -c` is PID 1 | ❌ No — `sh` doesn't forward signals |

### Why This Matters for Graceful Shutdown

```dockerfile
# BAD — Shell form: sh is PID 1, node never receives SIGTERM
CMD npm start
# docker stop sends SIGTERM to /bin/sh -c "npm start"
# sh does NOT forward the signal to node
# After 10s timeout, Docker sends SIGKILL (ungraceful)

# GOOD — Exec form: node is PID 1, receives SIGTERM directly
CMD ["node", "server.js"]
# docker stop sends SIGTERM directly to node process
# Node handles it gracefully (closes connections, flushes logs)
```

### Demonstration

```bash
# Shell form — container takes 10s to stop (SIGKILL after timeout)
docker run -d --name shell-test alpine sh -c "sleep infinity"
time docker stop shell-test
# real    0m10.3s  (waited for timeout, then SIGKILL)

# Exec form — container stops instantly
docker run -d --name exec-test alpine sleep infinity
time docker stop exec-test
# real    0m0.2s  (sleep received SIGTERM, exited immediately)
```

### When to Use Each Form

| Use Case | Recommended Form | Why |
|----------|-----------------|-----|
| `CMD` / `ENTRYPOINT` | **Exec form** always | Signal forwarding for graceful shutdown |
| `RUN` (build-time) | Shell form is fine | No signals during build; shell features useful |
| Variable expansion needed | Shell form + `SHELL` | `$VAR` expansion requires shell |

```dockerfile
# If you NEED variable expansion in CMD, use exec form with sh explicitly:
CMD ["sh", "-c", "node server.js --port=$PORT"]
# Or better — use ENTRYPOINT script that handles signals:
COPY entrypoint.sh /entrypoint.sh
ENTRYPOINT ["/entrypoint.sh"]
```

### ONBUILD — Triggers for Child Images

`ONBUILD` defers an instruction to execute when another Dockerfile uses this image as a base (`FROM`).

```dockerfile
# Base image (team-standard Python image)
FROM python:3.11-slim
ONBUILD COPY requirements.txt /app/
ONBUILD RUN pip install --no-cache-dir -r /app/requirements.txt
ONBUILD COPY . /app
WORKDIR /app

# Child image (developer's Dockerfile — dead simple)
FROM team/python-base:latest
CMD ["python", "app.py"]
# ONBUILD instructions execute automatically here!
```

| Use Case | Example |
|----------|---------|
| Standardized build patterns | Team base images with `ONBUILD COPY` + `ONBUILD RUN` |
| Language-specific base images | Official `node:onbuild` (deprecated but pattern is useful) |

> **Caution:** `ONBUILD` hides build steps from the child Dockerfile. Use sparingly and document clearly.

## Part 4: Instruction Deep Dive

### FROM – The Base Image

```dockerfile
# Basic
FROM ubuntu:22.04

# Alpine (small, musl libc)
FROM alpine:3.18

# Distroless (no shell, minimal)
FROM gcr.io/distroless/static-debian11

# Scratch (completely empty)
FROM scratch
```

### SHELL – Changing Default Shell

By default, Docker uses `/bin/sh -c` for shell form commands. Use `SHELL` to change it:

```dockerfile
# Change to bash (needed for bash-specific syntax)
SHELL ["/bin/bash", "-c"]
RUN source ~/.bashrc && echo $PATH

# PowerShell on Windows
SHELL ["powershell", "-Command"]
RUN Get-Process
```

### RUN – Executing Commands

```dockerfile
# Shell form (uses /bin/sh -c)
RUN apt update && apt install -y curl

# Exec form (no shell – more secure)
RUN ["/usr/bin/apt", "update"]

# Best practice: combine commands (reduce layers)
RUN apt update && \
    apt install -y curl wget && \
    apt clean && \
    rm -rf /var/lib/apt/lists/*
```

### COPY vs ADD

| Feature          | COPY         | ADD                                   |
| ---------------- | ------------ | ------------------------------------- |
| Copy local files | Yes          | Yes                                   |
| Auto-extract tar | No           | Yes                                   |
| Remote URLs      | No           | Yes (avoid – use `curl` instead)      |
| Recommendation   | **Use COPY** | Use ADD only for local tar extraction |

```dockerfile
# Good – COPY for local files
COPY app.py /app/

# Acceptable – ADD for tar extraction
ADD app.tar.gz /app/

# Avoid – ADD for URLs (use RUN curl instead)
# ADD https://example.com/file.tar.gz /tmp/   # Not recommended
```

### ENV vs ARG

| Feature                 | ENV                                   | ARG                 |
| ----------------------- | ------------------------------------- | ------------------- |
| Available at build time | Yes                                   | Yes                 |
| Available at runtime    | Yes                                   | No                  |
| Override at build       | No (but can be overridden at runtime) | Yes (`--build-arg`) |
| Persists in final image | Yes                                   | No                  |

```dockerfile
# Build-time only (ARG)
ARG VERSION=1.0
RUN echo "Building version ${VERSION}"

# Runtime environment variable (ENV)
ENV DATABASE_URL=postgres://localhost:5432/mydb
ENV NODE_ENV=production

# Override ARG at build
# docker build --build-arg VERSION=2.0 -t myapp .
```

### CMD vs ENTRYPOINT

| Feature             | CMD                                | ENTRYPOINT                   |
| ------------------- | ---------------------------------- | ---------------------------- |
| Default command     | Yes                                | Yes                          |
| Override at runtime | Yes (`docker run myapp /bin/bash`) | No (must use `--entrypoint`) |
| Combined use        | Provides defaults to ENTRYPOINT    | Main executable              |

```dockerfile
# ENTRYPOINT + CMD pattern
ENTRYPOINT ["/app/entrypoint.sh"]
CMD ["--help"]

# When container runs:
# /app/entrypoint.sh --help
```

**Real-world example:**

```dockerfile
# Nginx image pattern
ENTRYPOINT ["/docker-entrypoint.sh"]
CMD ["nginx", "-g", "daemon off;"]

# Users can override CMD:
# docker run nginx nginx -g "daemon off;" -c /custom/nginx.conf
```

## Part 5: Layer Caching – The Key to Fast Builds

### How Caching Works

Docker caches each instruction's result. If the instruction hasn't changed, Docker reuses the cache.

```dockerfile
# 1. FROM – cached if same base image
FROM node:18-alpine

# 2. WORKDIR – cached (no change)
WORKDIR /app

# 3. COPY package.json – CACHE INVALIDATED if package.json changes!
COPY package.json package-lock.json ./

# 4. RUN npm install – will rerun if previous layer changed
RUN npm ci

# 5. COPY source code – invalidates cache
COPY . .

# 6. CMD – cached (no effect)
CMD ["npm", "start"]
```

### Optimizing for Cache

**Bad (cache invalidated on any source change):**

```dockerfile
FROM node:18-alpine
WORKDIR /app
COPY . .
RUN npm install
CMD ["npm", "start"]
```

**Good (dependencies cached separately):**

```dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
CMD ["npm", "start"]
```

### Viewing Build Cache

```bash
# Build with cache (default)
docker build -t myapp .

# Build without cache
docker build --no-cache -t myapp .

# Show build cache usage
docker system df
```

## Part 6: Writing Your First Dockerfile

### Example: Python Web Application

```dockerfile
# Dockerfile
FROM python:3.11-slim

# Set environment variables
ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1

# Set work directory
WORKDIR /app

# Install dependencies
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Copy project
COPY . .

# Create non-root user
RUN addgroup --system app && adduser --system --group app
USER app

# Expose port
EXPOSE 8000

# Run the application
CMD ["gunicorn", "--bind", "0.0.0.0:8000", "myapp.wsgi"]
```

### Build and Run

```bash
# Build image
docker build -t myapp:latest .

# Build with specific tag
docker build -t myapp:1.0.0 -t myapp:latest .

# Run container
docker run -d -p 8000:8000 --name myapp myapp:latest

# Test
curl http://localhost:8000
```

## Part 7: Dockerfile Best Practices

### 1. Order Instructions by Frequency of Change

```dockerfile
# Good: Least frequent changes first
FROM python:3.11-slim
ENV PYTHONUNBUFFERED=1
RUN pip install --upgrade pip
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
CMD ["python", "app.py"]
```

### 2. Combine RUN Commands (Reduce Layers)

```dockerfile
# Bad – 3 layers
RUN apt update
RUN apt install -y curl
RUN apt clean

# Good – 1 layer
RUN apt update && \
    apt install -y curl && \
    apt clean && \
    rm -rf /var/lib/apt/lists/*
```

### 3. Use Specific Tags (Not `latest`)

```dockerfile
# Bad
FROM ubuntu:latest

# Good
FROM ubuntu:22.04
```

### 4. Clean Up in Same Layer

```dockerfile
RUN apt update && \
    apt install -y build-essential && \
    make install && \
    apt remove -y build-essential && \
    apt autoremove -y && \
    rm -rf /var/lib/apt/lists/*
```

### 5. Use .dockerignore

```dockerfile
# .dockerignore
.git
__pycache__
*.pyc
.env
node_modules
Dockerfile
README.md
.gitignore
```

## Quick Task: Create a Dockerfile

*Write a Dockerfile for a simple static website.*

1. Create a directory `static-site` with an `index.html` file.
2. Write a Dockerfile using `nginx:alpine` as base.
3. Copy `index.html` to `/usr/share/nginx/html/`.
4. Expose port 80.
5. Build and run the container.

> **Ready Solution:**
>
> ```bash
> # Task 1: Create files
> mkdir static-site
> cd static-site
> echo "<h1>Hello from Docker!</h1>" > index.html
>
> # Task 2: Dockerfile
> cat > Dockerfile << 'EOF'
> FROM nginx:alpine
> COPY index.html /usr/share/nginx/html/
> EXPOSE 80
> CMD ["nginx", "-g", "daemon off;"]
> EOF
>
> # Task 3: Build
> docker build -t static-site .
>
> # Task 4: Run
> docker run -d -p 8080:80 --name site static-site
>
> # Test
> curl http://localhost:8080
> # <h1>Hello from Docker!</h1>
>
> # Clean up
> docker stop site && docker rm site
> ```

## Summary Table: Dockerfile Instructions

| Instruction   | Purpose            | When to Use                          |
| ------------- | ------------------ | ------------------------------------ |
| `FROM`        | Base image         | Every Dockerfile (first instruction) |
| `RUN`         | Execute commands   | Installing packages, building code   |
| `COPY`        | Copy files         | Adding application code, configs     |
| `ADD`         | Copy + extract     | Extracting local tar files           |
| `WORKDIR`     | Set working dir    | Before COPY/CMD (avoid `cd`)         |
| `ENV`         | Runtime env vars   | Configuration, paths                 |
| `ARG`         | Build-time vars    | Versions, build flags                |
| `EXPOSE`      | Document ports     | Informational (doesn't publish)      |
| `CMD`         | Default command    | Main process                         |
| `ENTRYPOINT`  | Main executable    | Wrapper scripts                      |
| `USER`        | Non-root user      | Security                             |
| `LABEL`       | Metadata           | Version, maintainer                  |
| `HEALTHCHECK` | Health probe       | Production readiness                 |
| `VOLUME`      | Persistent storage | Database, uploads                    |

### Layer Caching Rules

| Instruction          | Cache Invalidated When |
| -------------------- | ---------------------- |
| `FROM`               | Different base image   |
| `RUN`                | Command string changes |
| `COPY` / `ADD`       | File content changes   |
| `ENV`                | Value changes          |
| `ARG`                | Value changes          |
| `WORKDIR`            | Path changes           |
| `CMD` / `ENTRYPOINT` | Command changes        |
