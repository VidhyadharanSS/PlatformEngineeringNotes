# 4.4.1 Volumes, Bind Mounts, and tmpfs: Persistent and Temporary Storage

#### Why Storage Matters in Containers

By default, containers are **ephemeral** – when a container is removed, all data inside it is lost. This is a problem for:

* Databases (PostgreSQL, MySQL data)

* User uploads

* Configuration files

* Logs that must persist

Docker provides three types of storage for data that outlives containers:

* **Volumes** – Managed by Docker, stored in `/var/lib/docker/volumes/`

* **Bind mounts** – Map any host directory into container

* **tmpfs** – In-memory storage (lost on container stop)

This note covers all three. Note 4.4.2 covers Docker Compose; note 4.4.3 is the subchapter review.

### Comparison

| Feature               | Volume                        | Bind Mount                | tmpfs                    |
| --------------------- | ----------------------------- | ------------------------- | ------------------------ |
| **Managed by Docker** | Yes                           | No                        | Yes                      |

| **Persistence**       | Until volume deleted          | Until files deleted       | Until container stops    |
| **Use case**          | Database, persistent app data | Config files, source code | Secrets, temporary cache |

## Part 2: Volumes – Docker-Managed Persistent Storage

### Creating and Managing Volumes

```bash
# Create a volume
docker volume create mydata

# List volumes
docker volume ls
# DRIVER    VOLUME NAME
# local     mydata

# Inspect volume
docker volume inspect mydata
# [
#   {
#     "CreatedAt": "2024-01-16T10:00:00Z",
#     "Driver": "local",
#     "Labels": {},
#     "Mountpoint": "/var/lib/docker/volumes/mydata/_data",
#     "Name": "mydata",
#     "Options": {},
#     "Scope": "local"
#   }
# ]

# Remove volume
docker volume rm mydata

# Remove unused volumes
docker volume prune
```

### Using Volumes with Containers

```bash
# Method 1: -v (older syntax)
docker run -d --name db -v mydata:/var/lib/postgresql/data postgres

# Method 2: --mount (preferred, more explicit)
docker run -d --name db \
  --mount source=mydata,target=/var/lib/postgresql/data \
  postgres

# Read-only volume
docker run -d --name app \
  --mount source=config,target=/app/config,readonly \
  myapp

# Anonymous volume (Docker generates name)
docker run -d --name db -v /var/lib/postgresql/data postgres
```

### Volume Drivers

Docker supports volume drivers for various storage backends:

| Driver    | Use Case                    |
| --------- | --------------------------- |
| `local`   | Default, host filesystem    |
| `nfs`     | Network File System         |
| `cloud`   | AWS EBS, Azure Disk, GCE PD |
| `vsphere` | VMware vSAN                 |

```bash
# Create NFS volume
docker volume create \
  --driver local \
  --opt type=nfs \
  --opt o=addr=192.168.1.100,rw \
  --opt device=:/exports/data \
  nfs-data
```

### Backup and Restore Volumes

```bash
# Backup volume to tar file
docker run --rm -v mydata:/source -v $(pwd):/backup alpine \
  tar czf /backup/mydata-backup.tar.gz -C /source .

# Restore volume from backup
docker run --rm -v mydata:/target -v $(pwd):/backup alpine \
  tar xzf /backup/mydata-backup.tar.gz -C /target

# Copy between volumes
docker run --rm -v source:/from -v dest:/to alpine \
  sh -c "cp -a /from/. /to/"
```

## Part 3: Bind Mounts – Host Filesystem Access

### Basic Bind Mounts

Bind mounts map a host directory directly into the container.

```bash
# Method 1: -v (absolute path required)
docker run -d --name app -v /host/path:/container/path nginx

# Method 2: --mount (preferred)
docker run -d --name app \
  --mount type=bind,source=/host/path,target=/container/path \
  nginx

# Read-only bind mount
docker run -d --name app \
  --mount type=bind,source=/host/config,target=/app/config,readonly \
  myapp
```

### Common Bind Mount Use Cases

**1. Development (live code reload):**

```bash
# Mount source code for hot reload
docker run -d --name dev \
  -v $(pwd):/app \
  -v /app/node_modules \  # Anonymous volume for node_modules
  node npm run dev
```

**2. Configuration files:**

```bash
# Mount custom nginx config
docker run -d --name nginx \
  -v /etc/nginx/custom.conf:/etc/nginx/conf.d/default.conf:ro \
  nginx
```

**3. Logs access:**

```bash
# Mount log directory to host
docker run -d --name app \
  -v /var/log/myapp:/var/log/app \
  myapp
```

### Bind Mount vs Volume – When to Use Which

| Use Case            | Recommended              | Why                                |
| ------------------- | ------------------------ | ---------------------------------- |
| Database data       | **Volume**               | Docker manages, better performance |
| Configuration files | **Bind mount**           | Easy to edit on host               |
| Source code (dev)   | **Bind mount**           | Live reload                        |
| Secrets             | **Volume or bind mount** | Either works with restrictions     |
| Backup/restore      | **Volume**               | Built-in commands                  |
| Cross-host sharing  | **Volume** (with driver) | NFS, cloud drivers                 |

## Part 4: tmpfs – In-Memory Storage

tmpfs stores data in RAM. Data is lost when container stops.

```bash
# Mount tmpfs (memory only)
docker run -d --name app --tmpfs /tmp:rw,noexec,nosuid,size=100m nginx

# Using --mount (tmpfs)
docker run -d --name app \
  --mount type=tmpfs,destination=/tmp,tmpfs-size=100m \
  nginx
```

**Use cases for tmpfs:**

* Temporary cache (re-creatable)

* Session data (short-lived)

* Secrets (avoid writing to disk)

* High-performance temporary storage

**Limitations:**

* Not persistent (lost on container stop)

* Limited by available RAM

* Not shared between containers

## Part 5: Practical Examples

### Example 1: PostgreSQL with Persistent Volume

```bash
# Create volume
docker volume create pgdata

# Run PostgreSQL with volume
docker run -d \
  --name postgres \
  -e POSTGRES_PASSWORD=secret \
  -v pgdata:/var/lib/postgresql/data \
  postgres:15

# Data persists even after container removal
docker stop postgres
docker rm postgres
docker run -d --name postgres-new -v pgdata:/var/lib/postgresql/data postgres:15
# Data still there!
```

### Example 2: Development Environment with Bind Mounts

```bash
# Project structure
# .
# ├── Dockerfile
# ├── app/
# ├── node_modules/
# └── package.json

# Development container with live reload
docker run -d \
  --name dev \
  -p 3000:3000 \
  -v $(pwd):/app \
  -v /app/node_modules \  # Preserve container's node_modules
  node:18 \
  npm run dev
```

### Example 3: Nginx with Custom Config and Static Files

```bash
# Bind mount config and static files
docker run -d \
  --name web \
  -p 80:80 \
  -v /opt/nginx/html:/usr/share/nginx/html:ro \
  -v /opt/nginx/conf.d:/etc/nginx/conf.d:ro \
  nginx
```

### Example 4: Redis with tmpfs for Cache

```bash
# Redis with persistence disabled, tmpfs for data
docker run -d \
  --name redis-cache \
  --tmpfs /data:rw,noexec,nosuid,size=256m \
  redis:alpine \
  redis-server --appendonly no --save ""
```

## Part 6: Volume Management Best Practices

### Naming Conventions

```bash
# Good volume names
docker volume create postgres_production_data
docker volume create redis_cache
docker volume create app_uploads

# Avoid anonymous volumes (hard to manage)
docker run -v /data postgres  # Creates anonymous volume
```

### Labeling Volumes

```bash
# Add labels for organization
docker volume create \
  --label environment=production \
  --label application=postgres \
  --label backup=daily \
  pgdata
```

### Cleaning Up Unused Volumes

```bash
# List volumes
docker volume ls

# Remove unused volumes
docker volume prune

# Remove all unused volumes (force)
docker volume prune -f

# Remove specific volume
docker volume rm pgdata
```

### Volume Drivers for Production

```bash
# NFS volume (shared across hosts)
docker volume create \
  --driver local \
  --opt type=nfs \
  --opt o=addr=192.168.1.100,rw,soft,timeo=30 \
  --opt device=:/exports/data \
  nfs-shared

# AWS EBS (via Rex-Ray or Cloudstor)
docker volume create \
  --driver rexray/ebs \
  --opt size=100 \
  --opt volumetype=gp3 \
  ebs-volume
```

## Part 7: Troubleshooting Storage

### Common Issues

**Permission denied on bind mount:**

```bash
# Check host directory permissions
ls -la /host/path

# Fix with proper user (container's user)
docker run -u $(id -u):$(id -g) -v /host/path:/container/path myapp
```

**Volume not mounting:**

```bash
# Verify volume exists
docker volume ls

# Check container's mount points
docker inspect container | grep -A 10 Mounts
```

**Disk space full:**

```bash
# Check Docker disk usage
docker system df

# Find large volumes
du -sh /var/lib/docker/volumes/* | sort -rh | head -10
```

**Backup volume content:**

```bash
# List files in volume
docker run --rm -v mydata:/data alpine ls -la /data
```

## Quick Task: Storage Practice

*Practice using volumes and bind mounts.*

1. Create a volume named `app-data`.
2. Run an alpine container that writes a file to the volume.
3. Verify the file persists by running a second container.
4. Create a bind mount from a host directory to a container.
5. Run a container with a tmpfs mount and verify it's in RAM.

> **Ready Solution:**
>
> ```bash
> # Task 1
> docker volume create app-data
>
> # Task 2
> docker run --rm -v app-data:/data alpine sh -c "echo 'Hello Volume' > /data/test.txt"
>
> # Task 3
> docker run --rm -v app-data:/data alpine cat /data/test.txt
> # Hello Volume
>
> # Task 4
> mkdir -p /tmp/bind-test
> echo "Hello Bind" > /tmp/bind-test/file.txt
> docker run --rm -v /tmp/bind-test:/data alpine cat /data/file.txt
> # Hello Bind
>
> # Task 5
> docker run --rm --tmpfs /tmpfs:size=64m alpine sh -c "mount | grep tmpfs"
> # tmpfs on /tmpfs type tmpfs (rw,nosuid,nodev,noexec,relatime,size=65536k)
>
> # Cleanup
> docker volume rm app-data
> rm -rf /tmp/bind-test
> ```

## Summary Table: Storage Types

| Feature              | Volume                                                | Bind Mount                                    | tmpfs                                  |
| -------------------- | ----------------------------------------------------- | --------------------------------------------- | -------------------------------------- |
| **Syntax**           | `-v vol:/path`                                        | `-v /host:/path`                              | `--tmpfs /path`                        |
| **Preferred syntax** | `--mount source=vol,target=/path`                     | `--mount type=bind,source=/host,target=/path` | `--mount type=tmpfs,destination=/path` |
| **Host location**    | `/var/lib/docker/volumes/`                            | Any path                                      | RAM                                    |
| **Backup**           | `docker run --rm -v vol:/data alpine tar czf - /data` | `rsync`, `cp`                                 | N/A                                    |
| **Performance**      | Good                                                  | Good                                          | Excellent (RAM)                        |
| **Persistence**      | Yes                                                   | Yes                                           | No                                     |
| **Use case**         | Databases, persistent data                            | Config, source code                           | Cache, secrets                         |

### Volume Commands

| Command                 | Purpose               |
| ----------------------- | --------------------- |
| `docker volume create`  | Create volume         |
| `docker volume ls`      | List volumes          |
| `docker volume inspect` | Show volume details   |
| `docker volume rm`      | Remove volume         |
| `docker volume prune`   | Remove unused volumes |

### Mount Flag Comparison

| Syntax                                             | Example                                                 | Read-only                                                  |
| -------------------------------------------------- | ------------------------------------------------------- | ---------------------------------------------------------- |
| `-v /host:/container`                              | `-v ./data:/app/data`                                   | `-v ./conf:/app/conf:ro`                                   |
| `--mount type=bind,source=/host,target=/container` | `--mount type=bind,source=$(pwd)/data,target=/app/data` | `--mount type=bind,source=/conf,target=/app/conf,readonly` |
| `-v vol:/container`                                | `-v mydata:/var/lib/postgresql/data`                    | `-v mydata:/data:ro`                                       |
| `--mount source=vol,target=/container`             | `--mount source=mydata,target=/var/lib/postgresql/data` | `--mount source=mydata,target=/data,readonly`              |
