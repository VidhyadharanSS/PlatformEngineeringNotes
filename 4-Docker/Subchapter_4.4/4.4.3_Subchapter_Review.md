## Cheatsheet: Volumes and Docker Compose

### Storage Types Comparison

| Feature               | Volume                     | Bind Mount          | tmpfs          |
| --------------------- | -------------------------- | ------------------- | -------------- |
| **Managed by Docker** | Yes                        | No                  | Yes            |
| **Host location**     | `/var/lib/docker/volumes/` | Any path            | RAM            |
| **Backup**            | Easy (Docker commands)     | Manual              | N/A            |
| **Persistence**       | Yes                        | Yes                 | No             |
| **Use case**          | Databases, persistent data | Config, source code | Cache, secrets |

### Volume Commands

| Command                      | Purpose               |
| ---------------------------- | --------------------- |
| `docker volume create NAME`  | Create volume         |
| `docker volume ls`           | List volumes          |
| `docker volume inspect NAME` | Show details          |
| `docker volume rm NAME`      | Remove volume         |
| `docker volume prune`        | Remove unused volumes |

### Mount Syntax

| Method                 | Syntax                                        | Read-only                                              |
| ---------------------- | --------------------------------------------- | ------------------------------------------------------ |
| `-v` (volume)          | `-v vol:/container/path`                      | `-v vol:/path:ro`                                      |
| `--mount` (volume)     | `--mount source=vol,target=/path`             | `--mount source=vol,target=/path,readonly`             |
| `-v` (bind mount)      | `-v /host/path:/container/path`               | `-v /host/path:/path:ro`                               |
| `--mount` (bind mount) | `--mount type=bind,source=/host,target=/path` | `--mount type=bind,source=/host,target=/path,readonly` |
| `--tmpfs`              | `--tmpfs /path`                               | N/A                                                    |
| `--mount` (tmpfs)      | `--mount type=tmpfs,destination=/path`        | N/A                                                    |

### Volume Backup and Restore

```bash
# Backup
docker run --rm -v mydata:/source -v $(pwd):/backup alpine \
  tar czf /backup/backup.tar.gz -C /source .

# Restore
docker run --rm -v mydata:/target -v $(pwd):/backup alpine \
  tar xzf /backup/backup.tar.gz -C /target
```

### Docker Compose Commands

| Command                           | Purpose               |
| --------------------------------- | --------------------- |
| `docker compose up -d`            | Start in background   |
| `docker compose down`             | Stop and remove       |
| `docker compose down -v`          | Also remove volumes   |
| `docker compose stop`             | Stop without removing |
| `docker compose start`            | Start stopped         |
| `docker compose restart`          | Restart               |
| `docker compose ps`               | List containers       |
| `docker compose logs`             | View logs             |
| `docker compose exec SERVICE CMD` | Run command           |
| `docker compose build`            | Build images          |
| `docker compose config`           | Validate config       |

### docker-compose.yml Structure

```yaml
version: '3.8'

services:
  web:
    build: .
    image: myapp:latest
    ports:
      - "8080:80"
    volumes:
      - ./src:/app
      - node_modules:/app/node_modules
    environment:
      - NODE_ENV=production
    env_file:
      - .env
    depends_on:
      - db
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost/health"]
      interval: 30s
      timeout: 3s
      retries: 3

  db:
    image: postgres:15
    volumes:
      - db_data:/var/lib/postgresql/data
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD}

volumes:
  db_data:
  node_modules:

networks:
  frontend:
    driver: bridge
```

### Service Options Quick Reference

| Option        | Example                 | Purpose               |
| ------------- | ----------------------- | --------------------- |
| `build`       | `build: ./app`          | Build from Dockerfile |
| `image`       | `image: nginx:alpine`   | Use existing image    |
| `ports`       | `- "8080:80"`           | Port mapping          |
| `volumes`     | `- ./data:/data`        | Mount volumes         |
| `environment` | `- NODE_ENV=prod`       | Environment variables |
| `env_file`    | `env_file: .env`        | Load env from file    |
| `depends_on`  | `- db`                  | Startup order         |
| `restart`     | `restart: always`       | Restart policy        |
| `command`     | `command: npm start`    | Override command      |
| `healthcheck` | `test: ["CMD", "curl"]` | Health check          |

## Comparison Tables

### Volume vs Bind Mount vs tmpfs

| Aspect             | Volume                 | Bind Mount           | tmpfs           |
| ------------------ | ---------------------- | -------------------- | --------------- |
| **Performance**    | Good                   | Good                 | Excellent       |
| **Portability**    | High                   | Low (path-dependent) | High            |
| **Backup ease**    | High                   | Medium               | N/A             |
| **Security**       | High (Docker isolated) | Medium (host access) | High (RAM only) |
| **Use in Compose** | `volumes:` section     | Direct path          | `tmpfs:` option |

### Compose vs Docker Run

| Feature                    | Docker Compose                 | Docker Run             |
| -------------------------- | ------------------------------ | ---------------------- |
| **Multi-container**        | Yes (one command)              | No (multiple commands) |
| **Configuration**          | YAML file                      | Command line flags     |
| **Reproducibility**        | High (file in version control) | Low (manual commands)  |
| **Environment management** | Built-in (`.env` file)         | Manual                 |
| **Networking**             | Automatic                      | Manual                 |
| **Scaling**                | `--scale` flag                 | Manual                 |

## Interview Questions (Scenario-Based)

These questions assume only knowledge from Subchapter 4.4. Answers reference only concepts from 4.4.1 and 4.4.2.

### Question 1

**Scenario:** A production PostgreSQL container was stopped and removed. When a new container was started with the same volume, all data was missing. The original container was run with `docker run -d --name db -v /var/lib/postgresql/data postgres`.

**Question:** Why was the data lost? How should the container have been run to preserve data?

**Answer:**

**Why data was lost:**

* The command used an **anonymous volume** (no name specified)

* Anonymous volumes are created with random names like `0a1b2c3d4e5f...`

* When the container was removed, the anonymous volume may have been orphaned or not reattached

* The new container created a new anonymous volume (different random name)

**Correct way – use named volume:**

```bash
# Create named volume first
docker volume create pgdata

# Run with named volume
docker run -d --name db -v pgdata:/var/lib/postgresql/data postgres
```

**Alternative using** **`--mount`** **(preferred):**

```bash
docker run -d --name db \
  --mount source=pgdata,target=/var/lib/postgresql/data \
  postgres
```

**Verification data persisted:**

```bash
# First container
docker run -d --name db1 -v pgdata:/var/lib/postgresql/data postgres
docker exec db1 createdb -U postgres mydb

# Remove container (data in volume persists)
docker stop db1 && docker rm db1

# New container using same volume
docker run -d --name db2 -v pgdata:/var/lib/postgresql/data postgres
docker exec db2 psql -U postgres -l | grep mydb  # Database still there!
```

**Best practices:**

1. Always use **named volumes** for production data
2. Use `--mount` syntax (more explicit than `-v`)
3. Include volumes in Docker Compose for clarity
4. Regularly backup volumes
5. Label volumes for organization: `--label environment=production`

### Question 2

**Scenario:** A developer cannot get Docker Compose to read environment variables from a `.env` file. The `docker-compose.yml` has `${DB_PASSWORD}` but the variable is not expanded.

**Question:** What are the possible causes? How would you debug and fix?

**Answer:**

**Possible causes:**

1. **`.env`** **file not in correct location** – Must be in same directory as `docker-compose.yml`

2. **Variable not defined in** **`.env`** – Check spelling and syntax

3. **Variable used in wrong place** – Some fields don't support variable substitution

4. **Quoting issues** – Quotes can prevent expansion

5. **Using legacy** **`docker-compose`** **(with hyphen) vs** **`docker compose`**

**Debugging steps:**

```bash
# 1. Check .env file exists and has content
ls -la .env
cat .env
# DB_PASSWORD=SecurePass123

# 2. Validate compose config (shows expanded values)
docker compose config

# 3. Check for syntax errors
docker compose config --resolve-image-digests

# 4. Test variable substitution with simple command
docker compose run --rm web env | grep DB_PASSWORD
```

**Fixes:**

**Fix 1 – Ensure correct** **`.env`** **location:**

```bash
# Move .env to correct directory
mv /wrong/path/.env /project/docker-compose.yml
```

**Fix 2 – Correct .env syntax:**

```env
# Correct
DB_PASSWORD=SecurePass123
DB_USER=appuser

# Wrong (spaces around =)
DB_PASSWORD = SecurePass123

# Wrong (quotes included in value)
DB_PASSWORD="SecurePass123"  # Quotes become part of value
```

**Fix 3 – Use in compatible fields:**

```yaml
# Works
services:
  web:
    environment:
      - PASSWORD=${DB_PASSWORD}  # Works
    image: myapp:${TAG}          # Works

# Doesn't work in all versions
    ports:
      - "${HOST_PORT}:80"  # May not work in older versions
```

**Fix 4 – Explicit env file (if multiple environments):**

```yaml
services:
  web:
    env_file:
      - .env.production
```

**Fix 5 – Pass variables explicitly:**

```bash
# Instead of .env file
DB_PASSWORD=SecurePass123 docker compose up -d

# Or export first
export DB_PASSWORD=SecurePass123
docker compose up -d
```

**Best practices:**

* Keep `.env` in version control (with example values, not secrets)

* Use `.env.example` for template, `.env` for actual (gitignored)

* Validate with `docker compose config` before starting

### Question 3

**Scenario:** A development environment needs to run a Node.js application with live code reload. The source code is on the host and must be synced to the container. However, `node_modules` should NOT be synced from host (it may be OS-specific).

**Question:** How would you configure Docker Compose to achieve this? Show the `docker-compose.yml` configuration.

**Answer:**

**Solution:** Use a bind mount for source code and an anonymous volume for `node_modules`.

```yaml
# docker-compose.yml
version: '3.8'

services:
  web:
    build: .
    ports:
      - "3000:3000"
    volumes:
      # Bind mount source code (live reload)
      - ./src:/app
      # Anonymous volume for node_modules (prevents host sync)
      - /app/node_modules
    environment:
      - NODE_ENV=development
    command: npm run dev
```

**Why this works:**

* `./src:/app` – Bind mount syncs host source code to container (changes reflected immediately)

* `/app/node_modules` – Anonymous volume (no host path) keeps container's `node_modules` intact

* The anonymous volume takes precedence over the bind mount for the `node_modules` subdirectory

**Alternative using named volume:**

```yaml
services:
  web:
    build: .
    volumes:
      - ./src:/app
      - node_modules_vol:/app/node_modules

volumes:
  node_modules_vol:
```

**Dockerfile (for reference):**

```dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
CMD ["npm", "start"]
```

**Running with hot reload (nodemon):**

```yaml
services:
  web:
    build: .
    volumes:
      - ./src:/app
      - /app/node_modules
    command: npx nodemon --legacy-watch src/server.js
```

**Verification:**

```bash
# Check mounts
docker compose exec web mount | grep /app

# Verify node_modules is not empty
docker compose exec web ls -la /app/node_modules

# Test live reload
echo "console.log('test')" >> src/test.js
# Container sees the change immediately
```

### Question 4

**Scenario:** A team shares a `docker-compose.yml` file, but developers have different UIDs/GIDs on their host machines. Bind-mounted files get created with the wrong ownership inside containers, causing permission errors.

**Question:** How would you solve this without requiring developers to change their host UIDs? Provide the Compose configuration.

**Answer:**

**Solution:** Pass the host user's UID/GID into the container and create a user with matching IDs.

**docker-compose.yml:**

```yaml
version: '3.8'

services:
  web:
    build:
      context: .
      args:
        - UID=${UID:-1000}
        - GID=${GID:-1000}
    volumes:
      - ./src:/app
      - /app/node_modules
    environment:
      - UID=${UID:-1000}
      - GID=${GID:-1000}
    user: "${UID:-1000}:${GID:-1000}"
```

**.env file:**

```env
# Each developer sets their own UID/GID
UID=1001
GID=1001
```

**Dockerfile:**

```dockerfile
FROM node:18-alpine

ARG UID=1000
ARG GID=1000

# Create group and user with host UID/GID
RUN addgroup -g ${GID} appgroup || true && \
    adduser -u ${UID} -G appgroup -D appuser || true

WORKDIR /app
COPY --chown=appuser:appgroup package*.json ./
RUN npm ci
COPY --chown=appuser:appgroup . .

USER appuser

CMD ["npm", "start"]
```

**Alternative: Use** **`user:`** **field without building image (simpler):**

```yaml
services:
  web:
    image: node:18-alpine
    user: "${UID:-1000}:${GID:-1000}"
    volumes:
      - ./src:/app
      - /app/node_modules
    working_dir: /app
    command: sh -c "npm ci && npm start"
```

**Alternative: Fix permissions at runtime (entrypoint script):**

```bash
#!/bin/sh
# entrypoint.sh
if [ -n "$HOST_UID" ] && [ -n "$HOST_GID" ]; then
    # Change existing user's UID/GID (requires root)
    usermod -u $HOST_UID node
    groupmod -g $HOST_GID node
    chown -R node:node /app
fi
exec su -c "$@" node
```

**One-liner for developers to get their UID/GID:**

```bash
echo "UID=$(id -u)" > .env
echo "GID=$(id -g)" >> .env
```

**Best practice:**

* Add `.env` to `.gitignore` (personal settings)

* Provide `.env.example` in repository

* Document how to set UID/GID in README

### Question 5

**Scenario:** A production deployment uses Docker Compose with 5 services. After a server reboot, all containers failed to start because the database container takes 30 seconds to become healthy, but the web app tries to connect immediately.

**Question:** What is the problem? How would you fix it using Docker Compose features?

**Answer:**

**Problem:** `depends_on` only waits for containers to **start**, not for them to be **healthy/ready**. The web app attempts to connect before the database is ready to accept connections.

**Fix 1 – Add healthcheck and condition (Compose 3.8+):**

```yaml
version: '3.8'

services:
  db:
    image: postgres:15
    healthcheck:
      test: ["CMD", "pg_isready", "-U", "postgres"]
      interval: 5s
      timeout: 3s
      retries: 5
      start_period: 10s

  web:
    build: .
    depends_on:
      db:
        condition: service_healthy  # Wait for healthy, not just started
    ports:
      - "8080:80"
```

**Fix 2 – Add start\_period to web app (allow time to connect):**

```yaml
services:
  web:
    build: .
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost/health"]
      start_period: 60s  # Give app time to establish DB connection
    depends_on:
      db:
        condition: service_started
```

**Fix 3 – Use wait-for-it script (fallback for older Compose):**

```yaml
services:
  web:
    build: .
    command: ["./wait-for-it.sh", "db:5432", "--", "npm", "start"]
    depends_on:
      - db
```

**Fix 4 – Implement retry logic in application (most robust):**

```javascript
// Node.js example – retry database connection
const connectWithRetry = () => {
  return new Promise((resolve, reject) => {
    let retries = 10;
    const tryConnect = () => {
      db.connect().then(resolve).catch(err => {
        if (retries > 0) {
          retries--;
          console.log(`DB connection failed, ${retries} retries left...`);
          setTimeout(tryConnect, 3000);
        } else {
          reject(err);
        }
      });
    };
    tryConnect();
  });
};
```

**Fix 5 – Use** **`restart: on-failure`** **with exponential backoff:**

```yaml
services:
  web:
    restart: on-failure  # Will retry if DB connection fails
```

**Verification of health checks:**

```bash
# Check container health status
docker compose ps
# NAME   STATUS
# db     Up 5 seconds (healthy)
# web    Up 5 seconds (starting)

# Wait for web to become healthy
watch -n 1 'docker compose ps'
```

**Best practice for production:**

1. Implement health checks on all services that can report readiness
2. Use `condition: service_healthy` for critical dependencies
3. Add retry logic in application code (defense in depth)
4. Set appropriate `start_period` to avoid premature health check failures
5. Monitor health check failures in production
