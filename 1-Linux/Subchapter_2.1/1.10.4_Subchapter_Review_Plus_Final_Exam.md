## Cheatsheet: Capstone Troubleshooting

### Boot Process Stages

| Stage | Component | Recovery Method                      |
| ----- | --------- | ------------------------------------ |
| 1     | BIOS/UEFI | Check boot order, reset CMOS         |
| 2     | GRUB      | Rescue shell, reinstall from live CD |
| 3     | Kernel    | Edit GRUB command line               |
| 4     | initramfs | `rd.break`, regenerate initramfs     |
| 5     | systemd   | Emergency/single-user mode           |
| 6     | Login     | Reset root password                  |

### Boot Parameters

| Parameter                    | Effect                 |
| ---------------------------- | ---------------------- |
| `single` or `1`              | Single-user mode       |
| `emergency`                  | Emergency mode         |
| `rd.break`                   | Break before initramfs |
| `init=/bin/bash`             | Use bash as init       |
| `systemd.unit=rescue.target` | Rescue target          |

### GRUB Rescue Commands

```bash
# Find boot partition
ls
ls (hd0,msdos1)/grub

# Set root and prefix
set root=(hd0,msdos1)
set prefix=(hd0,msdos1)/grub

# Load normal mode
insmod normal
normal
```

### Space vs Inode Troubleshooting

| Check         | Command                | Full Symptom    | Cleanup                                   |
| ------------- | ---------------------- | --------------- | ----------------------------------------- |
| Space         | `df -h`                | 100% used       | `find -size +100M`, `journalctl --vacuum` |
| Inodes        | `df -i`                | 100% used       | `find \| wc -l`, delete temp files        |
| Deleted files | `lsof \| grep deleted` | Space not freed | `truncate -s 0 /proc/PID/fd/FD`           |

### Disk Cleanup Commands

| Command                       | Effect                      |
| ----------------------------- | --------------------------- |
| `apt clean` / `dnf clean all` | Clear package cache         |
| `journalctl --vacuum-time=7d` | Keep last 7 days of logs    |
| `truncate -s 0 file`          | Empty file without deleting |
| `find -size +100M -delete`    | Delete large files          |

### SSH Error Diagnosis

| Error                           | Likely Cause                | Fix                    |
| ------------------------------- | --------------------------- | ---------------------- |
| `Connection refused`            | sshd not running            | `systemctl start sshd` |
| `Connection timed out`          | Firewall/network            | Check firewall         |
| `Host key changed`              | Server reinstalled          | `ssh-keygen -R host`   |
| `Permission denied (publickey)` | Key not in authorized\_keys | Add key, check perms   |
| `Bad ownership or modes`        | Wrong permissions           | `chmod 700 ~/.ssh`     |
| `SELinux denial`                | Wrong context               | `restorecon -R ~/.ssh` |

### SSH Permission Requirements

| File/Directory                       | Required Permission |
| ------------------------------------ | ------------------- |
| `~/.ssh`                             | `700` (drwx------)  |
| `~/.ssh/authorized_keys`             | `600` (-rw-------)  |
| Private key (`id_rsa`, `id_ed25519`) | `600` (-rw-------)  |
| Public key (`*.pub`)                 | `644` (-rw-r--r--)  |

### SELinux SSH Contexts

| Path                   | Correct Context | Fix Command                       |
| ---------------------- | --------------- | --------------------------------- |
| `~/.ssh`               | `ssh_home_t`    | `restorecon -R ~/.ssh`            |
| `/etc/ssh/sshd_config` | `etc_t`         | `restorecon /etc/ssh/sshd_config` |

### Question 1: Boot Recovery (15 minutes)

**Scenario:** A RHEL 9 server fails to boot. The last visible message is:

```
ALERT! UUID=abc123... does not exist. Dropping to a shell.
```

You are at the `dracut:/#` prompt. The server has LVM with `/dev/mapper/rhel-root` as the root filesystem.

**Questions:**

1. What is the most likely cause of this error?
2. How would you identify the correct root device?
3. How would you boot the system without permanently fixing the issue?
4. How would you make the fix permanent after boot?

**Answer:**

1. **Most likely cause:** The `root=` parameter in the kernel command line (from GRUB) has an incorrect UUID. This often happens after cloning a disk, restoring from backup, or modifying partitions without updating GRUB.

2. **Identify correct root device:**

```bash
# At dracut:/# prompt
# List available block devices
lsblk
# or
blkid

# Look for the LVM root volume
lvs
# Should show something like: root rhel

# The correct device path is typically:
/dev/mapper/rhel-root
# or
/dev/rhel/root
```

1. **Boot without permanent fix:**

```bash
# At dracut:/# prompt, exit to continue boot but with modified parameters
# First, mount the root filesystem manually
mount /dev/mapper/rhel-root /sysroot

# Exit dracut shell (continues boot)
exit
```

**Alternative – Edit GRUB at boot:**

* Reboot, at GRUB menu press `e` to edit

* Find the `linux` line with `root=UUID=...`

* Replace with `root=/dev/mapper/rhel-root`

* Press `Ctrl+X` to boot

1. **Make permanent:**

```bash
# After successful boot
# Get correct UUID
sudo blkid /dev/mapper/rhel-root

# Update GRUB configuration
# For RHEL/Rocky:
sudo vi /etc/default/grub
# Update GRUB_CMDLINE_LINUX with correct root=UUID=...
sudo grub2-mkconfig -o /boot/grub2/grub.cfg

# For Debian/Ubuntu:
sudo vi /etc/default/grub
sudo update-grub

# Also update fstab if needed
sudo blkid >> check UUIDs in /etc/fstab
```

### Question 2: Disk Full Emergency (15 minutes)

**Scenario:** A production web server alerts that `/var` is at 98% capacity. The server is running Nginx, MySQL, and a custom application. You cannot reboot.

**Tasks:**

1. Identify all locations in `/var` that could be consuming space.
2. Write commands to find the top 5 largest files in `/var`.
3. Write commands to safely free at least 20% space without breaking services.
4. What additional check should you perform beyond `df -h`?

**Answer:**

1. **Common space consumers in** **`/var`:**

   * `/var/log` – System and application logs

   * `/var/lib/mysql` – Database data (if MySQL)

   * `/var/cache` – Package cache, application cache

   * `/var/tmp` – Persistent temporary files

   * `/var/lib/docker` – Docker storage (if used)

   * `/var/log/nginx` – Web server logs

   * `/var/spool` – Mail spool, cron spool

2. **Find top 5 largest files in** **`/var`:**

```bash
# Method 1: find with sort
sudo find /var -type f -exec du -h {} \; 2>/dev/null | sort -rh | head -5

# Method 2: find with size filter and ls
sudo find /var -type f -size +100M -exec ls -lh {} \; 2>/dev/null | awk '{print $5, $9}' | sort -rh | head -5

# Method 3: Using ncdu (interactive, if installed)
sudo ncdu /var
```

1. **Safely free 20% space:**

```bash
# Step 1: Clean package cache
sudo apt clean 2>/dev/null || sudo dnf clean all 2>/dev/null

# Step 2: Rotate and clean logs
sudo logrotate --force /etc/logrotate.conf
sudo find /var/log -name "*.gz" -delete
sudo find /var/log -name "*.1" -delete
sudo find /var/log -name "*.old" -delete

# Step 3: Clean journal logs (keep last 7 days)
sudo journalctl --vacuum-time=7d

# Step 4: Clean temporary files
sudo rm -rf /var/tmp/*

# Step 5: If Docker is running
docker system prune -af --volumes 2>/dev/null

# Step 6: Check for deleted but open files
sudo lsof | grep deleted | awk '{print $2, $4}' | while read pid fd; do
    fd_num=$(echo $fd | sed 's/[^0-9]//g')
    sudo truncate -s 0 /proc/$pid/fd/$fd_num 2>/dev/null
done

# Step 7: Verify space freed
df -h /var
```

1. **Additional check beyond** **`df -h`:**

```bash
# Check inode usage – may be full even if space is available
df -i /var

# If inodes are exhausted (100% IUse), the problem is too many small files
# Find directories with most files:
sudo find /var -xdev -type f -printf "%h\n" 2>/dev/null | sort | uniq -c | sort -rn | head -10
```

### Question 3: SSH Lockout Recovery (15 minutes)

**Scenario:** You are locked out of a production Ubuntu 22.04 server after a security team member ran a hardening script. SSH now returns:

```
Permission denied (publickey).
```

You have console access via the cloud provider's web console.

**Tasks:**

1. List the step-by-step recovery procedure.
2. What SSH configuration settings should you check?
3. What permission and SELinux (or AppArmor) issues should you verify?
4. How would you test the fix before disconnecting?

**Answer:**

1. **Recovery procedure:**

```bash
# Step 1: Access via cloud console (root or sudo user)
# Log in using the console

# Step 2: Check SSH daemon status
sudo systemctl status ssh

# Step 3: Check SSH configuration
sudo cat /etc/ssh/sshd_config | grep -E "PubkeyAuthentication|PasswordAuthentication|PermitRootLogin"

# Step 4: Fix configuration if needed
sudo vi /etc/ssh/sshd_config
# Ensure:
# PubkeyAuthentication yes
# PasswordAuthentication yes (temporarily for recovery)
# PermitRootLogin prohibit-password

# Step 5: Check your public key is present
cat ~/.ssh/authorized_keys
# If missing, add it
mkdir -p ~/.ssh
chmod 700 ~/.ssh
echo "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI..." >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys

# Step 6: Check permissions
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys

# Step 7: Check AppArmor (Ubuntu's equivalent to SELinux)
sudo aa-status | grep ssh
# If AppArmor is blocking, check logs
sudo journalctl -u ssh -n 50

# Step 8: Restart SSH
sudo systemctl restart ssh

# Step 9: Test from another terminal before closing console
# (Keep console open as backup)
```

1. **SSH configuration settings to check:**

```bash
PubkeyAuthentication yes      # Must be yes
PasswordAuthentication yes    # Temporarily for recovery
PermitRootLogin prohibit-password  # Or yes temporarily
AllowUsers                    # Check if your user is listed
DenyUsers                     # Check if your user is blocked
AllowGroups                   # Check group membership
MaxAuthTries                  # May be too low
AuthenticationMethods         # May be too restrictive
```

1. **Permission and security issues to verify:**

```bash
# File permissions
ls -la ~/.ssh
# Must be drwx------ (700)
ls -la ~/.ssh/authorized_keys
# Must be -rw------- (600)

# Home directory permissions
ls -ld ~
# Should not be world-writable (should not end with 'w')

# AppArmor (Ubuntu)
sudo aa-status | grep -E "ssh|complain|enforce"
sudo journalctl -u apparmor | grep ssh

# Check if home directory is encrypted (ecryptfs)
ls -la ~/.ecryptfs
```

1. **Test before disconnecting:**

```bash
# Method 1: Open a second console session (if cloud console allows multiple tabs)
# Keep original console open as backup

# Method 2: Test SSH to localhost from console
ssh localhost
# or
ssh -v localhost

# Method 3: Use a different terminal (if you have another machine)
ssh -v user@server-ip

# Method 4: After confirming SSH works, disable password auth if desired
sudo vi /etc/ssh/sshd_config
# Set PasswordAuthentication no
sudo systemctl restart ssh
# Test again before closing console
```

### Question 4: GRUB Rescue (10 minutes)

**Scenario:** After resizing partitions, a server boots to:

```
grub rescue>
```

**Tasks:**

1. What commands would you use to boot the system from the rescue prompt?
2. After booting, what commands would permanently fix GRUB?

**Answer:**

1. **Boot from GRUB rescue prompt:**

```bash
# Step 1: Find the boot partition
ls
# Output example: (hd0) (hd0,msdos1) (hd0,msdos2)

# Step 2: Check each partition for GRUB files
ls (hd0,msdos1)/
# Look for grub/ directory, vmlinuz*, initrd*

# Step 3: Set root and prefix
set root=(hd0,msdos1)
set prefix=(hd0,msdos1)/grub

# Step 4: Load normal module
insmod normal
normal

# Step 5: Now in normal GRUB menu, boot normally
# Select kernel and boot
```

1. **Permanently fix GRUB after boot:**

```bash
# Debian/Ubuntu
sudo update-grub
sudo grub-install /dev/sda

# RHEL/Rocky
sudo grub2-mkconfig -o /boot/grub2/grub.cfg
sudo grub2-install /dev/sda

# If /boot is on a separate partition
sudo mount /dev/sda1 /boot
# Then reinstall GRUB

# For UEFI systems
sudo grub-install --target=x86_64-efi --efi-directory=/boot/efi --bootloader-id=GRUB
sudo update-grub
```

### Question 5: Permission Denied Chain (15 minutes)

**Scenario:** A new developer, `alice`, cannot SSH to a RHEL 9 server. You have root access via console. The error is `Permission denied (publickey)`.

**Tasks:**

1. Write a systematic diagnostic script to check all possible causes.
2. For each cause, provide the fix command.

**Answer:**

**Diagnostic script:**

```bash
#!/bin/bash
# ssh_diagnostic.sh – Run as root on the server

USER="alice"
HOME_DIR="/home/$USER"

echo "=== SSH Diagnostic for $USER ==="

# 1. Check SSH daemon
echo -e "\n1. SSH Daemon Status:"
systemctl is-active sshd

# 2. Check SSH configuration
echo -e "\n2. SSH Config Key Settings:"
grep -E "^PubkeyAuthentication|^PasswordAuthentication|^PermitRootLogin|^AllowUsers|^DenyUsers" /etc/ssh/sshd_config

# 3. Check user exists
echo -e "\n3. User Existence:"
id $USER 2>/dev/null || echo "User $USER does not exist"

# 4. Check home directory
echo -e "\n4. Home Directory:"
ls -ld $HOME_DIR

# 5. Check .ssh directory permissions
echo -e "\n5. .ssh Directory:"
ls -ld $HOME_DIR/.ssh 2>/dev/null || echo ".ssh directory missing"

# 6. Check authorized_keys
echo -e "\n6. authorized_keys:"
ls -l $HOME_DIR/.ssh/authorized_keys 2>/dev/null || echo "authorized_keys missing"
if [ -f "$HOME_DIR/.ssh/authorized_keys" ]; then
    echo "Number of keys: $(wc -l < $HOME_DIR/.ssh/authorized_keys)"
    echo "First key fingerprint:"
    ssh-keygen -lf $HOME_DIR/.ssh/authorized_keys 2>/dev/null | head -1
fi

# 7. Check SELinux
echo -e "\n7. SELinux Status:"
getenforce
echo "SELinux contexts:"
ls -Z $HOME_DIR/.ssh 2>/dev/null

# 8. Check recent SSH auth failures
echo -e "\n8. Recent SSH Authentication Failures:"
sudo journalctl -u sshd -n 20 | grep -i "failed\|$USER"

# 9. Check if user is locked
echo -e "\n9. User Account Status:"
passwd -S $USER 2>/dev/null
```

**Fixes for each cause:**

| Cause                    | Diagnostic Output                     | Fix Command                                                                                                                                                     |
| ------------------------ | ------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| sshd not running         | `inactive` or `failed`                | `systemctl start sshd; systemctl enable sshd`                                                                                                                   |
| PubkeyAuthentication no  | `PubkeyAuthentication no`             | `sed -i 's/^PubkeyAuthentication no/PubkeyAuthentication yes/' /etc/ssh/sshd_config; systemctl restart sshd`                                                    |
| User doesn't exist       | `User alice does not exist`           | `useradd -m -s /bin/bash alice`                                                                                                                                 |
| Wrong home perms         | `drwxrwxr-x` (not 700)                | `chmod 700 /home/alice`                                                                                                                                         |
| .ssh missing             | `.ssh directory missing`              | `mkdir -p /home/alice/.ssh; chmod 700 /home/alice/.ssh; chown alice:alice /home/alice/.ssh`                                                                     |
| authorized\_keys missing | `authorized_keys missing`             | `echo "ssh-ed25519 AAAA..." > /home/alice/.ssh/authorized_keys; chmod 600 /home/alice/.ssh/authorized_keys; chown alice:alice /home/alice/.ssh/authorized_keys` |
| SELinux wrong context    | `home_root_t` instead of `ssh_home_t` | `restorecon -R /home/alice/.ssh`                                                                                                                                |
| User account locked      | `Password locked`                     | `passwd -u alice`                                                                                                                                               |
| AllowUsers missing alice | `AllowUsers bob` only                 | `echo "AllowUsers bob alice" >> /etc/ssh/sshd_config; systemctl restart sshd`                                                                                   |

### Question 6: Inode Exhaustion (10 minutes)

**Scenario:** `df -h` shows 40% space used, but applications report "No space left on device". `df -i` shows `/` at 100% IUse.

**Tasks:**

1. Explain the difference between space and inode exhaustion.
2. Find which directory has the most files.
3. Write commands to resolve the issue.

**Answer:**

1. **Difference between space and inode exhaustion:**

   * **Space exhaustion** – Too many bytes of data. Caused by large files.

   * **Inode exhaustion** – Too many files/directories. Each file (even empty) consumes one inode. Caused by millions of tiny files (cache, temp files, mail spool).

2. **Find directory with most files:**

```bash
# Method 1: Count files per directory (slow but accurate)
sudo find / -xdev -type d -print0 2>/dev/null | while IFS= read -r -d '' dir; do
    count=$(find "$dir" -maxdepth 1 -type f | wc -l)
    echo "$count $dir"
done | sort -rn | head -10

# Method 2: Faster using awk
sudo find / -xdev -type f -printf "%h\n" 2>/dev/null | sort | uniq -c | sort -rn | head -10

# Method 3: Check common culprits first
echo "Mail spool: $(ls /var/spool/mail/ 2>/dev/null | wc -l)"
echo "Temp directory: $(ls /tmp/ 2>/dev/null | wc -l)"
echo "Systemd journal: $(ls /var/log/journal/ 2>/dev/null | wc -l)"
echo "Docker overlay2: $(ls /var/lib/docker/overlay2/ 2>/dev/null | wc -l)"
```

1. **Resolve the issue:**

```bash
# Identify the problematic directory (example: /var/spool/mail)
sudo du -sh /var/spool/mail
sudo ls /var/spool/mail | wc -l

# Clean up based on culprit:

# For mail spool (after notifying users)
sudo rm -f /var/spool/mail/*

# For temp directory
sudo rm -rf /tmp/*
sudo rm -rf /var/tmp/*

# For journal logs
sudo journalctl --vacuum-size=500M
sudo journalctl --rotate

# For Docker (if applicable)
docker system prune -a --volumes

# For package cache
sudo apt clean 2>/dev/null || sudo dnf clean all

# For session files
sudo find /var/lib -name "sess_*" -type f -mtime +1 -delete

# Verify fix
df -i /
```

### Question 7: Deleted File Recovery (10 minutes)

**Scenario:** A developer deleted `/var/log/app/app.log` while the application was still writing to it. Space is not freed. The application cannot be restarted.

**Tasks:**

1. How do you verify the file is still open?
2. How do you recover the content?
3. How do you free the space without restarting the application?

**Answer:**

1. **Verify file is still open:**

```bash
# Find deleted open files
sudo lsof | grep deleted | grep app.log

# Output example:
# java    1234 root    4w   REG  8,17  1048576 12345 /var/log/app/app.log (deleted)

# Check the process and file descriptor
sudo ls -la /proc/1234/fd/
# l-wx------ 1 root root 64 Jan 16 10:00 4 -> /var/log/app/app.log (deleted)
```

1. **Recover the content:**

```bash
# Recover to new location
sudo cp /proc/1234/fd/4 /tmp/app.log.recovered

# View the content (tail)
sudo tail -100 /proc/1234/fd/4

# Verify recovery
wc -l /tmp/app.log.recovered
```

1. **Free space without restarting:**

```bash
# Truncate the file via /proc
sudo truncate -s 0 /proc/1234/fd/4

# Or using redirection
sudo sh -c 'echo > /proc/1234/fd/4'

# Or using dd
sudo dd if=/dev/null of=/proc/1234/fd/4

# Verify space freed
df -h /var

# Verify no more deleted file
sudo lsof | grep deleted | grep app.log
# Should return nothing
```
