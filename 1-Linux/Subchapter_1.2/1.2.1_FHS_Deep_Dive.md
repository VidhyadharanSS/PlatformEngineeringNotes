## 1.2.1 FHS Deep Dive: Where Everything Lives

#### The Blueprint of a Linux System

The **Filesystem Hierarchy Standard (FHS)** defines the directory structure and contents of Linux and Unix-like operating systems. It ensures that regardless of distribution (RHEL, Debian, Arch), you can predict where to find executables, configuration files, logs, and device files.

**Core principle:** The FHS distinguishes between:

* **Shareable vs unshareable** data – Can be shared over a network (`/usr`) or must be local (`/etc`, `/var`)?

* **Static vs variable** data – Does it change without administrator intervention (`/usr/bin` – static) or does it change frequently (`/var/log` – variable)?

## Top-Level Directories (Essential for Platform Engineers)

### `/` – The Root Directory

The starting point of the entire filesystem. All other directories branch from here. Only `root` user has write permission in `/` itself (except for `/tmp`, `/home`, etc.).

```bash
# View root directory contents
ls -la /
```

### `/bin` – Essential User Binaries

Contains executable commands required for the system to boot and run in single-user mode (recovery). Available to all users.

**Examples:** `ls`, `cp`, `mv`, `cat`, `echo`, `bash`, `sh`

**Modern note:** Many distributions now symlink `/bin` to `/usr/bin`. But the FHS still recognizes `/bin` as a separate entity.

```bash
# Check if /bin is a symlink
ls -l /bin
# Output (on modern Ubuntu): /bin -> /usr/bin
```

### `/sbin` – System Binaries (Administrative)

Executables intended for system administration, typically run by `root` or with `sudo`. Not usually in a regular user's `PATH`.

**Examples:** `fdisk`, `mkfs`, `mount`, `systemctl`, `iptables`

> **Important:** When troubleshooting a Linux host, always ask three questions first: **what binary is running, what config in `/etc` controls it, and what state/logs under `/var` explain its behavior?** This mental model solves many incidents faster.

### `/etc` – Configuration Files (The Brain)

Contains system-wide configuration files. **No binaries here** – only text configuration. Host-specific configuration (not shared over network).

**Critical files you will encounter:**

* `/etc/passwd` – User account database

* `/etc/shadow` – Encrypted passwords (only readable by root)

* `/etc/group` – Group definitions

* `/etc/fstab` – Filesystem mount table (covered in 1.5.2)

* `/etc/hostname` – System's hostname

* `/etc/hosts` – Static DNS lookup table

* `/etc/ssh/sshd_config` – SSH daemon configuration

* `/etc/systemd/` – Systemd unit files and overrides

```bash
# View critical config (safe to read)
cat /etc/hostname

# List all config directories
ls -d /etc/*/
```

### `/home` – User Home Directories

Each regular user (non-root) gets a subdirectory here: `/home/username`. Contains user's personal files, configuration dotfiles (`.bashrc`, `.profile`), and application data.

```bash
# Navigate to your home directory
cd ~
# Which is equivalent to
cd /home/username
```

### `/root` – Root User's Home Directory

Separate from `/home` for security. Only accessible by the root user and members of certain privileged groups.

### `/var` – Variable Data

Data that changes frequently in size or content. **Crucial for monitoring disk space.**

| Subdirectory | Purpose                                  | Example                                          |
| ------------ | ---------------------------------------- | ------------------------------------------------ |
| `/var/log`   | Log files (syslog, auth.log, kern.log)   | `tail -f /var/log/syslog`                        |
| `/var/tmp`   | Temporary files preserved across reboots | Unlike `/tmp`, not cleared on boot               |
| `/var/lib`   | State information for applications       | Databases, package manager state, Docker volumes |
| `/var/spool` | Queued data (print jobs, mail)           | `/var/spool/cron` for user crontabs              |
| `/var/cache` | Application cache data                   | `apt` package cache, font cache                  |

```bash
# Check largest directories in /var
sudo du -sh /var/* | sort -rh | head -10
```

### `/tmp` – Temporary Space

World-writable directory for temporary files. **Cleared on reboot** (on most modern systems using tmpfs – stored in RAM). Files not accessed in 10 days are auto-deleted by systemd-tmpfiles.

```bash
# Create a temp file that disappears on reboot
touch /tmp/my_temp_file.txt

# Safe practice: use mktemp for unique names
tempfile=$(mktemp)
echo "temporary data" > $tempfile

> **Warning:** Do not hardcode predictable filenames in `/tmp` like `/tmp/app.log`. Use `mktemp` to avoid race conditions and accidental overwrites by other users or processes.
```

### `/usr` – User System Resources (Read-Only)

The largest directory hierarchy. Contains most user utilities and applications. **Should be shareable** across network (in theory) and **read-only** except for `/usr/local`.

| Subdirectory   | Purpose                                                                 | Example                                        |
| -------------- | ----------------------------------------------------------------------- | ---------------------------------------------- |
| `/usr/bin`     | Most user commands (primary location)                                   | `grep`, `awk`, `sed`, `python`, `vim`          |
| `/usr/sbin`    | Non-essential system binaries (not required for boot)                   | `sshd`, `httpd`, `useradd`                     |
| `/usr/lib`     | Libraries for binaries in `/usr/bin` and `/usr/sbin`                    | Shared objects (`.so` files)                   |
| `/usr/local`   | Locally compiled or installed software (not managed by package manager) | `/usr/local/bin/my_script.sh`                  |
| `/usr/share`   | Architecture-independent data                                           | Man pages, icons, documentation, timezone data |
| `/usr/include` | C/C++ header files for development                                      | `stdio.h`, `stdlib.h`                          |

**Historical note:** `/bin`, `/sbin`, and `/lib` were separate from `/usr` to allow booting with minimal `/usr` unavailable (e.g., network-mounted `/usr`). Modern systems often merge them.

```bash
# See merge on modern Linux
ls -l /bin /sbin /lib
# Typically symlinks to /usr/bin, /usr/sbin, /usr/lib
```

### `/proc` – Virtual Process Filesystem

**Not a real filesystem** – exists only in memory. Provides kernel and process information as virtual files. Introduced in 1.1.2.

```bash
# CPU information
cat /proc/cpuinfo

# Memory information
cat /proc/meminfo

# Kernel command line
cat /proc/cmdline

# List all processes (each PID is a directory)
ls /proc | grep -E '^[0-9]+$'
```

### `/sys` – Sysfs Virtual Filesystem

Similar to `/proc` but exposes kernel objects, devices, drivers, and power management features. Used extensively by systemd and monitoring tools.

```bash
# List block devices (disks)
ls /sys/block/

# Check power state of a network interface
cat /sys/class/net/eth0/operstate
```

### `/dev` – Device Files

Contains device nodes that represent hardware. Introduced in 1.1.2 with "everything is a file".

| Device Type   | Example                  | Purpose                |
| ------------- | ------------------------ | ---------------------- |
| Hard disks    | `/dev/sda`, `/dev/sdb`   | First SCSI/SATA disk   |
| Partitions    | `/dev/sda1`, `/dev/sda2` | Partitions on disk     |
| Random number | `/dev/urandom`           | Pseudorandom generator |
| Null device   | `/dev/null`              | Discards all input     |
| Zero device   | `/dev/zero`              | Returns null bytes     |
| Terminal      | `/dev/tty`, `/dev/pts/0` | Current terminal       |

```bash
# Write zeroes to a file (creates a 1MB file of null bytes)
dd if=/dev/zero of=zeros.bin bs=1M count=1

# Discard a command's output entirely
my_noisy_command > /dev/null 2>&1
```

### `/boot` – Bootloader Files

Contains static files required to boot the system: kernel (`vmlinuz`), initial RAM disk (`initrd.img` or `initramfs`), bootloader configuration (GRUB). **Keep it clean – old kernels consume space.**

| File              | Purpose                                         |
| ----------------- | ----------------------------------------------- |
| `vmlinuz-*`       | Compressed Linux kernel image                   |
| `initrd.img-*`    | Initial RAM disk (drivers to mount root fs)     |
| `grub/`           | GRUB bootloader configuration and modules       |
| `System.map-*`    | Kernel symbol table (debugging)                 |
| `config-*`        | Kernel build configuration                      |

```bash
# List kernels installed
ls /boot/vmlinuz-*

# Check available space on /boot (critical for updates)
df -h /boot

# Count installed kernels
ls /boot/vmlinuz-* | wc -l
```

**Tip:** On systems with small `/boot` partitions (common in older setups), remove old kernels to prevent update failures. See [1.10.1 Boot Process and Recovery](../Subchapter_1.10/1.10.1_Boot_Process_and_Recovery.md) for troubleshooting.

### `/lib` and `/lib64` – Shared Libraries

Essential libraries for binaries in `/bin` and `/sbin`. `/lib64` is for 64-bit libraries on some architectures. Often symlinked to `/usr/lib`.

```bash
# List loaded libraries for a command
ldd /bin/ls
```

### `/opt` – Optional Add-on Software

For third-party, manually installed software that does not conform to FHS standards. Each package gets its own subdirectory.

**Examples:** `/opt/bitnami/`, `/opt/teamcity/`, `/opt/google/chrome`

### `/srv` – Service Data

Data for services served by the system (web servers, FTP, Git repos). Distribution-dependent – many systems leave this empty.

```bash
# Typical use (if used)
/srv/www/    # Web server content
/srv/git/    # Git repositories
```

### `/run` – Runtime Variable Data

Temporary filesystem stored in RAM (tmpfs) that represents system state since last boot. Replaces `/var/run` (now a symlink). Contains PID files, socket files, and user session data.

```bash
# Check systemd runtime directory
ls /run/systemd/

# Current user runtime directory (for socket activation)
echo $XDG_RUNTIME_DIR
# Typically /run/user/1000

# Common runtime files
ls /run/*.pid           # PID files for daemons
ls /run/lock/           # Lock files
```

**PID files:** Daemons write their process ID to `/run/<service>.pid` so management tools (systemd, init scripts) can track and signal them. Example: `/run/sshd.pid` contains the SSH daemon's PID.

### `/media` and `/mnt` – Mount Points

* `/media` – Automounted removable media (USB drives, CD-ROMs). Subdirectories created automatically.

* `/mnt` – Temporarily mounted filesystems (manual, administrator use).

```bash
# Mount a USB drive manually (older style)
sudo mount /dev/sdb1 /mnt/usb
```

### `/lost+found` – Recovered Files

Present on every ext2/ext3/ext4 filesystem. Contains file fragments recovered by `fsck` (filesystem check) after an unclean shutdown or disk error. **Only accessible by root.**

```bash
# Check lost+found (usually empty on healthy systems)
sudo ls -la /lost+found

# Each mounted partition has its own lost+found
ls -la /home/lost+found  # If /home is a separate partition
```

**Platform engineering relevance:** If you find files in `lost+found` after a crash, they may be corrupted or incomplete. Examine carefully before restoring.

## Viewing Mount Hierarchy: findmnt and lsblk

### `findmnt` – Mount Point Tree

```bash
# Show all mount points as a tree
findmnt

# Show only real filesystems (exclude virtual)
findmnt -t ext4,xfs,btrfs

# Show mount options for a specific path
findmnt /home

# Output as JSON (for scripting)
findmnt --json

# Find what filesystem a path belongs to
findmnt --target /var/log/syslog
```

### `lsblk` – Block Device Tree

```bash
# Show block devices with size and mount points
lsblk

# Include filesystem type
lsblk -f

# Show device paths and UUIDs (useful for fstab)
lsblk -o NAME,SIZE,FSTYPE,UUID,MOUNTPOINT

# Show only SCSI devices (physical disks)
lsblk --scsi
```

**Example output:**

```
NAME   MAJ:MIN RM   SIZE RO TYPE MOUNTPOINT
sda      8:0    0   100G  0 disk 
├─sda1   8:1    0   512M  0 part /boot/efi
├─sda2   8:2    0     1G  0 part /boot
└─sda3   8:3    0  98.5G  0 part /
nvme0n1 259:0   0   500G  0 disk 
└─nvme0n1p1 259:1 0 500G  0 part /data
```

## XDG Base Directory Specification

Modern Linux applications follow the **XDG Base Directory Specification** for user configuration, data, and cache files. This standardizes where applications store their data.

| Variable          | Default            | Purpose                                     |
| ----------------- | ------------------ | ------------------------------------------- |
| `$XDG_CONFIG_HOME` | `~/.config`        | User-specific configuration files           |
| `$XDG_DATA_HOME`   | `~/.local/share`   | User-specific data files                    |
| `$XDG_CACHE_HOME`  | `~/.cache`         | Non-essential cached data                   |
| `$XDG_STATE_HOME`  | `~/.local/state`   | State data (logs, history, recent files)    |
| `$XDG_RUNTIME_DIR` | `/run/user/$UID`   | Runtime files (sockets, locks) – RAM-based  |

```bash
# Check XDG directories
echo $XDG_CONFIG_HOME   # Often unset, defaults to ~/.config
echo $XDG_RUNTIME_DIR   # e.g., /run/user/1000

# Modern applications follow XDG
ls ~/.config/           # Contains: git/, Code/, kubectl/, etc.
ls ~/.local/share/      # Contains: applications/, fonts/, etc.
ls ~/.cache/            # Contains: pip/, npm/, thumbnails/
```

**Platform engineering relevance:**
- When backing up user configs, include `~/.config` and `~/.local/share`
- When clearing disk space, `~/.cache` can usually be safely deleted
- Container applications should respect XDG paths for configuration

## FHS in Containers: What Changes?

In containerized environments (Docker, Kubernetes), the FHS is simplified but still applies:

| Directory | Container Behavior                                                    |
| --------- | --------------------------------------------------------------------- |
| `/`       | Read-only root filesystem (best practice)                             |
| `/etc`    | Often mounted as ConfigMaps or secrets                                |
| `/var`    | `/var/log` and `/var/lib` may be volumes for persistence              |
| `/proc`   | Mounted read-only, may be masked for security                         |
| `/sys`    | Usually masked or read-only                                           |
| `/tmp`    | Often a tmpfs volume with size limits                                 |
| `/dev`    | Minimal devices (no access to host hardware)                          |
| `/home`   | Typically doesn't exist – containers run as specific user             |

```bash
# Inside a minimal container (Alpine-based)
ls /
# bin  dev  etc  home  lib  media  mnt  opt  proc  root  run  sbin  srv  sys  tmp  usr  var

# Check if root filesystem is read-only
mount | grep ' / '
# overlay on / type overlay (ro,relatime,...)  # 'ro' = read-only
```

**Distroless containers** (Google's approach) remove even more:
- No shell (`/bin/sh`)
- No package manager
- Only the application and its dependencies

## Why FHS Matters for Platform Engineering

| Scenario                       | FHS Knowledge Required                                                                        |
| ------------------------------ | --------------------------------------------------------------------------------------------- |
| **Docker container security**  | Knowing that `/proc` and `/sys` should be read-only or masked in containers                   |
| **Log rotation configuration** | `/var/log` location and why you monitor disk usage there                                      |
| **Binary path resolution**     | Why `which python` returns `/usr/bin/python` and not `/bin/python`                            |
| **Backup strategies**          | Excluding `/proc`, `/sys`, `/dev`, `/tmp` from backups (virtual or ephemeral)                 |
| **System recovery**            | Knowing that `/etc` holds critical config and `/var` holds state – separate backup strategies |
| **Kernel upgrades**            | Free space in `/boot` must be managed to avoid failed updates                                 |

## Quick Task: Map Your System

*Using only commands from Subchapters 1.1 and 1.2, explore your Linux system.*

1. Determine whether `/bin` is a symlink or a real directory on your system.
2. Find three different log files in `/var/log` that have been modified in the last 24 hours.
3. List all block devices (disks) visible through `/sys/block/`.
4. Identify your kernel version by reading the appropriate virtual file in `/proc`.
5. Check how much space is used in `/var` versus `/usr`.

> **Ready Solution (example from Ubuntu 22.04 LTS):**
>
> ```bash
> # Task 1
> ls -l /bin
> # Output: lrwxrwxrwx 1 root root 7 ... /bin -> usr/bin
> # Conclusion: /bin is a symlink to /usr/bin
>
> # Task 2
> find /var/log -type f -mtime -1 | head -3
> # Output (example):
> # /var/log/syslog
> # /var/log/auth.log
> # /var/log/kern.log
>
> # Task 3
> ls /sys/block/
> # Output (example): loop0 loop1 sda sdb
> # (loop devices are virtual, sda/sdb are real disks)
>
> # Task 4
> cat /proc/version
> # Output: Linux version 5.15.0-91-generic ...
>
> # Task 5
> du -sh /var /usr
> # Output (example):
> # 4.2G    /var
> # 12G     /usr
> ```

## Summary Table: Critical FHS Directories

| Directory        | Content                                  | Persistence                 | Writeable by               |
| ---------------- | ---------------------------------------- | --------------------------- | -------------------------- |
| `/`              | Root of entire tree                      | Persistent                  | root only (except subdirs) |
| `/bin`           | Essential user binaries                  | Persistent                  | root (via package manager) |
| `/sbin`          | System admin binaries                    | Persistent                  | root                       |
| `/etc`           | Configuration files                      | Persistent                  | root                       |
| `/home`          | User personal files                      | Persistent                  | respective user            |
| `/root`          | root's home                              | Persistent                  | root                       |
| `/var`           | Variable data (logs, spool)              | Persistent                  | root, daemons              |
| `/tmp`           | Temporary files                          | RAM/disk, cleared on boot   | all users                  |
| `/usr`           | User system resources (mostly read-only) | Persistent                  | root (rarely)              |
| `/proc`          | Kernel and process info                  | Virtual (in memory)         | kernel only                |
| `/sys`           | Kernel objects, devices                  | Virtual (in memory)         | kernel only                |
| `/dev`           | Device nodes                             | Virtual (managed by udev)   | kernel, root               |
| `/boot`          | Bootloader, kernel images                | Persistent                  | root                       |
| `/opt`           | Third-party software                     | Persistent                  | root                       |
| `/run`           | Runtime state                            | RAM (tmpfs), lost on reboot | root, users                |
| `/media`, `/mnt` | Mount points                             | Persistent (empty dirs)     | root                       |
