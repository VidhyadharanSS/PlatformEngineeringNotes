## Cheatsheet: FHS, Permissions, tar, and rsync

### FHS Critical Directories Quick Reference

| Directory | Purpose                     | Backup Priority                     |
| --------- | --------------------------- | ----------------------------------- |
| `/etc`    | System configuration        | High – includes all service configs |
| `/home`   | User data                   | High – user files                   |
| `/var`    | Logs, spool, variable data  | Medium – logs can be regenerated    |
| `/usr`    | System software (read-only) | Low – can be reinstalled            |
| `/boot`   | Kernel and bootloader       | Medium – keep old kernels           |
| `/proc`   | Virtual process info        | Never – do not backup               |
| `/sys`    | Virtual kernel objects      | Never – do not backup               |
| `/dev`    | Device nodes                | Never – recreated on boot           |
| `/tmp`    | Temporary files             | Never – cleared on reboot           |

### File Types (First Character of `ls -l`)

| Char | Type             | Example                |
| ---- | ---------------- | ---------------------- |
| `-`  | Regular file     | `/etc/hostname`        |
| `d`  | Directory        | `/home`                |
| `l`  | Symbolic link    | `/bin → /usr/bin`      |
| `c`  | Character device | `/dev/null`            |
| `b`  | Block device     | `/dev/sda`             |
| `s`  | Socket           | `/var/run/docker.sock` |
| `p`  | Named pipe       | Created with `mkfifo`  |

### Permission Octal Reference

| Octal | Symbolic | Meaning              |
| ----- | -------- | -------------------- |
| 7     | `rwx`    | Read, write, execute |
| 6     | `rw-`    | Read, write          |
| 5     | `r-x`    | Read, execute        |
| 4     | `r--`    | Read only            |
| 3     | `-wx`    | Write, execute       |
| 2     | `-w-`    | Write only           |
| 1     | `--x`    | Execute only         |
| 0     | `---`    | No permissions       |

### Common Permission Patterns

```bash
chmod 755 file   # -rwxr-xr-x (executable for all, write only for owner)
chmod 644 file   # -rw-r--r-- (default for text files)
chmod 600 file   # -rw------- (private key, password file)
chmod 700 dir    # drwx------ (private directory)
chmod 755 dir    # drwxr-xr-x (shared directory, no write for others)
chmod 1777 dir   # drwxrwxrwt (/tmp pattern – sticky bit)
```

### Ownership Commands

```bash
chown user file           # Change owner
chown user:group file     # Change owner and group
chgrp group file          # Change group only
chown -R user:group dir/  # Recursive
```

### umask Quick Reference

```bash
umask                    # Show current mask
umask 022                # Files: 644, Dirs: 755 (default)
umask 027                # Files: 640, Dirs: 750 (group read, no others)
umask 077                # Files: 600, Dirs: 700 (private)
```

**Formula:** `result = base - umask` (where base = 666 for files, 777 for directories)

### tar Command Reference

| Operation               | Command                                           |
| ----------------------- | ------------------------------------------------- |
| Create uncompressed     | `tar -cvf archive.tar /path`                      |
| Create gzip compressed  | `tar -czvf archive.tar.gz /path`                  |
| Create bzip2 compressed | `tar -cjvf archive.tar.bz2 /path`                 |
| Create xz compressed    | `tar -cJvf archive.tar.xz /path`                  |
| Extract any             | `tar -xvf archive.tar.*`                          |
| Extract to directory    | `tar -xzvf archive.tar.gz -C /target`             |
| List contents           | `tar -tvf archive.tar.gz`                         |
| Exclude patterns        | `tar -czvf backup.tar.gz /home --exclude="*.log"` |
| Preserve permissions    | `tar -xzvpf archive.tar.gz -C /`                  |

**Flag order matters:** `-f` must be immediately followed by the filename.

### rsync Command Reference

| Pattern           | Command                                                 |
| ----------------- | ------------------------------------------------------- |
| Local sync        | `rsync -av source/ dest/`                               |
| Remote push       | `rsync -avz local/ user@host:/remote/`                  |
| Remote pull       | `rsync -avz user@host:/remote/ local/`                  |
| Dry run           | `rsync -avzn source/ dest/`                             |
| Delete extraneous | `rsync -avz --delete source/ dest/`                     |
| Show progress     | `rsync -avP source/ dest/`                              |
| Limit bandwidth   | `rsync -avz --bwlimit=1000 source/ dest/`               |
| Exclude patterns  | `rsync -av --exclude='*.log' source/ dest/`             |
| Custom SSH port   | `rsync -avz -e "ssh -p 2222" local/ user@host:/remote/` |

**Trailing slash rule:**

* `rsync -av source/ dest/` → copies contents of source into dest

* `rsync -av source dest/` → copies source directory itself into dest

### Essential rsync Flags

| Flag        | Meaning                                                                     |
| ----------- | --------------------------------------------------------------------------- |
| `-a`        | Archive (recursive, preserve perms, times, owner, group, symlinks, devices) |
| `-v`        | Verbose                                                                     |
| `-z`        | Compress during transfer                                                    |
| `-P`        | Show progress + keep partial files                                          |
| `-n`        | Dry run (no changes)                                                        |
| `--delete`  | Remove destination files not in source                                      |
| `--exclude` | Skip matching patterns                                                      |

## Interview Questions (Scenario-Based)

These questions assume only knowledge from Subchapter 1.2. Answers reference only concepts from 1.2.1, 1.2.2, and 1.2.3.

### Question 1

**Scenario:** You run `df -h` and see that `/var` is at 95% capacity. You need to free space immediately without restarting any services.

**Question:** Which specific directories inside `/var` would you investigate first, and why? What commands would you use to identify the largest consumers of space?

**Answer:**

Investigate these directories in order:

1. **`/var/log`** – Log files are the most common cause of sudden space exhaustion. A misconfigured application or log rotation can fill the disk rapidly.

2. **`/var/cache`** – Package manager caches (`apt` cache at `/var/cache/apt/archives/`, `dnf` at `/var/cache/dnf/`) can grow large over time.

3. **`/var/lib`** – Database storage (PostgreSQL at `/var/lib/postgresql/`, MySQL at `/var/lib/mysql/`) or Docker volumes (`/var/lib/docker/`) can consume significant space.

Commands to identify largest consumers:

```bash
# Show top 10 largest directories under /var (human-readable, sort reverse)
sudo du -sh /var/* | sort -rh | head -10

# Find files larger than 100MB in /var/log
sudo find /var/log -type f -size +100M -exec ls -lh {} \;

# Show log directory sizes sorted
sudo du -sh /var/log/* | sort -rh | head -10
```

To free space without restarting services:

* Rotate logs: `sudo logrotate -f /etc/logrotate.conf` (forces immediate rotation)

* Clear package cache: On Debian/Ubuntu: `sudo apt clean`; on RHEL: `sudo dnf clean all`

* Truncate large log files (if safe): `sudo truncate -s 0 /var/log/huge.log`

**Important:** Never delete open log files without truncating or rotating – the space may not be freed until the process releases the file handle. Use `lsof | grep deleted` to find such files (covered in 1.9.2).

### Question 2

**Scenario:** You downloaded a script `deploy.sh` from a trusted internal repository. When you try to run it with `./deploy.sh`, you get "Permission denied". Running `ls -l deploy.sh` shows `-rw-r--r--`.

**Question:** What is the problem, and how do you fix it? After fixing, what would the new `ls -l` output show?

**Answer:**

The problem is that the file lacks execute permission for the owner. The permission string `-rw-r--r--` means:

* File type: `-` (regular file)

* Owner: `rw-` (read and write, but no execute)

* Group: `r--` (read only)

* Others: `r--` (read only)

You cannot execute a file that does not have the execute bit (`x`) set for your user (or group/others if you are not the owner).

**Fix:** Add execute permission for the owner:

```bash
chmod u+x deploy.sh
```

Or using octal mode (adds execute to owner while preserving existing permissions):

```bash
chmod 744 deploy.sh   # rwxr--r--
```

After the fix, `ls -l deploy.sh` would show:

```
-rwxr--r-- 1 user group 1234 Jan 15 10:00 deploy.sh
```

The owner now has `rwx` (7), group remains `r--` (4), others remain `r--` (4).

**Alternative approach** if you cannot change permissions (e.g., read-only filesystem): Run the script by passing it as an argument to the shell:

```bash
bash deploy.sh
# or
sh deploy.sh
```

This works because the shell reads the script file (requires read permission, which you have) and executes its contents without needing the execute bit on the file itself.

### Question 3

**Scenario:** You need to transfer a 50GB database backup from a production server to a disaster recovery server over a slow (10 Mbps) WAN link. The transfer will take hours, and the network is unreliable.

**Question:** Which tool would you choose and why? What specific flags would you use to handle the unreliable connection and minimize data transferred on retries?

**Answer:**

Choose **`rsync`** over `scp` or `tar` because:

* `rsync` supports resuming interrupted transfers with the `-P` flag.

* `rsync` performs delta transfers – if the transfer fails after 40GB, it will resume from where it left off, not restart from zero.

* `rsync` can compress on the fly with `-z` to reduce bandwidth usage.

**Command:**

```bash
rsync -avPz --bwlimit=8000 /path/to/backup.db user@dr-server:/backup/location/
```

**Explanation of flags:**

* `-a` – Archive mode (preserves permissions, timestamps – important for database backups)

* `-v` – Verbose (shows progress, useful for long transfers)

* `-P` – Combines `--partial` (keeps partially transferred files) and `--progress` (shows transfer rate and percent)

* `-z` – Compresses during transfer (reduces data over the slow link)

* `--bwlimit=8000` – Limits bandwidth to 8000 KBps (8 Mbps), leaving 2 Mbps for other production traffic

**Why not** **`tar`?** A single `tar` archive would require restarting from scratch if the transfer fails. You could split the `tar` into smaller chunks, but `rsync` handles this automatically.

**Why not** **`scp`?** `scp` does not resume and does not compress by default. It would waste bandwidth retransmitting already-transferred data.

**Additional recommendation:** Run the command inside `screen` or `tmux` (covered in 1.9.1) to prevent session disconnect from killing the transfer.

### Question 4

**Scenario:** A junior engineer runs `chmod 666 /etc/shadow` by accident. This file stores hashed passwords. Immediately after, users cannot change their passwords.

**Question:** What are the correct permissions for `/etc/shadow`? Why did the mistake break password changes? How would you fix it?

**Answer:**

**Correct permissions for** **`/etc/shadow`:**

```bash
-rw-r----- (640) or sometimes ---------- (000) with capabilities
# Owner: root, Group: shadow (on Debian) or root (on RHEL)
```

On most distributions:

* Owner: `root` – can read and write

* Group: `shadow` (Debian/Ubuntu) or `root` (RHEL) – can read only

* Others: no access

**Why** **`chmod 666`** **broke password changes:**

The file `/etc/shadow` stores password hashes. The `passwd` command (which users run to change their passwords) is a SUID binary (setuid root). It needs to write to `/etc/shadow` to update the password hash. When permissions became `666` (rw-rw-rw-), any user could theoretically write to the file – but more critically, the system's security policies (or the `passwd` binary itself) may reject world-writable sensitive files. Additionally, the `shadow` group or root ownership was changed or the group write permission caused the `passwd` command to fail for safety.

**Fix (as root):**

```bash
# Restore correct permissions
chmod 640 /etc/shadow

# Restore correct ownership if also changed
chown root:shadow /etc/shadow   # Debian/Ubuntu
# OR
chown root:root /etc/shadow     # RHEL/CentOS

# Verify
ls -l /etc/shadow
```

**Prevention:** Never use `666` or `777` on system files. Understand that sensitive files should be readable only by necessary users and writable only by root. Use `umask 027` or `077` for system administration to avoid accidentally creating over-permissive files.

### Question 5

**Scenario:** You are tasked with backing up a web application's configuration directory `/opt/webapp/config` every hour. You want to store only changed files in each backup to save disk space, and you want to be able to restore any version from the last 30 days.

**Question:** Design a backup strategy using tools from this subchapter. Explain your choice of tool(s) and the specific commands you would use.

**Answer:**

Use **`rsync`** **with the** **`--link-dest`** **option** to create incremental backups with hard links. This stores only changed files while making each backup appear as a full directory tree.

**Strategy:**

1. Create a base directory `/backups/webapp/`
2. Inside, store timestamped backups: `20240115_1200/`, `20240115_1300/`, etc.
3. First backup: full copy
4. Subsequent backups: use `--link-dest` pointing to the previous backup – unchanged files become hard links (no additional space), changed files are copied new.

**Backup script:**

```bash
#!/bin/bash
SOURCE="/opt/webapp/config"
BACKUP_ROOT="/backups/webapp"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
LATEST=$(ls -1 $BACKUP_ROOT | grep -E '^[0-9]{8}_[0-9]{6}$' | sort -r | head -1)

mkdir -p $BACKUP_ROOT

if [ -z "$LATEST" ]; then
    # First backup – full copy
    rsync -av $SOURCE/ $BACKUP_ROOT/$TIMESTAMP/
else
    # Incremental backup using previous as link destination
    rsync -av --link-dest=$BACKUP_ROOT/$LATEST $SOURCE/ $BACKUP_ROOT/$TIMESTAMP/
fi

# Optional: delete backups older than 30 days
find $BACKUP_ROOT -maxdepth 1 -type d -name "[0-9]*_[0-9]*" -mtime +30 -exec rm -rf {} \;
```

**Why this works:**

* `--link-dest` creates hard links to unchanged files instead of copying them

* Each backup directory is a complete snapshot – you can restore by simply copying any timestamped directory

* Space usage equals one full backup plus only the changed files for each incremental

**Restoration example:**

```bash
# Restore config from 2024-01-15 at 13:00
cp -r /backups/webapp/20240115_1300/* /opt/webapp/config/
```

**Alternative using** **`tar`:** You could create full `.tar.gz` backups each hour, but that would consume far more disk space. `tar` does not natively support incremental backups without using `--listed-incremental` (more complex). `rsync --link-dest` is simpler and more space-efficient.
