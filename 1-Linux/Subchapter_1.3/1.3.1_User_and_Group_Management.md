## 1.3.1 User and Group Management: The Foundation of Access Control

#### Why User Management Matters for Platform Engineering

Every process in Linux runs as a specific user. Containers, system daemons, cron jobs, and human users all have associated user IDs (UIDs) and group IDs (GIDs). Understanding user management allows you to:

* Enforce the **principle of least privilege** (services run as non-root)

* Audit who can do what on production systems

* Debug permission issues (e.g., "Why can't my web server write to /var/www/uploads?")

* Automate user provisioning across fleets of servers

#### The Three Critical Files

| File          | Purpose                                  | Permissions (from 1.2.2)          | Editable By |
| ------------- | ---------------------------------------- | --------------------------------- | ----------- |
| `/etc/passwd` | User account information (no passwords!) | `644` (world-readable)            | root        |
| `/etc/shadow` | Encrypted passwords and expiry info      | `640` or `000` (root/shadow only) | root        |
| `/etc/group`  | Group definitions and memberships        | `644` (world-readable)            | root        |

**Security principle:** Passwords are never stored in `/etc/passwd` (which must be world-readable for programs like `ls` to map UIDs to usernames). Instead, a placeholder `x` appears there, and the real hash lives in `/etc/shadow`, readable only by root.

***

## `/etc/passwd` – The User Database

Each line represents one user account with seven colon-separated fields.

```bash
# View the file
cat /etc/passwd

# Example entries
root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
ubuntu:x:1000:1000:Ubuntu,,,:/home/ubuntu:/bin/bash
nobody:x:65534:65534:nobody:/nonexistent:/usr/sbin/nologin
```

**Field breakdown:**

| Field # | Name                 | Example        | Meaning                                                           |
| ------- | -------------------- | -------------- | ----------------------------------------------------------------- |
| 1       | Username             | `ubuntu`       | Login name (must be unique)                                       |
| 2       | Password placeholder | `x`            | `x` means hash is in `/etc/shadow`; empty means no password       |
| 3       | UID                  | `1000`         | User ID (0 = root, 1-999 = system, 1000+ = human users typically) |
| 4       | GID                  | `1000`         | Primary group ID (from `/etc/group`)                              |
| 5       | GECOS                | `Ubuntu,,,`    | Comment field (full name, contact info)                           |
| 6       | Home directory       | `/home/ubuntu` | Where user starts after login                                     |
| 7       | Shell                | `/bin/bash`    | Login shell; `/sbin/nologin` disables interactive login           |

**System users (UID < 1000):** Created by packages to run daemons (e.g., `www-data` for nginx, `mysql` for MySQL). They typically have `/usr/sbin/nologin` as their shell and no valid password.

```bash
# List only system users (UID 1-999)
awk -F: '$3 >= 1 && $3 <= 999 {print $1 ":" $3}' /etc/passwd

# List human users (UID >= 1000)
awk -F: '$3 >= 1000 {print $1}' /etc/passwd
```

## `/etc/shadow` – Password Hashes and Policies

Each line corresponds to a user in `/etc/passwd` and contains password information in nine colon-separated fields.

```bash
sudo cat /etc/shadow

# Example entry
ubuntu:$6$random_salt$long_hash_value:19450:0:99999:7:::
```

**Field breakdown:**

| Field # | Name          | Example        | Meaning                                                         |
| ------- | ------------- | -------------- | --------------------------------------------------------------- |
| 1       | Username      | `ubuntu`       | Matches `/etc/passwd`                                           |
| 2       | Password hash | `$6$salt$hash` | `$6$` = SHA-512, `$y$` = yescrypt, `!` = locked, `*` = no login |
| 3       | Last change   | `19450`        | Days since 1970-01-01 that password was last changed            |
| 4       | Minimum days  | `0`            | Days before password can be changed again (0 = anytime)         |
| 5       | Maximum days  | `99999`        | Days until password must be changed (99999 = never)             |
| 6       | Warn days     | `7`            | Days before expiry to warn user                                 |
| 7       | Inactive days | (blank)        | Days after expiry before account is locked                      |
| 8       | Expire date   | (blank)        | Date when account becomes invalid (days since 1970)             |
| 9       | Reserved      | (blank)        | Unused                                                          |

**Locked accounts:** A `!` or `*` in the password field means the account cannot authenticate via password. This is how system accounts are disabled.

```bash
# Lock a user account (prevents password login)
sudo passwd -l username

# Unlock
sudo passwd -u username
```

## `/etc/group` – Group Memberships

Each line defines a group with four colon-separated fields.

```bash
cat /etc/group

# Example entries
root:x:0:
sudo:x:27:ubuntu,alice
docker:x:999:ubuntu
developers:x:1001:bob,charlie
```

**Field breakdown:**

| Field # | Name                 | Example       | Meaning                                          |
| ------- | -------------------- | ------------- | ------------------------------------------------ |
| 1       | Group name           | `developers`  | Unique group identifier                          |
| 2       | Password placeholder | `x`           | Rarely used; group passwords are legacy          |
| 3       | GID                  | `1001`        | Group ID (0 = root group)                        |
| 4       | Members              | `bob,charlie` | Comma-separated usernames (supplemental members) |

**Primary vs supplemental groups:**

* **Primary group** – Defined in `/etc/passwd` (field 4). Files created by the user inherit this group.

* **Supplemental groups** – Listed in `/etc/group`. User inherits these permissions in addition to primary group.

```bash
# See a user's groups (primary + supplemental)
groups ubuntu
# Output: ubuntu : ubuntu sudo docker

# Or using id command (more detailed)
id ubuntu
# Output: uid=1000(ubuntu) gid=1000(ubuntu) groups=1000(ubuntu),27(sudo),999(docker)
```

## User Management Commands

### Adding Users

```bash
# Basic user creation
sudo useradd alice

# Better: create with home directory and default skeleton files
sudo useradd -m -s /bin/bash alice

# Set password immediately
sudo passwd alice

# Create user with specific UID and group
sudo useradd -m -s /bin/bash -u 1500 -g developers alice

# Create system user (no home, no login shell, UID in system range)
sudo useradd -r -s /usr/sbin/nologin mydaemon
```

**Important** **`useradd`** **flags:**

| Flag | Long form       | Meaning                                         |
| ---- | --------------- | ----------------------------------------------- |
| `-m` | `--create-home` | Create home directory at `/home/username`       |
| `-s` | `--shell`       | Set login shell (e.g., `/bin/bash`, `/bin/zsh`) |
| `-u` | `--uid`         | Specify UID (must be unique)                    |
| `-g` | `--gid`         | Specify primary group (name or GID)             |
| `-G` | `--groups`      | Supplemental groups (comma-separated)           |
| `-c` | `--comment`     | GECOS field (full name)                         |
| `-r` | `--system`      | Create system user (UID < 1000)                 |
| `-d` | `--home-dir`    | Custom home directory path                      |

**Note:** On Debian/Ubuntu, `adduser` (interactive wrapper) is often preferred over low-level `useradd`. For scripts, use `useradd` for consistency across distributions.

### Modifying Users

```bash
# Change user's home directory
sudo usermod -d /new/home alice

# Change primary group
sudo usermod -g developers alice

# Add to supplemental groups (without removing existing)
sudo usermod -aG docker,sudo alice   # -a = append, very important!

# Change login shell
sudo usermod -s /bin/zsh alice

# Lock/unlock account
sudo usermod -L alice   # Lock (prefixes ! to password hash)
sudo usermod -U alice   # Unlock

# Change username
sudo usermod -l newusername oldusername
```

**Critical warning:** Forgetting `-a` when using `-G` will **replace** all existing supplemental groups, potentially locking the user out of critical groups like `sudo`.

```bash
# DANGEROUS: removes user from all groups except docker
sudo usermod -G docker alice

# CORRECT: adds docker while preserving existing groups
sudo usermod -aG docker alice
```

### Deleting Users

```bash
# Remove user but keep home directory and mail spool
sudo userdel alice

# Remove user and home directory, mail spool
sudo userdel -r alice

# Force removal even if user is logged in
sudo userdel -f alice
```

### Group Management

```bash
# Create group
sudo groupadd developers

# Create group with specific GID
sudo groupadd -g 1500 developers

# Delete group
sudo groupdel developers

# Add user to group (already covered via usermod)
sudo usermod -aG developers alice

# Remove user from group (edit /etc/group directly or use gpasswd)
sudo gpasswd -d alice developers

# Change group's GID (requires updating files owned by old GID)
sudo groupmod -g 1501 developers

# Change group name
sudo groupmod -n devteam developers
```

## Password Management with `passwd`

```bash
# Change your own password
passwd

# Root can change any user's password
sudo passwd alice

# Force password change on next login
sudo passwd -e alice

# Lock account (same as usermod -L)
sudo passwd -l alice

# Unlock
sudo passwd -u alice

# Set password expiry policies
sudo passwd -x 90 alice    # Maximum 90 days
sudo passwd -n 7 alice     # Minimum 7 days between changes
sudo passwd -w 14 alice    # Warn 14 days before expiry
```

## Understanding User IDs (UIDs) and Reserved Ranges

| UID Range   | Type             | Description                                               |
| ----------- | ---------------- | --------------------------------------------------------- |
| 0           | root             | Superuser – bypasses all permissions                      |
| 1-99        | System (static)  | Pre-allocated for system daemons (bin, daemon, sys, etc.) |
| 100-999     | System (dynamic) | Assigned by package installation for service accounts     |
| 1000-60000  | Regular users    | Human users (configurable in `/etc/login.defs`)           |
| 60000-65533 | Reserved         | Not typically used                                        |
| 65534       | nobody           | Unprivileged user (NFS, some daemons)                     |
| 65535       | 65535            | Some systems use for `nobody` fallback                    |

**Check your system's UID\_MIN and UID\_MAX:**

```bash
grep -E "^(UID_MIN|UID_MAX)" /etc/login.defs
```

## Real-World Platform Engineering Scenarios

### Scenario 1: Creating a Service Account for a Custom Application

```bash
# Create system user for a Java app
sudo useradd -r -s /usr/sbin/nologin -d /var/lib/myapp -m myapp

# The -m creates the home directory (even for system users)
# The -r ensures UID in system range (100-999)

# Verify
id myapp
# Output: uid=998(myapp) gid=998(myapp) groups=998(myapp)

# Run application as this user
sudo -u myapp java -jar /opt/myapp/app.jar
```

### Scenario 2: Auditing All Human Users on a Production Server

```bash
# List all human users with their home directories and shells
awk -F: '$3 >= 1000 && $3 < 60000 {print $1 ":" $6 ":" $7}' /etc/passwd

# Check when each user last logged in (using lastlog)
lastlog -u $(awk -F: '$3>=1000 {print $1}' /etc/passwd | tr '\n' ' ')
```

### Scenario 3: Batch Creating Users from a CSV File

```bash
# users.csv format: username,fullname,group
# alice,Alice Johnson,developers
# bob,Bob Smith,developers

while IFS=',' read -r username fullname group; do
    sudo useradd -m -s /bin/bash -c "$fullname" "$username"
    echo "$username:TempPass123" | sudo chpasswd
    sudo usermod -aG "$group" "$username"
    sudo passwd -e "$username"  # Force password change on first login
done < users.csv
```

## Quick Task: User Management Practice

*Create a safe test environment for practicing user commands.*

1. Create a new user named `testuser` with home directory and bash shell.
2. Set the password for `testuser` to `TempPass123`.
3. Create a group named `platform` and add `testuser` as a member (without removing other groups).
4. Verify the user's groups using `id testuser`.
5. Lock the `testuser` account. Try to switch to the user with `sudo -u testuser bash`. What happens?
6. Unlock the account and verify it works.
7. Delete the user and their home directory.

> **Ready Solution:**
>
> ```bash
> # Task 1
> sudo useradd -m -s /bin/bash testuser
>
> # Task 2
> echo "testuser:TempPass123" | sudo chpasswd
>
> # Task 3

>
> # Task 4

> # Result: sudo: unable to change to runas user: Permission denied

>
> # Task 7

> ```

## Summary Table: User Management Commands

| Command    | Purpose                | Common Flags                                  |
| ---------- | ---------------------- | --------------------------------------------- |
| `useradd`  | Create user            | `-m`, `-s`, `-u`, `-g`, `-G`, `-r`            |
| `usermod`  | Modify user            | `-aG`, `-d`, `-s`, `-l`, `-L`, `-U`           |
| `userdel`  | Delete user            | `-r` (remove home), `-f` (force)              |
| `groupadd` | Create group           | `-g` (specify GID)                            |
| `groupmod` | Modify group           | `-n` (rename), `-g` (change GID)              |
| `groupdel` | Delete group           | (none)                                        |
| `passwd`   | Set/change password    | `-l`, `-u`, `-e`, `-x`, `-n`, `-w`            |
| `gpasswd`  | Administer group       | `-a` (add), `-d` (delete), `-M` (set members) |
| `id`       | Display user/group IDs | (none)                                        |
| `groups`   | Show group memberships | (none)                                        |
