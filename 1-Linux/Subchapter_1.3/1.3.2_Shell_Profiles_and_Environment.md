## 1.3.2 Shell Profiles and Environment Variables

#### Why Shell Configuration Matters

Every time you log into a Linux system or open a terminal, your shell reads configuration files that set up your environment: the `PATH` variable that determines where commands are found, aliases for common commands, prompt appearance, and environment variables that influence program behavior (e.g., `EDITOR`, `JAVA_HOME`). Understanding the order in which these files are loaded is essential for:

* Debugging why a command is not found or an alias doesn't work

* Setting consistent environments for automation scripts (cron, systemd, CI/CD)

* Understanding the difference between interactive and non-interactive shells

***

## Types of Shells

Before understanding profiles, you must distinguish between shell types:

| Shell Type                | Description                                   | Examples                                                                       |
| ------------------------- | --------------------------------------------- | ------------------------------------------------------------------------------ |
| **Login shell**           | Requires authentication (username/password)   | SSH session, TTY login (`Ctrl+Alt+F2`), `su - username`                        |
| **Non-login shell**       | Started from an existing session              | Opening a terminal emulator in GUI, `bash` command from another shell, scripts |
| **Interactive shell**     | User can type commands, sees a prompt         | Most terminal sessions, SSH (unless command passed)                            |
| **Non-interactive shell** | Reads commands from a file or pipe, no prompt | Shell scripts, `bash -c "command"`                                             |

**Combinations matter:** A login shell is almost always interactive (except `ssh host command`). A non-login shell can be interactive (terminal emulator) or non-interactive (script).

```bash
# Check if shell is login shell
shopt -q login_shell && echo "Login shell" || echo "Not login shell"

# Check if shell is interactive
[[ $- == *i* ]] && echo "Interactive" || echo "Not interactive"
```

```mermaid
    F -->|Yes| G[/etc/bash.bashrc]
    G --> H[~/.bashrc]
    F -->|No| I[No user config files read]
    D --> J[Logout: ~/.bash_logout]

    style G fill:#9f9,stroke:#333
    style H fill:#9f9,stroke:#333
```

### For Bash (Most Common Shell)

| File               | When Read                                    | Typical Use                                        |
| ------------------ | -------------------------------------------- | -------------------------------------------------- |
| `/etc/profile`     | Login shells (system-wide)                   | System-wide `PATH`, `umask`, environment variables |
| `/etc/bash.bashrc` | Interactive non-login shells (system-wide)   | System-wide aliases, prompt settings               |
| `~/.bash_profile`  | Login shells (user-specific)                 | User's environment variables, startup commands     |
| `~/.bash_login`    | Login shells (if `~/.bash_profile` missing)  | Rare ŌĆō legacy                                      |
| `~/.profile`       | Login shells (if above missing)              | Cross-shell compatibility (read by sh, ksh, bash)  |
| `~/.bashrc`        | Interactive non-login shells (user-specific) | Aliases, functions, prompt, shell options          |
| `~/.bash_logout`   | When exiting login shell                     | Cleanup commands, clearing history                 |

**Critical point for platform engineers:** `~/.bashrc` is **not** read by login shells by default. SSH sessions (login shells) will not see aliases or functions defined in `~/.bashrc` unless you source it from `~/.bash_profile`.

```bash
# Common pattern in ~/.bash_profile (if exists)
if [ -f ~/.bashrc ]; then
    source ~/.bashrc
fi
```

## Environment Variables vs Shell Variables

| Type                     | Scope                           | Persistence               | Example                |
| ------------------------ | ------------------------------- | ------------------------- | ---------------------- |
| **Shell variable**       | Current shell only              | Lost when shell exits     | `myvar="hello"`        |
| **Environment variable** | Current shell + child processes | Inherited by subprocesses | `export MYVAR="hello"` |

```bash
# Shell variable (not exported)
MYVAR="local only"
echo $MYVAR          # Works
bash -c 'echo $MYVAR' # Empty ŌĆō child process doesn't see it

# Environment variable (exported)
export MYVAR="available everywhere"
bash -c 'echo $MYVAR' # "available everywhere"
```

### Viewing Variables

```bash
# Show all environment variables (exported)
env
printenv

# Show all variables (shell + environment)
set

# Show specific variable
echo $PATH
printenv HOME

# Show all variables containing "PATH"
set | grep -i path
```

### Common Environment Variables You Must Know

| Variable            | Purpose                               | Typical Value                                                  |
| ------------------- | ------------------------------------- | -------------------------------------------------------------- |
| `PATH`              | Directories to search for commands    | `/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin` |
| `HOME`              | User's home directory                 | `/home/username`                                               |
| `USER` or `LOGNAME` | Current username                      | `ubuntu`                                                       |
| `SHELL`             | Current shell binary                  | `/bin/bash`                                                    |
| `TERM`              | Terminal type                         | `xterm-256color`, `screen`, `vt100`                            |
| `LANG`              | Locale (language, character encoding) | `en_US.UTF-8`                                                  |
| `PWD`               | Current working directory             | `/var/log`                                                     |
| `OLDPWD`            | Previous working directory            | `/home/username`                                               |
| `EDITOR`            | Default text editor                   | `vim`, `nano`, `code --wait`                                   |
| `VISUAL`            | Visual editor (preferred over EDITOR) | `vim`                                                          |
| `PS1`               | Primary prompt string                 | `\u@\h:\w\$ `                                                  |
| `TMPDIR`            | Temporary directory for this user     | `/tmp` or `/var/tmp`                                           |

### Modifying PATH (Most Common Operation)

```bash
# Add directory to beginning (overrides system commands)
export PATH=/opt/myapp/bin:$PATH

# Add directory to end (fallback)
export PATH=$PATH:/opt/myapp/bin

# Remove a directory (complex, but possible)
export PATH=$(echo $PATH | tr ':' '\n' | grep -v '/bad/dir' | tr '\n' ':' | sed 's/:$//')
```

**Why order matters:** The shell searches directories in `PATH` from left to right. If `/usr/local/bin` appears before `/usr/bin`, a command in `/usr/local/bin` will be found first, even if the same name exists in `/usr/bin`.

```bash
# See which version of a command will be executed
which -a python
# /usr/local/bin/python
# /usr/bin/python
```

## Setting Environment Variables Permanently

### System-Wide (All Users)

| File                  | Purpose                                                                   |
| --------------------- | ------------------------------------------------------------------------- |
| `/etc/environment`    | Simple `KEY=value` format, read by PAM (all shells, even non-interactive) |
| `/etc/profile`        | Shell script, login shells only                                           |
| `/etc/bash.bashrc`    | Interactive non-login shells (bash only)                                  |
| `/etc/profile.d/*.sh` | Preferred system-wide location (each setting in its own file)             |

```bash
# Create system-wide environment variable
echo 'export MYAPP_HOME=/opt/myapp' | sudo tee /etc/profile.d/myapp.sh
sudo chmod +x /etc/profile.d/myapp.sh
```

### User-Specific

| File              | When Read                                         | Best For                                                               |
| ----------------- | ------------------------------------------------- | ---------------------------------------------------------------------- |
| `~/.bash_profile` | Login shells (SSH, TTY)                           | Environment variables, `PATH` modifications, running commands at login |
| `~/.bashrc`       | Interactive non-login shells (terminal emulators) | Aliases, functions, prompt, shell options                              |
| `~/.profile`      | Login shells (cross-shell, read by sh/dash)       | Simpler environment variables (avoid bash-specific syntax)             |

**Best practice for platform engineers:**

* Put **environment variables** and `PATH` modifications in `~/.profile` (works for all POSIX shells)

* Put **aliases, functions, and bash-specific settings** in `~/.bashrc`

* Source `~/.bashrc` from `~/.bash_profile` if you want the same environment everywhere

**Example** **`~/.profile`:**

```bash
# ~/.profile ŌĆō executed for login shells (bash, sh, dash)

# Set PATH for local user binaries
export PATH="$HOME/.local/bin:$PATH"

# Set editor
export EDITOR=vim
export VISUAL=vim

# Application paths
export JAVA_HOME=/usr/lib/jvm/java-11-openjdk-amd64
export PATH="$JAVA_HOME/bin:$PATH"
```

**Example** **`~/.bashrc`:**

```bash
# ~/.bashrc ŌĆō executed for interactive non-login shells

# Aliases
alias ll='ls -alF'
alias la='ls -A'
alias l='ls -CF'
alias grep='grep --color=auto'

# Prompt (\u=user, \h=hostname, \w=working dir, \$=# for root)
PS1='\u@\h:\w\$ '

# History settings
HISTSIZE=10000
HISTFILESIZE=20000
HISTCONTROL=ignoreboth:erasedups

# Custom functions
mkcd() {
    mkdir -p "$1" && cd "$1"
}
```

## Shell Options with `shopt` and `set`

Bash provides two mechanisms for controlling shell behavior: `shopt` (bash-specific) and `set` (POSIX-compatible).

### `shopt` ŌĆō Bash-Specific Options

```bash
# List all shopt options and their status
shopt

# List only enabled options
shopt -s

# Enable an option
shopt -s option_name

# Disable an option
shopt -u option_name
```

**Essential shopt options for platform engineers:**

| Option        | Default | Description                                                          |
| ------------- | ------- | -------------------------------------------------------------------- |
| `extglob`     | off     | Extended globbing patterns (`+(pattern)`, `?(pattern)`, etc.)        |
| `globstar`    | off     | `**` matches all files and directories recursively                   |
| `dotglob`     | off     | Globbing includes files starting with `.`                            |
| `nocaseglob`  | off     | Case-insensitive pathname expansion                                  |
| `cdspell`     | off     | Auto-correct minor typos in `cd` arguments                           |
| `histappend`  | off     | Append to history file instead of overwriting                        |
| `checkjobs`   | off     | Warn about running jobs before exiting shell                         |
| `autocd`      | off     | A directory name as command changes to that directory                |

```bash
# Enable useful options in ~/.bashrc
shopt -s extglob      # Extended globs: rm !(keep.txt) removes all except keep.txt
shopt -s globstar     # Recursive glob: ls **/*.py finds all Python files
shopt -s histappend   # Don't lose history when multiple terminals open
shopt -s cdspell      # cd /ect/nginx auto-corrects to /etc/nginx
```

### `set` ŌĆō POSIX Shell Options

```bash
# List all set options
set -o

# Enable an option
set -o option_name
set -flag        # Short form

# Disable an option
set +o option_name
set +flag        # Short form
```

**Critical `set` options for scripting (preview of Module 3):**

| Option/Flag | Description                                           |
| ----------- | ----------------------------------------------------- |
| `-e`        | Exit immediately if command exits with non-zero       |
| `-u`        | Treat unset variables as error                        |
| `-o pipefail` | Pipeline fails if any command fails                 |
| `-x`        | Print commands before execution (debugging)           |
| `-n`        | Read commands but don't execute (syntax check)        |

```bash
# Strict mode for scripts (recommended)
set -euo pipefail

# In interactive shells, be careful with -e (can exit accidentally)
# Better to use these only in scripts
```

## Readline Configuration with `~/.inputrc`

The readline library handles command-line editing in bash. Customize it via `~/.inputrc`.

```bash
# Create or edit ~/.inputrc
cat > ~/.inputrc << 'EOF'
# Case-insensitive tab completion
set completion-ignore-case on

# Show all completions on first tab (don't require double-tab)
set show-all-if-ambiguous on

# Append / to directories when completing
set mark-directories on

# Color completions by file type (like ls)
set colored-stats on

# Don't expand ~ to full path when completing
set expand-tilde off

# Treat hyphens and underscores as equivalent in completion
set completion-map-case on

# Vi mode (uncomment if preferred over emacs mode)
# set editing-mode vi

# Custom key bindings
"\e[A": history-search-backward  # Up arrow searches history
"\e[B": history-search-forward   # Down arrow searches history
EOF

# Reload without logging out
bind -f ~/.inputrc
```

**System-wide readline configuration:** `/etc/inputrc`

## `dircolors` and `LS_COLORS` Customization

The `LS_COLORS` environment variable controls how `ls` colorizes output.

```bash
# View current LS_COLORS (very long string)
echo $LS_COLORS

# View default database
dircolors --print-database

# Generate LS_COLORS from database
eval "$(dircolors -b)"

# Customize colors ŌĆō create ~/.dircolors
dircolors --print-database > ~/.dircolors
# Edit ~/.dircolors to customize

# Example customizations in ~/.dircolors:
# .log 00;33        # Log files in yellow
# .yaml 00;36       # YAML files in cyan
# .tf 00;32         # Terraform files in green

# Load custom colors in ~/.bashrc
if [ -f ~/.dircolors ]; then
    eval "$(dircolors -b ~/.dircolors)"
fi
```

**Color codes:**

| Code | Effect          |
| ---- | --------------- |
| 00   | Default         |
| 01   | Bold            |
| 04   | Underlined      |
| 31   | Red             |
| 32   | Green           |
| 33   | Yellow          |
| 34   | Blue            |
| 35   | Magenta         |
| 36   | Cyan            |

## XDG Base Directory Specification

Modern applications follow XDG standards for configuration and data storage (introduced in 1.2.1, expanded here).

```bash
# Key XDG environment variables
echo $XDG_CONFIG_HOME    # User config (~/.config if unset)
echo $XDG_DATA_HOME      # User data (~/.local/share if unset)
echo $XDG_CACHE_HOME     # User cache (~/.cache if unset)
echo $XDG_RUNTIME_DIR    # Runtime files (/run/user/UID)
echo $XDG_STATE_HOME     # State data (~/.local/state if unset)

# Set explicitly in ~/.profile for consistency
export XDG_CONFIG_HOME="$HOME/.config"
export XDG_DATA_HOME="$HOME/.local/share"
export XDG_CACHE_HOME="$HOME/.cache"
export XDG_STATE_HOME="$HOME/.local/state"

# Applications that respect XDG:
# - vim stores undo files in $XDG_STATE_HOME/vim/undo
# - nvim uses $XDG_CONFIG_HOME/nvim/init.vim
# - git can use $XDG_CONFIG_HOME/git/config
```

**Platform engineering benefit:** Cleaning up user environments is easier when apps follow XDG ŌĆō just `rm -rf ~/.cache/*` to clear all caches.

## The `/etc/default/` Directory

This directory contains default settings for system services and scripts.

```bash
ls /etc/default/
# console-setup  grub  keyboard  locale  networking  useradd  ...

# Example: /etc/default/useradd (defaults for useradd command)
cat /etc/default/useradd
# GROUP=100
# HOME=/home
# INACTIVE=-1
# EXPIRE=
# SHELL=/bin/sh
# SKEL=/etc/skel
# CREATE_MAIL_SPOOL=yes

# Example: /etc/default/grub (GRUB bootloader settings)
cat /etc/default/grub
# GRUB_TIMEOUT=5
# GRUB_CMDLINE_LINUX_DEFAULT="quiet splash"

# After modifying /etc/default/grub, regenerate config:
sudo update-grub
```

**How services use /etc/default:**

```bash
# Service init script typically sources its default file:
# /etc/init.d/myservice contains:
# [ -r /etc/default/myservice ] && . /etc/default/myservice

# Modern systemd services can use EnvironmentFile:
# [Service]
# EnvironmentFile=-/etc/default/myservice
```

## The `$BASH_ENV` Variable

For **non-interactive non-login shells** (scripts), bash reads `$BASH_ENV` if set.

```bash
# Set in ~/.profile to affect all scripts
export BASH_ENV=~/.bash_noninteractive

# Create the file with common script settings
cat > ~/.bash_noninteractive << 'EOF'
# Functions available to all scripts
log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" >&2
}

# Common paths
export SCRIPTS_DIR=/opt/scripts
EOF

# Now scripts run by this user have access to log() function
```

**Caution:** `$BASH_ENV` is evaluated for every script, which can slow down script execution and introduce unexpected behavior. Use sparingly.

**Example** **`~/.bash_profile`** **(simple):**

```bash
# ~/.bash_profile ŌĆō executed for login shells

# Source .profile for environment variables
if [ -f ~/.profile ]; then
    source ~/.profile
fi

# Source .bashrc for aliases and functions
if [ -f ~/.bashrc ]; then
    source ~/.bashrc
fi
```

## Aliases: Shortcuts for Common Commands

Aliases are shell shortcuts that replace a command with another string.

```bash
# List current aliases
alias

# Create alias (temporary)
alias ll='ls -la'

# Create alias with arguments (works naturally)
alias gs='git status'

# Remove alias
unalias ll

# Bypass alias (use command prefix)
\ll
command ll
/bin/ls -la
```

**Permanent aliases:** Put them in `~/.bashrc` (not `~/.profile` ŌĆō aliases are bash-specific).

```bash
# Useful aliases for platform engineers
alias dfh='df -h'
alias duh='du -sh'
alias psg='ps aux | grep'
alias tailf='tail -f'
alias reload='source ~/.bashrc'
alias v='vim'
alias ..='cd ..'
alias ...='cd ../..'
alias ports='netstat -tulanp' 2>/dev/null || ss -tulanp
```

***

## Functions vs Aliases

Functions are more powerful than aliases (can accept arguments, use conditionals, multiple commands).

```bash
# Function (in ~/.bashrc)
backup() {
    local src="$1"
    local dest="${src}.$(date +%Y%m%d).bak"
    cp -r "$src" "$dest"
    echo "Backed up $src to $dest"
}

# Usage
backup /etc/nginx

# Multi-line function with error handling
deploy() {
    local env="$1"
    if [[ -z "$env" ]]; then
        echo "Usage: deploy <environment>"
        return 1
    fi
    echo "Deploying to $env..."
    # actual deployment commands
}
```

## The `source` Command (`.`) ŌĆō Executing Files in Current Shell

`source` reads and executes commands from a file in the **current** shell (not a subshell). This is how profile changes take effect without logging out.

```bash
# Two equivalent syntaxes
source ~/.bashrc
. ~/.bashrc   # POSIX standard

# Why source matters without it:
export TEST=hello   # Set variable in current shell
./script.sh        # Runs in subshell ŌĆō changes don't persist
source script.sh   # Runs in current shell ŌĆō changes persist
```

**Practical use:** After editing `~/.bashrc`, run `source ~/.bashrc` to apply changes immediately.

### Testing with Minimal Environment: `env -i`

Use `env -i` to run a command with an empty (or near-empty) environment ŌĆō extremely useful for debugging and testing what happens in cron-like contexts:

```bash
# Run bash with completely empty environment
env -i bash --noprofile --norc -c 'echo "PATH is: $PATH"'
# Output: PATH is:     (empty!)

# Test a script with minimal environment (like cron)
env -i PATH=/usr/bin:/bin HOME=/home/user /opt/app/backup.sh

# Compare with current environment
echo "Current PATH: $PATH"
env -i bash -c 'echo "Empty PATH: $PATH"'
```

**Why this matters:** When debugging why scripts work manually but fail in cron, `env -i` lets you simulate the minimal environment.

## Real-World Platform Engineering Scenarios

### Scenario 1: Debugging PATH Issues in CI/CD

**Problem:** A CI pipeline script runs `terraform --version`, but the system reports `terraform: command not found`.

**Diagnosis:**

```bash
# In the CI script, add debugging
echo "Current PATH: $PATH"
which -a terraform
ls -la /usr/local/bin/terraform
```

**Fix:** Modify the CI job to set PATH explicitly or install terraform to a directory already in PATH.

```yaml
# GitHub Actions example
- name: Set PATH
  run: echo "/opt/terraform/bin" >> $GITHUB_PATH
```

### Scenario 2: Cron Jobs Missing Environment Variables

**Problem:** A cron job runs a script that works fine when executed manually but fails with `command not found` when run by cron.

**Why:** Cron runs with a minimal environment (only a few variables like `HOME`, `LOGNAME`, `SHELL`). It does **not** read `~/.profile` or `~/.bashrc`.

**Solutions:**

```bash
# Solution 1: Set PATH explicitly in crontab
# crontab -e
PATH=/usr/local/bin:/usr/bin:/bin
0 2 * * * /home/user/backup.sh

# Solution 2: Source profile inside script
#!/bin/bash
source ~/.profile
/usr/local/bin/myapp --backup

# Solution 3: Use absolute paths for all commands
0 2 * * * /usr/local/bin/python3 /home/user/script.py
```

### Scenario 3: Container Entrypoint Scripts

In Docker containers (Module 4), entrypoint scripts often need to source profiles to set up the environment.

```bash
#!/bin/bash
# entrypoint.sh
source /etc/profile
source ~/.bashrc

exec "$@"
```

## Quick Task: Profile Experimentation

*Create a test environment to understand profile loading order.*

1. Create a test user `proftest` with home directory.
2. As `proftest`, add unique echo statements to each profile file:

   * `~/.bash_profile`: `echo "Loading .bash_profile"`

   * `~/.profile`: `echo "Loading .profile"`

   * `~/.bashrc`: `echo "Loading .bashrc"`
3. Log in via SSH (or `su - proftest` for login shell). What order do you see?
4. Open a terminal emulator (non-login interactive shell). What order now?
5. Run `bash -c "echo test"` (non-interactive non-login). What profile files are loaded?

> **Ready Solution:**
>
> ```bash
> # Task 1-2
> sudo useradd -m -s /bin/bash proftest
> sudo -u proftest bash -c 'echo "echo \"Loading .bash_profile\"" > ~/.bash_profile'
> sudo -u proftest bash -c 'echo "echo \"Loading .profile\"" > ~/.profile'
> sudo -u proftest bash -c 'echo "echo \"Loading .bashrc\"" > ~/.bashrc'
>
> # Task 3 (login shell)
> sudo su - proftest
> # Output:
> # Loading .bash_profile
> # (if .bash_profile sources .profile and .bashrc, you see those too)
>
> # Task 4 (non-login interactive)
> sudo -u proftest bash
> # Output: Loading .bashrc
> # (.profile and .bash_profile NOT loaded)
>
> # Task 5 (non-interactive non-login)
> sudo -u proftest bash -c "echo test"
> # Output: test
> # No profile files loaded at all
> ```
