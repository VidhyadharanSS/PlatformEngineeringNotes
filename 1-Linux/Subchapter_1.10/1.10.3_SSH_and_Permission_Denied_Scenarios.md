## 1.10.3 SSH and Permission Denied Scenarios

#### Why SSH Troubleshooting Matters

SSH is the lifeline to production servers. When it fails, you are locked out. Understanding the complete authentication chain allows you to:

* Diagnose "Permission denied (publickey)" errors systematically

* Recover from broken SSH configurations

* Understand how SELinux affects SSH

* Troubleshoot key permission issues (from 1.4.2)

* Restore access when locked out

This note synthesizes SSH knowledge from Subchapter 1.4 with additional troubleshooting techniques for production emergencies.

***

## Part 2: Common SSH Error Messages and Causes

### Error 1: `ssh: connect to host hostname port 22: Connection refused`

**Cause:** SSH daemon not running or not listening on port 22.

**Diagnosis:**

```bash
# Check if SSH daemon is running (from console or out-of-band)
systemctl status sshd    # RHEL
systemctl status ssh     # Debian

# Check if port 22 is listening
sudo netstat -tlnp | grep :22
sudo ss -tlnp | grep :22

# Check firewall
sudo iptables -L -n | grep :22
sudo firewall-cmd --list-all   # firewalld
```

**Fix:**

```bash
# Start SSH daemon
sudo systemctl start sshd
sudo systemctl enable sshd

# Open firewall port
sudo firewall-cmd --add-service=ssh --permanent
sudo firewall-cmd --reload
```

### Error 2: `Connection timed out`

**Cause:** Network issue, firewall blocking, or host unreachable.

**Diagnosis:**

```bash
# Check basic connectivity
ping hostname

# Check if port is reachable (using nc or telnet)
nc -zv hostname 22
telnet hostname 22

# Trace route
traceroute hostname
mtr hostname
```

### Error 3: `WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED`

**Cause:** Server's host key has changed (OS reinstall, new SSH keys, man-in-the-middle attack).

**Diagnosis:**

```bash
# View stored host key
ssh-keygen -H -F hostname

# Compare with actual host key (from out-of-band)
ssh-keyscan hostname 2>/dev/null | ssh-keygen -l -f -
```

**Fix (if legitimate change):**

```bash
# Remove old host key
ssh-keygen -R hostname
# Or edit ~/.ssh/known_hosts manually

# Add new key on next connection (will prompt)
ssh hostname
```

### Error 4: `Permission denied (publickey)`

**Cause:** Key authentication failed. Multiple possible sub-causes.

**This is the most common and complex error.** See Part 3 for complete diagnosis.

### Error 5: `Permission denied (publickey,password)`

**Cause:** Both publickey and password authentication failed. Password authentication may be disabled in sshd\_config.

**Diagnosis:**

```bash
# Check server SSH configuration (if you have out-of-band access)
sudo grep -E "PasswordAuthentication|PubkeyAuthentication" /etc/ssh/sshd_config
```

### Error 6: `Authentication refused: bad ownership or modes`

**Cause:** Incorrect permissions on `~/.ssh`, `authorized_keys`, or private key files.

**Diagnosis:**

```bash
# Check SSH-related permissions
ls -la ~/.ssh
# Must be drwx------ (700)

ls -la ~/.ssh/authorized_keys
# Must be -rw------- (600)

ls -la ~/.ssh/id_*
# Private keys: -rw------- (600)
# Public keys: -rw-r--r-- (644)
```

**Fix:**

```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
chmod 600 ~/.ssh/id_* 2>/dev/null
chmod 644 ~/.ssh/*.pub 2>/dev/null
```

### Error 7: `Permission denied (publickey)` with SELinux

**Cause:** SELinux context on `~/.ssh` or `authorized_keys` is incorrect.

**Diagnosis:**

```bash
# Check SELinux status
getenforce
# Enforcing or Permissive

# Check SELinux context
ls -Z ~/.ssh
# Should be ssh_home_t or unconfined_u:object_r:ssh_home_t

# Check audit log for denials
sudo ausearch -m avc -ts recent | grep ssh
sudo grep ssh /var/log/audit/audit.log
```

**Fix:**

```bash
# Restore default SELinux context
restorecon -R -v ~/.ssh

# If still failing, temporarily set permissive for testing
sudo setenforce 0
# Test SSH, then re-enable
sudo setenforce 1
```

## Part 3: Systematic Public Key Authentication Debugging

### Debug Workflow (Client-Side)

```bash
# Step 1: Connect with maximum verbosity
ssh -vvv user@hostname

# Key lines to look for:
debug1: Authentications that can continue: publickey
debug1: Next authentication method: publickey
debug1: Offering public key: /home/user/.ssh/id_ed25519
debug1: Server accepts key: pkalg ssh-ed25519 blen 53
debug1: Authentication succeeded (publickey).

# If failing:
debug1: Authentications that can continue: publickey
debug1: Offering public key: /home/user/.ssh/id_ed25519
debug1: send_pubkey_test: no mutual signature algorithm
# → Key type not supported by server

debug1: Offering public key: /home/user/.ssh/id_ed25519
debug1: Authentications that can continue: publickey,password
# → Key offered but not accepted by server (not in authorized_keys or permissions wrong)
```

### Debug Workflow (Server-Side)

If you have out-of-band access (console, iLO, iDRAC, cloud console), check server logs:

```bash
# Debian/Ubuntu – SSH logs in syslog
sudo grep sshd /var/log/auth.log | tail -20

# RHEL/Rocky – SSH logs in secure
sudo grep sshd /var/log/secure | tail -20

# Any system with journald
sudo journalctl -u sshd -n 50

# Key successful authentication lines:
Accepted publickey for user from x.x.x.x port xxxxx ssh2: ED25519 SHA256:abc123

# Key failure lines:
Failed publickey for user from x.x.x.x port xxxxx ssh2: ED25519 SHA256:abc123
# Means key was offered but not in authorized_keys

# Permission issues:
Authentication refused: bad ownership or modes for file /home/user/.ssh/authorized_keys
```

### Complete Diagnosis Checklist

| Check                        | Command                                                      | Expected Result    |
| ---------------------------- | ------------------------------------------------------------ | ------------------ |
| SSH daemon running           | `systemctl status sshd`                                      | `active (running)` |
| Port listening               | `ss -tlnp \| grep :22`                                       | `LISTEN`           |
| Firewall allows              | `iptables -L -n \| grep :22`                                 | `ACCEPT`           |
| SELinux context              | `ls -Z ~/.ssh`                                               | `ssh_home_t`       |
| \~/.ssh permissions          | `ls -ld ~/.ssh`                                              | `drwx------`       |
| authorized\_keys permissions | `ls -l ~/.ssh/authorized_keys`                               | `-rw-------`       |
| Key in authorized\_keys      | `grep "$(cat ~/.ssh/id_ed25519.pub)" ~/.ssh/authorized_keys` | Match found        |
| SELinux (if enforcing)       | `sudo ausearch -m avc -ts recent \| grep ssh`                | No denials         |

## Part 4: Recovery When Locked Out

### Scenario A: You have console access (physical or cloud console)

```bash
# Access via cloud provider's web console or out-of-band management

# Step 1: Check SSH daemon status
systemctl status sshd

# Step 2: If not running, start it
systemctl start sshd

# Step 3: Check SSH configuration
cat /etc/ssh/sshd_config | grep -E "PermitRootLogin|PasswordAuthentication|PubkeyAuthentication"

# Step 4: Fix configuration if needed
sudo vi /etc/ssh/sshd_config
# Ensure:
# PermitRootLogin prohibit-password (or yes temporarily)
# PasswordAuthentication yes (temporarily for recovery)
# PubkeyAuthentication yes

# Step 5: Restart SSH
systemctl restart sshd

# Step 6: Add your public key if missing
mkdir -p ~/.ssh
chmod 700 ~/.ssh
echo "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI..." >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys

# Step 7: Fix SELinux if needed
restorecon -R ~/.ssh
```

### Scenario B: You have root password via console

```bash
# Log in as root via console

# Add a new user with SSH access
useradd -m -s /bin/bash tempadmin
passwd tempadmin
mkdir -p /home/tempadmin/.ssh
echo "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI..." > /home/tempadmin/.ssh/authorized_keys
chmod 700 /home/tempadmin/.ssh
chmod 600 /home/tempadmin/.ssh/authorized_keys
chown -R tempadmin:tempadmin /home/tempadmin/.ssh

# Test SSH as tempadmin, then disable password if desired
```

### Scenario C: Rescue mode from boot (cloud or VM)

Many cloud providers offer rescue mode or recovery console:

```bash
# Boot into rescue mode
# Mount the root filesystem
mount /dev/sda1 /mnt

# Chroot into the system
chroot /mnt /bin/bash

# Now fix SSH as above
vi /etc/ssh/sshd_config
systemctl enable sshd
exit
reboot
```

### Scenario D: Single-user mode (physical or VM)

```bash
# At GRUB, add to kernel command line:
single
# or
init=/bin/bash

# System boots to root shell
# Remount root as read-write
mount -o remount,rw /

# Fix SSH configuration or add keys
# Reboot
reboot -f
```

## Part 5: SELinux and AppArmor SSH Issues

### Ubuntu/Debian: AppArmor

AppArmor is Ubuntu's mandatory access control system (similar to SELinux on RHEL).

**Checking AppArmor status:**

```bash
# Check if AppArmor is active
sudo aa-status

# Output shows profiles in enforce/complain mode
# apparmor module is loaded.
# 50 profiles are loaded.
# 50 profiles are in enforce mode.

# Check if SSH is affected by AppArmor
sudo aa-status | grep ssh
```

**Diagnosing AppArmor SSH issues:**

```bash
# Check AppArmor logs for denials
sudo journalctl | grep apparmor | grep DENIED | tail -20

# Or check audit log
sudo dmesg | grep apparmor | grep DENIED
```

**Temporarily disable AppArmor for testing:**

```bash
# Set SSH profile to complain mode (logs but doesn't block)
sudo aa-complain /etc/apparmor.d/usr.sbin.sshd

# Test SSH connection

# Re-enable enforce mode
sudo aa-enforce /etc/apparmor.d/usr.sbin.sshd
```

### RHEL/Rocky: SELinux

### Common SELinux Contexts for SSH

| File/Directory           | Correct Context | Command to Fix                      |
| ------------------------ | --------------- | ----------------------------------- |
| `~/.ssh`                 | `ssh_home_t`    | `restorecon -R ~/.ssh`              |
| `~/.ssh/authorized_keys` | `ssh_home_t`    | `restorecon ~/.ssh/authorized_keys` |
| `~/.ssh/id_rsa`          | `ssh_home_t`    | `restorecon ~/.ssh/id_rsa`          |
| `/etc/ssh/sshd_config`   | `etc_t`         | `restorecon /etc/ssh/sshd_config`   |
| `/var/empty/sshd`        | `var_t`         | `restorecon /var/empty/sshd`        |

### Diagnosing SELinux SSH Denials

```bash
# Check for SELinux denials
sudo ausearch -m avc -ts recent | grep sshd

# Example denial:
type=AVC msg=audit(1234567890.123:456): avc:  denied  { read } for  pid=1234 comm="sshd" name="authorized_keys" dev="dm-0" ino=12345 scontext=system_u:system_r:sshd_t tcontext=unconfined_u:object_r:home_root_t tclass=file

# This means ~/.ssh/authorized_keys has wrong context (home_root_t instead of ssh_home_t)

# Fix with restorecon
restorecon -R /home/user/.ssh

# If restorecon doesn't fix, manually set
chcon -R -t ssh_home_t /home/user/.ssh
```

### Temporarily Disable SELinux (for testing only)

```bash
# Set to permissive (logs but doesn't enforce)
sudo setenforce 0

# Test SSH
ssh user@host

# If works, fix contexts and re-enable
sudo setenforce 1
```

## Part 6: Key Restriction Issues

### Common `authorized_keys` Restrictions That Break Access

| Restriction                | Effect                   | Symptom                                         |
| -------------------------- | ------------------------ | ----------------------------------------------- |
| `command="/bin/backup.sh"` | Only runs that command   | Connection closes immediately after command     |
| `from="192.168.1.0/24"`    | Only allows specific IPs | Connection from wrong IP gets permission denied |
| `no-pty`                   | No terminal allocation   | `ssh user@host` fails with "not a terminal"     |
| `expiry-time="20240101"`   | Key expires on date      | Permission denied after expiry                  |

**Diagnosis:**

```bash
# Check authorized_keys for restrictions
cat ~/.ssh/authorized_keys

# Look for lines starting with:
# command=, from=, no-pty, no-agent-forwarding, etc.

# Remove or modify restrictions as needed
vi ~/.ssh/authorized_keys
```

### Key Revocation (When Key is Compromised)

```bash
# Remove the compromised key from authorized_keys
vi ~/.ssh/authorized_keys
# Delete the line containing the compromised public key

# Or comment it out
# ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAI... compromised-key

# Check all user's authorized_keys
sudo find /home -name authorized_keys -exec grep -l "compromised-fingerprint" {} \;
```

## Part 7: SSH Configuration Audit Script

```bash
#!/bin/bash
# ssh_audit.sh – Check SSH configuration for common issues

echo "=== SSH Daemon Status ==="
systemctl is-active sshd 2>/dev/null || systemctl is-active ssh 2>/dev/null

echo -e "\n=== SSH Listening Port ==="
ss -tlnp | grep -E ":(22|ssh)" | head -1

echo -e "\n=== Key Authentication Settings ==="
grep -E "^PubkeyAuthentication|^PasswordAuthentication|^PermitRootLogin" /etc/ssh/sshd_config 2>/dev/null

echo -e "\n=== SELinux Status ==="
getenforce 2>/dev/null || echo "SELinux not installed"

echo -e "\n=== User SSH Directory Permissions ==="
for user in $(ls /home); do
    if [ -d "/home/$user/.ssh" ]; then
        perm=$(ls -ld "/home/$user/.ssh" | awk '{print $1}')
        if [ "$perm" != "drwx------" ]; then
            echo "WARNING: /home/$user/.ssh has bad permissions: $perm"
        fi
    fi
done

echo -e "\n=== Root SSH Access ==="
grep "^PermitRootLogin" /etc/ssh/sshd_config 2>/dev/null || echo "PermitRootLogin not explicitly set (default: prohibit-password)"

echo -e "\n=== Recent SSH Auth Failures ==="
sudo journalctl -u sshd -n 20 2>/dev/null | grep -i "failed\|denied" | tail -5
```

## Quick Task: SSH Troubleshooting Simulation

*Practice diagnosing common SSH failures.*

1. Simulate permission error: Change `~/.ssh` permissions to `755` and try SSH to localhost. Observe error.
2. Fix permissions and test again.
3. Simulate missing key: Rename `~/.ssh/authorized_keys` and attempt SSH. Observe error.
4. Restore and test.
5. (If SELinux enabled) Simulate context issue: Change context on `~/.ssh` and test, then fix.

> **Ready Solution:**
>
> ```bash
> # Task 1: Simulate permission error
> chmod 755 ~/.ssh
> ssh localhost
> # Error: Authentication refused: bad ownership or modes for directory /home/user/.ssh
>
> # Task 2: Fix permissions
> chmod 700 ~/.ssh
> ssh localhost  # Should work
>
> # Task 3: Simulate missing key
> mv ~/.ssh/authorized_keys ~/.ssh/authorized_keys.bak
> ssh localhost
> # Error: Permission denied (publickey)
>
> # Task 4: Restore
> mv ~/.ssh/authorized_keys.bak ~/.ssh/authorized_keys
> ssh localhost  # Should work
>
> # Task 5: SELinux context issue (if SELinux is enforcing)
> sudo chcon -R -t home_root_t ~/.ssh
> ssh localhost
> # Error: Permission denied (publickey)
> # Check audit log: sudo ausearch -m avc -ts recent | grep ssh
> sudo restorecon -R ~/.ssh
> ssh localhost  # Should work
> ```

## Summary Table: SSH Error Diagnosis

| Error Message                   | Most Likely Cause           | Diagnostic Command      | Fix                         |
| ------------------------------- | --------------------------- | ----------------------- | --------------------------- |
| `Connection refused`            | sshd not running            | `systemctl status sshd` | `systemctl start sshd`      |
| `Connection timed out`          | Firewall/network            | `nc -zv host 22`        | Check firewall/network      |
| `Host key changed`              | Server reinstalled          | `ssh-keygen -R host`    | Remove old key              |
| `Permission denied (publickey)` | Key not in authorized\_keys | `ssh -vvv`              | Add key to authorized\_keys |
| `Bad ownership or modes`        | Wrong permissions           | `ls -la ~/.ssh`         | `chmod 700 ~/.ssh`          |
| `SELinux denial`                | Wrong context               | `ausearch \| grep ssh`  | `restorecon -R ~/.ssh`      |

### SSH Debugging Command Quick Reference

| Command                            | Purpose                       |
| ---------------------------------- | ----------------------------- |
| `ssh -vvv user@host`               | Maximum verbosity client-side |
| `sudo journalctl -u sshd -n 50`    | Server-side logs (systemd)    |
| `sudo grep sshd /var/log/auth.log` | Server logs (Debian)          |
| `sudo grep sshd /var/log/secure`   | Server logs (RHEL)            |
| `sudo ausearch -m avc \| grep ssh` | SELinux denials               |
| `restorecon -R ~/.ssh`             | Fix SELinux context           |
| `chmod 700 ~/.ssh`                 | Fix directory permissions     |
| `ssh-keyscan hostname`             | Get remote host key           |
