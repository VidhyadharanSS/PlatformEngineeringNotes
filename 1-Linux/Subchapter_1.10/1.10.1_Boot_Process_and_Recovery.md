## 1.10.1 Boot Process and Recovery: From Power-On to Login

#### Why Boot Recovery Matters

When a server fails to boot, you are often under extreme pressure to restore service. Understanding each stage of the boot process allows you to:

* Diagnose where the failure occurs (BIOS, bootloader, kernel, initramfs, systemd)

* Boot into recovery modes (single-user, emergency, rescue)

* Repair broken bootloaders (GRUB)

* Recover from corrupted filesystems or misconfigured fstab

* Reset lost root passwords

This note covers the **complete Linux boot sequence** and recovery techniques. Note 1.10.2 covers disk full scenarios, 1.10.3 covers SSH troubleshooting, and 1.10.4 is the final review with comprehensive exam.

## Part 1: The Linux Boot Process (6 Stages)

```mermaid
graph TD
    A[1. BIOS/UEFI] -->|Loads bootloader| B[2. Bootloader GRUB]
    B -->|Loads kernel + initramfs| C[3. Kernel]
    C -->|Mounts initramfs| D[4. initramfs]
    D -->|Mounts real root| E[5. systemd (PID 1)]
    E -->|Starts targets| F[6. Login Prompt]
    
    style A fill:#f99,stroke:#333
    style B fill:#ff9,stroke:#333
    style C fill:#9f9,stroke:#333
    style D fill:#9cf,stroke:#333
    style E fill:#cf9,stroke:#333
    style F fill:#9c9,stroke:#333
```

### Stage 1: BIOS/UEFI (Power-On Self-Test)

| Aspect              | BIOS (Legacy)              | UEFI (Modern)                |
| ------------------- | -------------------------- | ---------------------------- |
| Partition table     | MBR                        | GPT                          |
| Bootloader location | First sector of disk (MBR) | EFI System Partition (FAT32) |
| Disk limit          | 2 TB                       | 9.4 ZB                       |
| Boot speed          | Slower                     | Faster                       |

**Common failures:**

* No bootable device found → Check boot order in BIOS/UEFI

* CMOS battery failure → BIOS settings reset, date/time wrong

### Stage 2: Bootloader (GRUB)

GRUB (Grand Unified Bootloader) is the most common Linux bootloader.

**GRUB configuration:** `/boot/grub2/grub.cfg` (RHEL) or `/boot/grub/grub.cfg` (Debian)

**Common GRUB failures:**

* `grub rescue>` prompt → Bootloader corrupted or partition moved

* Kernel panic immediately after GRUB → Wrong kernel or initramfs

### Stage 3: Kernel

The kernel is loaded into memory and initializes:

* CPU, memory, interrupt handlers

* Basic device drivers (built-in)

* Mounts initramfs as temporary root

**Kernel command line parameters** (can be edited in GRUB):

| Parameter                        | Effect                                                 |
| -------------------------------- | ------------------------------------------------------ |
| `single` or `1`                  | Boot to single-user mode (root shell, no network)      |
| `emergency`                      | Boot to emergency mode (minimal, before most services) |
| `rescue`                         | Boot to rescue target (basic services)                 |
| `rd.break`                       | Break before initramfs hands over to systemd           |
| `systemd.unit=multi-user.target` | Boot to specific target                                |
| `init=/bin/bash`                 | Use bash as init (bypass systemd)                      |

### Stage 4: initramfs (Initial RAM Filesystem)

initramfs is a temporary root filesystem loaded into RAM that:

* Contains necessary kernel modules (disk drivers, filesystem drivers, LVM)

* Mounts the real root filesystem

* Hands control to systemd

**Regenerating initramfs:**

```bash
# Debian/Ubuntu
sudo update-initramfs -u

# RHEL/Rocky
sudo dracut --force
```

**Common initramfs failures:**

* `ALERT! UUID=... does not exist` → Wrong UUID in fstab or kernel command line

* `mdadm: No arrays found` → RAID configuration missing

* `LVM: Volume group not found` → LVM not activated

### Stage 5: systemd (PID 1)

systemd mounts the real root from initramfs and starts targets.

**Boot targets:**

| Target              | Equivalent Runlevel | Services Started     |
| ------------------- | ------------------- | -------------------- |
| `poweroff.target`   | 0                   | Shutdown             |
| `rescue.target`     | 1                   | Single-user, minimal |
| `multi-user.target` | 3                   | Multi-user, no GUI   |
| `graphical.target`  | 5                   | Multi-user with GUI  |
| `reboot.target`     | 6                   | Reboot               |

### Stage 6: Login Prompt

Display Manager (graphical) or getty (text) presents login prompt.

## Part 2: GRUB Recovery

### Accessing GRUB Menu

```bash
# During boot, press (usually):
# - Esc (BIOS systems)
# - Shift (hold during boot for GRUB menu)
# - F8 or F10 (some UEFI systems)
```

### Editing GRUB at Boot

1. At GRUB menu, press `e` to edit the boot entry
2. Find the line starting with `linux` or `linuxefi`
3. Add recovery parameters at the end of the line
4. Press `Ctrl+X` or `F10` to boot

**Example: Add** **`rd.break`** **to break before initramfs:**

```
linux /vmlinuz-5.14.0-... root=UUID=... ro rd.break
```

### GRUB Rescue Shell (When GRUB Can't Find Its Files)

```bash
# At grub rescue> prompt:

# Step 1: Find the partition containing /boot
ls
# (hd0) (hd0,msdos1) (hd0,msdos2)

# Step 2: Check each partition
ls (hd0,msdos1)/
# If you see grub/, vmlinuz*, initrd*, that's the boot partition

# Step 3: Set root and prefix
set root=(hd0,msdos1)
set prefix=(hd0,msdos1)/grub

# Step 4: Load normal module
insmod normal
normal

# Step 5: Now in normal GRUB, boot normally or repair
```

### Reinstalling GRUB (From Live CD/USB)

```bash
# Boot from live Linux USB

# Step 1: Identify disk and partitions
lsblk
# Assume /dev/sda is system disk, root partition is /dev/sda2, boot partition (if separate) is /dev/sda1

# Step 2: Mount system
sudo mount /dev/sda2 /mnt
sudo mount /dev/sda1 /mnt/boot   # If separate boot partition
sudo mount --bind /dev /mnt/dev
sudo mount --bind /proc /mnt/proc
sudo mount --bind /sys /mnt/sys

# Step 3: Chroot into system
sudo chroot /mnt

# Step 4: Reinstall GRUB
# For BIOS
grub-install /dev/sda
update-grub        # Debian/Ubuntu
grub2-mkconfig -o /boot/grub2/grub.cfg   # RHEL/Rocky

# For UEFI
grub-install --target=x86_64-efi --efi-directory=/boot/efi --bootloader-id=GRUB
update-grub

# Step 5: Exit chroot and reboot
exit
sudo umount -R /mnt
sudo reboot
```

## Part 3: Recovery Modes

### Single-User Mode (Runlevel 1)

Single-user mode boots to a root shell with minimal services. Network is usually disabled.

**Access methods:**

```bash
# Method 1: At GRUB, add to kernel command line
single
# or
1

# Method 2: From working system, set next boot to single-user
sudo systemctl isolate rescue.target
# or for next boot only
sudo systemctl set-default rescue.target
# Reboot, then change back
sudo systemctl set-default multi-user.target
```

**Use cases:**

* Reset root password

* Fix fstab errors

* Repair filesystems

* Disable broken services

### Emergency Mode (Minimal)

Even more minimal than single-user – only root filesystem mounted read-only.

**Access:**

```
# At GRUB, add:
emergency
```

**Emergency mode tasks:**

```bash
# Remount root as read-write
mount -o remount,rw /

# Check filesystem
fsck /dev/sda2

# Fix fstab
vi /etc/fstab

# Exit to continue boot
exit
```

### Rescue Mode (Systemd)

Rescue mode is similar to single-user but with systemd's rescue target.

```bash
# From working system
sudo systemctl rescue

# At GRUB
systemd.unit=rescue.target
```

### Chroot – The Ultimate Recovery Tool

Chroot changes the root directory for a process, allowing you to work on a broken system from a live environment.

**Full chroot procedure:**

```bash
# Boot from live USB

# 1. Mount the broken system's root
sudo mkdir /mnt/root
sudo mount /dev/sda2 /mnt/root

# 2. Mount necessary virtual filesystems
sudo mount --bind /dev /mnt/root/dev
sudo mount --bind /proc /mnt/root/proc
sudo mount --bind /sys /mnt/root/sys

# 3. If /boot is separate
sudo mount /dev/sda1 /mnt/root/boot

# 4. If UEFI, mount EFI partition
sudo mount /dev/sda1 /mnt/root/boot/efi

# 5. Chroot
sudo chroot /mnt/root /bin/bash

# 6. Set up environment
source /etc/profile
export PS1="(chroot) $PS1"

# 7. Now you can run recovery commands as if on the broken system
# Examples:
passwd root
grub-install /dev/sda
update-grub
systemctl enable my-service

# 8. Exit and unmount
exit
sudo umount -R /mnt/root
sudo reboot
```

## Part 4: Common Boot Failures and Fixes

### Failure 1: "Kernel panic – not syncing: VFS: Unable to mount root fs"

**Cause:** Kernel can't find root filesystem (wrong UUID, missing driver)

**Fix from GRUB:**

```bash
# At GRUB, press 'e' to edit
# Find the 'root=UUID=...' parameter
# Replace with correct device (temporary)
root=/dev/sda2

# Or list available UUIDs
# In initramfs shell (rd.break), run:
blkid
# Copy correct UUID and update kernel command line
```

**Permanent fix:**

```bash
# Update GRUB configuration
# Get correct UUID
sudo blkid /dev/sda2
# Update /etc/default/grub
GRUB_CMDLINE_LINUX="root=UUID=correct-uuid"
# Regenerate GRUB config
sudo update-grub   # Debian
sudo grub2-mkconfig -o /boot/grub2/grub.cfg   # RHEL
```

### Failure 2: "Give root password for maintenance" or "Entering emergency mode"

**Cause:** fstab error (missing partition, wrong UUID, wrong filesystem type)

**Fix:**

```bash
# At emergency shell (root password may be required)
# Remount root as read-write
mount -o remount,rw /

# Check fstab
cat /etc/fstab

# Comment out suspicious lines (add # at beginning)
vi /etc/fstab

# Test mounting all entries
mount -a

# Exit to continue boot
exit
```

### Failure 3: GRUB Rescue Shell

**Cause:** GRUB modules missing or partition moved

**Fix:**

```bash
# At grub rescue>
# Find boot partition
ls
ls (hd0,msdos1)/grub
ls (hd0,msdos2)/grub

# Set boot partition
set root=(hd0,msdos1)
set prefix=(hd0,msdos1)/grub

# Load normal mode
insmod normal
normal

# Then from GRUB menu, boot normally and reinstall GRUB
```

### Failure 4: "Failed to start Load Kernel Modules"

**Cause:** Missing or corrupted kernel modules, often after kernel update without initramfs rebuild

**Fix:**

```bash
# Boot to single-user or from live CD
# Rebuild initramfs
sudo dracut --force   # RHEL
sudo update-initramfs -u   # Debian

# Reinstall kernel if needed
sudo dnf reinstall kernel   # RHEL
sudo apt install --reinstall linux-image-$(uname -r)   # Debian
```

### Failure 5: "Dependency failed for /mount/point"

**Cause:** fstab entry for a mount point fails (network drive offline, disk missing)

**Fix:**

```bash
# Boot to emergency mode
# Comment out failing mount in fstab
vi /etc/fstab
# Add 'nofail' option to prevent future boot failures
# Example: UUID=... /data ext4 defaults,nofail 0 2
```

## Part 5: Resetting Root Password

### Method 1: Single-User Mode (If GRUB is accessible)

```bash
# At GRUB, press 'e' to edit
# Find the 'linux' line, add at the end:
rd.break

# Press Ctrl+X to boot
# You'll be dropped into initramfs shell

# Remount sysroot as read-write
mount -o remount,rw /sysroot

# Chroot into sysroot
chroot /sysroot

# Reset root password
passwd root

# If SELinux is enabled, create autorelabel file
touch /.autorelabel

# Exit and reboot
exit
exit
reboot
```

### Method 2: Live CD/USB

```bash
# Boot from live Linux USB
# Mount the root partition
sudo mount /dev/sda2 /mnt

# Chroot
sudo chroot /mnt

# Reset password
passwd root

# Exit and reboot
exit
sudo reboot
```

## Part 6: Boot Performance Analysis

```bash
# Show boot time
systemd-analyze time

# Show which services take longest
systemd-analyze blame
# 5.234s docker.service
# 2.345s cloud-init.service
# 1.234s network.service

# Show critical chain (services that delayed boot)
systemd-analyze critical-chain

# Generate boot chart (SVG)
systemd-analyze plot > boot.svg

# Show kernel and initrd times
systemd-analyze timestamp
```

## Summary Table: Boot Stages and Recovery

| Stage | Component | Recovery Method                          |
| ----- | --------- | ---------------------------------------- |
| 1     | BIOS/UEFI | Check boot order, reset CMOS             |
| 2     | GRUB      | Rescue shell, reinstall from live CD     |
| 3     | Kernel    | Edit GRUB command line, different kernel |
| 4     | initramfs | `rd.break`, regenerate initramfs         |
| 5     | systemd   | Emergency/single-user mode, fix fstab    |
| 6     | Login     | Reset root password, fix PAM             |

### Boot Parameters Quick Reference

| Parameter                        | Effect                                    |
| -------------------------------- | ----------------------------------------- |
| `single` or `1`                  | Single-user mode                          |
| `emergency`                      | Emergency mode (minimal)                  |
| `rescue`                         | Rescue target                             |
| `rd.break`                       | Break before initramfs handoff            |
| `init=/bin/bash`                 | Use bash as init                          |
| `systemd.unit=multi-user.target` | Boot to specific target                   |
| `quiet`                          | Suppress kernel messages                  |
| `nomodeset`                      | Disable graphics drivers (for GPU issues) |

### Recovery Commands Quick Reference

| Command                                  | Purpose                                 |
| ---------------------------------------- | --------------------------------------- |
| `mount -o remount,rw /`                  | Remount root as read-write              |
| `fsck /dev/sda2`                         | Check filesystem                        |
| `grub-install /dev/sda`                  | Reinstall GRUB                          |
| `update-grub` / `grub2-mkconfig`         | Regenerate GRUB config                  |
| `update-initramfs -u` / `dracut --force` | Regenerate initramfs                    |
| `chroot /mnt`                            | Change root to mounted system           |
| `systemctl rescue`                       | Boot to rescue mode from running system |
