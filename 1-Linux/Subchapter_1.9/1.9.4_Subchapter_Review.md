## Interview Questions (Scenario-Based)

These questions assume only knowledge from Subchapter 1.9. Answers reference only concepts from 1.9.1, 1.9.2, and 1.9.3.

### Question 1

**Scenario:** You are running a long database migration script on a production server over SSH. Your internet connection is unstable and frequently drops. The migration cannot be restarted if interrupted.

**Question:** How would you ensure the migration continues even after SSH disconnects? Provide the complete solution using terminal multiplexing.

**Answer:**

**Solution:** Use `screen` or `tmux` to run the migration in a persistent session that survives SSH disconnection.

**Using screen:**

```bash
# Step 1: Start a named screen session
screen -S db_migration

# Step 2: Run the migration script
./migrate_database.sh

# Step 3: Detach from session (Ctrl+A d)
# Screen session continues running in background

# Step 4: If SSH disconnects, reattach later
screen -r db_migration

# Step 5: Monitor progress (can reattach from anywhere)
screen -ls                    # List sessions
screen -x db_migration        # Attach to session (even if attached elsewhere)
```

**Using tmux:**

```bash
# Step 1: Start a named tmux session
tmux new -s db_migration

# Step 2: Run the migration script
./migrate_database.sh

# Step 3: Detach (Ctrl+B d)

# Step 4: Reattach after disconnect
tmux attach -t db_migration
```

**Best practices for this scenario:**

1. **Run in a dedicated session:**

   ```bash
   screen -S migration_2024_01_15
   ```

2. **Log all output:**

   ```bash
   # Inside screen session
   script migration.log
   ./migrate_database.sh
   exit
   # Or redirect output
   ./migrate_database.sh 2>&1 | tee migration.log
   ```

3. **Use verbose mode for visibility:**

   ```bash
   ./migrate_database.sh --verbose
   ```

4. **Set up notification on completion:**

   ```bash
   ./migrate_database.sh && echo "Migration complete" | mail -s "Migration Status" admin@example.com
   ```

5. **Prevent accidental detach by changing prefix (optional in tmux):**

   ```bash
   # In ~/.tmux.conf
   set-option -g prefix C-a
   unbind-key C-b
   bind-key C-a send-prefix
   ```

**Verification:**

```bash
# After reattaching, check progress
tail -f migration.log

# Verify process is still running
ps aux | grep migrate_database
```

### Question 2

**Scenario:** A web server is not responding. You suspect something is already using port 80. You need to identify what process is listening on port 80 and terminate it if necessary.

**Question:** What commands would you use to identify the process, and how would you terminate it gracefully? Provide the complete workflow.

**Answer:**

**Identify the process using port 80:**

```bash
# Method 1: Using lsof
sudo lsof -i :80
# Output:
# COMMAND  PID  USER   FD   TYPE DEVICE SIZE/OFF NODE NAME
# nginx   1234  root    6u  IPv4  12345      0t0  TCP *:http (LISTEN)

# Method 2: Using ss (from Module 2, but available now)
sudo ss -tlnp | grep :80
# LISTEN 0 128 0.0.0.0:80 0.0.0.0:* users:(("nginx",pid=1234,fd=6))

# Method 3: Using netstat
sudo netstat -tlnp | grep :80

# Method 4: Using fuser
sudo fuser 80/tcp
# 80/tcp: 1234
```

**Extract PID:**

```bash
# Using lsof with -t (terse, PID only)
sudo lsof -t -i :80
# 1234

# Store in variable
PID=$(sudo lsof -t -i :80)
echo $PID
```

**Terminate the process gracefully:**

```bash
# Step 1: Try graceful termination (SIGTERM)
sudo kill -15 $PID    # Or kill $PID (default is SIGTERM)

# Step 2: Check if process still exists
ps -p $PID

# Step 3: If still running, try more forceful (SIGINT)
sudo kill -2 $PID

# Step 4: Last resort (SIGKILL) – avoid unless necessary
sudo kill -9 $PID

# Step 5: Verify port is free
sudo lsof -i :80
# Should return nothing
```

**Complete one-liner to kill process on port 80:**

```bash
# Graceful kill
sudo kill $(sudo lsof -t -i :80)

# Force kill (if graceful fails)
sudo kill -9 $(sudo lsof -t -i :80)

# Safer: check before killing
if PID=$(sudo lsof -t -i :80); then
    echo "Killing process $PID"
    sudo kill $PID
else
    echo "No process found on port 80"
fi
```

**If process won't die (stuck in uninterruptible sleep):**

```bash
# Check process state
ps aux | grep $PID
# STAT column shows 'D' for uninterruptible

# Check what it's waiting on
sudo cat /proc/$PID/stack
sudo strace -p $PID

# May need to reboot if truly stuck
# Or reset the GPU/device if it's a hardware issue
```

### Question 3

**Scenario:** A production server has high disk I/O and you suspect a process is writing large amounts of data. You need to identify which process and which file is being written to.

**Question:** Using tools from this subchapter, how would you identify the offending process and file? Provide the commands and explain what each shows.

**Answer:**

**Step 1: Identify top I/O processes using lsof (point-in-time):**

```bash
# List open files sorted by size (largest first)
sudo lsof | awk '{print $7, $9}' | sort -rn | head -20
# $7 = size, $9 = filename

# Count open files per process (high FD count may indicate file leak)
sudo lsof | awk '{print $2}' | sort | uniq -c | sort -rn | head -10
```

**Step 2: Find processes with many writes using sysdig (if installed):**

```bash
# Top processes by file I/O
sudo sysdig -c top_file

# Spy on file writes in real-time
sudo sysdig -c spy_file

# Show all write operations
sudo sysdig "evt.type=write" -c echo_fds
```

**Step 3: Identify large or rapidly growing files:**

```bash
# Find files modified in last 10 minutes, sorted by size
find /var -type f -mmin -10 -exec ls -lh {} \; 2>/dev/null | sort -k5 -rh | head -10

# Watch directory for changes
watch -n 1 'ls -la /var/log/ | sort -k5 -rh | head -10'
```

**Step 4: Use lsof to see what a specific process is writing:**

```bash
# Once you have suspect PID
sudo lsof -p 1234 | grep -E "(REG|DIR)"
# Shows all regular files and directories opened by PID 1234
```

**Step 5: Monitor specific file growth in real-time:**

```bash
# Watch a specific file
watch -n 1 'ls -lh /var/log/huge.log'

# Check if file is deleted but still open (space not freed)
sudo lsof | grep deleted
```

**Complete diagnostic script:**

```bash
#!/bin/bash
echo "=== Top 10 Processes by Open File Count ==="
sudo lsof | awk '{print $2}' | sort | uniq -c | sort -rn | head -10 | while read count pid; do
    cmd=$(ps -p $pid -o comm= 2>/dev/null)
    echo "$count files: PID $pid ($cmd)"
done

echo -e "\n=== Large Files Modified Recently ==="
find /var/log -type f -size +100M -mmin -30 -exec ls -lh {} \; 2>/dev/null

echo -e "\n=== Deleted Files Still Open ==="
sudo lsof 2>/dev/null | grep deleted | head -10
```

### Question 4

**Scenario:** You manage a GPU cluster for machine learning training. Users report that their training jobs fail with "CUDA out of memory" errors, but `nvidia-smi` shows available memory. The server has 8 GPUs (Tesla V100, 32GB each).

**Question:** How would you investigate and resolve this issue? What commands would you use to monitor GPU usage and identify the root cause?

**Answer:**

**Investigation steps:**

**Step 1: Check overall GPU status:**

```bash
nvidia-smi
# Look for:
# - Memory usage per GPU
# - Running processes
# - Persistence mode status
# - Temperature (overheating can cause throttling)
```

**Step 2: Identify which GPUs are busy and by whom:**

```bash
# Show processes per GPU
nvidia-smi pmon -c 1

# Query specific GPU
nvidia-smi --query-compute-apps=pid,used_memory,gpu_uuid --format=csv

# Sample output:
# pid, used_gpu_memory [MiB], gpu_uuid
# 12345, 28192, GPU-abc123
# 12346, 28144, GPU-abc123
# Total would exceed 32GB!
```

**Step 3: Check if multiple processes are using the same GPU:**

```bash
# Group by GPU
for gpu in $(nvidia-smi --query-gpu=index --format=csv,noheader); do
    echo "GPU $gpu:"
    nvidia-smi --query-compute-apps=gpu_uuid,used_memory --format=csv | grep "GPU-$gpu" || echo "  No processes"
done
```

**Step 4: Check for zombie GPU processes:**

```bash
# Processes that are gone but GPU memory not freed
nvidia-smi | grep -E "python|java" | awk '{print $5}' | while read pid; do
    if ! ps -p $pid > /dev/null 2>&1; then
        echo "Stale PID $pid still showing in nvidia-smi"
    fi
done
```

**Resolutions:**

**Option 1: Kill offending processes**

```bash
# Kill specific PID
kill -9 12345

# Kill all python processes on GPU 0
nvidia-smi | grep "python" | grep "Default" | awk '{print $5}' | xargs kill -9
```

**Option 2: Reset the GPU (if processes won't die)**

```bash
# Reset GPU 0 (requires root)
sudo nvidia-smi -i 0 -r
```

**Option 3: Enable Exclusive Compute Mode (prevent future over-subscription)**

```bash
# Check current mode
nvidia-smi --query-gpu=compute_mode --format=csv

# Set to exclusive mode (one process per GPU)
sudo nvidia-smi -i 0 -c 1

# Modes:
# 0 = Default (multiple processes share)
# 1 = Exclusive (one process per GPU)
# 2 = Prohibited (no compute)
# 3 = Exclusive Process (one process tree per GPU)
```

**Option 4: Implement GPU quotas (using MIG or Kubernetes):**

```bash
# For NVIDIA MIG (if supported by GPU)
sudo nvidia-smi mig -i 0 -cgi 19,19,19,19 -C  # Create 4 instances
```

**Option 5: Enable persistence mode for consistent behavior:**

```bash
sudo nvidia-smi -pm 1
sudo systemctl enable nvidia-persistenced
sudo systemctl start nvidia-persistenced
```

**Monitoring script for production:**

```bash
#!/bin/bash
# gpu_monitor.sh

THRESHOLD_MEM=90   # 90% memory usage alert
THRESHOLD_TEMP=85  # 85°C alert

while true; do
    for gpu in 0 1 2 3 4 5 6 7; do
        MEM_USED=$(nvidia-smi --id=$gpu --query-gpu=memory.used --format=csv,noheader | tr -d ' ')
        MEM_TOTAL=$(nvidia-smi --id=$gpu --query-gpu=memory.total --format=csv,noheader | tr -d ' ')
        MEM_PERCENT=$((MEM_USED * 100 / MEM_TOTAL))
        
        TEMP=$(nvidia-smi --id=$gpu --query-gpu=temperature.gpu --format=csv,noheader)
        
        if [ $MEM_PERCENT -gt $THRESHOLD_MEM ]; then
            echo "WARNING: GPU $gpu memory ${MEM_PERCENT}%"
            nvidia-smi --id=$gpu --query-compute-apps=pid,used_memory --format=csv
        fi
        
        if [ $TEMP -gt $THRESHOLD_TEMP ]; then
            echo "WARNING: GPU $gpu temperature ${TEMP}°C"
        fi
    done
    sleep 60
done
```

### Question 5

**Scenario:** A system administrator accidentally deleted `/var/log/messages` while `rsyslog` was still writing to it. The disk space is still showing as used (`df` shows no free space increase). The server has critical services running and cannot be rebooted.

**Question:** How would you recover the disk space without restarting any services? Provide the complete recovery procedure.

**Answer:**

**Problem explanation:** When a file is deleted but still open by a process, the space is not freed until the process closes the file or terminates. The file's inode remains, and the process holds a reference to it.

**Recovery procedure:**

**Step 1: Identify the deleted file still in use:**

```bash
# Find deleted open files
sudo lsof | grep deleted | grep /var/log/messages

# Output example:
# rsyslogd  1234  root    4w   REG   8,17  1048576 12345 /var/log/messages (deleted)
```

**Step 2: Verify the process and file descriptor:**

```bash
# Show details
sudo lsof -p 1234 | grep deleted

# Check /proc for the file descriptor
ls -la /proc/1234/fd/
# lr-x------ 1 root root 64 Jan 16 10:00 4 -> /var/log/messages (deleted)
```

**Step 3: Recover the file content (if needed):**

```bash
# Recover before truncating
sudo cp /proc/1234/fd/4 /tmp/messages.recovered

# Or view content
sudo cat /proc/1234/fd/4 | tail -100
```

**Step 4: Truncate the file (free space without killing process):**

```bash
# Method 1: Truncate via /proc
sudo truncate -s 0 /proc/1234/fd/4

# Method 2: Redirect null
sudo sh -c 'echo > /proc/1234/fd/4'

# Method 3: Use dd
sudo dd if=/dev/null of=/proc/1234/fd/4
```

**Step 5: Verify space was freed:**

```bash
# Check disk space
df -h /var

# Verify no more deleted references
sudo lsof | grep deleted | grep /var/log/messages
# Should return nothing
```

**Step 6: Restart rsyslog to create new log file (optional but recommended):**

```bash
sudo systemctl restart rsyslog

# Verify new file created
ls -la /var/log/messages
# -rw-r----- 1 root adm 0 Jan 16 10:05 /var/log/messages
```

**Complete automated script for this scenario:**

```bash
#!/bin/bash
# recover_deleted_space.sh

LOG_FILE="/var/log/messages"
SERVICE="rsyslog"

# Find process holding deleted file
INFO=$(sudo lsof | grep deleted | grep "$LOG_FILE")
if [ -z "$INFO" ]; then
    echo "No deleted file found for $LOG_FILE"
    exit 0
fi

# Extract PID and FD
PID=$(echo "$INFO" | awk '{print $2}')
FD=$(echo "$INFO" | awk '{print $4}' | sed 's/[^0-9]//g')

echo "Found process $PID holding deleted $LOG_FILE on FD $FD"

# Recover content
sudo cp "/proc/$PID/fd/$FD" "/tmp/$(basename $LOG_FILE).recovered"
echo "Recovered content to /tmp/$(basename $LOG_FILE).recovered"

# Truncate the file
sudo truncate -s 0 "/proc/$PID/fd/$FD"
echo "Truncated deleted file, space freed"

# Restart service
sudo systemctl restart $SERVICE
echo "Restarted $SERVICE"

# Verify
df -h /var | grep -v Filesystem
sudo lsof | grep deleted | grep "$LOG_FILE" && echo "WARNING: Still have deleted file" || echo "Success: No deleted file remaining"
```

**Why this works:**

* The process's `/proc/PID/fd/FD` is a symlink to the open file (even if deleted)

* Writing to this symlink writes to the open file handle

* `truncate -s 0` empties the file, releasing disk space

* The process continues running uninterrupted

* After restart, a new log file is created normally

**Prevention for the future:**

```bash
# Use logrotate with copytruncate option
cat /etc/logrotate.d/rsyslog
/var/log/messages {
    copytruncate   # Truncates after copy instead of moving
    daily
    rotate 7
    compress
}
```
