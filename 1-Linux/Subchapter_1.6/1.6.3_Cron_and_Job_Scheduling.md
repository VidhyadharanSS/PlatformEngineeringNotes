## 1.6.3 Cron and Job Scheduling: Automating Recurring Tasks

#### Why Cron Matters

Cron is the traditional time-based job scheduler on Unix-like systems. Despite systemd timers (covered in 1.6.2), cron remains ubiquitous for:

* Running maintenance tasks (log rotation, database backups, certificate renewal)

* Scheduling monitoring checks

* Automating batch processing (nightly reports, data syncs)

* Running user-specific scheduled tasks (without root)

For platform engineers, cron is still the standard for simple recurring tasks across nearly all Linux distributions, especially in legacy environments or minimal containers.

### Cron Components

| Component        | Location                                      | Purpose                                |
| ---------------- | --------------------------------------------- | -------------------------------------- |
| `crond` daemon   | System service                                | Reads crontabs, executes jobs          |
| System crontab   | `/etc/crontab`                                | System-wide jobs (specify user)        |
| Cron directories | `/etc/cron.d/`                                | Package-specific crontabs              |
| User crontabs    | `/var/spool/cron/crontabs/`                   | Per-user jobs (edit with `crontab -e`) |
| Cron scripts     | `/etc/cron.hourly/`, `/etc/cron.daily/`, etc. | Simpler drop-in scripts                |

## Part 2: Crontab Syntax

### The 5 Time Fields

```
* * * * * command_to_execute
│ │ │ │ │
│ │ │ │ └─── Day of week (0-7, 0 or 7 = Sunday)
│ │ │ └───── Month (1-12)
│ │ └─────── Day of month (1-31)
│ └───────── Hour (0-23)
└─────────── Minute (0-59)
```

### Field Values and Special Characters

| Character | Meaning                              | Example                                              |
| --------- | ------------------------------------ | ---------------------------------------------------- |
| `*`       | Any value (every minute, hour, etc.) | `* * * * *` = every minute                           |
| `*/N`     | Every N units                        | `*/15 * * * *` = every 15 minutes                    |
| `N`       | Specific value                       | `30 2 * * *` = 2:30 AM daily                         |
| `N-M`     | Range                                | `9-17 * * * *` = hours 9 through 17                  |
| `N,M`     | List                                 | `1,15,30 * * * *` = minutes 1,15,30                  |
| `N-M/X`   | Range with step                      | `0-30/5 * * * *` = every 5 min, first 30 min of hour |

### Common Scheduling Examples

```bash
# Every minute
* * * * * /script.sh

# Every 5 minutes
*/5 * * * * /script.sh

# Every hour (at minute 0)
0 * * * * /script.sh

# Every day at 2:30 AM
30 2 * * * /script.sh

# Every Monday at 5 AM
0 5 * * 1 /script.sh

# Weekdays (Mon-Fri) at 9 AM
0 9 * * 1-5 /script.sh

# First day of every month at midnight
0 0 1 * * /script.sh

# Every Sunday at midnight (0 or 7 for Sunday)
0 0 * * 0 /script.sh

# Every 6 hours (at 0, 6, 12, 18)
0 */6 * * * /script.sh

# At 8:30 AM and 8:30 PM
30 8,20 * * * /script.sh
```

### Command Field

The command can be:

* Any executable with full path (recommended)

* Shell built-in (less reliable)

* Multiple commands separated by `;` or `&&`

* Redirected output: `>/dev/null 2>&1` to suppress emails

```bash
# Multiple commands
30 2 * * * /usr/bin/backup.sh && /usr/bin/verify.sh

# With output redirection
0 * * * * /usr/bin/monitor.sh > /var/log/monitor.log 2>&1

# With environment variables
0 2 * * * PATH=/usr/local/bin:/usr/bin /opt/app/job.sh
```

## Part 3: User Crontabs (`crontab` command)

### Managing User Crontabs

```bash
# Edit current user's crontab
crontab -e

# Edit specific user's crontab (root only)
sudo crontab -u alice -e

# List current user's crontab
crontab -l

# List specific user's crontab (root only)
sudo crontab -u alice -l

# Remove all cron jobs for current user
crontab -r

# Remove specific user's crontab (root only)
sudo crontab -u alice -r
```

### Example User Crontab

```bash
# Example crontab for user alice
# Format: MIN HOUR DOM MON DOW COMMAND

# Environment variable (affects all jobs)
MAILTO=alice@example.com
SHELL=/bin/bash
PATH=/usr/local/bin:/usr/bin:/bin

# Backup home directory daily at 1 AM
0 1 * * * /usr/bin/rsync -av /home/alice/ /backup/alice/

# Clean temp files every Sunday at 2 AM
0 2 * * 0 find /tmp -type f -atime +7 -delete

# Check disk space every hour
0 * * * * df -h / > /tmp/disk_usage.txt

# Notification when script fails (using &&
30 3 * * * /usr/local/bin/daily_report.sh || echo "Report failed" | mail -s "Cron failure" alice
```

### Default Editor for `crontab -e`

```bash
# Set editor for current session
export EDITOR=vim
crontab -e

# Set permanently (add to ~/.bashrc)
echo "export EDITOR=vim" >> ~/.bashrc

# Alternative: select-editor (Debian/Ubuntu)
sudo select-editor
```

## Part 4: System Crontabs

### `/etc/crontab` – System-wide (specifies user)

```bash
# /etc/crontab
SHELL=/bin/bash
PATH=/usr/local/sbin:/usr/local/bin:/sbin:/bin:/usr/sbin:/usr/bin
MAILTO=root

# Minute Hour Day Month Day-of-week User Command
25 6 * * * root test -x /usr/sbin/anacron || ( cd / && run-parts --report /etc/cron.daily )
47 6 * * 7 root test -x /usr/sbin/anacron || ( cd / && run-parts --report /etc/cron.weekly )
52 6 1 * * root test -x /usr/sbin/anacron || ( cd / && run-parts --report /etc/cron.monthly )
```

**Difference from user crontab:** The `/etc/crontab` file has an **extra field** (the user field) between time and command.

### `/etc/cron.d/` – Drop-in System Crontabs

```bash
# /etc/cron.d/logrotate
SHELL=/bin/sh
PATH=/usr/local/sbin:/usr/local/bin:/sbin:/bin:/usr/sbin:/usr/bin

# Run logrotate hourly
0 * * * * root /usr/sbin/logrotate /etc/logrotate.conf

# /etc/cron.d/backup
30 1 * * * backup /opt/backup_script.sh
```

### `/etc/cron.hourly/`, `/etc/cron.daily/`, etc. – Script Directories

These directories contain scripts that run at specific intervals via `run-parts`.

```bash
# List scripts in daily directory
ls -la /etc/cron.daily/
# -rwxr-xr-x 1 root root 1234 Jan 15 apt-compat
# -rwxr-xr-x 1 root root 5678 Jan 15 logrotate

# Add custom daily script
sudo tee /etc/cron.daily/backup << 'EOF'
#!/bin/bash
/usr/local/bin/backup.sh
EOF
sudo chmod +x /etc/cron.daily/backup
```

**Execution times (default on Debian/Ubuntu):**

* Hourly: 17 minutes past each hour

* Daily: 6:25 AM

* Weekly: 6:47 AM on Sunday

* Monthly: 6:52 AM on 1st day

## Part 5: Cron Environment and Logging

### Cron's Minimal Environment

Cron provides a minimal environment (different from interactive shell):

```bash
# Default cron environment (typical)
HOME=/home/alice
LOGNAME=alice
PATH=/usr/bin:/bin
SHELL=/bin/sh
```

**Why this matters:**

* `PATH` is limited – use full paths (`/usr/bin/rsync` instead of `rsync`)

* `SHELL` is `/bin/sh`, not `/bin/bash` (some bash features unavailable)

* No interactive profile files (`~/.bashrc`, `~/.profile`) are read

**Solutions:**

```bash
# Solution 1: Set PATH in crontab
PATH=/usr/local/bin:/usr/bin:/bin
0 2 * * * backup_script.sh

# Solution 2: Use full paths
0 2 * * * /usr/local/bin/backup_script.sh

# Solution 3: Source profile in script
#!/bin/bash
source /home/alice/.profile
/usr/local/bin/backup_script.sh
```

### Cron Logging

```bash
# Check if cron is running
systemctl status cron   # Debian/Ubuntu
systemctl status crond  # RHEL/Rocky

# View cron logs (syslog)
sudo grep CRON /var/log/syslog      # Debian/Ubuntu
sudo grep CRON /var/log/cron        # RHEL/Rocky

# Follow cron logs
sudo tail -f /var/log/syslog | grep CRON

# Check for errors
sudo journalctl -u cron -p err
```

### Preventing Overlapping Jobs with `flock`

A common production problem is a job starting again before the previous run finishes. `flock` solves this with a lock file.

```bash
# Skip new run if lock is already held
0 1 * * * /usr/bin/flock -n /tmp/backup.lock /opt/backup/backup.sh

# Exit with non-zero status if another instance is running
*/5 * * * * /usr/bin/flock -n /tmp/report.lock /usr/local/bin/report.sh
```

Use this for backups, sync jobs, or anything that must not overlap.

### Mail Output from Cron

By default, cron emails output (stdout/stderr) to the user. If mail is not configured, output goes to syslog.

```bash
# Suppress all output (no email)
0 2 * * * backup.sh > /dev/null 2>&1

# Log to file instead
0 2 * * * backup.sh >> /var/log/backup.log 2>&1

# Send email to specific address (set in crontab)
MAILTO=admin@example.com
0 2 * * * backup.sh
```

## Part 6: Anacron – For Systems Not Always On

Anacron runs jobs even if the system was off during scheduled time. It's used for laptops, desktops, or systems that are not 24/7.

```bash
# Check if anacron is installed
which anacron

# Anacron configuration (/etc/anacrontab)
# Period in days Delay-in-minutes Job-identifier Command
1    5    cron.daily    run-parts /etc/cron.daily
7    10   cron.weekly   run-parts /etc/cron.weekly
30   15   cron.monthly  run-parts /etc/cron.monthly
```

**How anacron works:**

* Runs at boot or via cron

* Checks when each job last ran

* Runs job if period has elapsed (e.g., 1 day for daily)

* Delay randomizes execution time

**Note:** Anacron cannot run jobs at exact times (e.g., 2:30 AM) – only daily, weekly, monthly intervals.

## Part 7: Troubleshooting Cron

### Problem 1: Job Doesn't Run

```bash
# 1. Check cron daemon is running
systemctl status cron

# 2. Check crontab syntax
crontab -l

# 3. Check cron logs
sudo grep CRON /var/log/syslog | tail -20

# 4. Check script permissions
ls -l /path/to/script.sh
# Must be readable and executable

# 5. Run script manually to test
/path/to/script.sh

# 6. Check PATH in script or use full paths
which command_name   # Get full path
```

### Problem 2: Environment Variable Missing

```bash
# Debug: Capture cron environment
# In crontab:
* * * * * env > /tmp/cron_env.txt

# Compare with interactive environment
env > /tmp/shell_env.txt
diff /tmp/cron_env.txt /tmp/shell_env.txt
```

### Problem 3: Script Runs But Fails Silently

```bash
# Redirect output for debugging
0 2 * * * /path/to/script.sh > /tmp/cron_debug.log 2>&1

# Add logging to script
#!/bin/bash
exec >> /var/log/myscript.log 2>&1
echo "$(date): Starting script"
# ... rest of script
```

### Problem 4: "Permission denied" in Crontab

```bash
# Ensure crontab file permissions are correct
ls -la /var/spool/cron/crontabs/alice
# Must be 600 (-rw-------)

# Fix permissions
sudo chmod 600 /var/spool/cron/crontabs/alice
```

## Part 8: Cron vs Systemd Timers

| Feature                   | Cron                          | Systemd Timers                |
| ------------------------- | ----------------------------- | ----------------------------- |
| Syntax                    | Simple 5-field time           | Calendar events, monotonic    |
| Second precision          | No (minute minimum)           | Yes                           |
| Run after missed schedule | No (anacron for daily/weekly) | Yes (with `Persistent=true`)  |
| Environment control       | Limited (set in crontab)      | Full unit file control        |
| Logging                   | Via syslog/mail               | Via journald (structured)     |
| Dependencies              | No                            | Yes (network, other services) |
| Resource limits           | No                            | Yes (CPU, memory via cgroups) |
| Complexity                | Low                           | Medium                        |

**When to use cron:**

* Simple, traditional environments

* User-specific schedules without root

* Legacy systems

* One-liner commands

**When to use systemd timers:**

* Need precise scheduling (seconds, missed runs)

* Complex dependencies (e.g., run after network is up)

* Integration with other systemd features (logging, resource limits)

* Modern systemd-based distributions (most today)

## Quick Task: Create Cron Jobs

*Practice creating and managing cron jobs.*

1. Create a script `/tmp/cron_test.sh` that appends the current date to `/tmp/cron_output.txt`.
2. Add a crontab entry to run this script every 5 minutes.
3. Verify it's running by checking `/tmp/cron_output.txt` after 10 minutes.
4. List your crontab and then remove the job.
5. (Optional) Add a system crontab entry to `/etc/cron.d/test` that runs as root every hour.

> **Ready Solution:**
>
> ```bash
> # Task 1
> tee /tmp/cron_test.sh << 'EOF'
> #!/bin/bash
> echo "$(date): Cron job ran" >> /tmp/cron_output.txt
> EOF
> chmod +x /tmp/cron_test.sh
>
> # Task 2
> (crontab -l 2>/dev/null; echo "*/5 * * * * /tmp/cron_test.sh") | crontab -
>
> # Task 3
> # Wait 10 minutes, then:
> cat /tmp/cron_output.txt
> # Output (example):
> # Mon Jan 15 10:00:01 UTC 2024: Cron job ran
> # Mon Jan 15 10:05:01 UTC 2024: Cron job ran
>
> # Task 4
> crontab -l
> crontab -r
>
> # Task 5 (optional)
> sudo tee /etc/cron.d/test << 'EOF'
> # Run every hour at minute 0
> 0 * * * * root /tmp/cron_test.sh
> EOF
> sudo systemctl restart cron
> ```

### Crontab Time Field Reference

| Field        | Range              | Special                   |
| ------------ | ------------------ | ------------------------- |
| Minute       | 0-59               | `*/N` for every N minutes |
| Hour         | 0-23               | `*/N` for every N hours   |
| Day of month | 1-31               | `1-15` for range          |
| Month        | 1-12               | `1,6,12` for list         |
| Day of week  | 0-7 (0=Sun, 7=Sun) | `1-5` for weekdays        |

### Common Cron Patterns

| Desired Schedule      | Cron Expression |
| --------------------- | --------------- |
| Every minute          | `* * * * *`     |
| Every 5 minutes       | `*/5 * * * *`   |
| Every hour (minute 0) | `0 * * * *`     |
| Every day at 2:30 AM  | `30 2 * * *`    |
| Every Sunday at 5 AM  | `0 5 * * 0`     |
| Weekdays at 9 AM      | `0 9 * * 1-5`   |
| First day of month    | `0 0 1 * *`     |
| Every 6 hours         | `0 */6 * * *`   |
