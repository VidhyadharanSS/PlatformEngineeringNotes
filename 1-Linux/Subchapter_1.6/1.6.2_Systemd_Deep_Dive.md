## 1.6.2 Systemd Deep Dive: The Modern Init System

#### Why Systemd Matters

Systemd is the first process (PID 1) on virtually every modern Linux distribution. It replaced legacy init systems (SysV init, Upstart) and is now the standard for:

* Booting the system (starting services in parallel)

* Managing services (starting, stopping, restarting, enabling at boot)

* Logging (via `journald`)

* Timers (replacing cron for many use cases)

* Socket activation (starting services on-demand)

* Resource control (cgroups – covered in Module 4)

For platform engineers, understanding systemd is essential for writing production service definitions, debugging startup failures, and managing container runtimes (Docker, containerd, kubelet all run as systemd services).

## Part 2: Systemctl – Managing Services

### Basic Service Operations

```bash
# View status of a service
systemctl status sshd
# Output:
# ● sshd.service - OpenSSH server daemon
#    Loaded: loaded (/usr/lib/systemd/system/sshd.service; enabled; preset: enabled)
#    Active: active (running) since Mon 2024-01-15 10:00:00 UTC; 2h ago
#      Docs: man:sshd(8)
#  Main PID: 1234 (sshd)
#     Tasks: 1 (limit: 23456)
#    Memory: 2.5M
#       CPU: 100ms
#    CGroup: /system.slice/sshd.service
#            └─1234 /usr/sbin/sshd -D

# Start a service
sudo systemctl start nginx

# Stop a service
sudo systemctl stop nginx

# Restart (stop then start)
sudo systemctl restart nginx

# Reload (re-read config without restarting – not all services support this)
sudo systemctl reload nginx

# Enable at boot (creates symlink in /etc/systemd/system/multi-user.target.wants/)
sudo systemctl enable nginx

# Disable at boot
sudo systemctl disable nginx

# Check if enabled at boot
systemctl is-enabled nginx
# enabled, disabled, static, masked

# Check if running
systemctl is-active nginx
# active, inactive, failed, activating

# Mask a service (prevent manual or automatic start – stronger than disable)
sudo systemctl mask nginx
sudo systemctl unmask nginx
```

### Understanding `systemctl status` Output

| Field      | Meaning                              | Example                                                  |
| ---------- | ------------------------------------ | -------------------------------------------------------- |
| `Loaded`   | Unit file location and enable status | `loaded (/usr/lib/systemd/system/sshd.service; enabled)` |
| `Active`   | Running state and timestamp          | `active (running) since Mon 2024-01-15 10:00:00 UTC`     |
| `Main PID` | Primary process ID                   | `1234 (sshd)`                                            |
| `Tasks`    | Number of threads/processes          | `1 (limit: 23456)`                                       |
| `Memory`   | Memory usage                         | `2.5M`                                                   |
| `CGroup`   | Control group path                   | `/system.slice/sshd.service`                             |

### Listing Services

```bash
# List all active units (services, sockets, timers, etc.)
systemctl list-units

# List all service units (including inactive)
systemctl list-units --type=service --all

# List failed services
systemctl list-units --state=failed

# List enabled services
systemctl list-unit-files --state=enabled

# List all unit files (not just loaded)
systemctl list-unit-files --type=service
```

## Part 3: Systemd Unit Files

Unit files define how systemd manages a resource. They are stored in:

* `/usr/lib/systemd/system/` – Package-provided units (do not edit directly)

* `/etc/systemd/system/` – Custom units (overrides package units)

* `/run/systemd/system/` – Runtime units (temporary)

### Service Unit Example (`/etc/systemd/system/myapp.service`)

```ini
[Unit]
Description=My Custom Application
After=network.target
Requires=network.target
Documentation=https://docs.myapp.com

[Service]
Type=simple
User=myapp
Group=myapp
WorkingDirectory=/opt/myapp
ExecStart=/usr/bin/python3 /opt/myapp/app.py
ExecReload=/bin/kill -HUP $MAINPID
ExecStop=/bin/kill -TERM $MAINPID
Restart=on-failure
RestartSec=5
StandardOutput=journal
StandardError=journal
Environment="APP_ENV=production"
EnvironmentFile=/etc/myapp/env.conf
LimitNOFILE=65536

[Install]
WantedBy=multi-user.target
```

### Unit File Section Breakdown

#### `[Unit]` – Metadata and Dependencies

| Directive     | Meaning                                     | Example                               |
| ------------- | ------------------------------------------- | ------------------------------------- |
| `Description` | Human-readable description                  | `Description=My App`                  |
| `After`       | Start after these units                     | `After=network.target docker.service` |
| `Before`      | Start before these units                    | `Before=shutdown.target`              |
| `Requires`    | Hard dependency (fails if dependency fails) | `Requires=network.target`             |
| `Wants`       | Soft dependency (start if possible)         | `Wants=time-sync.target`              |
| `Conflicts`   | Cannot run with these units                 | `Conflicts=myapp-old.service`         |

#### `[Service]` – How to Run the Service

| Directive          | Meaning                            | Example                              |
| ------------------ | ---------------------------------- | ------------------------------------ |
| `Type`             | Process behavior (see table below) | `Type=simple`                        |
| `User` / `Group`   | Run as specific user/group         | `User=nginx`                         |
| `WorkingDirectory` | Set before ExecStart               | `WorkingDirectory=/var/www`          |
| `ExecStart`        | Command to start                   | `ExecStart=/usr/bin/node server.js`  |
| `ExecStop`         | Command to stop (optional)         | `ExecStop=/bin/kill -TERM $MAINPID`  |
| `ExecReload`       | Command to reload config           | `ExecReload=/bin/kill -HUP $MAINPID` |
| `Restart`          | When to restart (see table)        | `Restart=on-failure`                 |
| `RestartSec`       | Delay before restart (seconds)     | `RestartSec=10`                      |
| `Environment`      | Set environment variable           | `Environment="PATH=/usr/local/bin"`  |
| `EnvironmentFile`  | Load env vars from file            | `EnvironmentFile=/etc/myapp/env`     |
| `LimitNOFILE`      | Max open file descriptors          | `LimitNOFILE=65536`                  |
| `LimitMEMLOCK`     | Max locked memory                  | `LimitMEMLOCK=infinity`              |

**Service** **`Type`** **values:**

| Type      | Behavior                            | Use Case                           |
| --------- | ----------------------------------- | ---------------------------------- |
| `simple`  | ExecStart is main process (default) | Most applications                  |
| `forking` | ExecStart forks, parent exits       | Traditional daemons (httpd)        |
| `oneshot` | Runs once, no lingering process     | Setup scripts, backups             |
| `dbus`    | Registers with D-Bus                | D-Bus activated services           |
| `notify`  | Notifies systemd when ready         | Services with startup notification |
| `idle`    | Runs when no other jobs active      | Low-priority services              |

**Restart values:**

| Value         | Behavior                                     |
| ------------- | -------------------------------------------- |
| `no`          | Never restart (default)                      |
| `on-success`  | Restart on clean exit (code 0)               |
| `on-failure`  | Restart on non-zero exit, signal, or timeout |
| `on-abnormal` | Restart on signal or timeout                 |
| `always`      | Always restart                               |
| `on-abort`    | Restart on signal only                       |

#### `[Install]` – Installation (Enable) Behavior

| Directive    | Meaning                        | Example                        |
| ------------ | ------------------------------ | ------------------------------ |
| `WantedBy`   | Targets that want this unit    | `WantedBy=multi-user.target`   |
| `RequiredBy` | Targets that require this unit | `RequiredBy=multi-user.target` |
| `Alias`      | Alternative name for unit      | `Alias=app.service`            |

### Target Units (Runlevels)

Targets group other units. They replace SysV init runlevels.

| Target              | Runlevel | Purpose             |
| ------------------- | -------- | ------------------- |
| `poweroff.target`   | 0        | Shutdown            |
| `rescue.target`     | 1        | Single-user mode    |
| `multi-user.target` | 3        | Multi-user, no GUI  |
| `graphical.target`  | 5        | Multi-user with GUI |
| `reboot.target`     | 6        | Reboot              |

```bash
# Get current default target (boot target)
systemctl get-default
# multi-user.target

# Change default target (change runlevel)
sudo systemctl set-default multi-user.target

# Boot into rescue mode
sudo systemctl rescue

# Boot into emergency mode (minimal)
sudo systemctl emergency
```

## Part 4: Creating and Managing Custom Services

### Step-by-Step: Create a Custom Service

```bash
# 1. Create service file
sudo vi /etc/systemd/system/myapp.service

# 2. Reload systemd to recognize new unit
sudo systemctl daemon-reload

# 3. Enable at boot
sudo systemctl enable myapp.service

# 4. Start now
sudo systemctl start myapp.service

# 5. Check status
sudo systemctl status myapp.service

# 6. View logs
sudo journalctl -u myapp.service -f
```

### Overriding Package Services

Never edit `/usr/lib/systemd/system/*.service` directly – updates will overwrite changes.

```bash
# Create override directory
sudo mkdir -p /etc/systemd/system/nginx.service.d/

# Create override file
sudo tee /etc/systemd/system/nginx.service.d/override.conf << EOF
[Service]
ExecStartPre=
ExecStartPre=/usr/local/bin/pre-check.sh
Restart=always
EOF

# Reload and restart
sudo systemctl daemon-reload
sudo systemctl restart nginx

# View merged configuration
systemctl cat nginx
```

### Validating and Inspecting Unit Definitions

These commands are useful when a unit fails to start or behaves differently than expected.

```bash
# Validate unit file syntax before reload/restart
systemd-analyze verify /etc/systemd/system/myapp.service

# Show all known properties for a unit
systemctl show myapp

# Filter specific properties
systemctl show myapp -p Environment -p FragmentPath -p ExecStart -p User
```

Use `systemd-analyze verify` when editing custom units or drop-ins. Use `systemctl show` when you need the effective values after overrides are merged.

## Part 5: Systemd Timers (Cron Replacement)

Timers are systemd units that trigger services at scheduled times.

### Timer Unit Example (`/etc/systemd/system/backup.timer`)

```ini
[Unit]
Description=Run backup daily at 2 AM
Requires=backup.service

[Timer]
OnCalendar=daily
OnCalendar=02:00:00
Persistent=true

[Install]
WantedBy=timers.target
```

### Service Unit for Timer (`/etc/systemd/system/backup.service`)

```ini
[Unit]
Description=Backup script

[Service]
Type=oneshot
ExecStart=/opt/scripts/backup.sh
User=backup
```

### Timer Commands

```bash
# Enable and start timer (not the service directly)
sudo systemctl enable backup.timer
sudo systemctl start backup.timer

# List active timers
systemctl list-timers
# NEXT                         LEFT          LAST                         PASSED
# Tue 2024-01-16 02:00:00 UTC  5h left       Mon 2024-01-15 02:00:00 UTC  19h ago

# List all timers (including inactive)
systemctl list-timers --all

# Show timer details
systemctl status backup.timer
```

**Essential timer note:** a timer triggers a service unit. You normally `enable` and `start` `backup.timer`, while `backup.service` contains the actual job command.

### Calendar Event Syntax

| Expression          | Meaning            |
| ------------------- | ------------------ |
| `daily`             | Every day at 00:00 |
| `hourly`            | Every hour at :00  |
| `weekly`            | Monday 00:00       |
| `monthly`           | 1st day 00:00      |
| `*-*-* 02:00:00`    | Every day at 2 AM  |
| `Mon..Fri 09:00:00` | Weekdays at 9 AM   |
| `*-*-01 00:00:00`   | 1st of every month |
| `*:0/15`            | Every 15 minutes   |

## Part 6: Systemd Journal (`journalctl`)

Journald collects logs from systemd services, kernel, and other sources.

### Basic Journalctl Usage

```bash
# View all logs (oldest to newest)
journalctl

# View all logs (newest to oldest)
journalctl -r

# Follow new logs (like tail -f)
journalctl -f

# Show last N lines
journalctl -n 50

# Show logs for a specific service
journalctl -u sshd

# Show logs since last boot
journalctl -b

# Show logs since 2 hours ago
journalctl --since "2 hours ago"

# Show logs in a time range
journalctl --since "2024-01-15 10:00:00" --until "2024-01-15 12:00:00"

# Filter by priority (1=emerg, 2=alert, 3=crit, 4=err, 5=warning, 6=notice, 7=info, 8=debug)
journalctl -p err -u nginx

# Show kernel messages
journalctl -k

# Show logs for a specific PID
journalctl _PID=1234

# Show logs with JSON output (for scripting)
journalctl -o json-pretty -u nginx

# Show disk usage of journal
journalctl --disk-usage
# Archived and active journals take up 512M on disk.

# Rotate journal (force archival of active journal)
sudo journalctl --rotate

# Vacuum old logs (keep last 2 days)
sudo journalctl --vacuum-time=2d

# Vacuum to specific size
sudo journalctl --vacuum-size=500M
```

### Journalctl Output Formatting

| Flag             | Output Format                               |
| ---------------- | ------------------------------------------- |
| `-o short`       | Default (timestamp, hostname, PID, message) |
| `-o verbose`     | All fields (key=value)                      |
| `-o json`        | JSON                                        |
| `-o json-pretty` | Pretty JSON                                 |
| `-o cat`         | Message only (no metadata)                  |

### Journal Configuration (`/etc/systemd/journald.conf`)

```ini
[Journal]
# Maximum disk space for logs
SystemMaxUse=1G

# Maximum file size per journal file
SystemMaxFileSize=100M

# Keep logs for 1 week
MaxRetentionSec=1week

# Forward to syslog (if needed)
ForwardToSyslog=no

# Forward to console
ForwardToConsole=no

# Compress old logs
Compress=yes
```

After changes: `sudo systemctl restart systemd-journald`

## Quick Task: Create a Systemd Service

*Create a simple systemd service that runs a script.*

1. Create a script `/usr/local/bin/health_check.sh` that outputs "OK" and exits.
2. Create a systemd service file `/etc/systemd/system/health.service` that runs this script every 5 minutes (use `Type=oneshot`).
3. Create a timer file `/etc/systemd/system/health.timer` that triggers every 5 minutes.
4. Enable and start the timer.
5. Check the logs with `journalctl -u health.service -f`.

> **Ready Solution:**
>
> ```bash
> # Task 1
> sudo tee /usr/local/bin/health_check.sh << 'EOF'
> #!/bin/bash
> echo "OK - $(date)"
> EOF
> sudo chmod +x /usr/local/bin/health_check.sh
>
> # Task 2
> sudo tee /etc/systemd/system/health.service << 'EOF'
> [Unit]
> Description=Health check script
>
> [Service]
> Type=oneshot
> ExecStart=/usr/local/bin/health_check.sh
> User=root
> EOF
>
> # Task 3
> sudo tee /etc/systemd/system/health.timer << 'EOF'
> [Unit]
> Description=Run health check every 5 minutes
>
> [Timer]
> OnCalendar=*:0/5
> Persistent=true
>
> [Install]
> WantedBy=timers.target
> EOF
>
> # Task 4
> sudo systemctl daemon-reload
> sudo systemctl enable health.timer
> sudo systemctl start health.timer
>
> # Task 5
> journalctl -u health.service -f
> # Wait 5 minutes, see output:
> # OK - Mon 2024-01-15 10:05:00 UTC
> ```

### timedatectl – System Time and Timezone

The `timedatectl` command manages system time, timezone, and NTP synchronization.

```bash
# View current time and timezone settings
timedatectl
# Output shows:
#   Local time, Universal time, RTC time
#   Timezone, NTP service status, RTC in local TZ

# List available timezones
timedatectl list-timezones | grep America

# Set timezone
sudo timedatectl set-timezone America/New_York
sudo timedatectl set-timezone UTC   # Common for servers

# Enable/disable automatic time synchronization (NTP)
sudo timedatectl set-ntp true

# Set time manually (requires NTP disabled)
sudo timedatectl set-ntp false
sudo timedatectl set-time "2024-01-15 10:30:00"

# Check NTP synchronization status
timedatectl show --property=NTPSynchronized
```

**Platform engineering relevance:**
- All servers should use UTC timezone for consistent log timestamps
- NTP synchronization is critical for distributed systems, TLS certificates, and cron jobs
- Time drift can cause authentication failures and log ordering issues

### localectl – System Locale and Keyboard

The `localectl` command manages system locale, keyboard layout, and console settings.

```bash
# View current locale and keyboard
localectl

# List available locales
localectl list-locales | grep -i en_us

# Set system locale
sudo localectl set-locale LANG=en_US.UTF-8

# List available keyboard layouts
localectl list-keymaps

# Set keyboard layout
sudo localectl set-keymap us
sudo localectl set-x11-keymap us   # For X11/Wayland
```

### System Power Commands

These commands control system power state through systemd:

```bash
# Shutdown immediately
sudo shutdown now

# Shutdown with delay (minutes)
sudo shutdown +5 "Server shutting down in 5 minutes for maintenance"

# Shutdown at specific time
sudo shutdown 23:00 "Nightly maintenance"

# Cancel a scheduled shutdown
sudo shutdown -c

# Reboot immediately
sudo reboot

# Reboot with systemctl (equivalent)
sudo systemctl reboot

# Power off (halt + power down)
sudo poweroff

# Halt (stop system but don't power off)
sudo halt

# Suspend to RAM
sudo systemctl suspend

# Hibernate (suspend to disk)
sudo systemctl hibernate
```

**Best practices:**
- Use `shutdown +N` with message for production to warn users
- `systemctl reboot` and `reboot` are equivalent on systemd systems
- Check for active users with `who` before shutting down

### Service Type Comparison

| Type      | Main Process                  | Forking | Use Case                       |
| --------- | ----------------------------- | ------- | ------------------------------ |
| `simple`  | ExecStart                     | No      | Most apps (Node, Python, Go)   |
| `forking` | Parent exits, child continues | Yes     | Traditional daemons (httpd)    |
| `oneshot` | Runs once, exits              | No      | Setup scripts, backups         |
| `notify`  | Notifies when ready           | No      | Apps with startup notification |
